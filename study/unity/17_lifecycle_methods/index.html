Awake에서 초기화했는데 왜 null이 뜰까?

Awake, Start, Update, FixedUpdate, LateUpdate - 호출 시점과 차이 완전 정리

Part 1: Awake vs Start

Awake와 Start는 둘 다 초기화용인데, 무엇이 다른가요?
구분 Awake Start
호출 시점 GameObject 인스턴스화 또는 스크립트 로드 시 컴포넌트 활성화 시 (첫 Update 전)
비활성 상태 호출됨 (오브젝트가 비활성이어도) 호출 안 됨
주 용도 자기 자신의 참조 생성 다른 오브젝트 참조를 통한 접근
호출 순서 무작위 Awake 이후

Awake의 호출 순서는 무작위입니다! Awake에서 다른 스크립트의 참조를 통해 접근하면 NullReferenceException이 발생할 수 있습니다.

따라서 Awake에서는 자기 자신의 컴포넌트 참조를 잡고, Start에서 다른 오브젝트에 접근하는 패턴을 사용합니다.

Part 2: OnEnable vs Start

OnEnable과 Start 모두 "활성화 시 호출"인데, 차이점은?

Start는 한 번만 호출되지만, OnEnable은 활성화될 때마다 호출됩니다.

초기화 작업에 OnEnable을 사용하면 안 됩니다! OnEnable은 오브젝트 풀링 등에서 재활성화 시 처리할 로직에 적합합니다.

Part 3: Update vs FixedUpdate vs LateUpdate

세 가지 Update의 차이점은?
함수 호출 주기 주 용도
Update 프레임 단위 (가변) 일반 게임 로직, 입력 처리
FixedUpdate 고정 간격 (물리 엔진 기반) Rigidbody 조작, 물리 연산
LateUpdate Update 후 카메라 추적, 후처리

FixedUpdate의 '고정 단위'는 물리 엔진에 의해 결정됩니다. 컴퓨터 성능에 따라 프레임이 달라지는 Update와 달리 항상 일정한 간격으로 호출됩니다.

Rigidbody를 조작할 때는 반드시 FixedUpdate를 사용해야 합니다. Update에서 물리 조작을 하면 프레임 드랍 시 물리 시뮬레이션이 불안정해집니다.

📌 핵심 정리

  • Awake는 무작위 순서로 호출되므로 자기 자신의 초기화에만 사용한다
  • Start는 Awake 이후 호출되므로 다른 오브젝트 참조에 안전하다
  • OnEnable은 활성화될 때마다 호출되므로 초기화가 아닌 재활성화 로직에 사용한다
  • FixedUpdate는 물리 엔진 기반 고정 간격으로 호출되며, Rigidbody 조작에 필수이다
  • LateUpdate는 Update 이후 호출되며, 카메라 추적 등 후처리에 사용한다
