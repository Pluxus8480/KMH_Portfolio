매번 새로 만들면 GC가 울린다

오브젝트 풀링으로 Instantiate/Destroy 지옥 탈출

Part 1: 왜 Instantiate/Destroy가 문제인가?

총알을 발사할 때마다 Instantiate로 생성하고, 벽에 닿으면 Destroy하면 안 되나요?

기능적으로는 문제없지만, 성능 측면에서 심각한 문제를 일으킵니다.

Instantiate는 메모리 할당, 컴포넌트 초기화, Awake/Start 호출 등 무거운 작업을 수반합니다. Destroy는 오브젝트를 메모리에서 해제하며 GC(Garbage Collector) 부담을 유발합니다.

초당 30발을 발사하는 기관총이 있다면, 매 초 30번의 할당과 30번의 해제가 발생합니다. 이것이 쌓이면 GC Spike로 인한 프레임 드랍이 나타납니다.

GC Spike가 정확히 뭔가요?
🎯 비유로 이해하기

책상 위에 종이를 계속 쌓다가, 더 이상 공간이 없으면 한꺼번에 정리하는 것과 같습니다.

이 "한꺼번에 정리"하는 순간이 GC Spike이며, 이 동안 게임이 잠깐 멈추는 현상(스터터링)이 발생합니다.

모바일 게임에서 GC Spike는 치명적입니다. 60fps 기준 1프레임은 16.6ms인데, GC가 수십 ms를 잡아먹으면 눈에 보이는 끊김이 발생합니다.

Part 2: 오브젝트 풀링의 기본 원리

오브젝트 풀링은 어떤 아이디어인가요?

핵심 아이디어: 미리 만들어 두고, 쓰고, 반납하고, 다시 쓴다.

오브젝트를 파괴하는 대신 비활성화(SetActive(false))하여 풀에 반환하고, 필요할 때 다시 활성화하여 재사용합니다.

두 가지 전략이 있습니다:

Part 3: Unity 내장 ObjectPool<T>

Unity에 이미 풀링 시스템이 내장되어 있나요?

Unity 2021부터 UnityEngine.Pool 네임스페이스에 ObjectPool<T>가 내장되었습니다.

4개의 콜백으로 오브젝트의 생명주기를 제어합니다: createFunc, actionOnGet, actionOnRelease, actionOnDestroy

C# - Unity ObjectPool 기본 사용법
using UnityEngine;
using UnityEngine.Pool;

public class BulletSpawner : MonoBehaviour
{
    [SerializeField] private GameObject bulletPrefab;
    private ObjectPool<GameObject> pool;

    void Awake()
    {
        pool = new ObjectPool<GameObject>(
            createFunc: () => Instantiate(bulletPrefab),
            actionOnGet: bullet => {
                bullet.SetActive(true);
                bullet.transform.position = transform.position;
            },
            actionOnRelease: bullet => bullet.SetActive(false),
            actionOnDestroy: bullet => Destroy(bullet),
            collectionCheck: false,
            defaultCapacity: 20,
            maxSize: 100
        );
    }

    public GameObject GetBullet() => pool.Get();

    public void ReturnBullet(GameObject bullet) => pool.Release(bullet);
}
collectionCheck와 maxSize는 무엇인가요?

collectionCheck: 같은 오브젝트를 두 번 반환하는 실수를 감지합니다. 디버깅 시 true로 설정하되, 릴리즈 빌드에서는 false로 두세요.

maxSize: 풀의 최대 크기입니다. 이를 초과하면 반환된 오브젝트는 actionOnDestroy를 통해 파괴됩니다.

maxSize를 너무 작게 잡으면 풀링 효과가 줄어들고, 너무 크게 잡으면 불필요한 메모리를 차지합니다. 프로파일링을 통해 적절한 값을 설정하세요.

Part 4: 커스텀 오브젝트 풀 구현하기

내장 ObjectPool 대신 직접 구현해야 할 때도 있나요?

여러 종류의 프리팹을 하나의 매니저에서 관리하거나, 사전 워밍업(Pre-warm), 씬 전환 시 풀 유지 같은 세밀한 제어가 필요할 때 커스텀 구현이 유리합니다.

C# - 제네릭 커스텀 풀 매니저
using System.Collections.Generic;
using UnityEngine;

public class GenericPool<T> where T : Component
{
    private readonly T prefab;
    private readonly Transform parent;
    private readonly Queue<T> available = new Queue<T>();

    public GenericPool(T prefab, Transform parent, int preWarmCount = 10)
    {
        this.prefab = prefab;
        this.parent = parent;
        PreWarm(preWarmCount);
    }

    private void PreWarm(int count)
    {
        for (int i = 0; i < count; i++)
        {
            T obj = Object.Instantiate(prefab, parent);
            obj.gameObject.SetActive(false);
            available.Enqueue(obj);
        }
    }

    public T Get(Vector3 position, Quaternion rotation)
    {
        T obj = available.Count > 0
            ? available.Dequeue()
            : Object.Instantiate(prefab, parent);

        obj.transform.SetPositionAndRotation(position, rotation);
        obj.gameObject.SetActive(true);
        return obj;
    }

    public void Release(T obj)
    {
        obj.gameObject.SetActive(false);
        available.Enqueue(obj);
    }

    public int CountAvailable => available.Count;
}
IPoolable 인터페이스를 만들면 더 좋지 않나요?

맞습니다. 풀에서 꺼내고 반환할 때 오브젝트가 스스로 초기화/정리할 수 있도록 인터페이스를 정의하면 코드가 깔끔해집니다.

C# - IPoolable 인터페이스
public interface IPoolable
{
    void OnSpawn();   // 풀에서 꺼낼 때
    void OnDespawn(); // 풀로 반환할 때
}

// 총알 클래스에서 구현
public class Bullet : MonoBehaviour, IPoolable
{
    private Rigidbody rb;
    private TrailRenderer trail;

    void Awake()
    {
        rb = GetComponent<Rigidbody>();
        trail = GetComponent<TrailRenderer>();
    }

    public void OnSpawn()
    {
        rb.linearVelocity = Vector3.zero;
        trail.Clear();
    }

    public void OnDespawn()
    {
        rb.linearVelocity = Vector3.zero;
    }
}

특히 TrailRenderer.Clear()를 호출하지 않으면, 재활용된 총알이 이전 위치에서 현재 위치까지 잔상을 남기는 버그가 발생합니다.

Part 5: 총알과 파티클 풀링 실전

실제 슈팅 게임에서 총알 풀링은 어떻게 구성하나요?

총알의 라이프사이클 전체를 풀 기반으로 설계해야 합니다.

C# - 풀링 기반 슈팅 시스템
public class Weapon : MonoBehaviour
{
    [SerializeField] private Bullet bulletPrefab;
    [SerializeField] private ParticleSystem hitEffectPrefab;

    private GenericPool<Bullet> bulletPool;
    private ObjectPool<ParticleSystem> hitEffectPool;

    void Start()
    {
        bulletPool = new GenericPool<Bullet>(bulletPrefab, transform, 30);

        hitEffectPool = new ObjectPool<ParticleSystem>(
            () => Instantiate(hitEffectPrefab),
            ps => { ps.gameObject.SetActive(true); ps.Play(); },
            ps => { ps.Stop(); ps.gameObject.SetActive(false); },
            ps => Destroy(ps.gameObject),
            false, 15, 50
        );
    }

    public void Fire(Vector3 direction)
    {
        var bullet = bulletPool.Get(transform.position, Quaternion.LookRotation(direction));
        bullet.Init(direction, 30f, OnBulletHit);
    }

    private void OnBulletHit(Bullet bullet, Vector3 hitPoint)
    {
        bulletPool.Release(bullet);
        var effect = hitEffectPool.Get();
        effect.transform.position = hitPoint;
        StartCoroutine(ReturnEffectAfterDelay(effect, 1.5f));
    }
}
파티클 이펙트를 풀링할 때 주의할 점은?

ParticleSystem은 재생 상태를 가지고 있으므로 반환 전 반드시 Stop()과 Clear()를 호출해야 합니다.

ParticleSystem.main.stopAction을 Destroy로 설정하지 마세요! 풀링과 충돌합니다. 대신 Callback이나 None으로 설정하고, 수동으로 반환 타이밍을 관리하세요.

Part 6: 풀링 최적화와 모범 사례

풀 사이즈는 어떻게 결정하나요?

프로파일링으로 동시 활성 오브젝트 최대치를 측정한 뒤, 그 값에 20~30% 여유를 두세요.

오브젝트 유형추천 풀 크기근거
권총 총알20~30발사 속도 낮음, 수명 짧음
기관총 총알100~150초당 15~30발
피격 이펙트15~301~2초 후 자동 반환
적 NPC30~50웨이브 기반 스폰
씬 전환 시 풀은 어떻게 관리하나요?

DontDestroyOnLoad로 풀 매니저를 유지하거나, 씬별로 새 풀을 생성하는 방법이 있습니다.

Addressables와 함께 사용한다면, 풀 해제 시 Addressables.ReleaseInstance도 호출해야 레퍼런스 카운트가 올바르게 관리됩니다.

📌 핵심 정리

  • Instantiate/Destroy 반복은 GC Spike를 유발하여 프레임 드랍의 주범이 된다
  • 오브젝트 풀링은 미리 생성한 오브젝트를 비활성/활성 전환으로 재사용하는 패턴이다
  • Unity 2021+는 ObjectPool<T>를 내장하며, 4개의 콜백으로 생명주기를 제어한다
  • 재활용 시 TrailRenderer.Clear(), Rigidbody velocity 초기화 등 상태 리셋이 필수다
  • 풀 크기는 프로파일링 기반으로 결정하고, IPoolable 인터페이스로 초기화 로직을 캡슐화하라
