Part 1: 프로파일링의 필요성

게임이 느린데, 어디서부터 손봐야 할지 모르겠어요.

"느리다"는 감각적인 판단입니다. 프로파일링은 그 감각을 수치로 바꿔주는 도구입니다.

최적화의 첫 번째 원칙: 측정하지 않으면 최적화하지 마라. 추측 기반 최적화는 시간 낭비의 주범입니다.

Unity에서 프로파일링 도구는 뭐가 있나요?

Part 2: CPU 병목 vs GPU 병목 구분하기

프레임이 떨어지면 CPU 문제인가요, GPU 문제인가요?

둘 다 가능합니다. 핵심은 어디가 병목인지 구분하는 것입니다.

🎯 비유로 이해하기

레스토랑을 생각해보세요. 주방(CPU)이 요리를 너무 느리게 만들면, 서빙(GPU)이 아무리 빨라도 손님은 기다립니다.

반대로 주방은 빠른데 서빙이 느리면? 역시 손님은 기다립니다. 어디가 병목인지 찾아야 합니다.

Profiler에서 CPU 프레임 시간과 GPU 프레임 시간을 비교하세요. 더 큰 쪽이 병목입니다.

CPU 병목의 대표적인 원인은?

CPU 병목은 대부분 스크립트 로직, 물리 연산, 또는 드로우콜 준비 과정에서 발생합니다.

C# - CPU 병목 유발 코드 예시
// 매 프레임 FindObjectOfType 호출 - 매우 느림
void Update()
{
    // BAD: 매 프레임 전체 씬을 탐색
    var player = FindObjectOfType<Player>();

    // BAD: 매 프레임 GetComponent 호출
    var rb = GetComponent<Rigidbody>();

    // BAD: 매 프레임 새 배열 할당 (GC 유발)
    var enemies = new List<Enemy>();
    foreach (var obj in FindObjectsOfType<Enemy>())
        enemies.Add(obj);
}

// GOOD: 캐싱 패턴
private Player _player;
private Rigidbody _rb;
private List<Enemy> _enemies = new List<Enemy>();

void Start()
{
    _player = FindObjectOfType<Player>();
    _rb = GetComponent<Rigidbody>();
}

void Update()
{
    // 캐싱된 참조 사용 - 빠름
    transform.LookAt(_player.transform);
}
GPU 병목은 어떻게 확인하나요?

GPU 병목은 셰이더 복잡도, 오버드로, 해상도가 주원인입니다.

모바일에서 GPU 병목은 발열과 직결됩니다. 프레임 드랍뿐 아니라 써멀 스로틀링까지 유발할 수 있습니다.

Part 3: 배칭과 드로우콜 이해하기

드로우콜이 뭔가요? 왜 줄여야 하나요?

드로우콜은 CPU가 GPU에게 "이 메시를 그려라"라고 명령하는 단위입니다. 각 드로우콜마다 상태 변경 비용이 발생합니다.

드로우콜 자체보다 SetPass Call(머티리얼/셰이더 전환)이 더 비쌉니다. 같은 머티리얼을 쓰는 오브젝트끼리 묶으면 비용이 줄어듭니다.

배칭은 어떻게 동작하나요?
구분Static BatchingDynamic BatchingSRP Batcher
대상움직이지 않는 오브젝트작은 메시 (300 vertex 이하)SRP 호환 셰이더
조건같은 머티리얼 + Static 체크같은 머티리얼같은 셰이더 Variant
메모리추가 메모리 사용CPU 부하 증가최소 오버헤드
권장배경, 건물파티클, 작은 소품URP/HDRP 프로젝트

Part 4: 메모리 프로파일링

메모리 누수는 어떻게 찾나요?

Unity Memory Profiler 패키지를 사용하면 메모리 스냅샷을 찍고 비교할 수 있습니다.

메모리 프로파일링 워크플로
// 1. Memory Profiler 패키지 설치
// Package Manager > Memory Profiler

// 2. 스냅샷 비교 패턴
// 씬 로드 전 스냅샷 A
// 씬 로드 후 스냅샷 B
// 씬 언로드 후 스냅샷 C
// A와 C를 비교 → 차이가 있으면 누수!
GC(가비지 컬렉션)가 프레임을 끊는데 어떻게 해야 하나요?

GC 스파이크는 힙 메모리 할당이 누적되어 발생합니다. 핵심은 런타임 할당을 줄이는 것입니다.

C# - GC 최소화 패턴
// BAD: 매 프레임 문자열 할당
void Update()
{
    scoreText.text = "Score: " + score.ToString(); // 매번 새 string
}

// GOOD: StringBuilder 또는 캐싱
private StringBuilder _sb = new StringBuilder(32);
void Update()
{
    _sb.Clear();
    _sb.Append("Score: ").Append(score);
    scoreText.text = _sb.ToString();
}

// GOOD: Incremental GC 활성화
// Project Settings > Player > Use Incremental GC ✓

LINQ 쿼리는 내부적으로 힙 할당을 유발합니다. 성능 크리티컬 경로에서는 for 루프를 사용하세요.

Part 5: Profiler 실전 활용법

Profiler를 열면 데이터가 너무 많은데, 뭘 봐야 하나요?

단계적으로 접근하세요.

3. Deep Profile

4. 수정 후 비교
개선 전/후 수치 비교
에디터에서 프로파일링하면 정확한가요?

에디터 프로파일링은 참고용입니다. 에디터 자체의 오버헤드(인스펙터 갱신, 씬뷰 렌더링)가 결과를 왜곡합니다. 반드시 빌드된 기기에서 프로파일링하세요.

원격 프로파일링 설정
// Build Settings에서 Development Build + Autoconnect Profiler 체크
// 1. Development Build ✓
// 2. Autoconnect Profiler ✓
// 3. 빌드 후 기기에서 실행
// 4. Profiler 창에서 Target 드롭다운 → 기기 선택

Part 6: 최적화 체크리스트

프로파일링 후 최적화 우선순위는 어떻게 잡나요?

가장 큰 병목부터 해결하세요. 1%짜리 10개보다 30%짜리 1개가 우선입니다.

셰이더모바일용 셰이더, 연산 복잡도 줄이기높음

📌 핵심 정리

  • 측정 우선: 추측하지 말고 Profiler로 병목을 측정한 뒤 최적화하라
  • CPU vs GPU: 프레임 시간 비교로 병목 위치를 먼저 파악하라
  • 드로우콜: SetPass Call을 줄이는 것이 핵심 — 배칭과 머티리얼 통합 활용
  • GC 스파이크: 런타임 힙 할당을 최소화하고 Incremental GC를 활성화하라
  • 실기기 프로파일링: 에디터가 아닌 빌드된 기기에서 측정해야 정확하다
