MonoBehaviour 생명주기와 실행 순서 완벽 정리

Part 1: 생명주기의 전체 그림

MonoBehaviour에 왜 이렇게 많은 콜백 함수가 있나요?

Unity의 게임 오브젝트는 생성부터 파괴까지 명확한 생명주기(Lifecycle)를 따릅니다. 각 단계마다 적절한 초기화, 업데이트, 정리 작업이 필요하기 때문에 다양한 콜백이 존재합니다.

생명주기를 이해하지 못하면 "왜 변수가 null인가?", "왜 이 코드가 먼저 실행되는가?" 같은 문제에 끊임없이 시달리게 됩니다.

Part 2: Awake vs Start — 무엇이 다른가?

Awake와 Start 둘 다 초기화인데 왜 두 개가 필요한가요?

핵심 차이는 호출 시점과 활성 상태 의존성입니다.

구분AwakeStart
호출 시점스크립트 인스턴스 로드 직후첫 번째 Update 직전
활성 상태비활성이어도 호출됨활성 상태일 때만 호출
호출 횟수평생 1회평생 1회
주요 용도자기 자신의 참조 설정다른 오브젝트 참조, 초기 상태 설정
🎯 비유로 이해하기

Awake는 아침에 눈을 뜨는 것(알람이 울리면 강제로 깨어남)이고, Start는 침대에서 일어나 하루를 시작하는 것(준비가 되었을 때)입니다.

눈을 떴다고 바로 출근하지 않듯이, Awake에서 자기 자신을 준비하고 Start에서 다른 오브젝트와 상호작용을 시작합니다.

그러면 초기화 코드를 어디에 넣어야 하나요?

원칙: Awake에서 자기 자신을 설정하고, Start에서 외부 의존성을 연결하세요.

C# - 올바른 초기화 패턴
public class PlayerController : MonoBehaviour
{
    private Rigidbody rb;
    private GameManager gameManager;

    void Awake()
    {
        // 자기 자신의 컴포넌트 참조 (항상 성공)
        rb = GetComponent<Rigidbody>();
    }

    void Start()
    {
        // 다른 오브젝트 참조 (Awake가 모두 끝난 후)
        gameManager = GameManager.Instance;
        gameManager.RegisterPlayer(this);
    }
}

주의: Start에서 참조한 다른 오브젝트가 아직 Start를 실행하지 않았을 수 있습니다. 순서가 중요한 경우 Script Execution Order를 설정하세요.

Part 3: Update 삼형제 — 언제 무엇을 쓸까?

Update, FixedUpdate, LateUpdate는 어떻게 구분해서 사용하나요?
함수호출 주기용도예시
Update매 프레임 (가변)입력 처리, 일반 로직키 입력, UI 갱신
FixedUpdate고정 주기 (0.02초)물리 연산Rigidbody 이동, 힘 적용
LateUpdate매 프레임 (Update 후)후처리 로직카메라 추적, IK 보정
물리 이동을 Update에서 하면 안 되는 이유가 뭔가요?

Update는 프레임 속도에 따라 호출 빈도가 달라집니다. 60fps에서는 초당 60번, 30fps에서는 초당 30번 호출됩니다.

물리 엔진은 일정한 시간 간격(Fixed Timestep)을 기준으로 시뮬레이션합니다. Update에서 Rigidbody를 직접 조작하면 물리 시뮬레이션과 타이밍이 어긋나 떨림(jitter) 현상이 발생합니다.

C# - 올바른 물리 이동
// ❌ 잘못된 방법 - Update에서 물리 조작
void Update()
{
    rb.AddForce(Vector3.forward * speed); // 프레임 의존적!
}

// ✅ 올바른 방법 - FixedUpdate에서 물리 조작
void FixedUpdate()
{
    rb.AddForce(Vector3.forward * speed); // 일정한 주기
}

// ✅ 입력은 Update, 적용은 FixedUpdate
private Vector3 moveInput;

void Update()
{
    moveInput = new Vector3(Input.GetAxis("Horizontal"), 0,
                            Input.GetAxis("Vertical"));
}

void FixedUpdate()
{
    rb.MovePosition(rb.position + moveInput * speed * Time.fixedDeltaTime);
}
LateUpdate는 왜 카메라에 사용하나요?

캐릭터가 Update에서 이동한 뒤, 카메라가 같은 Update에서 따라가면 캐릭터의 이동이 아직 완료되지 않은 상태일 수 있습니다.

LateUpdate는 모든 Update가 끝난 뒤 호출되므로, 캐릭터의 최종 위치를 기준으로 카메라를 배치할 수 있습니다.

C# - 카메라 따라가기
public class CameraFollow : MonoBehaviour
{
    public Transform target;
    public Vector3 offset = new Vector3(0, 5, -10);

    void LateUpdate()
    {
        // 모든 캐릭터 이동이 끝난 후 카메라 위치 갱신
        transform.position = target.position + offset;
        transform.LookAt(target);
    }
}

Part 4: OnEnable / OnDisable — 반복되는 생명주기

OnEnable과 OnDisable은 언제 사용하나요?

오브젝트 풀링(Object Pooling)처럼 오브젝트를 활성/비활성하며 재사용하는 패턴에서 매우 중요합니다.

OnEnable은 오브젝트가 활성화될 때마다, OnDisable은 비활성화될 때마다 호출됩니다. Awake/Start와 달리 여러 번 호출될 수 있습니다.

C# - 이벤트 구독 패턴
public class EnemyAI : MonoBehaviour
{
    void OnEnable()
    {
        // 활성화될 때 이벤트 구독
        GameEvents.OnPlayerDeath += HandlePlayerDeath;
        GameEvents.OnWaveStart += HandleWaveStart;
    }

    void OnDisable()
    {
        // 비활성화될 때 구독 해제 (메모리 누수 방지!)
        GameEvents.OnPlayerDeath -= HandlePlayerDeath;
        GameEvents.OnWaveStart -= HandleWaveStart;
    }
}

주의: OnDisable에서 이벤트 구독을 해제하지 않으면 오브젝트가 파괴된 후에도 이벤트가 호출되어 NullReferenceException이 발생합니다.

호출 순서를 정리하면 어떻게 되나요?

오브젝트 풀링에서의 전체 흐름입니다:

최초 생성
Awake → OnEnable → Start
→
비활성화
OnDisable
→
재활성화
OnEnable (Start 안 호출!)

재활성화 시 Start는 다시 호출되지 않습니다. 오브젝트 풀링에서 매번 초기화가 필요하면 OnEnable에 초기화 코드를 넣으세요.

Part 5: 실행 순서의 함정

여러 스크립트의 Awake 호출 순서는 어떻게 되나요?

기본적으로 보장되지 않습니다. Unity는 스크립트 로딩 순서를 명시적으로 지정하지 않으면 임의의 순서로 Awake를 호출합니다.

C# - 순서 문제 예시
// GameManager.cs
public class GameManager : MonoBehaviour
{
    public static GameManager Instance;
    void Awake() { Instance = this; }
}

// PlayerController.cs
public class PlayerController : MonoBehaviour
{
    void Awake()
    {
        // GameManager의 Awake가 먼저 실행되었을까?
        // 보장할 수 없다! NullReferenceException 위험!
        GameManager.Instance.RegisterPlayer(this);
    }
}
실행 순서를 제어하려면 어떻게 해야 하나요?

세 가지 방법이 있습니다:

  • Script Execution Order: Edit → Project Settings → Script Execution Order에서 우선순위를 지정
  • Awake/Start 분리: 자기 초기화는 Awake, 외부 참조는 Start에서 수행
  • 지연 초기화: 필요한 시점에 처음 접근할 때 초기화
C# - 지연 초기화 패턴
public class GameManager : MonoBehaviour
{
    private static GameManager instance;
    public static GameManager Instance
    {
        get
        {
            if (instance == null)
                instance = FindObjectOfType<GameManager>();
            return instance;
        }
    }

    void Awake()
    {
        instance = this;
    }
}

Part 6: 코루틴과 생명주기

코루틴은 생명주기에서 어디에 위치하나요?

코루틴의 yield return null은 Update 직후, LateUpdate 직전에 실행됩니다.

프레임 실행 순서
FixedUpdateUpdateyield nullLateUpdate
기타 yield
WaitForEndOfFrame렌더링 후

주의: 오브젝트가 비활성화(SetActive(false))되면 해당 오브젝트의 코루틴이 즉시 중단되고 다시 활성화해도 자동 재개되지 않습니다.

📌 핵심 정리

  • Awake는 자기 자신 초기화, Start는 외부 의존성 연결에 사용한다
  • 물리 연산은 반드시 FixedUpdate에서, 카메라 추적은 LateUpdate에서 처리한다
  • OnEnable/OnDisable은 여러 번 호출될 수 있으며, 이벤트 구독/해제에 활용한다
  • 여러 스크립트의 실행 순서는 기본적으로 보장되지 않으므로 Script Execution Order나 Awake/Start 분리로 해결한다
  • 오브젝트 비활성화 시 코루틴이 중단되며 자동 재개되지 않는다
