UI 하나가 전체 프레임을 깎는다고?

UGUI와 UI Toolkit — Unity UI 시스템 완벽 비교

Part 1: Canvas 리빌드 문제

UI가 왜 성능 문제를 일으키나요?

UGUI(Unity의 기본 UI 시스템)는 Canvas 단위로 메시를 생성합니다. Canvas 하위 요소가 하나라도 변경되면 전체 Canvas의 메시를 다시 계산합니다.

🎯 비유로 이해하기

1000피스 퍼즐을 완성했는데, 한 조각을 바꾸려면 전체를 다시 맞춰야 하는 상황입니다.

Canvas 리빌드가 바로 이것입니다. 텍스트 한 글자만 바꿔도 Canvas 전체가 다시 계산됩니다.

Canvas 리빌드는 Layout 재계산 + Mesh 재생성 + 배칭 재수행의 3단계로 이루어집니다. 요소가 많을수록 비용이 급증합니다.

리빌드를 유발하는 원인은 뭔가요?
원인설명빈도
Text 변경점수, 타이머 등 텍스트 업데이트매우 높음
Image 변경색상, sprite, fillAmount 변경높음
SetActive요소 활성/비활성높음
Transform 변경위치, 크기, 회전 변경중간
계층구조 변경자식 추가/제거, 순서 변경낮음 (비용 높음)

매 프레임 Text.text를 업데이트하는 것은 매 프레임 Canvas 리빌드를 유발합니다. 값이 실제로 변경될 때만 업데이트하세요.

C# - 불필요한 리빌드 방지
// BAD: 매 프레임 텍스트 갱신
void Update()
{
    scoreText.text = score.ToString();  // 값이 안 바뀌어도 리빌드 발생
}

// GOOD: 값이 변경될 때만 갱신
private int _lastScore = -1;
void Update()
{
    if (score != _lastScore)
    {
        _lastScore = score;
        scoreText.text = score.ToString();
    }
}

Part 2: Canvas 분리 전략

Canvas 리빌드 비용을 줄이려면 어떻게 해야 하나요?

자주 변경되는 요소와 정적인 요소를 별도의 Canvas(또는 Sub-Canvas)로 분리하세요.

Static Canvas
배경, 프레임, 아이콘
|
Dynamic Canvas
점수, 타이머, HP바
|
Popup Canvas
팝업, 대화창

Sub-Canvas를 사용하면 부모 Canvas의 리빌드와 독립적으로 동작합니다.

Canvas 분리 구조 예시
// 권장 Canvas 계층 구조
HUD (Canvas)
├── StaticLayer (Sub-Canvas)  → 변경 없음
│   ├── Background
│   ├── SkillSlotFrames
│   └── MinimapFrame
├── DynamicLayer (Sub-Canvas)  → 자주 변경
│   ├── HPBar
│   ├── ScoreText
│   └── Timer
└── OverlayLayer (Sub-Canvas)  → 가끔 활성화
    ├── DamageNumbers
    └── Notifications
Canvas를 너무 많이 나누면 안 되나요?

Canvas가 늘어나면 드로우콜이 증가할 수 있습니다. Canvas마다 독립적으로 배칭되므로, 과도한 분리는 오히려 역효과입니다.

실무에서는 "정적 / 동적 / 오버레이" 정도의 2~4개 Sub-Canvas로 분리하는 것이 균형점입니다.

Part 3: UGUI 배칭 규칙

UI 배칭은 어떤 규칙으로 동작하나요?

UGUI는 같은 Canvas 내에서 동일한 머티리얼/텍스처를 사용하는 요소를 하나의 드로우콜로 묶습니다. 하지만 배칭이 깨지는 경우가 많습니다.

배칭이 깨지는 경우해결 방법
서로 다른 텍스처/머티리얼Sprite Atlas로 텍스처 통합
UI 요소가 겹칠 때 (Z-order 문제)계층 순서 정리, 겹침 최소화
Mask 컴포넌트 사용RectMask2D로 대체 (Stencil 불사용)
다른 Canvas 소속필요한 경우에만 Canvas 분리
Sprite Atlas는 왜 중요한가요?

개별 스프라이트를 각각 로드하면 각각이 별도의 드로우콜을 필요로 합니다. Sprite Atlas로 묶으면 하나의 텍스처로 합쳐져 배칭됩니다.

Sprite Atlas 설정
// 1. Assets > Create > 2D > Sprite Atlas
// 2. Objects for Packing에 스프라이트 폴더 추가
// 3. Pack Preview로 결과 확인

// 코드에서 Atlas 스프라이트 로드
using UnityEngine.U2D;

SpriteAtlas atlas = Resources.Load<SpriteAtlas>("UIAtlas");
Sprite icon = atlas.GetSprite("sword_icon");

같은 화면에 동시에 보이는 UI 스프라이트는 하나의 Atlas에 묶으세요. 서로 다른 화면의 스프라이트를 같은 Atlas에 넣으면 메모리 낭비입니다.

Part 4: UI Toolkit 소개

UI Toolkit은 UGUI와 뭐가 다른가요?

UI Toolkit은 웹 개발의 HTML/CSS와 유사한 방식으로 UI를 구성합니다. Unity 2021부터 런타임 지원이 본격화되었습니다.

구분UGUIUI Toolkit
구조 정의GameObject 계층UXML (HTML과 유사)
스타일컴포넌트 속성USS (CSS와 유사)
렌더링Canvas 메시 기반독자 렌더링 파이프라인
리빌드Canvas 전체 리빌드변경된 요소만 재계산
에디터 도구InspectorUI Builder (시각적 편집)
성숙도안정적, 에셋 풍부발전 중, 에셋 부족
UI Toolkit의 코드는 어떻게 생겼나요?
UXML - UI 구조 정의
<ui:UXML xmlns:ui="UnityEngine.UIElements">
    <ui:VisualElement class="hud-container">
        <ui:Label name="score-label" text="Score: 0" class="score" />
        <ui:ProgressBar name="hp-bar" value="100" class="hp-bar" />
        <ui:Button name="pause-btn" text="Pause" class="btn-pause" />
    </ui:VisualElement>
</ui:UXML>
USS - UI 스타일 정의
.hud-container {
    flex-direction: row;
    justify-content: space-between;
    padding: 10px;
}

.score {
    font-size: 24px;
    color: white;
    -unity-font-style: bold;
}

.hp-bar {
    width: 200px;
    height: 20px;
}

.btn-pause:hover {
    background-color: rgba(255, 255, 255, 0.2);
}
C# - UI Toolkit 이벤트 바인딩
using UnityEngine.UIElements;

public class HUDController : MonoBehaviour
{
    private Label _scoreLabel;
    private ProgressBar _hpBar;

    void OnEnable()
    {
        var root = GetComponent<UIDocument>().rootVisualElement;

        _scoreLabel = root.Q<Label>("score-label");
        _hpBar = root.Q<ProgressBar>("hp-bar");

        var pauseBtn = root.Q<Button>("pause-btn");
        pauseBtn.RegisterCallback<ClickEvent>(OnPauseClicked);
    }

    public void UpdateScore(int score)
    {
        _scoreLabel.text = $"Score: {score}";
    }

    private void OnPauseClicked(ClickEvent evt)
    {
        Time.timeScale = 0f;
    }
}

Unity는 UI Toolkit을 차세대 UI 시스템으로 밀고 있습니다. 장기적으로는 UI Toolkit을 학습해두는 것이 유리합니다.

Part 6: UI 성능 최적화 체크리스트

UGUI 최적화 실전 팁을 정리해주세요.
  • Raycast Target 끄기: 클릭이 필요 없는 Image, Text의 Raycast Target을 해제하세요. 이벤트 시스템이 불필요하게 순회합니다.
  • Canvas.ForceUpdateCanvases() 금지: 이 함수는 모든 Canvas를 강제 리빌드합니다.
  • Layout 컴포넌트 최소화: LayoutGroup은 편리하지만 매 리빌드마다 재계산됩니다. 위치가 고정된 요소는 수동 배치하세요.
  • Animator 대신 DOTween: UI 애니메이션에 Animator를 사용하면 매 프레임 dirty 플래그가 설정됩니다. 트윈 라이브러리를 사용하세요.
C# - UI 오브젝트 풀링
// 데미지 숫자 UI 풀링 예시
public class DamageNumberPool : MonoBehaviour
{
    [SerializeField] private GameObject _prefab;
    private Queue<GameObject> _pool = new Queue<GameObject>();

    public GameObject Get()
    {
        GameObject obj;
        if (_pool.Count > 0)
        {
            obj = _pool.Dequeue();
            obj.SetActive(true);
        }
        else
        {
            obj = Instantiate(_prefab, transform);
        }
        return obj;
    }

    public void Return(GameObject obj)
    {
        obj.SetActive(false);
        _pool.Enqueue(obj);
    }
}

SetActive(false/true)도 Canvas 리빌드를 유발합니다. 자주 토글되는 요소는 Canvas Group의 alpha를 0으로 하고 blocksRaycasts를 false로 설정하는 것이 더 가볍습니다.

📌 핵심 정리

  • Canvas 리빌드: 하나의 요소 변경이 전체 Canvas를 재계산시킨다 — Sub-Canvas로 분리하라
  • 배칭 규칙: Sprite Atlas로 텍스처를 통합하고, Mask 대신 RectMask2D를 사용하라
  • Raycast Target: 클릭 불필요한 요소는 반드시 해제하라
  • UI Toolkit: USS/UXML 기반 차세대 시스템 — 리빌드 범위가 좁고 스타일링이 유연하다
  • 선택 기준: 안정적 출시는 UGUI, 장기 프로젝트와 에디터 도구는 UI Toolkit을 고려하라
