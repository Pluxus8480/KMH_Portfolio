게임도 테스트 코드가 필요한가?

Unity Test Framework로 안정적인 게임 개발하기

Part 1: 게임에서 테스트가 필요한 이유

게임은 직접 플레이해보면 되는데, 왜 테스트 코드를 써야 하나요?

직접 플레이해서 찾는 버그는 빙산의 일각입니다. 수동 테스트로는 모든 경우의 수를 커버할 수 없습니다.

🎯 비유로 이해하기

건물을 지을 때 "눈으로 보니까 괜찮아"라고 하지 않습니다. 구조 계산서, 안전 검사를 통해 체계적으로 검증합니다.

테스트 코드는 코드의 안전 검사입니다. 변경할 때마다 자동으로 "이전에 되던 것이 여전히 되는가?"를 확인합니다.

테스트의 핵심 가치는 "회귀 방지"입니다. 새 기능을 추가할 때 기존 기능이 깨지지 않았음을 자동으로 검증합니다.

어떤 부분을 테스트해야 하나요?
테스트 적합테스트 부적합
데미지 계산 공식카메라 앵글이 예쁜지
인벤토리 아이템 추가/삭제UI 레이아웃이 자연스러운지
스킬 쿨다운 로직파티클 이펙트가 화려한지
세이브/로드 데이터 무결성BGM 분위기가 맞는지
상태 전환 (FSM) 로직조작감이 좋은지

규칙이 명확하고 결과가 예측 가능한 로직이 테스트하기 좋습니다. "느낌"이나 "감성"은 테스트 코드로 검증할 수 없습니다.

Part 2: EditMode vs PlayMode 테스트

Unity Test Framework의 두 가지 모드는 뭔가요?
구분EditMode TestPlayMode Test
실행 환경에디터 모드 (Play 없이)Play 모드 진입
속도빠름 (밀리초 단위)느림 (씬 로드 필요)

코루틴불가가능 (UnityTest)

EditMode 테스트 코드는 어떻게 작성하나요?
C# - EditMode 테스트 예시
using NUnit.Framework;

[TestFixture]
public class DamageCalculatorTests
{
    private DamageCalculator _calculator;

    [SetUp]
    public void SetUp()
    {
        _calculator = new DamageCalculator();
    }

    [Test]
    public void CalculateDamage_BasicAttack_ReturnsCorrectDamage()
    {
        // Arrange
        float attackPower = 100f;
        float defense = 30f;

        // Act
        float damage = _calculator.Calculate(attackPower, defense);

        // Assert
        Assert.AreEqual(70f, damage);
    }

    [Test]
    public void CalculateDamage_NegativeResult_ReturnMinimumOne()
    {
        // 방어력이 공격력보다 높아도 최소 1 데미지
        float damage = _calculator.Calculate(10f, 999f);
        Assert.GreaterOrEqual(damage, 1f);
    }

    [TestCase(100f, 0f, 100f)]
    [TestCase(100f, 50f, 50f)]
    [TestCase(0f, 50f, 1f)]
    public void CalculateDamage_VariousInputs(
        float atk, float def, float expected)
    {
        float damage = _calculator.Calculate(atk, def);
        Assert.AreEqual(expected, damage);
    }
}

TestCase 어트리뷰트를 사용하면 하나의 테스트 메서드로 여러 입력 조합을 검증할 수 있습니다.

PlayMode 테스트는 어떻게 다른가요?
C# - PlayMode 테스트 예시
using System.Collections;
using NUnit.Framework;
using UnityEngine;
using UnityEngine.TestUtils;

public class PlayerHealthTests
{
    private GameObject _playerObj;
    private PlayerHealth _health;

    [SetUp]
    public void SetUp()
    {
        _playerObj = new GameObject("TestPlayer");
        _health = _playerObj.AddComponent<PlayerHealth>();
        _health.Initialize(maxHp: 100);
    }

    [TearDown]
    public void TearDown()
    {
        Object.Destroy(_playerObj);
    }

    [UnityTest]  // PlayMode에서는 UnityTest 사용
    public IEnumerator TakeDamage_ReducesHealth()
    {
        _health.TakeDamage(30);

        yield return null;  // 한 프레임 대기

        Assert.AreEqual(70, _health.CurrentHp);
    }

    [UnityTest]
    public IEnumerator TakeDamage_WhenDead_TriggersDeathEvent()
    {
        bool deathTriggered = false;
        _health.OnDeath += () => deathTriggered = true;

        _health.TakeDamage(200);

        yield return null;

        Assert.IsTrue(deathTriggered);
        Assert.AreEqual(0, _health.CurrentHp);
    }
}

PlayMode 테스트에서 생성한 GameObject는 반드시 TearDown에서 Destroy해야 합니다. 그렇지 않으면 다음 테스트에 영향을 줍니다.

Part 3: 테스트 가능한 설계

MonoBehaviour는 new로 생성할 수 없어서 테스트하기 어려운데요?

맞습니다. 그래서 핵심 로직을 MonoBehaviour에서 분리하는 것이 중요합니다.

게임 로직은 순수 C# 클래스(POCO)로 작성하고, MonoBehaviour는 Unity 이벤트를 받아 로직 클래스에 전달하는 역할만 합니다.

MonoBehaviour
Unity 이벤트 수신 (Update, OnCollision)
→
Logic Class (POCO)
핵심 로직 (데미지, 인벤토리, FSM)
←
EditMode Test
new로 직접 생성하여 테스트
C# - 테스트 가능한 구조 분리
// 순수 로직 클래스 (테스트 가능)
public class Inventory
{
    private List<Item> _items = new List<Item>();
    public int MaxSlots { get; }

    public Inventory(int maxSlots) { MaxSlots = maxSlots; }

    public bool TryAddItem(Item item)
    {
        if (_items.Count >= MaxSlots) return false;
        _items.Add(item);
        return true;
    }

    public bool RemoveItem(string itemId)
    {
        return _items.RemoveAll(i => i.Id == itemId) > 0;
    }

    public int ItemCount => _items.Count;
}

// MonoBehaviour는 브릿지 역할만
public class InventoryController : MonoBehaviour
{
    private Inventory _inventory;

    void Awake()
    {
        _inventory = new Inventory(maxSlots: 20);
    }

    public void OnItemPickup(Item item)
    {
        if (_inventory.TryAddItem(item))
            ShowPickupEffect(item);
        else
            ShowInventoryFullMessage();
    }
}

// EditMode 테스트 (빠르고 안정적)
[Test]
public void TryAddItem_WhenFull_ReturnsFalse()
{
    var inventory = new Inventory(maxSlots: 2);
    inventory.TryAddItem(new Item("sword"));
    inventory.TryAddItem(new Item("shield"));

    bool result = inventory.TryAddItem(new Item("potion"));

    Assert.IsFalse(result);
    Assert.AreEqual(2, inventory.ItemCount);
}
의존성 주입은 어떻게 활용하나요?

인터페이스를 통해 의존성을 주입하면, 테스트 시 Mock 객체로 대체할 수 있습니다.

C# - 인터페이스와 Mock 활용
// 인터페이스 정의
public interface ISaveSystem
{
    void Save(string key, string data);
    string Load(string key);
}

// 실제 구현
public class PlayerPrefsSaveSystem : ISaveSystem
{
    public void Save(string key, string data)
        => PlayerPrefs.SetString(key, data);
    public string Load(string key)
        => PlayerPrefs.GetString(key, "");
}

// 게임 로직은 인터페이스에 의존
public class GameProgressManager
{
    private readonly ISaveSystem _save;
    public GameProgressManager(ISaveSystem save) { _save = save; }

    public void SaveProgress(int level, int score)
    {
        string data = JsonUtility.ToJson(
            new ProgressData { Level = level, Score = score }
        );
        _save.Save("progress", data);
    }
}

// 테스트용 Mock
public class MockSaveSystem : ISaveSystem
{
    public string LastKey;
    public string LastData;
    public void Save(string key, string data)
    {
        LastKey = key;
        LastData = data;
    }
    public string Load(string key) => LastData ?? "";
}

// 테스트
[Test]
public void SaveProgress_StoresCorrectData()
{
    var mock = new MockSaveSystem();
    var manager = new GameProgressManager(mock);

    manager.SaveProgress(level: 5, score: 1000);

    Assert.AreEqual("progress", mock.LastKey);
    StringAssert.Contains("1000", mock.LastData);
}

Part 4: 테스트 패턴과 팁

좋은 테스트의 조건은 뭔가요?

좋은 테스트는 FIRST 원칙을 따릅니다.

원칙의미설명
Fast빠르게수백 개의 테스트가 수 초 안에 완료
Isolated독립적으로다른 테스트에 영향 없이 단독 실행 가능
Repeatable반복 가능하게몇 번을 실행해도 같은 결과
Self-validating자체 검증성공/실패가 자동으로 판별
Timely적시에코드 작성과 함께 테스트 작성
테스트 메서드 이름은 어떻게 짓나요?

테스트 이름은 "메서드_시나리오_기대결과" 패턴을 사용합니다. 이름만 보고 무엇을 검증하는지 알 수 있어야 합니다.

Part 5: AAA 패턴과 Assertion

테스트 코드의 구조는 어떻게 짜야 하나요?

AAA (Arrange-Act-Assert) 패턴을 따르면 일관되고 읽기 쉬운 테스트를 작성할 수 있습니다.

C# - AAA 패턴
[Test]
public void ApplyBuff_DoubleAttack_DoublesAttackPower()
{
    // Arrange - 테스트 환경 준비
    var stats = new CharacterStats(baseAttack: 50);
    var buff = new Buff(BuffType.DoubleAttack, duration: 10f);

    // Act - 테스트 대상 실행
    stats.ApplyBuff(buff);

    // Assert - 결과 검증
    Assert.AreEqual(100, stats.AttackPower);
}
NUnit에서 자주 쓰는 Assert 메서드는?
Assert 메서드용도
Assert.AreEqual(expected, actual)값이 같은지
Assert.IsTrue(condition)조건이 참인지
Assert.IsNull(obj)null인지
Assert.Throws<T>(action)예외가 발생하는지
Assert.That(actual, Is.InRange(a, b))범위 내 값인지
CollectionAssert.Contains(list, item)컬렉션에 포함되는지
C# - 다양한 Assert 활용
// float 비교 시 오차 허용
Assert.AreEqual(3.14f, result, 0.01f);

// 예외 발생 검증
Assert.Throws<ArgumentException>(() => {
    inventory.AddItem(null);
});

// 컬렉션 검증
var loot = lootTable.GenerateLoot();
CollectionAssert.IsNotEmpty(loot);
Assert.That(loot.Count, Is.LessThanOrEqualTo(5));

Part 6: CI 연동과 자동화

테스트를 자동으로 실행할 수는 없나요?

CI/CD 파이프라인에 Unity 테스트를 통합하면 코드 푸시마다 자동으로 테스트가 실행됩니다.

Git Push
코드 변경 커밋
→
CI Trigger
GitHub Actions / Jenkins
→
Unity Test Runner
EditMode + PlayMode 테스트
→
결과 리포트
성공/실패 알림
Unity CLI 테스트 실행 명령어
# EditMode 테스트 실행
Unity.exe -batchmode -runTests \
  -projectPath ./MyProject \
  -testPlatform EditMode \
  -testResults ./results/editmode.xml

# PlayMode 테스트 실행
Unity.exe -batchmode -runTests \
  -projectPath ./MyProject \
  -testPlatform PlayMode \
  -testResults ./results/playmode.xml
테스트 커버리지는 얼마나 목표해야 하나요?

게임 프로젝트에서 100% 커버리지는 비현실적이고 불필요합니다.

핵심 게임 로직(전투 공식, 경제 시스템, 상태 전환)의 커버리지를 80% 이상 유지하는 것이 현실적인 목표입니다. 렌더링, 입력, UI는 수동 QA가 더 효과적입니다.

영역테스트 방식커버리지 목표
전투/스킬 로직EditMode 단위 테스트80%+
인벤토리/경제EditMode 단위 테스트80%+
씬 전환/로딩PlayMode 통합 테스트핵심 경로만
UI/UX수동 QA-
그래픽/이펙트수동 QA-

📌 핵심 정리

  • 테스트의 가치: 회귀 버그를 방지하고, 리팩토링 시 안전망 역할을 한다
  • EditMode vs PlayMode: 순수 로직은 EditMode(빠름), Unity 기능은 PlayMode(느림)
  • 테스트 가능한 설계: 핵심 로직을 POCO로 분리하고, MonoBehaviour는 브릿지만 담당
  • 의존성 주입: 인터페이스와 Mock으로 외부 의존성을 격리하여 테스트
  • CI 자동화: Git Push마다 자동 테스트를 실행하여 코드 품질을 지속적으로 검증
