Resources.Load는 이제 그만!

Addressables로 에셋을 스마트하게 관리하기

Part 1: Resources 폴더의 문제점

Resources.Load가 편한데 왜 쓰면 안 되나요?

Resources 폴더는 Unity 초기부터 있던 에셋 로딩 방식이지만, 심각한 문제들이 있습니다:

  • Resources 폴더의 모든 에셋이 빌드에 포함됨 (사용 여부와 무관)
  • 앱 시작 시 인덱싱으로 로딩 시간 증가
  • 세밀한 메모리 관리 불가 — 언로드 타이밍 제어 어려움
  • 패치/DLC 대응 불가 — 에셋 변경 시 전체 빌드 필요
🎯 비유로 이해하기

Resources.Load는 이사할 때 짐을 전부 트럭에 싣는 것과 같습니다. 여름옷, 겨울옷, 요리 도구, 운동 기구... 당장 필요 없어도 전부 싣습니다.

Addressables는 필요한 짐만 택배로 받는 것입니다. 여름이면 여름옷만, 겨울이면 겨울옷만 배송받죠.

Unity 공식 문서에서도 Resources 폴더 사용을 권장하지 않습니다. 프로토타입이나 아주 작은 프로젝트가 아니라면 Addressables를 사용하세요.

Part 3: 비동기 로딩 실전

코드에서 어떻게 에셋을 로드하나요?
C# - 기본 비동기 로딩
using UnityEngine.AddressableAssets;
using UnityEngine.ResourceManagement.AsyncOperations;

public class AssetLoader : MonoBehaviour
{
    [SerializeField] private AssetReference weaponRef;

    private GameObject loadedWeapon;

    // 방법 1: 콜백
    public void LoadWeaponCallback()
    {
        weaponRef.InstantiateAsync().Completed += (handle) =>
        {
            if (handle.Status == AsyncOperationStatus.Succeeded)
            {
                loadedWeapon = handle.Result;
                Debug.Log("무기 로드 완료!");
            }
            else
            {
                Debug.LogError("무기 로드 실패!");
            }
        };
    }

    // 방법 2: async/await
    public async void LoadWeaponAsync()
    {
        var handle = weaponRef.InstantiateAsync();
        await handle.Task;

        if (handle.Status == AsyncOperationStatus.Succeeded)
        {
            loadedWeapon = handle.Result;
        }
    }

    // 방법 3: 코루틴
    public IEnumerator LoadWeaponCoroutine()
    {
        var handle = weaponRef.InstantiateAsync();
        yield return handle;

        if (handle.Status == AsyncOperationStatus.Succeeded)
        {
            loadedWeapon = handle.Result;
        }
    }
}

AssetReference는 Inspector에서 에셋을 드래그 앤 드롭으로 할당합니다. 문자열 주소보다 안전하고 리팩토링에 강합니다.

문자열 주소로도 로드할 수 있나요?
C# - 주소/라벨 기반 로딩
// 주소로 로드
Addressables.LoadAssetAsync<GameObject>("Prefabs/Enemy_Goblin")
    .Completed += OnEnemyLoaded;

// 라벨로 여러 에셋 로드
Addressables.LoadAssetsAsync<GameObject>("level1_assets",
    (asset) => {
        // 각 에셋 로드 완료 시 호출
        Debug.Log($"로드됨: {asset.name}");
    }
);

주의: 문자열 주소는 오타에 취약합니다. 가능하면 AssetReference나 상수 클래스를 사용하세요.

Part 4: 메모리 관리 — 해제의 중요성

로드한 에셋은 언제 해제해야 하나요?

Addressables는 참조 카운팅 방식으로 메모리를 관리합니다. 로드할 때마다 참조 카운트가 올라가고, Release할 때 내려갑니다. 카운트가 0이 되면 메모리에서 해제됩니다.

C# - 올바른 메모리 관리
public class PropManager : MonoBehaviour
{
    private AsyncOperationHandle<GameObject> handle;
    private GameObject spawnedObject;

    public async void SpawnProp(AssetReference propRef)
    {
        // 로드 (참조 카운트 +1)
        handle = propRef.InstantiateAsync();
        spawnedObject = await handle.Task;
    }

    void OnDestroy()
    {
        // 해제 (참조 카운트 -1)
        if (spawnedObject != null)
        {
            Addressables.ReleaseInstance(spawnedObject);
            // 또는
            // Addressables.Release(handle);
        }
    }
}

// ❌ 흔한 실수
void BadExample()
{
    // handle을 저장하지 않으면 Release할 수 없음!
    someRef.InstantiateAsync(); // 메모리 누수!
}

주의: Addressables로 Instantiate한 오브젝트를 Destroy()로 파괴하면 참조 카운트가 줄지 않아 메모리 누수가 발생합니다. 반드시 Addressables.ReleaseInstance()를 사용하세요.

Part 5: 원격 번들과 패치

서버에서 에셋을 다운로드하려면 어떻게 하나요?

Addressables의 핵심 강점 중 하나가 원격 번들입니다. 에셋을 CDN 서버에 올려두고 런타임에 다운로드할 수 있습니다.

빌드 시
Remote 그룹을 CDN에 업로드
→

카탈로그 체크 → 다운로드
→
캐시
로컬에 저장, 재사용
C# - 다운로드 크기 확인 및 진행
public async void CheckAndDownload(string label)
{
    // 다운로드 크기 확인
    var sizeHandle = Addressables.GetDownloadSizeAsync(label);
    long size = await sizeHandle.Task;

    if (size > 0)
    {
        Debug.Log($"다운로드 필요: {size / 1024 / 1024}MB");

        // 다운로드 시작
        var downloadHandle = Addressables.DownloadDependenciesAsync(label);

        while (!downloadHandle.IsDone)
        {
            float progress = downloadHandle.PercentComplete;
            progressBar.value = progress;
            await Task.Yield();
        }

        Addressables.Release(downloadHandle);
        Debug.Log("다운로드 완료!");
    }
    else
    {
        Debug.Log("이미 캐시됨, 다운로드 불필요");
    }
}

카탈로그를 업데이트하면 클라이언트 빌드 없이 에셋 패치가 가능합니다. 이벤트 콘텐츠, 시즌 업데이트에 필수적입니다.

Part 6: 프로파일링과 최적화

Addressables의 메모리 사용을 어떻게 확인하나요?

Window → Asset Management → Addressables Event Viewer를 사용합니다.

  • 참조 카운트가 0이 되지 않는 에셋 → 메모리 누수 후보
  • 번들 로드/언로드 타이밍 확인 → 불필요한 로드 감지
  • 중복 에셋 검출 → 여러 번들에 같은 텍스처가 포함되는 경우

최적화 팁:

  • 자주 함께 사용되는 에셋을 같은 그룹으로 묶기
  • 씬 전환 시 이전 씬 에셋 Release 확실히 하기
  • 프리로드가 필요한 에셋은 로딩 화면에서 미리 로드
  • 텍스처, 오디오 등 큰 에셋은 별도 번들로 분리

Addressables Analyze 도구를 사용하면 번들 간 중복 에셋을 자동으로 검출하고, 최적의 그룹 구성을 제안받을 수 있습니다.

📌 핵심 정리

  • Resources 폴더는 모든 에셋을 빌드에 포함시키므로 프로덕션에서는 사용하지 않는다
  • Addressables는 비동기 로딩과 참조 카운팅으로 메모리를 효율적으로 관리한다
  • AssetReference를 사용하면 Inspector에서 안전하게 에셋을 참조할 수 있다
  • Addressables.ReleaseInstance로 반드시 해제해야 메모리 누수를 방지한다
  • 원격 번들을 활용하면 클라이언트 빌드 없이 에셋 패치가 가능하다
