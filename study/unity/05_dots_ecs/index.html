Part 1: OOP의 한계와 DOD의 등장

GameObject와 MonoBehaviour로 충분하지 않나요?

수백 개의 오브젝트까지는 문제없습니다. 하지만 수만~수십만 개의 유닛을 시뮬레이션해야 한다면? 전통적인 OOP 방식은 심각한 성능 병목에 부딪힙니다.

🎯 비유로 이해하기

OOP는 서랍장입니다. 각 서랍(오브젝트)에 양말, 속옷, 연필, 충전기가 뒤섞여 있습니다. "양말만 전부 꺼내"라고 하면 모든 서랍을 뒤져야 합니다.

DOD(Data-Oriented Design)는 칸이 나뉜 수납장입니다. 양말은 양말끼리, 속옷은 속옷끼리 모여 있어서 "양말만 전부" 가져오기가 훨씬 빠릅니다.

구분OOP (GameObject)DOD (ECS)
메모리 배치오브젝트별로 흩어짐같은 타입끼리 연속 배치
캐시 효율Cache Miss 빈번Cache Hit 극대화
오브젝트 수수백~수천 개수만~수십만 개
멀티스레드수동 관리자동 병렬화
Burst 호환제한적완벽 호환

DOTS(Data-Oriented Technology Stack)는 ECS + Job System + Burst Compiler를 결합한 Unity의 고성능 프레임워크입니다.

Part 2: Entity, Component, System

ECS의 세 가지 핵심 요소는 무엇인가요?
  • Entity: 고유 ID에 불과합니다. GameObject처럼 기능이 없고, 단순히 "이 컴포넌트 조합이 존재한다"는 식별자입니다.
  • Component: 순수한 데이터만 담은 struct입니다. 로직이 없습니다.
  • System: 특정 Component 조합을 가진 Entity들을 처리하는 로직입니다.
C# - Component 정의 (순수 데이터)
using Unity.Entities;
using Unity.Mathematics;

// 위치 데이터
public struct Position : IComponentData
{
    public float3 Value;
}

// 이동 속도 데이터
public struct MoveSpeed : IComponentData
{
    public float Value;
}

// 체력 데이터
public struct Health : IComponentData
{
    public int Current;
    public int Max;
}

// 태그 컴포넌트 (데이터 없이 "표시"만)
public struct EnemyTag : IComponentData { }

Component는 struct이고 로직이 없습니다. "이 Entity는 위치가 있고, 이동 속도가 있고, 체력이 있다"는 것을 데이터 조합으로 표현합니다.

System은 어떻게 작성하나요?
C# - System 정의
using Unity.Entities;
using Unity.Transforms;
using Unity.Burst;

[BurstCompile]
public partial struct MoveForwardSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        float dt = SystemAPI.Time.DeltaTime;

        // Position과 MoveSpeed를 가진 모든 Entity를 순회
        foreach (var (transform, speed)
            in SystemAPI.Query<RefRW<LocalTransform>,
                                RefRO<MoveSpeed>>())
        {
            transform.ValueRW.Position +=
                new float3(0, 0, speed.ValueRO.Value * dt);
        }
    }
}

// 적만 처리하는 System
[BurstCompile]
public partial struct EnemyDamageSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        foreach (var (health, entity)
            in SystemAPI.Query<RefRW<Health>>()
                        .WithAll<EnemyTag>()
                        .WithEntityAccess())
        {
            if (health.ValueRO.Current <= 0)
            {
                state.EntityManager.DestroyEntity(entity);
            }
        }
    }
}

Part 3: Archetype과 Chunk — 성능의 비밀

ECS가 빠른 진짜 이유는 무엇인가요?

핵심은 Archetype과 Chunk 기반 메모리 레이아웃입니다.

Archetype은 동일한 Component 조합을 가진 Entity의 "타입"입니다. 같은 Archetype의 Entity들은 메모리에서 연속으로 배치됩니다.

Archetype A
Position + MoveSpeed + Health
Archetype B
Position + MoveSpeed
Archetype C
Position + Health + EnemyTag
Chunk 메모리 레이아웃 (16KB)
Position[0..N]MoveSpeed[0..N]Health[0..N]
캐시 라인 접근
연속 읽기 → Cache Hit!

각 Chunk는 16KB 크기의 메모리 블록으로, 같은 Archetype의 Component 데이터가 배열처럼 연속으로 저장됩니다. CPU 캐시 라인에 딱 맞아서 Cache Miss가 거의 발생하지 않습니다.

Component를 추가/제거하면 어떻게 되나요?

주의: Entity에 Component를 추가하거나 제거하면 Archetype이 변경되어 Entity가 다른 Chunk로 이동합니다. 이를 "구조적 변경(Structural Change)"이라 하며, 비용이 큽니다.

런타임에 빈번한 구조적 변경은 피해야 합니다. 대신 Enable/Disable Component나 Shared Component 전환을 고려하세요.

Part 4: Entity 생성과 관리

Entity는 어떻게 생성하나요?
C# - Entity 생성
public partial struct SpawnSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        var ecb = new EntityCommandBuffer(Allocator.Temp);

        // 단일 Entity 생성
        Entity entity = ecb.CreateEntity();
        ecb.AddComponent(entity, new Position { Value = float3.zero });
        ecb.AddComponent(entity, new MoveSpeed { Value = 5f });
        ecb.AddComponent(entity, new Health { Current = 100, Max = 100 });
        ecb.AddComponent(entity, new EnemyTag());

        ecb.Playback(state.EntityManager);
        ecb.Dispose();
    }
}

// 대량 생성 (프리팹 기반)
public partial struct MassSpawnSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        // Entity 프리팹에서 1만 개 복제
        var prefab = SystemAPI.GetSingleton<EnemyPrefab>().Value;

        state.EntityManager.Instantiate(prefab, 10000,
                                         Allocator.Temp);
    }
}

EntityCommandBuffer(ECB)를 사용하면 구조적 변경을 모아서 한 번에 처리할 수 있어 성능이 좋습니다.

Part 5: 성능 비교 — GameObject vs ECS

실제로 얼마나 차이가 나나요?

1만 개의 오브젝트를 매 프레임 이동시키는 시나리오 비교입니다:

지표GameObject + MonoBehaviourDOTS ECS + Burst
오브젝트 수10,00010,000
프레임 시간~33ms (30fps)~0.5ms (2000fps+)
메모리~400MB~20MB
GC 할당매 프레임 발생0 (Zero GC)
CPU 코어 활용1개전체 코어

DOTS는 단순 이동 같은 작업에서 60~100배 이상 빠를 수 있습니다. 특히 Burst Compiler와 결합하면 SIMD 벡터화까지 적용되어 극한의 성능을 발휘합니다.

그러면 모든 프로젝트에서 ECS를 써야 하나요?

아닙니다. ECS는 만능이 아닙니다:

  • 학습 비용이 높고, 기존 Unity 워크플로우와 완전히 다릅니다
  • 복잡한 상태 머신, AI 로직은 ECS로 구현하기 까다롭습니다
  • UI, 에디터 통합 등 기존 기능과의 호환성 제한
  • 소규모 프로젝트에서는 오버엔지니어링

대규모 시뮬레이션(RTS 유닛, 탄막, 파티클 시뮬레이션)에 ECS를 적용하고, 나머지는 전통적인 방식을 유지하는 하이브리드 접근이 현실적입니다.

Part 6: 하이브리드 접근 — 공존의 기술

기존 GameObject와 ECS를 함께 쓸 수 있나요?

가능합니다. Unity는 GameObject를 ECS Entity로 변환(Baking)하는 워크플로우를 제공합니다.

C# - Baker를 사용한 변환
// MonoBehaviour (에디터용 오서링 컴포넌트)
public class EnemyAuthoring : MonoBehaviour
{
    public float moveSpeed;
    public int maxHealth;
}

// Baker (GameObject → Entity 변환)
public class EnemyBaker : Baker<EnemyAuthoring>
{
    public override void Bake(EnemyAuthoring authoring)
    {
        var entity = GetEntity(TransformUsageFlags.Dynamic);

        AddComponent(entity, new MoveSpeed
        {
            Value = authoring.moveSpeed
        });
        AddComponent(entity, new Health
        {
            Current = authoring.maxHealth,
            Max = authoring.maxHealth
        });
        AddComponent(entity, new EnemyTag());
    }
}
에디터
GameObject + Authoring
→
Baking
Baker가 변환
→
런타임
Entity + Components

에디터에서는 익숙한 GameObject 워크플로우로 작업하고, 런타임에서는 ECS의 성능을 누리는 것이 하이브리드 접근의 핵심입니다.

📌 핵심 정리

  • ECS는 Entity(ID) + Component(데이터) + System(로직) 분리로 캐시 효율을 극대화한다
  • Archetype과 Chunk 기반 메모리 레이아웃이 압도적 성능의 핵심이다
  • 구조적 변경(Component 추가/제거)은 비용이 크므로 EntityCommandBuffer로 모아서 처리한다
  • 모든 프로젝트에 ECS가 필요한 것은 아니다 — 대규모 시뮬레이션에 집중 적용한다
  • Baker를 활용한 하이브리드 접근으로 에디터 편의성과 런타임 성능을 모두 잡을 수 있다
