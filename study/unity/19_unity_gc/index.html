같은 C#인데 .NET과 Unity의 GC가 다르다고?

Unity와 C#의 가비지 컬렉션 - Mark and Sweep부터 Boehm GC까지

Part 1: C#의 GC 기본 원리 - Mark and Sweep

C#의 GC는 어떤 알고리즘을 사용하나요?

C#(.NET)의 GC는 Mark and Sweep 알고리즘을 사용합니다.

1. Root 설정
전역 변수, 로컬 변수 등
→
2. Mark
Root에서 참조를 추적하며 방문 표시
→
3. Sweep
Mark 안 된 객체를 가비지로 해제

Part 2: .NET GC vs Unity GC

.NET과 Unity의 GC는 어떻게 다른가요?

둘 다 Mark and Sweep을 기반으로 하지만, 그 이후 과정이 다릅니다.

구분 .NET GC Unity GC (Boehm)
알고리즘 세대별 GC (0~2세대) Boehm-Demers-Weiser
세대 구분 O (3개 세대) X
메모리 정렬 O (Compaction) X
최적화 방향 자동 세대 승격 점진적 GC (Unity 2019+)

Unity의 Boehm GC는 세대 구분도 메모리 정렬도 없기 때문에, 개발자가 직접 오브젝트 풀링 등의 최적화를 적극적으로 해줘야 합니다.

Part 3: 상호 참조 문제

두 객체가 서로 참조하고 있으면 GC가 해제하지 못하나요?

아닙니다. Mark and Sweep은 Root에서부터 참조를 추적합니다. 두 객체가 서로 참조하고 있더라도 외부(Root)에서 참조가 없어 Mark되지 않으면 Sweep 단계에서 해제됩니다.

🎯 비유로 이해하기

두 사람이 서로의 전화번호만 가지고 있고, 주소록에 이 두 사람의 번호가 없다면? Mark(주소록 검색) 과정에서 찾을 수 없으므로 Sweep(정리) 대상이 됩니다.

Part 4: Unity GC 최적화 전략

Unity에서 GC 부담을 줄이려면 어떻게 해야 하나요?
  • 오브젝트 풀링: Instantiate/Destroy 대신 풀에서 재사용
  • 점진적 GC: Unity 2019 이상에서 GC를 프레임에 분산 처리
  • 가비지 생성 최소화: string 연결 대신 StringBuilder, 박싱 회피 등
  • 캐싱: GetComponent 결과를 변수에 저장하여 반복 호출 방지

Unity의 Boehm GC는 .NET GC보다 최적화가 부족하므로, 개발자의 적극적인 메모리 관리가 필수입니다.

📌 핵심 정리

  • C#의 GC는 Mark and Sweep 알고리즘 기반이다
  • .NET GC는 세대별 관리(0~2세대)와 메모리 정렬을 지원한다
  • Unity GC(Boehm)는 세대 구분과 메모리 정렬이 없어 최적화가 부족하다
  • 상호 참조는 Root에서 접근 불가하면 GC에 의해 해제된다
  • 오브젝트 풀링, 점진적 GC, 가비지 생성 최소화로 최적화해야 한다
