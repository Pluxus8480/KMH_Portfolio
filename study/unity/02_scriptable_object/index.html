MonoBehaviour 없이 데이터를 담는 법

ScriptableObject로 설계하는 유연한 데이터 아키텍처

Part 1: ScriptableObject가 필요한 이유

MonoBehaviour에 데이터를 넣으면 안 되나요?

가능하지만 심각한 문제가 발생합니다. MonoBehaviour는 게임 오브젝트에 부착되어야 하고, 인스턴스마다 데이터를 복사합니다.

🎯 비유로 이해하기

RPG에서 "철검"의 공격력, 가격 정보가 있다고 합시다. MonoBehaviour에 넣으면 철검이 100개 있을 때 같은 데이터가 100번 복사됩니다.

ScriptableObject는 도서관의 책과 같습니다. 원본은 하나이고, 모두가 같은 책을 참조합니다.

ScriptableObject는 씬에 존재하지 않는 데이터 컨테이너입니다. 에셋으로 저장되어 여러 오브젝트가 동일한 데이터를 공유할 수 있습니다.

구체적으로 어떤 이점이 있나요?
구분MonoBehaviour 데이터ScriptableObject
메모리인스턴스마다 복사하나의 에셋을 공유
씬 의존성씬에 종속에셋으로 독립
에디터 수정프리팹 수정 필요에셋에서 직접 수정
런타임 변경인스턴스별 독립공유 데이터 (주의 필요)
직렬화씬 파일에 저장별도 에셋 파일

Part 2: ScriptableObject 생성과 활용

어떻게 만들고 사용하나요?
C# - 아이템 데이터 정의
[CreateAssetMenu(fileName = "NewWeapon", menuName = "Game Data/Weapon")]
public class WeaponData : ScriptableObject
{
    public string weaponName;
    public int attackPower;
    public float attackSpeed;
    public float range;
    public Sprite icon;
    public GameObject prefab;

    [TextArea(3, 5)]
    public string description;

    public float DPS => attackPower * attackSpeed;
}

CreateAssetMenu 어트리뷰트를 추가하면 Project 창에서 우클릭 → Create → Game Data → Weapon으로 에셋을 생성할 수 있습니다.

C# - 데이터 사용
public class WeaponController : MonoBehaviour
{
    [SerializeField] private WeaponData weaponData;

    void Start()
    {
        Debug.Log($"장착 무기: {weaponData.weaponName}");
        Debug.Log($"DPS: {weaponData.DPS}");
    }

    public void Attack(Enemy target)
    {
        target.TakeDamage(weaponData.attackPower);
    }
}

Inspector에서 무기 데이터 에셋을 드래그 앤 드롭으로 할당합니다. 같은 WeaponData를 여러 오브젝트가 공유할 수 있습니다.

Part 3: 이벤트 채널 패턴

ScriptableObject로 이벤트 시스템을 만들 수 있다고요?

ScriptableObject를 이벤트 채널로 활용하면 오브젝트 간의 직접 참조 없이 통신할 수 있습니다. 이를 ScriptableObject 이벤트 아키텍처라 부릅니다.

🎯 비유로 이해하기

라디오 주파수를 생각하세요. 방송국(발신자)은 특정 주파수로 송출하고, 라디오(수신자)는 그 주파수에 맞추면 됩니다. 두 쪽 모두 상대방을 알 필요가 없습니다.

ScriptableObject 이벤트 채널이 바로 이 "주파수"입니다.

C# - 이벤트 채널 정의
[CreateAssetMenu(menuName = "Events/Game Event")]
public class GameEvent : ScriptableObject
{
    private readonly List<GameEventListener> listeners = new();

    public void Raise()
    {
        // 역순 호출 (리스너 제거 안전)
        for (int i = listeners.Count - 1; i >= 0; i--)
            listeners[i].OnEventRaised();
    }

    public void Register(GameEventListener listener)
        => listeners.Add(listener);

    public void Unregister(GameEventListener listener)
        => listeners.Remove(listener);
}
C# - 이벤트 리스너
public class GameEventListener : MonoBehaviour
{
    [SerializeField] private GameEvent gameEvent;
    [SerializeField] private UnityEvent response;

    void OnEnable() => gameEvent.Register(this);
    void OnDisable() => gameEvent.Unregister(this);

    public void OnEventRaised() => response.Invoke();
}

이 패턴의 장점: 플레이어 사망 이벤트를 UI, 사운드, 게임 매니저가 각각 독립적으로 구독합니다. 서로를 전혀 알지 못해도 됩니다.

Part 4: 런타임 데이터 vs 설정 데이터

ScriptableObject 데이터를 런타임에 수정하면 어떻게 되나요?

주의: 에디터에서는 ScriptableObject의 런타임 변경이 영구 저장됩니다! 빌드에서는 원본이 보존되지만, 에디터에서 플레이 모드 중 값을 바꾸면 플레이 종료 후에도 변경이 유지됩니다.

이 문제를 해결하는 방법:

C# - 런타임 데이터 분리 패턴
// 설정 데이터 (변경 불가)
[CreateAssetMenu(menuName = "Game Data/Character Config")]
public class CharacterConfig : ScriptableObject
{
    public int maxHP;
    public float moveSpeed;
    public int baseAttack;
}

// 런타임 데이터 (인스턴스별 상태)
public class CharacterRuntime
{
    public int currentHP;
    public int level;
    public List<BuffData> activeBuffs;

    public CharacterRuntime(CharacterConfig config)
    {
        currentHP = config.maxHP;
        level = 1;
        activeBuffs = new List<BuffData>();
    }
}

// 사용
public class Character : MonoBehaviour
{
    [SerializeField] private CharacterConfig config;
    private CharacterRuntime runtime;

    void Awake()
    {
        runtime = new CharacterRuntime(config);
    }
}

설정(Config)은 ScriptableObject로, 상태(State)는 일반 클래스로 분리하는 것이 안전합니다.

Part 5: 에디터 활용과 Enum 대체

ScriptableObject로 Enum을 대체할 수 있다고요?

Enum은 값을 추가/삭제할 때 코드를 수정해야 하고, 순서가 바뀌면 기존 데이터가 깨집니다. ScriptableObject를 사용하면 이 문제를 해결할 수 있습니다.

C# - Enum 대체 패턴
// Enum 대신 ScriptableObject 사용
[CreateAssetMenu(menuName = "Game Data/Element Type")]
public class ElementType : ScriptableObject
{
    public Color color;
    public float damageMultiplier;
    public ParticleSystem effectPrefab;
}

// 사용: Inspector에서 드래그 앤 드롭
public class Spell : MonoBehaviour
{
    [SerializeField] private ElementType element;

    public void Cast()
    {
        // element.color, element.damageMultiplier 등 활용
        var fx = Instantiate(element.effectPrefab, transform.position,
                             Quaternion.identity);
    }
}

이 방식의 장점: 새로운 속성(예: "독" 원소)을 추가할 때 코드를 전혀 수정하지 않아도 됩니다. 에셋만 하나 더 만들면 됩니다.

Part 6: 실전 설계 — 스킬 시스템 예시

실제 게임에서 ScriptableObject를 어떻게 조합하나요?
SkillData (SO)
이름, 쿨타임, 데미지, 아이콘
→
SkillEffect (SO)
파티클, 사운드, 적용 로직
→
SkillExecutor (MB)
실행, 쿨다운 관리
C# - 스킬 데이터 구조
[CreateAssetMenu(menuName = "Game Data/Skill")]
public class SkillData : ScriptableObject
{
    public string skillName;
    public Sprite icon;
    public float cooldown;
    public float manaCost;
    public AnimationClip castAnimation;
    public SkillEffect[] effects; // 다중 효과 조합

    public bool CanUse(float currentMana, float lastUsedTime)
    {
        return currentMana >= manaCost
            && Time.time - lastUsedTime >= cooldown;
    }
}

public abstract class SkillEffect : ScriptableObject
{
    public abstract void Apply(Character caster, Character target);
}

[CreateAssetMenu(menuName = "Game Data/Skill Effects/Damage")]
public class DamageEffect : SkillEffect
{
    public float baseDamage;
    public ElementType element;

    public override void Apply(Character caster, Character target)
    {
        float damage = baseDamage * element.damageMultiplier;
        target.TakeDamage(damage);
    }
}

기획자가 에디터에서 스킬 데이터를 조합하고, 이펙트를 교체하고, 밸런스를 조정할 수 있습니다. 프로그래머의 개입 없이도요.

📌 핵심 정리

  • ScriptableObject는 씬 독립적인 데이터 컨테이너로, 메모리를 절약하고 데이터를 공유한다
  • 이벤트 채널 패턴으로 오브젝트 간 결합도를 제거할 수 있다
  • 설정 데이터(Config)와 런타임 상태(State)를 반드시 분리해야 한다
  • Enum 대신 ScriptableObject를 사용하면 코드 수정 없이 데이터 확장이 가능하다
  • 기획자가 에디터에서 직접 데이터를 관리할 수 있는 워크플로우를 만들 수 있다
