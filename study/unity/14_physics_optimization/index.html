Unity Physics 최적화 전략과 성능 튜닝

Part 1: Unity Physics의 동작 원리

Unity 물리 엔진은 어떻게 동작하나요?

Unity는 내부적으로 NVIDIA PhysX 엔진을 사용합니다. 물리 시뮬레이션은 FixedUpdate 주기마다 실행되며, 크게 세 단계로 나뉩니다.

1. Broadphase
AABB로 후보 쌍 필터링
→
2. Narrowphase
정밀 충돌 검사
→
3. Solver
충돌 응답 + 힘 적용

Broadphase에서 충돌 가능한 쌍을 빠르게 걸러내는 것이 성능의 핵심입니다. 이 단계에서 불필요한 쌍이 많으면 Narrowphase 비용이 폭증합니다.

FixedUpdate와 Update의 차이가 뭔가요?
구분UpdateFixedUpdate
호출 주기매 프레임 (가변)고정 간격 (기본 0.02초)
프레임 의존프레임레이트에 따라 변동일정한 시간 간격 보장
용도입력, 렌더링 로직물리 연산, Rigidbody 조작
deltaTimeTime.deltaTimeTime.fixedDeltaTime

Rigidbody에 힘을 가하는 코드를 Update()에 넣으면 프레임레이트에 따라 결과가 달라집니다. 반드시 FixedUpdate()에서 처리하세요.

Part 2: Layer Collision Matrix

Layer Collision Matrix가 뭔가요?

물리 엔진이 어떤 레이어끼리 충돌을 검사할지 정의하는 행렬입니다.

Project Settings > Physics에서 설정할 수 있습니다.

🎯 비유로 이해하기

학교에서 모든 반이 모든 반과 시합하면 경기 수가 폭발적으로 늘어납니다.

Layer Collision Matrix는 "1반은 2반하고만, 3반은 4반하고만 시합"처럼 불필요한 경기를 제거합니다.

어떻게 설정해야 효율적인가요?

기본값은 모든 레이어가 모든 레이어와 충돌합니다. 필요한 조합만 활성화하세요.

레이어PlayerEnemyBulletEnvironmentUI
Player-O-O-
EnemyO-OO-
Bullet-O-O-
EnvironmentOOO--
UI-----

이 예시에서 Player끼리, Bullet끼리, UI와 다른 레이어의 충돌은 아예 검사하지 않습니다.

C# - 코드에서 레이어 충돌 설정
// 특정 레이어 쌍의 충돌 무시
int playerLayer = LayerMask.NameToLayer("Player");
int bulletLayer = LayerMask.NameToLayer("Bullet");

// 아군 총알은 플레이어와 충돌하지 않음
Physics.IgnoreLayerCollision(playerLayer, bulletLayer, true);

Part 3: Rigidbody 최적화

Rigidbody 설정에서 성능에 영향을 주는 항목은?
설정설명최적화 팁
Interpolate프레임 간 보간으로 떨림 방지플레이어만 활성화, NPC는 None
Collision Detection충돌 검사 정밀도대부분 Discrete, 고속 물체만 Continuous
Is Kinematic물리 시뮬레이션 비활성화움직이지 않는 물체는 Kinematic으로
Sleeping정지 상태 오브젝트 자동 비활성화Sleep 임계값을 적절히 설정
Collision Detection 모드는 정확히 어떻게 다른가요?

Discrete는 프레임 사이 위치만 검사합니다. 빠르게 움직이는 물체가 다른 물체를 뚫고 지나가는 "터널링" 문제가 발생할 수 있습니다.

모드비용용도
Discrete낮음느린 물체, 대부분의 경우
Continuous중간빠른 물체 vs 정적 콜라이더
Continuous Dynamic높음빠른 물체 vs 빠른 물체
Continuous Speculative중간회전하는 물체에 적합

모든 Rigidbody에 Continuous를 설정하지 마세요. 총알처럼 정말 빠른 물체에만 사용하고, 나머지는 Discrete로 두세요.

움직이지 않는 콜라이더는 어떻게 처리해야 하나요?

Static Collider(Rigidbody 없는 콜라이더)를 런타임에 이동시키면 물리 엔진이 내부 자료구조를 매번 재구축합니다.

움직이는 오브젝트에는 반드시 Rigidbody를 붙이세요. 물리 시뮬레이션이 필요 없다면 Is Kinematic을 체크하세요.

Static Collider 이동 시 내부 동작
// BAD: Rigidbody 없이 콜라이더만 이동
// → PhysX가 Static Tree를 매번 재구축 (매우 비쌈)
transform.position = newPosition;

// GOOD: Kinematic Rigidbody 사용
// → Dynamic Tree에서 관리되어 이동 비용이 낮음
[RequireComponent(typeof(Rigidbody))]
public class MovingPlatform : MonoBehaviour
{
    private Rigidbody _rb;
    void Start()
    {
        _rb = GetComponent<Rigidbody>();
        _rb.isKinematic = true;
    }
    void FixedUpdate()
    {
        _rb.MovePosition(CalculateNextPosition());
    }
}

Part 4: Trigger vs Collision

Trigger와 Collision의 차이는 뭔가요?
구분CollisionTrigger
물리 반응있음 (밀어내기, 튕기기)없음 (통과)
콜백OnCollisionEnter/Stay/ExitOnTriggerEnter/Stay/Exit
비용높음 (접촉점 계산)낮음 (겹침 여부만)
용도물리적 충돌이 필요한 경우영역 감지, 아이템 획득

물리적 반응이 필요 없는 감지 영역(아이템 획득, 영역 진입 등)에는 Trigger를 사용하세요. Collision보다 가볍습니다.

콜라이더 모양도 성능에 영향을 주나요?

네, 콜라이더의 복잡도에 따라 충돌 검사 비용이 크게 달라집니다.

Sphere
가장 빠름
→
Capsule
빠름
→
Box
보통
→
Mesh Collider
가장 느림

Mesh Collider끼리의 충돌은 Convex가 아니면 지원되지 않으며, Convex Mesh Collider도 primitive보다 수배 느립니다. 가능하면 Box/Capsule 조합으로 대체하세요.

Part 5: Raycast 최적화

Raycast를 많이 쓰면 느려지나요?

Raycast 자체는 빠르지만, 매 프레임 수십 개씩 쏘면 부담이 됩니다. 특히 MeshCollider가 많은 씬에서 비용이 증가합니다.

C# - Raycast 최적화 패턴
// BAD: 레이어마스크 없이 모든 콜라이더 검사
Physics.Raycast(origin, direction, out hit, 100f);

// GOOD: 레이어마스크로 검사 범위 제한
int layerMask = LayerMask.GetMask("Enemy", "Environment");
Physics.Raycast(origin, direction, out hit, 100f, layerMask);

// GOOD: NonAlloc 버전으로 GC 방지
private RaycastHit[] _hits = new RaycastHit[10];
void CheckHits()
{
    int count = Physics.RaycastNonAlloc(
        origin, direction, _hits, 100f, layerMask
    );
    for (int i = 0; i < count; i++)
    {
        ProcessHit(_hits[i]);
    }
}

// GOOD: SphereCast/BoxCast로 정밀도와 성능 균형
Physics.SphereCast(origin, 0.5f, direction, out hit, 100f, layerMask);

RaycastAll 대신 RaycastNonAlloc을 사용하면 매 호출마다 배열을 할당하지 않아 GC 부담이 없습니다.

Part 6: Fixed Timestep 튜닝

Fixed Timestep을 바꾸면 어떤 영향이 있나요?

기본값은 0.02초(50Hz)입니다. 이 값을 바꾸면 물리 시뮬레이션의 정밀도와 비용이 변합니다.

Timestep초당 호출정밀도CPU 비용
0.01100회높음높음
0.02 (기본)50회보통보통
0.0425회낮음낮음

Maximum Allowed Timestep도 중요합니다. 프레임이 심하게 떨어지면 FixedUpdate가 한 프레임에 여러 번 호출되어 "죽음의 소용돌이"에 빠질 수 있습니다. 기본값 0.333초를 유지하거나 더 줄이세요.

모바일에서 물리 최적화 팁을 정리해주세요.

모바일은 CPU/GPU 모두 제한적이므로 더 공격적인 최적화가 필요합니다.

  • Fixed Timestep을 0.04로 올려 물리 연산 횟수를 절반으로 줄이기
  • Solver Iterations를 4 이하로 줄이기 (기본 6)
  • Auto Sync Transforms를 비활성화하고 필요할 때만 Physics.SyncTransforms() 호출
  • 사용하지 않는 레이어 쌍의 충돌을 Layer Collision Matrix에서 해제
  • Mesh Collider 대신 primitive 콜라이더 조합 사용

📌 핵심 정리

  • Layer Collision Matrix: 필요한 레이어 쌍만 충돌하도록 설정하여 Broadphase 비용을 줄여라
  • Rigidbody 규칙: 움직이는 물체에는 반드시 Rigidbody를 붙이고, 물리 불필요 시 Kinematic으로
  • 콜라이더 복잡도: Sphere > Capsule > Box > Mesh 순으로 빠르다. Mesh Collider는 최후의 수단
  • Raycast: 레이어마스크 필수, NonAlloc 버전으로 GC 방지
  • Fixed Timestep: 모바일에서는 0.04로 올려 물리 연산 비용을 반으로 줄여라
