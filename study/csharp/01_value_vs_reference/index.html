스택에 살까, 힙에 살까?

값 타입과 참조 타입의 메모리 전략과 Boxing의 함정

Part 1: 두 세계의 주민들

C#에서 값 타입과 참조 타입은 왜 구분되어 있나요?

C#은 모든 타입을 두 가지 세계로 분류합니다. 값 타입(Value Type)은 데이터를 직접 들고 다니고, 참조 타입(Reference Type)은 데이터가 있는 곳의 주소만 들고 다닙니다.

🎯 비유로 이해하기

값 타입은 종이에 적은 메모와 같습니다. 복사하면 새 종이에 내용을 그대로 베끼는 거죠. 원본을 수정해도 복사본에는 영향이 없습니다.

참조 타입은 도서관 좌석 번호표와 같습니다. 번호표를 복사해도 가리키는 좌석은 같은 곳입니다. 누가 그 좌석의 책을 바꾸면, 번호표를 가진 모든 사람이 바뀐 책을 보게 됩니다.

값 타입: int, float, bool, struct, enum — 스택에 직접 저장

참조 타입: class, string, array, delegate — 힙에 저장, 스택에 참조(주소) 보관

struct와 class는 어떤 차이가 있나요?
C#
// 값 타입 - struct
struct Vector3
{
    public float X, Y, Z;
}

// 참조 타입 - class
class Transform
{
    public Vector3 Position;
    public Vector3 Rotation;
}

void Example()
{
    Vector3 a = new Vector3 { X = 1, Y = 2, Z = 3 };
    Vector3 b = a;    // 값 복사! a와 b는 완전히 독립
    b.X = 100;        // a.X는 여전히 1

    Transform t1 = new Transform();
    Transform t2 = t1; // 참조 복사! t1과 t2는 같은 객체
    t2.Position.X = 100; // t1.Position.X도 100으로 변경됨
}
구분 struct (값 타입) class (참조 타입)
저장 위치 스택 (지역변수일 때) 힙
대입 시 값 전체 복사 참조(주소)만 복사
기본값 모든 필드 0/null null
상속 불가 가능
GC 대상 아님 (스택) 대상

Part 2: 메모리 레이아웃의 진실

스택과 힙은 실제로 어떻게 다른가요?

스택(Stack)은 함수 호출 시 자동으로 할당되고 반환 시 자동으로 해제됩니다. LIFO(Last In First Out) 구조로 매우 빠릅니다.

힙(Heap)은 프로그래머가 명시적으로 할당(new)하며, GC가 더 이상 참조되지 않는 객체를 수거합니다.

스택 (Stack)
빠른 할당/해제
크기 제한 (~1MB)
지역 변수, 매개변수
↔
힙 (Heap)
느린 할당, GC 수거
크기 유연
new로 생성된 객체

struct가 class의 필드로 포함되면, struct도 힙에 저장됩니다. 저장 위치는 "타입"이 아니라 "어디에 속해 있느냐"에 따라 달라집니다.

struct가 항상 스택에 있는 건 아니라고요?

맞습니다. struct가 스택에 저장되는 건 지역 변수일 때뿐입니다.

C#
class Monster
{
    // 이 Vector3는 Monster 객체와 함께 힙에 저장됨!
    public Vector3 Position;
}

void Update()
{
    Vector3 localPos = new Vector3(); // 스택에 저장
    Monster m = new Monster();        // Monster는 힙, Position도 힙
}

면접에서 "struct는 스택에 저장됩니다"라고 단정하면 감점 요소입니다. 정확히는 "지역 변수로 선언된 struct는 스택에, 참조 타입의 필드인 struct는 힙에 저장됩니다."

Part 3: Boxing과 Unboxing의 함정

Boxing이 뭐고 왜 위험한가요?

Boxing은 값 타입을 참조 타입(object)으로 변환하는 과정입니다. 이때 힙에 새 객체가 할당되고, 값이 복사됩니다.

Unboxing은 반대로 object에서 값 타입을 꺼내는 과정입니다.

C#
int score = 100;

// Boxing: int -> object (힙 할당 발생!)
object boxed = score;

// Unboxing: object -> int (타입 검사 + 값 복사)
int unboxed = (int)boxed;

// 잘못된 Unboxing -> InvalidCastException!
// double wrong = (double)boxed;
스택: int 100
4바이트
Boxing →
힙: object
타입 헤더 + 동기화 블록 + 값
약 20~24바이트

Boxing 한 번에 약 20바이트 이상의 힙 할당이 발생합니다. 매 프레임 수천 번 반복되면 GC 압박의 주범이 됩니다.

게임에서 Boxing이 몰래 발생하는 경우는 어디인가요?

의도치 않은 Boxing은 생각보다 흔합니다.

C# - 숨겨진 Boxing 사례들
// 1. string.Format에서 값 타입 전달
int hp = 100;
string text = string.Format("HP: {0}", hp); // hp가 Boxing됨!

// 2. 비제네릭 컬렉션 사용
ArrayList list = new ArrayList();
list.Add(42); // Boxing 발생!

// 3. 인터페이스로 캐스팅
struct Bullet : IUpdatable
{
    public void Update() { }
}
IUpdatable bullet = new Bullet(); // Boxing!

// 4. Enum.HasFlag (구버전 .NET)
MyFlags flags = MyFlags.A | MyFlags.B;
flags.HasFlag(MyFlags.A); // Boxing 발생 (구버전)

해결책: 제네릭(List<int>) 사용, string interpolation 대신 StringBuilder, struct의 인터페이스 캐스팅 회피, where T : struct 제약 조건 활용

Part 4: struct 설계 가이드라인

언제 struct를 쓰고 언제 class를 써야 하나요?

Microsoft 공식 가이드라인에 따르면, 다음 조건을 모두 만족할 때 struct를 사용합니다.

조건 struct 적합 class 적합
크기 16바이트 이하 16바이트 초과
불변성 불변(immutable)이 자연스러움 가변(mutable)이 필요
수명 짧은 수명, 자주 생성/소멸 긴 수명, 상태 유지
상속 필요 없음 계층 구조 필요

struct가 너무 크면 복사 비용이 참조 전달보다 비싸집니다. 64바이트짜리 struct를 매개변수로 넘기면 매번 64바이트가 복사됩니다!

큰 struct를 전달할 때 복사 비용을 줄이는 방법은?

C# 7.2부터 in, ref, ref readonly 키워드로 값 타입도 참조로 전달할 수 있습니다.

C#
struct LargeData
{
    public float A, B, C, D, E, F, G, H; // 32바이트
}

// in: 읽기 전용 참조 전달 (복사 없음, 수정 불가)
float Calculate(in LargeData data)
{
    return data.A + data.B;
    // data.A = 10; // 컴파일 에러!
}

// readonly struct: 구조체 자체를 불변으로 선언
readonly struct ImmutableVector
{
    public readonly float X, Y, Z;
    public ImmutableVector(float x, float y, float z)
        => (X, Y, Z) = (x, y, z);
}

Unity의 NativeArray, Burst 컴파일러에서는 struct 기반 데이터가 SIMD 최적화의 핵심입니다. ECS(Entity Component System) 패턴에서 struct Component가 필수인 이유입니다.

Part 5: 게임 실전 패턴

게임 개발에서 값 타입/참조 타입 선택이 성능에 미치는 실제 영향은?

총알 1000개를 매 프레임 업데이트하는 시나리오를 비교해봅시다.

C# - class 방식 (GC 압박)
class BulletClass
{
    public float X, Y, Z;
    public float VX, VY, VZ;
    public float Speed;
}

// 1000개 생성 -> 1000번 힙 할당
BulletClass[] bullets = new BulletClass[1000];
for (int i = 0; i < 1000; i++)
    bullets[i] = new BulletClass(); // 각각 힙 할당

// 메모리: 배열 참조 + 1000개 객체 (각 객체 헤더 포함)
// 캐시 미스: 객체가 힙 곳곳에 분산
C# - struct 방식 (캐시 친화적)
struct BulletStruct
{
    public float X, Y, Z;
    public float VX, VY, VZ;
    public float Speed;
}

// 1000개 생성 -> 1번 힙 할당 (배열 자체만)
BulletStruct[] bullets = new BulletStruct[1000];
// 구조체가 배열 안에 연속 배치!

// 메모리: 배열 하나에 모든 데이터가 연속
// 캐시 히트: CPU 캐시라인에 여러 총알이 함께 로드

struct 배열은 데이터가 메모리에 연속 배치되어 CPU 캐시 적중률이 높아집니다. 대량의 오브젝트를 처리할 때 2~5배 성능 차이가 날 수 있습니다.

📌 핵심 정리

  • 값 타입(struct)은 스택에, 참조 타입(class)은 힙에 저장 — 단, struct도 참조 타입의 필드이면 힙에 간다
  • Boxing은 값 타입을 object로 변환할 때 발생하며, 숨겨진 힙 할당의 주범이다
  • struct는 16바이트 이하, 불변, 짧은 수명일 때 사용하고, 큰 struct는 in 키워드로 참조 전달한다
  • 대량 오브젝트는 struct 배열로 캐시 친화적 메모리 레이아웃을 확보한다
  • 제네릭 컬렉션, in 매개변수, readonly struct로 Boxing과 복사 비용을 모두 회피할 수 있다
