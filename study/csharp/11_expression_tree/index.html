코드가 데이터가 된다면?

Expression Tree로 런타임에 코드를 조립하고 분석하기

Part 1: 코드를 데이터로 바라보기

람다식은 그냥 함수 아닌가요? 데이터라니 무슨 뜻이죠?

C#에서 람다식은 두 가지 형태로 해석됩니다. Func<T>에 대입하면 실행 가능한 델리게이트(코드)가 되지만, Expression<Func<T>>에 대입하면 트리 형태의 데이터 구조가 됩니다.

Expression Tree는 코드 자체를 데이터로 표현한 것입니다. 실행하는 게 아니라 "분석"하고 "변환"할 수 있는 구조입니다.

실제로 어떤 차이가 있나요?
C# - 델리게이트 vs Expression
// 1) 델리게이트 - 바로 실행 가능한 코드
Func<int, bool> isEven = x => x % 2 == 0;
bool result = isEven(4); // true

// 2) Expression Tree - 코드를 분석 가능한 데이터로 변환
Expression<Func<int, bool>> expr = x => x % 2 == 0;
// expr.Body  => (x % 2) == 0
// expr.Parameters[0] => x
// 실행하려면 컴파일 필요:
var compiled = expr.Compile();
bool result2 = compiled(4); // true

델리게이트는 IL 코드로 즉시 컴파일되지만, Expression은 트리 노드 객체로 저장됩니다.

Part 2: 트리 구조 들여다보기

Expression Tree의 내부 구조는 어떻게 생겼나요?

x => x * 2 + 1 이라는 식을 트리로 표현하면 다음과 같습니다.

Add (+)
Multiply (*)
Parameter: x
Constant: 2
Constant: 1

모든 연산자, 상수, 매개변수가 하나의 노드로 표현되며, 부모-자식 관계로 연결됩니다.

이 트리 노드들의 타입은 어떻게 되나요?
노드 타입 설명 예시
BinaryExpression 이항 연산 x + 1, a * b
ParameterExpression 매개변수 x, y
ConstantExpression 상수 값 42, "hello"
MethodCallExpression 메서드 호출 str.Contains("a")
MemberExpression 프로퍼티/필드 접근 player.Health
트리를 직접 순회하며 분석하는 코드를 보여주세요.
C# - Expression Tree 방문자
Expression<Func<int, int, int>> expr = (a, b) => a + b * 2;

void PrintTree(Expression node, int depth = 0)
{
    string indent = new string(' ', depth * 2);
    switch (node)
    {
        case BinaryExpression bin:
            Console.WriteLine($"{indent}{bin.NodeType}");
            PrintTree(bin.Left, depth + 1);
            PrintTree(bin.Right, depth + 1);
            break;
        case ParameterExpression param:
            Console.WriteLine($"{indent}Param: {param.Name}");
            break;
        case ConstantExpression constant:
            Console.WriteLine($"{indent}Const: {constant.Value}");
            break;
    }
}

PrintTree(expr.Body);
// 출력:
// Add
//   Param: a
//   Multiply
//     Param: b
//     Const: 2

Part 3: 코드를 조립하기 - Expression API

람다식 말고 직접 Expression Tree를 만들 수도 있나요?

네, Expression 클래스의 팩토리 메서드를 사용해 런타임에 동적으로 코드를 조립할 수 있습니다.

C# - 동적으로 Expression 생성
// 목표: (int x) => x * x + 10 을 동적으로 생성

var param = Expression.Parameter(typeof(int), "x");

// x * x
var multiply = Expression.Multiply(param, param);

// (x * x) + 10
var add = Expression.Add(multiply, Expression.Constant(10));

// 람다로 래핑
var lambda = Expression.Lambda<Func<int, int>>(add, param);

// 컴파일 후 실행
var func = lambda.Compile();
Console.WriteLine(func(5));  // 35
Console.WriteLine(func(3));  // 19

Expression API를 사용하면 런타임에 로직을 조립하고 컴파일할 수 있습니다. 설정 파일이나 사용자 입력에 기반한 동적 로직 생성에 유용합니다.

게임에서 이걸 어디에 쓸 수 있나요?
🎯 비유로 이해하기

Expression Tree는 레고 블록과 같습니다. 미리 완성된 장난감(컴파일된 코드)을 사거나, 블록 하나하나를 조립해서(Expression API) 원하는 모양을 만들 수 있습니다.

게임 데미지 공식을 예로 들면, "공격력 * 스킬배율 - 방어력" 같은 공식을 기획자가 텍스트로 입력하면 Expression Tree로 변환하여 실행할 수 있습니다.

C# - 동적 데미지 공식 생성
// 기획 데이터: "ATK * SkillRate - DEF"
// 이를 Expression Tree로 변환
static Func<Stats, float> BuildDamageFormula(string formula)
{
    var param = Expression.Parameter(typeof(Stats), "s");

    // 파싱 후 트리 구성 (간략화)
    var atk = Expression.Property(param, "ATK");
    var rate = Expression.Property(param, "SkillRate");
    var def = Expression.Property(param, "DEF");

    var mul = Expression.Multiply(atk, rate);
    var sub = Expression.Subtract(mul, def);

    var lambda = Expression.Lambda<Func<Stats, float>>(sub, param);
    return lambda.Compile(); // 한 번 컴파일하면 재사용
}

Part 4: ORM과 LINQ Provider에서의 활용

Entity Framework에서 Where절이 SQL로 변환되는 원리가 뭔가요?

LINQ to Entities의 핵심이 바로 Expression Tree입니다. IQueryable의 Where 메서드는 Expression<Func<T, bool>>을 받습니다.

C# 람다식
x => x.Level > 10
→
Expression Tree
트리 구조로 분석
→
SQL 변환
WHERE Level > 10

LINQ Provider는 Expression Tree를 순회하며 각 노드를 SQL 문법으로 변환합니다. 만약 델리게이트였다면 분석이 불가능해 모든 데이터를 메모리에 올린 뒤 필터링해야 합니다.

주의: IEnumerable의 Where는 Func<T, bool>을 받습니다. IQueryable의 Where와 혼동하면 전체 테이블을 메모리에 로드하는 성능 재앙이 발생합니다.

IQueryable과 IEnumerable의 차이를 정리해 주세요.
구분 IEnumerable<T> IQueryable<T>
Where 매개변수 Func<T, bool> Expression<Func<T, bool>>
실행 위치 클라이언트(메모리) 서버(DB 등)
코드 분석 불가능 Expression Tree로 가능
용도 인메모리 컬렉션 원격 데이터 소스

Part 5: ExpressionVisitor로 트리 변환하기

기존 Expression Tree를 수정할 수 있나요?

Expression Tree는 불변(immutable)입니다. 하지만 ExpressionVisitor를 상속해 트리를 순회하며 새로운 트리를 생성할 수 있습니다.

C# - 상수를 파라미터로 교체하는 Visitor
class ConstantReplacer : ExpressionVisitor
{
    protected override Expression VisitConstant(ConstantExpression node)
    {
        // 정수 상수를 2배로 변환
        if (node.Type == typeof(int))
        {
            return Expression.Constant((int)node.Value * 2);
        }
        return base.VisitConstant(node);
    }
}

// 사용 예시
Expression<Func<int, int>> original = x => x + 5;
var visitor = new ConstantReplacer();
var modified = (Expression<Func<int, int>>)visitor.Visit(original);
// 결과: x => x + 10

var func = modified.Compile();
Console.WriteLine(func(3)); // 13

ExpressionVisitor 패턴을 사용하면 쿼리 조건 추가, 최적화, 다른 언어로의 변환 등 강력한 기능을 구현할 수 있습니다.

Part 6: 성능 고려사항과 실전 팁

Expression.Compile()은 비용이 큰 작업인가요?

네, Compile()은 IL 코드를 동적으로 생성하고 JIT 컴파일하므로 상대적으로 비용이 큽니다.

주의: 매 프레임마다 Compile()을 호출하면 심각한 성능 문제가 발생합니다. 반드시 한 번 컴파일한 결과를 캐싱하세요.

C# - 컴파일 결과 캐싱 패턴
class FormulaCache
{
    private readonly Dictionary<string, Delegate> _cache = new();

    public Func<float, float> GetOrCompile(
        string key, Expression<Func<float, float>> expr)
    {
        if (!_cache.TryGetValue(key, out var cached))
        {
            cached = expr.Compile();
            _cache[key] = cached;
        }
        return (Func<float, float>)cached;
    }
}
FastExpressionCompiler 같은 대안도 있나요?

네, NuGet 패키지인 FastExpressionCompiler는 System.Reflection.Emit 대신 경량화된 방식으로 컴파일하여 Compile() 속도를 5~10배 향상시킵니다. IoC 컨테이너나 매퍼 라이브러리에서 많이 사용됩니다.

📌 핵심 정리

  • Expression Tree는 코드를 실행 가능한 트리 구조의 데이터로 표현한 것이다.
  • Expression<Func<T>>는 분석/변환 가능하지만, Func<T>는 블랙박스다.
  • ORM(Entity Framework)은 Expression Tree를 SQL로 변환하여 서버 사이드 실행을 가능하게 한다.
  • ExpressionVisitor로 트리를 순회하며 새로운 트리로 변환할 수 있다.
  • Compile()은 비용이 크므로 결과를 반드시 캐싱해야 한다.
