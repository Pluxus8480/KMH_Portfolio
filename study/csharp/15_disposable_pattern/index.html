다 쓴 자원은 누가 치우나?

IDisposable 패턴으로 비관리 리소스 안전하게 해제하기

Part 1: GC가 못 치우는 것들

C#에 GC가 있는데 왜 수동으로 자원을 해제해야 하나요?

GC는 관리 메모리(Managed Memory)만 해제합니다. 하지만 프로그램은 메모리 외에도 다양한 시스템 자원을 사용합니다.

🎯 비유로 이해하기

GC는 호텔의 객실 청소원과 같습니다. 방 안의 침대 정리(관리 메모리)는 해주지만, 투숙객이 빌린 렌터카(파일 핸들), 수영장 라커(네트워크 소켓), 연회장 예약(GPU 리소스)은 직접 반납해야 합니다.

자원 유형 GC 해제 수동 해제 필요 게임 예시
관리 메모리 O X 일반 C# 객체
파일 핸들 X O 세이브 파일, 로그
네트워크 소켓 X O 서버 연결
GPU 리소스 X O 텍스처, 셰이더, 버퍼
네이티브 메모리 X O Marshal.AllocHGlobal

IDisposable 패턴은 이러한 비관리 리소스(Unmanaged Resource)를 확정적으로 해제하기 위한 .NET의 표준 규약입니다.

Part 2: using 문의 마법

using 문은 어떻게 자원을 해제하나요?

using 문은 블록이 끝나면 자동으로 Dispose()를 호출합니다. 예외가 발생해도 반드시 호출됩니다.

C# - using 문 (try-finally의 문법 설탕)
// using 문
using (var stream = new FileStream("save.dat", FileMode.Open))
{
    // 파일 읽기/쓰기
    byte[] data = new byte[stream.Length];
    stream.Read(data, 0, data.Length);
} // 여기서 자동으로 stream.Dispose() 호출

// 위 코드는 아래와 완전히 동일
FileStream stream2 = null;
try
{
    stream2 = new FileStream("save.dat", FileMode.Open);
    byte[] data = new byte[stream2.Length];
    stream2.Read(data, 0, data.Length);
}
finally
{
    stream2?.Dispose(); // 예외가 발생해도 반드시 실행
}
C# 8.0의 using 선언은 뭐가 다른가요?
C# - using 선언 (C# 8.0+)
void SaveGame(GameState state)
{
    // using 선언 - 중괄호 블록 없이 사용
    using var stream = new FileStream("save.dat", FileMode.Create);
    using var writer = new BinaryWriter(stream);

    writer.Write(state.PlayerName);
    writer.Write(state.Level);
    writer.Write(state.Health);

    // 메서드 끝에서 writer, stream 순서로 자동 Dispose
    // (선언 역순으로 해제됨)
}

using 선언은 변수의 스코프가 감싸는 블록(보통 메서드) 끝까지입니다. 중첩을 줄여 코드 가독성을 높여줍니다.

Part 3: Dispose 패턴 정석 구현

IDisposable을 제대로 구현하려면 어떻게 해야 하나요?

Microsoft가 권장하는 표준 Dispose 패턴은 다음과 같습니다.

C# - 표준 Dispose 패턴
public class GameResourceManager : IDisposable
{
    // 비관리 리소스
    private IntPtr _nativeHandle;
    // 관리 리소스 (IDisposable 구현체)
    private FileStream? _logStream;

    private bool _disposed = false;

    public GameResourceManager()
    {
        _nativeHandle = NativeLib.CreateResource();
        _logStream = new FileStream("game.log", FileMode.Append);
    }

    // 공개 Dispose - 사용자가 직접 호출
    public void Dispose()
    {
        Dispose(disposing: true);
        GC.SuppressFinalize(this); // Finalizer 호출 방지
    }

    // 실제 해제 로직
    protected virtual void Dispose(bool disposing)
    {
        if (_disposed) return;

        if (disposing)
        {
            // 관리 리소스 해제
            _logStream?.Dispose();
            _logStream = null;
        }

        // 비관리 리소스 해제 (항상 실행)
        if (_nativeHandle != IntPtr.Zero)
        {
            NativeLib.DestroyResource(_nativeHandle);
            _nativeHandle = IntPtr.Zero;
        }

        _disposed = true;
    }

    // 안전장치: Dispose를 호출하지 않았을 때
    ~GameResourceManager()
    {
        Dispose(disposing: false);
    }
}
disposing 매개변수의 역할이 무엇인가요?
Dispose() 호출
disposing = true
→
Dispose(true)
관리 + 비관리 해제
~Finalizer 호출
disposing = false
→
Dispose(false)
비관리만 해제

주의: Finalizer에서는 다른 관리 객체를 참조하면 안 됩니다. GC가 이미 해당 객체를 수거했을 수 있기 때문입니다. 따라서 disposing=false일 때는 비관리 리소스만 해제합니다.

Part 4: Finalizer의 비용과 GC 영향

Finalizer(소멸자)가 있으면 GC에 어떤 영향을 주나요?

Finalizer가 있는 객체는 GC 과정이 복잡해집니다.

1. GC 발견
참조 없는 객체 감지
→
2. F-Queue 등록
Finalizer 큐에 추가
→
3. Finalizer 실행
별도 스레드에서 실행
→
4. 다음 GC
이제야 메모리 해제

Finalizer가 있는 객체는 최소 2번의 GC 사이클을 거쳐야 메모리가 해제됩니다. 또한 Gen 1 이상으로 승격되어 GC 부담이 커집니다.

그래서 Dispose()에서 GC.SuppressFinalize(this)를 호출하여 Finalizer를 건너뛰게 하는 것이 중요합니다.

그러면 Finalizer를 아예 안 쓰는 게 나은가요?

가능하면 Finalizer 대신 SafeHandle을 사용하는 것이 권장됩니다.

C# - SafeHandle 사용
// SafeHandle을 상속한 커스텀 핸들
class GameResourceHandle : SafeHandleZeroOrMinusOneIsInvalid
{
    public GameResourceHandle() : base(ownsHandle: true) { }

    protected override bool ReleaseHandle()
    {
        // 비관리 리소스 해제
        NativeLib.DestroyResource(handle);
        return true;
    }
}

// SafeHandle을 사용하면 Finalizer가 필요 없음!
public class GameResource : IDisposable
{
    private readonly GameResourceHandle _handle = new();

    public void Dispose()
    {
        _handle.Dispose();
        // GC.SuppressFinalize 불필요 - Finalizer 자체가 없으므로
    }
}

SafeHandle은 CriticalFinalizerObject를 상속하여 Finalizer보다 안정적으로 동작하며, P/Invoke 시 핸들 재활용 공격(handle recycling attack)도 방지합니다.

Part 5: IAsyncDisposable과 비동기 해제

비동기적으로 리소스를 해제해야 하는 경우가 있나요?

네트워크 연결 종료, DB 커넥션 반환, 비동기 스트림 플러시 등은 비동기 해제가 필요합니다.

C# - IAsyncDisposable 구현
public class GameNetworkSession : IAsyncDisposable, IDisposable
{
    private TcpClient? _client;
    private NetworkStream? _stream;
    private bool _disposed;

    public async ValueTask DisposeAsync()
    {
        if (_disposed) return;

        if (_stream is not null)
        {
            // 남은 데이터를 비동기로 전송
            await _stream.FlushAsync();
            await _stream.DisposeAsync();
        }

        _client?.Dispose();
        _disposed = true;

        GC.SuppressFinalize(this);
    }

    public void Dispose()
    {
        if (_disposed) return;
        _stream?.Dispose();
        _client?.Dispose();
        _disposed = true;
        GC.SuppressFinalize(this);
    }
}

// 사용
await using var session = new GameNetworkSession();
await session.SendDataAsync(packet);
// 스코프 종료 시 DisposeAsync() 자동 호출

await using 구문으로 비동기 해제를 자동화할 수 있습니다. C# 8.0부터 지원됩니다.

Part 6: 게임 리소스 관리 실전

게임에서 Dispose 패턴을 적용한 리소스 매니저 예시를 보여주세요.
C# - 게임 텍스처 풀 관리
public class TexturePool : IDisposable
{
    private readonly Dictionary<string, TextureHandle> _textures = new();
    private bool _disposed;

    public TextureHandle LoadTexture(string path)
    {
        ObjectDisposedException.ThrowIf(_disposed, this);

        if (_textures.TryGetValue(path, out var existing))
            return existing;

        var handle = GraphicsDevice.CreateTexture(path);
        _textures[path] = handle;
        return handle;
    }

    public void UnloadUnused()
    {
        var unused = _textures
            .Where(kv => kv.Value.ReferenceCount == 0)
            .Select(kv => kv.Key)
            .ToList();

        foreach (var key in unused)
        {
            _textures[key].Dispose();
            _textures.Remove(key);
        }
    }

    public void Dispose()
    {
        if (_disposed) return;

        foreach (var texture in _textures.Values)
        {
            texture.Dispose();
        }
        _textures.Clear();
        _disposed = true;
    }
}

// 씬 전환 시 리소스 정리
void OnSceneUnload()
{
    _texturePool.Dispose();
    _audioPool.Dispose();
    _meshPool.Dispose();
}

Dispose된 객체에 접근하면 ObjectDisposedException을 던져야 합니다. .NET 7+에서는 ObjectDisposedException.ThrowIf를 사용하면 간결합니다.

흔히 하는 실수는 뭐가 있나요?
실수 결과 해결책
using 누락 리소스 누수 CA2000 분석기 활성화
Dispose 후 사용 ObjectDisposedException _disposed 플래그 검사

📌 핵심 정리

  • GC는 관리 메모리만 해제한다. 파일, 소켓, GPU 리소스 등 비관리 리소스는 IDisposable로 명시적으로 해제해야 한다.
  • using 문/선언은 Dispose() 호출을 보장하는 문법 설탕이다.
  • 표준 Dispose 패턴은 Dispose(bool disposing) 오버로드로 관리/비관리 리소스를 분리 해제한다.
  • Finalizer는 GC 비용을 2배로 증가시키므로, 가능하면 SafeHandle을 대안으로 사용한다.
  • IAsyncDisposable과 await using으로 비동기 리소스 해제를 구현할 수 있다.
