컴파일러가 코드를 대신 써준다?

Source Generator로 보일러플레이트 자동 생성하기

Part 1: 보일러플레이트의 고통

게임 개발에서 반복적으로 작성하는 코드가 정말 많죠?

네트워크 패킷 직렬화, 에디터 인스펙터, 이벤트 바인딩, Enum 유틸리티 등 매번 비슷한 구조를 반복 작성해야 합니다.

C# - 반복되는 보일러플레이트 예시
// 모든 패킷마다 이런 코드를 반복 작성해야 함
public class MovePacket : IPacket
{
    public float X { get; set; }
    public float Y { get; set; }
    public float Z { get; set; }

    public void Serialize(BinaryWriter writer)
    {
        writer.Write(X);
        writer.Write(Y);
        writer.Write(Z);
    }

    public void Deserialize(BinaryReader reader)
    {
        X = reader.ReadSingle();
        Y = reader.ReadSingle();
        Z = reader.ReadSingle();
    }
}
// 패킷이 100개면 이런 코드가 100번...

Source Generator는 컴파일 시점에 소스 코드를 분석하고 새로운 코드를 자동 생성합니다. 리플렉션 없이, 런타임 비용 없이!

기존 코드 생성 방식과 뭐가 다른가요?
방식 시점 장점 단점
리플렉션 런타임 유연함 느림, AOT 불가

Source Generator는 똑똑한 비서와 같습니다. 당신이 클래스 설계도(어트리뷰트)만 그려주면, 비서가 나머지 서류 작업(직렬화, 유틸리티 메서드 등)을 컴파일 시점에 자동으로 완성해 줍니다.

Part 2: Source Generator 동작 원리

컴파일 파이프라인에서 어떤 위치에서 실행되나요?
1. 파싱
소스 코드 → Syntax Tree
→
2. Source Generator
트리 분석 → 코드 추가
→
3. 컴파일
원본 + 생성 코드 → IL

Source Generator는 Roslyn 컴파일러의 일부로 실행됩니다. 기존 코드의 Syntax Tree와 Semantic Model을 읽고, 새로운 소스 파일을 추가합니다.

주의: Source Generator는 기존 코드를 수정할 수 없습니다. 오직 새로운 코드를 추가만 가능합니다. 따라서 partial 클래스를 활용해야 합니다.

ISourceGenerator와 IIncrementalGenerator의 차이는?
구분 ISourceGenerator (V1) IIncrementalGenerator (V2)
도입 시기 .NET 5 .NET 6
성능 매번 전체 재생성 변경분만 증분 처리
IDE 영향 타이핑 시 버벅임 부드러운 편집 경험
권장 여부 레거시 권장

신규 프로젝트에서는 반드시 IIncrementalGenerator를 사용하세요. IDE 성능에 큰 영향을 줍니다.

Source Generator는 netstandard2.0을 타겟으로 해야 하며, Roslyn API 패키지를 참조합니다.

간단한 예제를 보여주세요.
C# - ToString 자동 생성 Generator
[Generator]
public class ToStringGenerator : IIncrementalGenerator
{
    public void Initialize(IncrementalGeneratorInitializationContext context)
    {
        // 1단계: [AutoToString] 어트리뷰트가 붙은 클래스 수집
        var classes = context.SyntaxProvider
            .ForAttributeWithMetadataName(
                "AutoToStringAttribute",
                predicate: (node, _) => node is ClassDeclarationSyntax,
                transform: (ctx, _) => GetClassInfo(ctx))
            .Where(info => info is not null);

        // 2단계: 수집된 정보로 코드 생성
        context.RegisterSourceOutput(classes, (spc, classInfo) =>
        {
            string source = GenerateToString(classInfo!);
            spc.AddSource($"{classInfo!.ClassName}.g.cs", source);
        });
    }

    static string GenerateToString(ClassInfo info)
    {
        var props = string.Join(", ",
            info.Properties.Select(p => $"{p}: {{{p}}}"));

        return $@"
// 자동 생성된 코드
partial class {info.ClassName}
{{
    public override string ToString()
        => $""{info.ClassName}({props})"";
}}";
    }
}
사용하는 쪽 코드는 어떻게 되나요?
C# - 사용 측 코드
// 어트리뷰트만 붙이면 ToString이 자동 생성됨!
[AutoToString]
public partial class PlayerData
{
    public string Name { get; set; }
    public int Level { get; set; }
    public float Health { get; set; }
}

// 자동 생성된 코드 (PlayerData.g.cs):
// partial class PlayerData
// {
//     public override string ToString()
//         => $"PlayerData(Name: {Name}, Level: {Level}, Health: {Health})";
// }

var player = new PlayerData { Name = "Hero", Level = 5, Health = 100 };
Console.WriteLine(player); // PlayerData(Name: Hero, Level: 5, Health: 100)

핵심은 partial 키워드입니다. 사용자가 작성한 부분과 Generator가 생성한 부분이 합쳐져 하나의 클래스를 구성합니다.

Part 4: Syntax와 Semantic 분석

Syntax Tree와 Semantic Model의 차이가 뭔가요?
구분 Syntax Tree Semantic Model
관점 코드의 구조/문법 코드의 의미/타입 정보
알 수 있는 것 클래스명, 메서드명, 문법 구조 타입, 상속 관계, 참조 해석
비용 저렴 (파싱만) 비쌈 (전체 분석 필요)
활용 빠른 필터링 정확한 정보 추출
C# - Syntax vs Semantic 분석 예시
// Syntax: "이 노드가 ClassDeclarationSyntax인가?"
static bool IsCandidateClass(SyntaxNode node, CancellationToken _)
    => node is ClassDeclarationSyntax { AttributeLists.Count: > 0 };

// Semantic: "이 클래스가 IPacket 인터페이스를 구현하는가?"
static ClassInfo? GetClassInfo(
    GeneratorAttributeSyntaxContext context)
{
    var symbol = context.TargetSymbol as INamedTypeSymbol;
    if (symbol is null) return null;

    // 타입 정보 분석
    bool implementsIPacket = symbol.AllInterfaces
        .Any(i => i.Name == "IPacket");

    // 프로퍼티 목록 추출
    var properties = symbol.GetMembers()
        .OfType<IPropertySymbol>()
        .Select(p => (p.Name, p.Type.ToDisplayString()))
        .ToList();

    return new ClassInfo(symbol.Name, properties);
}

Incremental Generator에서는 Syntax로 빠르게 후보를 필터링하고, Semantic으로 정확한 정보를 추출하는 2단계 파이프라인이 성능의 핵심입니다.

Part 5: 게임 개발 실전 활용 사례

게임에서 Source Generator를 어디에 활용할 수 있나요?

가장 큰 효과를 보는 분야는 다음과 같습니다:

C# - 네트워크 패킷 직렬화 자동 생성
// 사용자 코드 - 구조만 정의
[Packet(OpCode = 0x0010)]
public partial class MovePacket
{
    public float X { get; set; }
    public float Y { get; set; }
    public float Z { get; set; }
    public float Rotation { get; set; }
}

// Source Generator가 자동 생성하는 코드:
partial class MovePacket : IPacket
{
    public ushort OpCode => 0x0010;

    public void Serialize(Span<byte> buffer)
    {
        BinaryPrimitives.WriteSingleLittleEndian(buffer[0..], X);
        BinaryPrimitives.WriteSingleLittleEndian(buffer[4..], Y);
        BinaryPrimitives.WriteSingleLittleEndian(buffer[8..], Z);
        BinaryPrimitives.WriteSingleLittleEndian(buffer[12..], Rotation);
    }

    public void Deserialize(ReadOnlySpan<byte> buffer)
    {
        X = BinaryPrimitives.ReadSingleLittleEndian(buffer[0..]);
        Y = BinaryPrimitives.ReadSingleLittleEndian(buffer[4..]);
        Z = BinaryPrimitives.ReadSingleLittleEndian(buffer[8..]);
        Rotation = BinaryPrimitives.ReadSingleLittleEndian(buffer[12..]);
    }

    public int Size => 16;
}

이 외에도 Enum 확장 메서드, DI 등록 코드, 상태머신 전이 코드 등을 자동 생성할 수 있습니다.

Part 6: 디버깅과 테스트

Source Generator를 어떻게 디버깅하나요?
C# - 디버깅 팁
// 1. Debugger.Launch()로 디버거 연결
public void Initialize(...)
{
#if DEBUG
    if (!Debugger.IsAttached)
        Debugger.Launch();
#endif
}

// 2. 생성된 코드 파일로 출력
// csproj에 추가:
// <EmitCompilerGeneratedFiles>true</EmitCompilerGeneratedFiles>
// <CompilerGeneratedFilesOutputPath>Generated</CompilerGeneratedFilesOutputPath>

// 3. Diagnostic으로 경고/에러 출력
context.RegisterSourceOutput(errors, (spc, error) =>
{
    spc.ReportDiagnostic(Diagnostic.Create(
        new DiagnosticDescriptor(
            "SG001", "생성 오류", error.Message,
            "SourceGenerator", DiagnosticSeverity.Error, true),
        Location.None));
});

EmitCompilerGeneratedFiles 옵션을 켜면 생성된 .g.cs 파일을 직접 확인할 수 있습니다. 디버깅 시 가장 유용한 방법입니다.

📌 핵심 정리

  • Source Generator는 컴파일 시점에 Roslyn을 통해 소스 코드를 분석하고 새 코드를 추가한다.
  • IIncrementalGenerator를 사용해 증분 처리하면 IDE 성능 저하를 방지할 수 있다.
  • partial 클래스를 통해 사용자 코드와 생성 코드를 결합한다.
  • 패킷 직렬화, Enum 유틸리티, DI 등록 등 보일러플레이트를 제거하여 생산성을 높인다.
  • 리플렉션을 대체하여 AOT 호환성과 런타임 성능을 동시에 확보할 수 있다.
