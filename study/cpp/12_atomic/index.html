락 없이 스레드 안전하게?

std::atomic과 메모리 오더링의 세계

Part 1: 왜 atomic이 필요한가?

일반 변수는 왜 스레드 안전하지 않나요?

단순한 ++i도 읽기-수정-쓰기 3단계라 경쟁 상태가 발생합니다!

경쟁 상태 문제
int counter = 0;  // 공유 변수

void increment() {
    for (int i = 0; i < 100000; i++) {
        counter++;  // 스레드 안전하지 않음!
    }
}

// 두 스레드가 동시에 실행하면?
// 예상: 200000
// 실제: 그보다 작은 값 (경쟁 상태!)

// counter++의 실제 동작:
// 1. counter 값을 레지스터로 읽기 (LOAD)
// 2. 레지스터에서 1 증가 (ADD)
// 3. 레지스터 값을 counter에 쓰기 (STORE)

// 두 스레드가 동시에 실행되면:
// Thread A: LOAD (counter = 0)
// Thread B: LOAD (counter = 0)
// Thread A: ADD  (reg = 1)
// Thread B: ADD  (reg = 1)
// Thread A: STORE (counter = 1)
// Thread B: STORE (counter = 1)  // 1이 됨, 2가 아님!
std::atomic 해결책
#include <atomic>

std::atomic<int> counter{0};  // 원자적 변수

void increment() {
    for (int i = 0; i < 100000; i++) {
        counter++;  // 원자적 증가 (스레드 안전!)
    }
}

// 두 스레드 실행 후: 정확히 200000
🎯 비유로 이해하기

화장실 문에 자물쇠가 있으면 한 사람씩만 들어갈 수 있습니다. atomic은 변수 접근에 보이지 않는 자물쇠를 다는 것과 같습니다.

Part 2: std::atomic 기본 사용

atomic 연산들
std::atomic<int> value{0};

// 읽기/쓰기
int x = value.load();   // 원자적 읽기
value.store(42);        // 원자적 쓰기

// 교환
int old = value.exchange(100);  // 값 교환 후 이전 값 반환

// 비교-교환 (CAS: Compare-And-Swap)
int expected = 100;
bool success = value.compare_exchange_strong(expected, 200);
// value가 100이면 200으로 변경하고 true 반환
// value가 100이 아니면 expected에 현재 값 저장하고 false 반환

// 산술 연산
value.fetch_add(10);    // value += 10, 이전 값 반환
value.fetch_sub(5);     // value -= 5
value.fetch_and(0xFF);  // value &= 0xFF
value.fetch_or(0x80);   // value |= 0x80
value.fetch_xor(0x01);  // value ^= 0x01

// 연산자 오버로드
value++;    // fetch_add(1)
value--;    // fetch_sub(1)
value += 5; // fetch_add(5)
atomic 지원 타입
// 정수형
std::atomic<int> ai;
std::atomic<long> al;
std::atomic<uint64_t> au64;

// 포인터
std::atomic<int*> ptr;
ptr.fetch_add(1);  // 포인터 산술

// bool
std::atomic<bool> flag{false};
flag.store(true);

// 별칭
std::atomic_int ai;       // std::atomic<int>
std::atomic_bool ab;      // std::atomic<bool>
std::atomic_flag af;      // 특수: 락-프리 보장

// 사용자 정의 타입 (trivially copyable이어야 함)
struct Point { int x, y; };
std::atomic<Point> atomicPoint;  // 원자적 Point

// 락-프리 여부 확인
static_assert(std::atomic<int>::is_always_lock_free);
std::cout << atomicPoint.is_lock_free();  // 런타임 확인

Part 3: 메모리 순서 (Memory Order)

메모리 순서가 왜 중요한가요?

컴파일러와 CPU는 최적화를 위해 명령어 순서를 바꿀 수 있습니다!

재배열 문제
int data = 0;
bool ready = false;

// Thread A (생산자)
void producer() {
    data = 42;        // (1)
    ready = true;     // (2)
}

// Thread B (소비자)
void consumer() {
    while (!ready);   // (3)
    int x = data;     // (4) x가 42일까?
}

// 문제: (1)과 (2)의 순서가 바뀔 수 있음!
// CPU 캐시, 컴파일러 최적화로 인해
// (2)가 먼저 실행되면 x = 0이 될 수 있음!
메모리 순서 옵션

relaxed         가장 약함, 원자성만 보장
├── consume     읽기 의존성만 동기화 (거의 안 씀)
├── acquire     이 읽기 이후 명령 재배열 방지
├── release     이 쓰기 이전 명령 재배열 방지
├── acq_rel     acquire + release
└── seq_cst     가장 강함, 전역 순서 보장 (기본값)
메모리 순서 적용
std::atomic<int> data{0};
std::atomic<bool> ready{false};

// Thread A (생산자)
void producer() {
    data.store(42, std::memory_order_relaxed);    // 데이터 먼저
    ready.store(true, std::memory_order_release); // release로 쓰기
    // release 전의 모든 쓰기가 다른 스레드에 보임
}

// Thread B (소비자)
void consumer() {
    while (!ready.load(std::memory_order_acquire)); // acquire로 읽기
    // acquire 후의 모든 읽기는 최신 값
    int x = data.load(std::memory_order_relaxed);   // 42 보장!
}

// release-acquire 동기화
// producer의 release와 consumer의 acquire가 페어를 이룸
// release 이전의 모든 쓰기는 acquire 이후에 보임

Part 4: 메모리 순서 상세

순서 용도 성능
relaxed 카운터, 통계 가장 빠름
acquire 락 획득, 데이터 읽기 중간
release 락 해제, 데이터 쓰기 중간
seq_cst 단순함 우선, 디버깅 가장 느림
각 순서의 사용 예
// 1. relaxed: 단순 카운터
std::atomic<uint64_t> totalOps{0};

void worker() {
    // 정확한 값은 중요하지 않고 대략적인 통계만 필요
    totalOps.fetch_add(1, std::memory_order_relaxed);
}

// 2. acquire-release: 생산자-소비자
std::atomic<Node*> head{nullptr};

void push(Node* node) {
    Node* old = head.load(std::memory_order_relaxed);
    do {
        node->next = old;
    } while (!head.compare_exchange_weak(old, node,
        std::memory_order_release,  // 성공 시
        std::memory_order_relaxed   // 실패 시
    ));
}

Node* pop() {
    Node* old = head.load(std::memory_order_acquire);
    // acquire로 읽으면 해당 노드의 데이터도 최신
    // ...
}

// 3. seq_cst: 전역 순서 필요
std::atomic<bool> x{false}, y{false};
std::atomic<int> z{0};

// Thread 1
x.store(true, std::memory_order_seq_cst);

// Thread 2
y.store(true, std::memory_order_seq_cst);

// Thread 3
while (!x.load(std::memory_order_seq_cst));
if (y.load(std::memory_order_seq_cst)) z++;

// Thread 4
while (!y.load(std::memory_order_seq_cst));
if (x.load(std::memory_order_seq_cst)) z++;

// seq_cst면 z는 반드시 1 이상 (전역 순서 보장)

Part 5: Lock-Free 자료구조

Lock-Free 스택
template<typename T>
class LockFreeStack {
    struct Node {
        T data;
        Node* next;
        Node(T d) : data(std::move(d)), next(nullptr) {}
    };

    std::atomic<Node*> head{nullptr};

public:
    void push(T value) {
        Node* newNode = new Node(std::move(value));
        newNode->next = head.load(std::memory_order_relaxed);

        // CAS 루프: head가 예상값이면 newNode로 교체
        while (!head.compare_exchange_weak(
            newNode->next, newNode,
            std::memory_order_release,
            std::memory_order_relaxed
        ));
    }

    std::optional<T> pop() {
        Node* old = head.load(std::memory_order_acquire);

        while (old && !head.compare_exchange_weak(
            old, old->next,
            std::memory_order_acquire,
            std::memory_order_relaxed
        ));

        if (old) {
            T data = std::move(old->data);
            delete old;  // 주의: ABA 문제 가능!
            return data;
        }
        return std::nullopt;
    }
};
Spinlock 구현
class Spinlock {
    std::atomic_flag flag = ATOMIC_FLAG_INIT;

public:
    void lock() {
        while (flag.test_and_set(std::memory_order_acquire)) {
            // 스핀 대기
            // C++20: flag.wait(true);
        }
    }

    void unlock() {
        flag.clear(std::memory_order_release);
        // C++20: flag.notify_one();
    }
};

// RAII 래퍼와 함께
class SpinlockGuard {
    Spinlock& lock_;
public:
    SpinlockGuard(Spinlock& l) : lock_(l) { lock_.lock(); }
    ~SpinlockGuard() { lock_.unlock(); }
};

Spinlock spinlock;
void criticalSection() {
    SpinlockGuard guard(spinlock);
    // 보호된 코드
}

Part 6: 실전 팁과 주의사항

False Sharing 방지
// 문제: 같은 캐시 라인에 있으면 성능 저하
struct BadCounters {
    std::atomic<int> counter1;  // 같은 캐시 라인
    std::atomic<int> counter2;  // 서로 경쟁!
};

// 해결: 캐시 라인 정렬
struct alignas(64) GoodCounters {
    std::atomic<int> counter1;
    char padding[60];  // 캐시 라인 분리
    std::atomic<int> counter2;
};

// C++17: hardware_destructive_interference_size
struct alignas(std::hardware_destructive_interference_size) Counter {
    std::atomic<int> value;
};

📌 핵심 정리

  • std::atomic: 원자적 연산으로 경쟁 상태 방지
  • CAS: compare_exchange로 락-프리 알고리즘 구현
  • relaxed: 원자성만, 순서 보장 없음 (카운터용)
  • acquire-release: 생산자-소비자 동기화
  • seq_cst: 가장 안전, 기본값으로 시작
