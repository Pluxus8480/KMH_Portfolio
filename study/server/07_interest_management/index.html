1000명이 접속했는데 왜 999명 정보를 보내나요?

Interest Management와 Area of Interest로 대역폭 최적화

Part 1: 문제 정의

왜 모든 플레이어 정보를 보내면 안 되나요?

플레이어 수가 늘어나면 대역폭이 기하급수적으로 증가합니다.

대역폭 문제:

N명의 플레이어, 각자 M bytes의 상태 정보

전체 브로드캐스트 시:
- 서버 → 각 클라이언트: N * M bytes
- 전체 대역폭: N * N * M bytes/tick

예시 (1000명, 50 bytes/player, 20 tick/s):
= 1000 * 1000 * 50 * 20
= 1,000,000,000 bytes/s
= 1 GB/s (!!!)

실제로 필요한 정보:
- 내 주변 100명 정도만 보이면 됨
- 100 * 50 * 20 = 100 KB/s (1/10000 감소!)

Interest Management는 각 플레이어에게 관련 있는 정보만 보내는 기술입니다.

Area of Interest(AOI)란 무엇인가요?

플레이어가 "관심을 가져야 하는 영역"입니다. 보통 플레이어 주변 일정 거리입니다.

Area of Interest 개념:

┌────────────────────────────────────────────────┐
│                    World                       │
│                                                │
│     ○         ○                               │
│            ┌─────────┐                         │
│     ○      │ ◉ Player│        ○               │
│            │   AOI   │                         │
│      ○     │    ○    │                         │
│            │      ○  │     ○                   │
│            └─────────┘                         │
│                               ○                │
│        ○                                       │
│                    ○              ○            │
└────────────────────────────────────────────────┘

◉ : 플레이어 (자신)
○ : 다른 플레이어
AOI 안의 ○만 정보를 받음

Part 2: 공간 분할 기법

AOI 안의 엔티티를 빠르게 찾으려면?

공간 분할 자료구조를 사용합니다. 가장 간단한 것은 Grid입니다.

Grid 기반 AOI
class GridAOI {
    static const int CELL_SIZE = 100;  // 100 units per cell
    static const int GRID_WIDTH = 100; // 10000 units total

    // 각 셀에 있는 엔티티 목록
    std::unordered_set grid[GRID_WIDTH][GRID_WIDTH];

    // 엔티티의 현재 셀 위치
    std::unordered_map> entityCells;

public:
    void UpdatePosition(EntityID id, float x, float y) {
        int newCellX = (int)(x / CELL_SIZE);
        int newCellY = (int)(y / CELL_SIZE);

        auto it = entityCells.find(id);
        if (it != entityCells.end()) {
            auto [oldX, oldY] = it->second;
            if (oldX == newCellX && oldY == newCellY) {
                return;  // 같은 셀, 변경 없음
            }
            // 이전 셀에서 제거
            grid[oldX][oldY].erase(id);
        }

        // 새 셀에 추가
        grid[newCellX][newCellY].insert(id);
        entityCells[id] = {newCellX, newCellY};
    }

    std::vector GetEntitiesInAOI(float x, float y, float radius) {
        std::vector result;

        int centerX = (int)(x / CELL_SIZE);
        int centerY = (int)(y / CELL_SIZE);
        int cellRadius = (int)ceil(radius / CELL_SIZE);

        // AOI 범위 내의 셀들 순회
        for (int dx = -cellRadius; dx <= cellRadius; dx++) {
            for (int dy = -cellRadius; dy <= cellRadius; dy++) {
                int cx = centerX + dx;
                int cy = centerY + dy;

                if (cx < 0 || cx >= GRID_WIDTH ||
                    cy < 0 || cy >= GRID_WIDTH) continue;

                for (EntityID id : grid[cx][cy]) {
                    // 실제 거리 체크 (원형 AOI)
                    auto& entity = GetEntity(id);
                    float dist = Distance(x, y, entity.x, entity.y);
                    if (dist <= radius) {
                        result.push_back(id);
                    }
                }
            }
        }

        return result;
    }
};
더 효율적인 자료구조는 없나요?

엔티티 분포에 따라 Quadtree나 Spatial Hash가 더 효율적일 수 있습니다.

자료구조 장점 단점 적합한 경우
Grid 구현 간단, O(1) 셀 접근 고정 크기, 메모리 낭비 균일 분포
Quadtree 적응적 분할 업데이트 비용 밀집/희소 혼합
Spatial Hash 동적 크기 해시 충돌 동적 월드
R-Tree 범위 쿼리 최적 구현 복잡 GIS, 대규모
Spatial Hash 구현
class SpatialHash {
    float cellSize;
    std::unordered_map> cells;

    uint64_t HashPosition(float x, float y) {
        int cx = (int)floor(x / cellSize);
        int cy = (int)floor(y / cellSize);
        // Cantor pairing function
        return ((uint64_t)(cx + cy) * (cx + cy + 1)) / 2 + cy;
    }

public:
    SpatialHash(float cellSize) : cellSize(cellSize) {}

    void Insert(EntityID id, float x, float y) {
        uint64_t hash = HashPosition(x, y);
        cells[hash].push_back(id);
    }

    void Remove(EntityID id, float x, float y) {
        uint64_t hash = HashPosition(x, y);
        auto& cell = cells[hash];
        cell.erase(std::remove(cell.begin(), cell.end(), id), cell.end());
        if (cell.empty()) {
            cells.erase(hash);
        }
    }

    std::vector Query(float x, float y, float radius) {
        std::vector result;
        int cellRadius = (int)ceil(radius / cellSize);

        int cx = (int)floor(x / cellSize);
        int cy = (int)floor(y / cellSize);

        for (int dx = -cellRadius; dx <= cellRadius; dx++) {
            for (int dy = -cellRadius; dy <= cellRadius; dy++) {
                uint64_t hash = HashPosition(
                    (cx + dx) * cellSize,
                    (cy + dy) * cellSize
                );

                auto it = cells.find(hash);
                if (it != cells.end()) {
                    for (EntityID id : it->second) {
                        result.push_back(id);
                    }
                }
            }
        }
        return result;
    }
};

Part 3: Enter/Leave 이벤트

AOI에 들어오고 나가는 것을 어떻게 처리하나요?

매 틱마다 현재 AOI와 이전 AOI를 비교하여 Enter/Leave 이벤트를 발생시킵니다.

AOI Enter/Leave 관리
class AOIManager {
    struct PlayerAOI {
        std::unordered_set visibleEntities;
    };

    std::unordered_map playerAOIs;
    GridAOI spatialIndex;
    float aoiRadius = 500.0f;

public:
    struct AOIUpdate {
        std::vector entered;   // 새로 보이기 시작
        std::vector left;      // 더 이상 안 보임
        std::vector stayed;    // 계속 보임 (업데이트 필요)
    };

    AOIUpdate UpdatePlayerAOI(PlayerID playerId, float x, float y) {
        AOIUpdate update;

        // 현재 AOI 내 엔티티 조회
        auto currentVisible = spatialIndex.GetEntitiesInAOI(x, y, aoiRadius);
        std::unordered_set currentSet(
            currentVisible.begin(), currentVisible.end()
        );

        // 이전 가시 목록
        auto& previousSet = playerAOIs[playerId].visibleEntities;

        // Enter: 현재에 있고 이전에 없음
        for (EntityID id : currentSet) {
            if (previousSet.find(id) == previousSet.end()) {
                update.entered.push_back(id);
            } else {
                update.stayed.push_back(id);
            }
        }

        // Leave: 이전에 있고 현재에 없음
        for (EntityID id : previousSet) {
            if (currentSet.find(id) == currentSet.end()) {
                update.left.push_back(id);
            }
        }

        // 가시 목록 업데이트
        previousSet = std::move(currentSet);

        return update;
    }

    void SendAOIUpdates(PlayerID playerId, const AOIUpdate& update) {
        // Enter: 전체 스냅샷 전송 (새로 보이는 엔티티)
        for (EntityID id : update.entered) {
            SendEntitySpawn(playerId, id, GetFullSnapshot(id));
        }

        // Leave: 엔티티 제거 알림
        for (EntityID id : update.left) {
            SendEntityDespawn(playerId, id);
        }

        // Stayed: Delta 업데이트만 전송
        for (EntityID id : update.stayed) {
            SendEntityUpdate(playerId, id, GetDeltaSnapshot(id));
        }
    }
};
AOI 경계에서 깜빡이는 문제는 어떻게 해결하나요?

Hysteresis(이력 현상)를 적용합니다. 들어올 때와 나갈 때 거리를 다르게 합니다.

Hysteresis AOI:

              Enter Radius (500)
           ┌─────────────────────┐
           │                     │
           │   Leave Radius(550) │
           │  ┌───────────────┐  │
           │  │               │  │
           │  │      ◉        │  │
           │  │    Player     │  │
           │  │               │  │
           │  └───────────────┘  │
           │                     │
           └─────────────────────┘

- Enter: 500 이내로 들어와야 보임
- Leave: 550 밖으로 나가야 안 보임
- 500~550 사이에서 왔다 갔다 해도 상태 유지
Hysteresis 구현
class HysteresisAOI {
    float enterRadius = 500.0f;
    float leaveRadius = 550.0f;  // enterRadius보다 큰 값

public:
    AOIUpdate UpdatePlayerAOI(PlayerID playerId, float x, float y) {
        auto& playerAOI = playerAOIs[playerId];

        // 넓은 범위로 후보 검색
        auto candidates = spatialIndex.GetEntitiesInAOI(x, y, leaveRadius);

        AOIUpdate update;

        for (EntityID id : candidates) {
            auto& entity = GetEntity(id);
            float dist = Distance(x, y, entity.x, entity.y);

            bool wasVisible = playerAOI.visibleEntities.count(id) > 0;

            bool shouldBeVisible;
            if (wasVisible) {
                // 이미 보이던 것: leaveRadius 밖으로 나가야 사라짐
                shouldBeVisible = (dist <= leaveRadius);
            } else {
                // 안 보이던 것: enterRadius 안으로 들어와야 보임
                shouldBeVisible = (dist <= enterRadius);
            }

            if (shouldBeVisible && !wasVisible) {
                update.entered.push_back(id);
                playerAOI.visibleEntities.insert(id);
            } else if (!shouldBeVisible && wasVisible) {
                update.left.push_back(id);
                playerAOI.visibleEntities.erase(id);
            } else if (shouldBeVisible) {
                update.stayed.push_back(id);
            }
        }

        return update;
    }
};

Part 4: 우선순위 기반 업데이트

모든 엔티티를 같은 빈도로 업데이트해야 하나요?

아니요. 중요도에 따라 업데이트 빈도를 조절합니다.

Priority-based Update
class PriorityAOI {
    struct EntityPriority {
        EntityID id;
        float priority;
        float lastUpdateTime;
    };

public:
    float CalculatePriority(PlayerID viewer, EntityID target) {
        auto& viewerEntity = GetEntity(viewer);
        auto& targetEntity = GetEntity(target);

        float priority = 0.0f;

        // 1. 거리 (가까울수록 중요)
        float distance = Distance(viewerEntity, targetEntity);
        float distancePriority = 1.0f - (distance / aoiRadius);
        priority += distancePriority * 0.4f;

        // 2. 시야 방향 (보고 있는 방향에 있으면 중요)
        Vector3 toTarget = Normalize(targetEntity.pos - viewerEntity.pos);
        float dot = Dot(viewerEntity.forward, toTarget);
        float viewPriority = (dot + 1.0f) / 2.0f;  // 0~1
        priority += viewPriority * 0.3f;

        // 3. 상호작용 (전투 중이면 매우 중요)
        if (IsInCombatWith(viewer, target)) {
            priority += 0.5f;
        }

        // 4. 엔티티 타입 (플레이어 > NPC > 아이템)
        switch (targetEntity.type) {
            case EntityType::Player: priority += 0.2f; break;
            case EntityType::NPC:    priority += 0.1f; break;
            case EntityType::Item:   priority += 0.05f; break;
        }

        // 5. 마지막 업데이트 이후 시간 (오래됐으면 우선순위 상승)
        float timeSinceUpdate = GetTime() - lastUpdateTime[target];
        priority += std::min(timeSinceUpdate * 0.1f, 0.3f);

        return std::min(priority, 1.0f);
    }

    std::vector GetEntitiesToUpdate(
        PlayerID viewer,
        int maxUpdates,
        float deltaTime
    ) {
        auto& visibleEntities = playerAOIs[viewer].visibleEntities;
        std::vector priorities;

        for (EntityID id : visibleEntities) {
            float p = CalculatePriority(viewer, id);
            priorities.push_back({ id, p, lastUpdateTime[id] });
        }

        // 우선순위 높은 순으로 정렬
        std::sort(priorities.begin(), priorities.end(),
            [](const auto& a, const auto& b) {
                return a.priority > b.priority;
            });

        // 상위 N개만 업데이트
        std::vector result;
        for (int i = 0; i < std::min((int)priorities.size(), maxUpdates); i++) {
            result.push_back(priorities[i].id);
            lastUpdateTime[priorities[i].id] = GetTime();
        }

        return result;
    }
};
업데이트 빈도를 거리에 따라 조절하는 방법은?
Distance-based Update Rate:

         ◉ Player
         │
    ─────┼─────  0-50m:  60 Hz (매 틱)
         │
    ─────┼─────  50-100m: 30 Hz (2틱에 1번)
         │
    ─────┼─────  100-200m: 10 Hz (6틱에 1번)
         │
    ─────┼─────  200-500m:  5 Hz (12틱에 1번)
         │
Distance-based Rate
class DistanceBasedUpdate {
    struct UpdateBucket {
        float minDist, maxDist;
        int updateInterval;  // 틱 단위
    };

    std::vector buckets = {
        { 0,    50,   1 },   // 60 Hz
        { 50,   100,  2 },   // 30 Hz
        { 100,  200,  6 },   // 10 Hz
        { 200,  500,  12 },  //  5 Hz
    };

    std::unordered_map lastUpdateTick;

public:
    bool ShouldUpdate(EntityID id, float distance, int currentTick) {
        // 적절한 버킷 찾기
        int interval = buckets.back().updateInterval;
        for (const auto& bucket : buckets) {
            if (distance >= bucket.minDist && distance < bucket.maxDist) {
                interval = bucket.updateInterval;
                break;
            }
        }

        // 마지막 업데이트 이후 충분한 시간이 지났는지
        int lastTick = lastUpdateTick[id];
        if (currentTick - lastTick >= interval) {
            lastUpdateTick[id] = currentTick;
            return true;
        }
        return false;
    }
};

Part 5: Zone 기반 시스템

MMO에서는 어떤 방식을 사용하나요?

대규모 MMO는 Zone(채널) 기반 시스템을 많이 사용합니다.

Zone 기반 아키텍처:

┌────────────────────────────────────────────────────────┐
│                      World                             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐             │
│  │  Zone 1  │  │  Zone 2  │  │  Zone 3  │             │
│  │ Server A │  │ Server B │  │ Server C │             │
│  │          │  │          │  │          │             │
│  │  ○ ○ ○   │──│  ○ ○     │──│    ○     │             │
│  │    ○     │  │  ○ ○ ○   │  │  ○   ○   │             │
│  └──────────┘  └──────────┘  └──────────┘             │
│        │              │              │                 │
│        └──────────────┴──────────────┘                 │
│                Cross-Zone 통신                         │
└────────────────────────────────────────────────────────┘

장점:
- 각 Zone이 독립 서버 → 수평 확장
- Zone 내에서만 AOI 계산 → 효율적
- 장애 격리

단점:
- Zone 경계 처리 복잡
- Cross-Zone 상호작용 지연
Zone 경계에서는 어떻게 처리하나요?
Zone Boundary 처리
class ZoneBoundaryManager {
    static const float BOUNDARY_OVERLAP = 100.0f;  // 경계 중복 영역

public:
    struct ZoneTransfer {
        EntityID entityId;
        ZoneID fromZone;
        ZoneID toZone;
        EntityState state;
    };

    std::optional CheckBoundary(
        EntityID id, const EntityState& state, ZoneID currentZone
    ) {
        Zone& zone = GetZone(currentZone);

        // 엔티티가 Zone 경계를 넘었는지 확인
        if (!zone.bounds.Contains(state.position)) {
            // 새 Zone 결정
            ZoneID newZone = FindZoneContaining(state.position);

            if (newZone != currentZone && newZone != INVALID_ZONE) {
                return ZoneTransfer{id, currentZone, newZone, state};
            }
        }
        return std::nullopt;
    }

    void ExecuteTransfer(const ZoneTransfer& transfer) {
        // 1. 이전 Zone에서 엔티티 제거
        Zone& fromZone = GetZone(transfer.fromZone);
        fromZone.RemoveEntity(transfer.entityId);

        // 2. 새 Zone에 엔티티 추가
        Zone& toZone = GetZone(transfer.toZone);
        toZone.AddEntity(transfer.entityId, transfer.state);

        // 3. 클라이언트에게 Zone 전환 알림
        SendZoneTransfer(transfer.entityId, transfer.toZone);

        // 4. 인접 Zone에 Ghost 엔티티 생성 (경계 가시성)
        CreateGhostEntities(transfer.entityId, transfer.toZone);
    }

    void CreateGhostEntities(EntityID id, ZoneID primaryZone) {
        auto& entity = GetEntity(id);
        auto& zone = GetZone(primaryZone);

        // 경계 근처면 인접 Zone에 Ghost 생성
        for (ZoneID neighborId : zone.neighbors) {
            Zone& neighbor = GetZone(neighborId);

            float distToBoundary = neighbor.DistanceToBoundary(entity.position);
            if (distToBoundary < BOUNDARY_OVERLAP) {
                // Ghost: 읽기 전용 복제본
                neighbor.AddGhost(id, entity.state);
            }
        }
    }
};

Ghost 엔티티는 읽기 전용 복제본으로, 경계 근처 플레이어들이 다른 Zone의 엔티티를 볼 수 있게 합니다.

Part 6: 최적화 기법

AOI 계산을 더 최적화할 수 있나요?
Dirty Flag 패턴
class OptimizedAOI {
    // 이동하지 않은 엔티티는 재계산 안 함
    std::unordered_set dirtyEntities;

public:
    void OnEntityMoved(EntityID id, Vector3 oldPos, Vector3 newPos) {
        // 셀이 바뀌었을 때만 dirty
        auto oldCell = GetCell(oldPos);
        auto newCell = GetCell(newPos);

        if (oldCell != newCell) {
            dirtyEntities.insert(id);
            spatialIndex.Update(id, oldPos, newPos);
        }
    }

    void ProcessDirtyEntities() {
        for (EntityID id : dirtyEntities) {
            // 이 엔티티의 AOI 변화 처리
            // 이 엔티티를 보고 있던 플레이어들의 AOI 업데이트
            UpdateAffectedPlayers(id);
        }
        dirtyEntities.clear();
    }
};

멀티스레드로 AOI를 처리할 수 있나요?
병렬 AOI 처리
class ParallelAOI {
    // Zone 단위로 락 분리
    std::array zoneLocks;

public:
    void ParallelUpdate(const std::vector& players) {
        // 플레이어별로 병렬 처리
        std::for_each(
            std::execution::par_unseq,
            players.begin(), players.end(),
            [this](PlayerID playerId) {
                // 각 플레이어의 AOI 계산은 독립적
                auto& player = GetPlayer(playerId);
                ZoneID zone = GetZone(player.position);

                // Zone 단위 락
                std::lock_guard lock(zoneLocks[zone]);

                auto update = UpdatePlayerAOI(playerId,
                    player.position.x, player.position.y);

                // 결과 큐에 저장 (락 없이)
                playerUpdateQueues[playerId].Push(update);
            }
        );
    }

    // 메인 스레드에서 네트워크 전송
    void FlushUpdates() {
        for (auto& [playerId, queue] : playerUpdateQueues) {
            AOIUpdate update;
            while (queue.TryPop(update)) {
                SendAOIUpdates(playerId, update);
            }
        }
    }
};

📌 핵심 정리

  • AOI: 플레이어 주변 관심 영역만 정보 전송 (N² → N)
  • 공간 분할: Grid, Quadtree, Spatial Hash로 빠른 범위 검색
