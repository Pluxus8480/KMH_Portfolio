게임 장르별 TCP vs UDP 선택 기준

Part 1: TCP와 UDP의 근본적 차이

TCP와 UDP의 가장 큰 차이는 무엇인가요?

신뢰성과 속도의 트레이드오프입니다.

TCP - "확실하게, 순서대로 배달"
Sender
→ [1] Hello
Receiver
← ACK [1]
[2] World → X (손실) ... 타임아웃 ... 재전송
← ACK [2]
장점: 모든 패킷 도착 보장, 순서 보장
단점: 손실 시 대기, Head-of-Line Blocking
UDP - "빠르게, 최선을 다해 배달 (보장은 없음)"
Sender
→
Receiver
[1] Position → 도착
[2] Position → X (손실, 무시)
[3] Position → 도착 (계속 진행)
장점: 빠름, 지연 없음
단점: 손실/순서 미보장, 직접 처리 필요
Head-of-Line Blocking이 뭔가요?

TCP에서 하나의 패킷이 손실되면, 그 뒤의 모든 패킷이 대기해야 합니다.

Head-of-Line Blocking 문제:
[1]
OK
[2]
손실!
[3]
대기
[4]
대기
[5]
대기
↓
[2]가 올 때까지 [3][4][5]를 애플리케이션에 전달 못함
↓
결과
50ms 후 [2] 재전송 도착 → [2][3][4][5] 한꺼번에 전달
실제론 [3][4][5]도 50ms 지연됨!

FPS에서 위치 정보가 50ms 지연되면 캐릭터가 순간이동하는 것처럼 보입니다.

Part 2: 게임 데이터 특성 분석

게임에서 어떤 데이터가 오가나요?
데이터 유형 특성 손실 허용? 순서 중요?
위치/이동 빈번, 작음 ✅ 예 (최신 값이면 됨) ❌ 아니오
사격/스킬 중요 이벤트 ❌ 아니오 ⚠️ 상황에 따라
채팅 텍스트 ❌ 아니오 ✅ 예
인벤토리 변경 중요, 드묾 ❌ 아니오 ✅ 예
게임 결과 매우 중요 ❌ 아니오 ✅ 예

대부분의 게임은 "위치 데이터"가 가장 빈번합니다. 이게 UDP를 선호하는 이유입니다.

그러면 중요한 데이터는 어떻게 보내나요?

UDP 위에 신뢰성 계층을 직접 구현합니다.

Reliable UDP 패킷 구조
// 패킷 헤더 구조
struct PacketHeader {
    uint16_t sequence;      // 패킷 순번
    uint16_t ack;           // 마지막으로 받은 패킷 순번
    uint32_t ackBitfield;   // 최근 32개 패킷 수신 여부 비트마스크
    uint8_t channelId;      // 채널 (unreliable, reliable, ordered)
    uint8_t packetType;     // 패킷 타입
};

// 채널 타입
enum class ChannelType : uint8_t {
    Unreliable = 0,        // 손실 허용, 순서 무관 (위치)
    UnreliableSequenced,   // 손실 허용, 최신만 처리 (상태)
    Reliable,              // 손실 불허, 순서 무관 (이벤트)
    ReliableOrdered,       // 손실 불허, 순서 보장 (채팅, RPC)
};

Part 3: Reliable UDP 구현

ACK와 재전송은 어떻게 구현하나요?
Reliable UDP 핵심 로직
class ReliableUDP {
    static const int WINDOW_SIZE = 32;

    // 송신 측
    struct SendBuffer {
        uint16_t sequence;
        std::vector data;
        float sendTime;
        int retryCount;
    };

    std::deque pendingAcks;  // ACK 대기 중인 패킷
    uint16_t localSequence = 0;

    // 수신 측
    uint16_t remoteSequence = 0;
    uint32_t receivedBits = 0;  // 최근 32개 패킷 수신 비트맵

public:
    void SendReliable(const std::vector& data) {
        PacketHeader header;
        header.sequence = localSequence++;
        header.ack = remoteSequence;
        header.ackBitfield = receivedBits;
        header.channelId = (uint8_t)ChannelType::Reliable;

        // 패킷 전송
        SendPacket(header, data);

        // 재전송 큐에 추가
        pendingAcks.push_back({
            header.sequence,
            data,
            GetTime(),
            0
        });
    }

    void OnPacketReceived(const PacketHeader& header,
                          const std::vector& data) {
        // 1. ACK 처리 (상대가 내 패킷을 받았는지)
        ProcessAcks(header.ack, header.ackBitfield);

        // 2. 시퀀스 처리 (내가 상대 패킷을 받았음을 기록)
        if (IsNewer(header.sequence, remoteSequence)) {
            // 건너뛴 패킷들 비트 시프트
            int shift = header.sequence - remoteSequence;
            receivedBits = (receivedBits << shift) | 1;
            remoteSequence = header.sequence;
        } else {
            // 과거 패킷
            int diff = remoteSequence - header.sequence;
            if (diff < 32) {
                receivedBits |= (1 << diff);
            }
        }

        // 3. 데이터 처리 (채널 타입에 따라)
        ProcessData(header.channelId, header.sequence, data);
    }

private:
    void ProcessAcks(uint16_t ack, uint32_t ackBits) {
        // ack: 상대가 받은 가장 최근 시퀀스
        // ackBits: ack-1, ack-2, ... ack-32까지의 수신 여부

        auto it = pendingAcks.begin();
        while (it != pendingAcks.end()) {
            uint16_t seq = it->sequence;

            bool acked = false;
            if (seq == ack) {
                acked = true;
            } else if (IsOlder(seq, ack)) {
                int diff = ack - seq;
                if (diff <= 32 && (ackBits & (1 << (diff - 1)))) {
                    acked = true;
                }
            }

            if (acked) {
                it = pendingAcks.erase(it);
            } else {
                ++it;
            }
        }
    }

    void Update() {
        float now = GetTime();

        for (auto& pending : pendingAcks) {
            // 100ms 타임아웃 시 재전송
            if (now - pending.sendTime > 0.1f) {
                if (pending.retryCount < MAX_RETRIES) {
                    // 재전송
                    PacketHeader header;
                    header.sequence = pending.sequence;
                    header.ack = remoteSequence;
                    header.ackBitfield = receivedBits;

                    SendPacket(header, pending.data);

                    pending.sendTime = now;
                    pending.retryCount++;
                } else {
                    // 연결 끊김 처리
                    OnConnectionLost();
                }
            }
        }
    }
};
Ordered 채널은 어떻게 구현하나요?
Reliable Ordered 채널
class ReliableOrderedChannel {
    uint16_t nextExpectedSeq = 0;
    std::map> outOfOrderBuffer;

public:
    void OnDataReceived(uint16_t sequence,
                        const std::vector& data) {
        if (sequence == nextExpectedSeq) {
            // 순서대로 도착
            DeliverToApplication(data);
            nextExpectedSeq++;

            // 버퍼에 있던 다음 패킷들도 전달
            while (outOfOrderBuffer.count(nextExpectedSeq)) {
                DeliverToApplication(outOfOrderBuffer[nextExpectedSeq]);
                outOfOrderBuffer.erase(nextExpectedSeq);
                nextExpectedSeq++;
            }
        } else if (IsNewer(sequence, nextExpectedSeq)) {
            // 미래 패킷 → 버퍼에 저장
            outOfOrderBuffer[sequence] = data;
        }
        // 과거 패킷은 이미 처리됨, 무시
    }
};

Part 4: 게임 장르별 선택

FPS 게임은 뭘 사용하나요?

UDP가 압도적으로 유리합니다.

TCP 사용 시
t=0ms: 위치 패킷 1 전송
t=16ms: 위치 패킷 2 전송 (패킷 손실!)
t=32ms: 위치 패킷 3 전송
t=48ms: 위치 패킷 4 전송
...
t=100ms: 패킷 2 재전송 도착
이제서야 패킷 2,3,4 순서대로 처리
패킷 4의 실제 지연: 52ms 추가!
UDP 사용 시
t=0ms: 패킷 1 → 즉시 처리
t=16ms: 패킷 2 (손실) → 무시
t=32ms: 패킷 3 → 즉시 처리 (2 건너뜀)
t=48ms: 패킷 4 → 즉시 처리
패킷 2만 손실, 나머지는 정상 처리!

FPS에서 TCP vs UDP 비교

FPS에서 오래된 위치 정보는 의미가 없습니다. 최신 정보만 중요합니다.

MMORPG는 어떤가요?

대부분 TCP를 사용하거나, 하이브리드를 사용합니다.

🎯 MMORPG의 특수성

- 이동 속도가 느려서 16ms 지연이 덜 눈에 띔

- 동시 접속자가 많아서 UDP 구현 복잡도 부담

게임 장르 권장 프로토콜 이유
FPS/TPS UDP (+Reliable) 반응 속도 최우선
RTS UDP (Lockstep) 입력만 전송, 손실 불가
MMORPG TCP or 하이브리드 신뢰성 중요, 속도 덜 중요
모바일 캐주얼 TCP 개발 편의성, 불안정한 네트워크
턴제 게임 TCP 실시간 아님, 신뢰성 필수
격투 게임 UDP (롤백) 프레임 단위 정밀도 필요

Part 5: 하이브리드 접근법

TCP와 UDP를 동시에 사용할 수 있나요?

네, 많은 게임이 이중 연결을 사용합니다.

하이브리드 네트워크 구조
class HybridConnection {
    TCPConnection tcpConn;   // 신뢰성 데이터용
    UDPConnection udpConn;   // 실시간 데이터용

public:
    void Send(const Packet& packet) {
        switch (packet.priority) {
            case Priority::RealTime:
                // 위치, 회전, 상태
                udpConn.SendUnreliable(packet);
                break;

            case Priority::Important:
                // 사격, 스킬 사용, 히트 판정
                udpConn.SendReliable(packet);
                break;

            case Priority::Critical:
                // 거래, 결제, 인벤토리
                tcpConn.Send(packet);
                break;
        }
    }

    void OnUDPFailed() {
        // UDP 차단 시 TCP로 폴백
        useTCPOnly = true;
        Log("UDP unavailable, falling back to TCP");
    }
};

// 데이터 타입별 채널 매핑
std::map channelMap = {
    { PacketType::Position,     { UDP, Unreliable } },
    { PacketType::Rotation,     { UDP, UnreliableSequenced } },
    { PacketType::Shoot,        { UDP, Reliable } },
    { PacketType::Chat,         { TCP, - } },
    { PacketType::Trade,        { TCP, - } },
    { PacketType::Login,        { TCP, - } },
};

Part 6: 실전 고려사항

UDP가 막히는 네트워크 환경이 있나요?

네, 일부 기업/학교 네트워크, 일부 국가에서 UDP가 차단됩니다.

UDP 가용성 체크 및 폴백
class ConnectionManager {
    enum class ConnectionState {
        TryingUDP,
        UDPActive,
        FallingBackToTCP,
        TCPOnly
    };

    ConnectionState state = ConnectionState::TryingUDP;
    float udpTimeout = 5.0f;

public:
    void Connect(const std::string& serverAddr) {
        // TCP는 항상 연결 (기본 통신)
        tcpConn.Connect(serverAddr, TCP_PORT);

        // UDP 연결 시도
        udpConn.SendHandshake(serverAddr, UDP_PORT);
        state = ConnectionState::TryingUDP;

        // 5초 내 응답 없으면 TCP만 사용
        StartTimer(udpTimeout, [this]() {
            if (state == ConnectionState::TryingUDP) {
                state = ConnectionState::TCPOnly;
                Log("UDP unavailable, using TCP only");
            }
        });
    }

    void OnUDPHandshakeResponse() {
        state = ConnectionState::UDPActive;
        Log("UDP connection established");
    }

    void Send(const Packet& packet, bool reliable) {
        if (state == ConnectionState::UDPActive && !reliable) {
            udpConn.Send(packet);
        } else {
            // TCP 폴백
            tcpConn.Send(packet);
        }
    }
};
NAT 통과(NAT Traversal)는 어떻게 하나요?

P2P 연결이 필요할 때 NAT를 통과해야 합니다.

1. STUN (Simple Traversal of UDP through NATs)
Client A
(NAT)
→
STUN Server
"너의 공인 IP:Port는
1.2.3.4:5000이야"
←
Client B
(NAT)
이 정보를 상대에게 전달하여 직접 연결 시도
2. TURN (Traversal Using Relays around NAT)
Client A
↔
TURN Server
모든 트래픽 경유
(느림, 비용↑)
↔
Client B
STUN 실패 시 중계 서버 사용
3. UDP Hole Punching
양쪽이 동시에 상대 NAT에 패킷 전송 → 타이밍 맞으면 직접 연결 성공

NAT Traversal 기법 비교

게임 서버 연결은 보통 서버가 공인 IP라 NAT 문제 없음. P2P 필요 시에만 STUN/TURN 사용.

📌 핵심 정리

  • TCP: 신뢰성 보장, 순서 보장, 하지만 Head-of-Line Blocking
  • UDP: 빠르지만 손실/순서 미보장, 직접 구현 필요
  • Reliable UDP: 시퀀스, ACK, 재전송으로 TCP 수준 신뢰성 구현
  • FPS/격투: UDP 필수 (반응 속도 최우선)
  • MMO/턴제: TCP 적합 (신뢰성 중요, 속도 덜 중요)
  • 하이브리드: TCP + UDP 동시 사용, 데이터별 채널 분리
  • 폴백: UDP 차단 환경 대비 TCP 폴백 필수
