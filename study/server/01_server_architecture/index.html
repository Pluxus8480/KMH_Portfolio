Part 1: 게임 서버의 역할

게임 서버는 왜 필요한가요?

멀티플레이어 게임에서 서버는 신뢰할 수 있는 권위자(Authority) 역할을 합니다.

클라이언트가 조작할 수 없는 게임 상태를 관리하고, 모든 플레이어에게 일관된 경험을 제공합니다.

게임 서버의 핵심 역할:

┌─────────────────────────────────────────────────────────┐
│                      Game Server                         │
├─────────────────────────────────────────────────────────┤
│  1. 권위적 상태 관리  │  게임 월드의 "진짜" 상태        │
│  2. 입력 검증        │  클라이언트 요청 유효성 확인     │
│  3. 치트 방지        │  불가능한 행동 차단              │
│  4. 상태 동기화      │  모든 클라이언트에게 상태 전파   │
│  5. 매치 관리        │  게임 세션 생성/종료             │
└─────────────────────────────────────────────────────────┘
            │
            │ 동기화
            ▼
┌──────────┐  ┌──────────┐  ┌──────────┐
│ Client 1 │  │ Client 2 │  │ Client 3 │
│ 플레이어A│  │ 플레이어B│  │ 플레이어C│
└──────────┘  └──────────┘  └──────────┘

Part 2: 모놀리식 vs 마이크로서비스

게임 서버도 마이크로서비스로 만들어야 하나요?

게임 특성에 따라 다릅니다. 실시간 게임은 레이턴시가 중요해 모놀리식이 유리하고, 대규모 MMO는 기능별 분리가 필요합니다.

모놀리식 (Monolithic):

┌─────────────────────────────────────┐
│           Game Server               │
│  ┌─────────────────────────────────┐│
│  │ 로직 + 네트워크 + DB + 매치    ││
│  │ 모든 기능이 하나의 프로세스    ││
│  └─────────────────────────────────┘│
└─────────────────────────────────────┘

장점: 낮은 레이턴시, 간단한 배포
단점: 스케일링 어려움, 한 기능 장애 = 전체 장애


마이크로서비스 (Microservices):

┌──────────┐  ┌──────────┐  ┌──────────┐
│ Auth     │  │ Match    │  │ Chat     │
│ Service  │  │ Making   │  │ Service  │
└────┬─────┘  └────┬─────┘  └────┬─────┘
     │             │             │
     └─────────────┼─────────────┘
                   │
           ┌───────┴───────┐
           │   Game Server  │  (실시간 게임 로직)
           │   (Stateful)   │
           └───────────────┘

장점: 독립적 스케일링, 장애 격리
단점: 복잡한 인프라, 서비스 간 레이턴시

아키텍처 비교

항목 모놀리식 마이크로서비스
적합한 게임 FPS, 격투, 레이싱 MMO, 소셜 게임
레이턴시 최소 서비스 호출 오버헤드
스케일링 수직(장비 업그레이드) 수평(서비스 복제)
개발 복잡도 낮음 높음

Part 3: Zone/Room 기반 아키텍처

수천 명이 동시 접속하면 서버 하나로 감당이 되나요?

월드를 Zone이나 Room으로 분할해 각 서버가 담당 영역만 처리합니다.

Zone 기반 (MMO):

    ┌────────────────────────────────────────────┐
    │            게임 월드 (World)                │
    │  ┌──────────┐ ┌──────────┐ ┌──────────┐   │
    │  │  Zone 1  │ │  Zone 2  │ │  Zone 3  │   │
    │  │ Server A │ │ Server B │ │ Server C │   │
    │  │          │→│          │→│          │   │
    │  │ 100명    │ │ 150명    │ │ 80명     │   │
    │  └──────────┘ └──────────┘ └──────────┘   │
    └────────────────────────────────────────────┘

플레이어가 Zone 경계를 넘으면 → 서버 핸드오프


Room/Session 기반 (세션 게임):

┌─────────────────────────────────────────────────┐
│                Lobby Server                      │
│        (매치메이킹, 대기실 관리)                  │
└───────────────────┬─────────────────────────────┘
                    │ 매치 성사 시 Room 생성
        ┌───────────┼───────────┐
        ▼           ▼           ▼
   ┌─────────┐ ┌─────────┐ ┌─────────┐
   │ Room 1  │ │ Room 2  │ │ Room 3  │
   │ 10명    │ │ 10명    │ │ 10명    │
   │ 5vs5    │ │ 5vs5    │ │ 5vs5    │
   └─────────┘ └─────────┘ └─────────┘

Zone vs Room 아키텍처

Room 관리 예시 (개념)
class RoomManager
{
    std::unordered_map> rooms;
    std::mutex roomsMutex;

public:
    RoomId CreateRoom(const GameSettings& settings)
    {
        std::lock_guard lock(roomsMutex);

        RoomId newId = GenerateRoomId();
        rooms[newId] = std::make_unique(settings);

        return newId;
    }

    bool JoinRoom(RoomId roomId, PlayerId playerId)
    {
        std::lock_guard lock(roomsMutex);

        auto it = rooms.find(roomId);
        if (it == rooms.end()) return false;

        return it->second->AddPlayer(playerId);
    }

    void Update(float deltaTime)
    {
        // 각 Room 독립적으로 업데이트
        for (auto& [id, room] : rooms)
        {
            room->Update(deltaTime);

            // 게임 종료된 Room 정리
            if (room->IsFinished())
            {
                pendingRemoval.push_back(id);
            }
        }
    }
};

Part 4: 게임 루프와 Tick Rate

서버도 게임 루프가 있나요?

네! 서버는 일정 주기(Tick Rate)로 게임 상태를 업데이트합니다. 64Hz면 초당 64번 업데이트합니다.

서버 게임 루프
class GameServer
{
    static constexpr int TICK_RATE = 64;  // Hz
    static constexpr float TICK_INTERVAL = 1.0f / TICK_RATE;  // 15.625ms

public:
    void Run()
    {
        auto lastTime = std::chrono::high_resolution_clock::now();
        float accumulator = 0.0f;

        while (running)
        {
            auto currentTime = std::chrono::high_resolution_clock::now();
            float deltaTime = std::chrono::duration(
                currentTime - lastTime).count();
            lastTime = currentTime;

            accumulator += deltaTime;

            // 고정 시간 간격으로 업데이트 (Fixed Timestep)
            while (accumulator >= TICK_INTERVAL)
            {
                // 1. 클라이언트 입력 처리
                ProcessPendingInputs();

                // 2. 게임 로직 업데이트
                UpdateGameState(TICK_INTERVAL);

                // 3. 물리 시뮬레이션
                UpdatePhysics(TICK_INTERVAL);

                // 4. 상태 스냅샷 생성 & 전송
                BroadcastStateSnapshot();

                accumulator -= TICK_INTERVAL;
                currentTick++;
            }

            // CPU 휴식 (다음 틱까지)
            SleepUntilNextTick();
        }
    }
};
Tick Rate 간격 적합한 게임
20 Hz 50ms MMO, 전략 게임
30 Hz 33ms 배틀로얄, 레이싱
64 Hz 15.6ms FPS (표준)
128 Hz 7.8ms 경쟁 FPS (CS:GO 등)

Part 5: 서버 컴포넌트 구조

서버 내부는 어떤 컴포넌트로 구성되나요?

Part 6: 실제 사례 분석

유명한 게임들은 어떤 아키텍처를 사용하나요?

장르와 규모에 따라 다양한 아키텍처가 사용됩니다:

게임/엔진 아키텍처 특징
League of Legends Room + 마이크로서비스 매치당 독립 서버, 10명 세션
World of Warcraft Zone 기반 영역별 서버, 서버 간 이동
Fortnite Room + AWS 100명 세션, 동적 스케일링
Counter-Strike Dedicated Room 128 tick, 10명 세션
🎯 아키텍처 선택 가이드

세션 기반 (FPS, MOBA): Room 서버 + 로비 서버

오픈 월드 (MMO): Zone 서버 + 월드 서버

소셜/캐주얼: 마이크로서비스 + API 서버

주의: 초기에 과도한 복잡성은 피하고, 필요에 따라 분리

📌 핵심 정리

  • 게임 서버는 신뢰할 수 있는 권위자로서 게임 상태 관리
  • 모놀리식: 실시간 게임에 적합, 낮은 레이턴시
  • Zone/Room 분할로 부하 분산 및 스케일링
  • Tick Rate: 게임 특성에 맞게 20~128Hz 선택
  • 계층 분리로 유지보수성과 확장성 확보
