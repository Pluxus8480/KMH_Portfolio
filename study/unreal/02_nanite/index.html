폴리곤 수십억 개를 어떻게 렌더링할까?

Nanite 가상화 지오메트리의 원리

Part 1: 기존 방식의 한계

게임에서 고품질 3D 모델을 쓰기 어려운 이유가 무엇인가요?

전통적인 렌더링에서는 모든 메시의 모든 폴리곤을 GPU에 보내야 했습니다. 영화용 에셋(수백만~수천만 폴리곤)을 그대로 쓰면 GPU가 감당할 수 없죠.

그래서 아티스트들은 LOD(Level of Detail)를 수작업으로 만들어야 했습니다. 멀리 있으면 저폴리, 가까우면 고폴리. 하지만 이건 노동집약적이고, LOD 전환 시 "팝인" 현상이 발생합니다.

🎯 비유로 이해하기

기존 방식은 마치 도서관의 모든 책을 한 번에 책상에 올려놓는 것과 같습니다.

Nanite는 필요한 페이지만 그때그때 가져오는 것입니다. 책 전체를 올릴 필요가 없죠.

Part 2: Nanite의 핵심 아이디어

Nanite는 어떻게 이 문제를 해결하나요?

Nanite의 핵심은 가상화(Virtualization)입니다. 가상 메모리처럼, 필요한 데이터만 필요할 때 로드합니다.

메시를 작은 클러스터로 분할하고, 화면에서 차지하는 픽셀 수에 따라 적절한 디테일만 렌더링합니다.

원본 메시 (수백만 폴리곤)
        │
        ▼
┌───────────────────────────────────────┐
│          클러스터 분할                │
│  ┌──┐ ┌──┐ ┌──┐ ┌──┐ ┌──┐ ┌──┐      │
│  │C1│ │C2│ │C3│ │C4│ │C5│ │C6│ ...  │
│  └──┘ └──┘ └──┘ └──┘ └──┘ └──┘      │
│    (각 클러스터 ~128 삼각형)         │
└───────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────┐
│           LOD 계층 생성               │
│                                       │
│   LOD 0: ████████████ (원본)         │
│   LOD 1: ████████                    │
│   LOD 2: ████                        │
│   LOD 3: ██                          │
│                                       │
└───────────────────────────────────────┘
        │
        ▼
    GPU 컬링 + 렌더링

Nanite의 처리 파이프라인

Part 3: 클러스터와 DAG 구조

클러스터는 구체적으로 어떻게 구성되나요?

Nanite는 메시를 약 128개의 삼각형으로 구성된 클러스터로 분할합니다. 이 클러스터들은 DAG(Directed Acyclic Graph) 구조로 계층화됩니다.

클러스터 계층 구조
// 개념적 구조
struct NaniteCluster {
    uint32 TriangleCount;     // ~128개
    BoundingBox Bounds;       // 바운딩 박스
    float LODError;           // 이 LOD의 오차

    // 부모/자식 관계 (DAG)
    NaniteCluster* Parent;    // 더 단순한 버전
    Array Children; // 더 디테일한 버전
};

// 화면에서 클러스터가 차지하는 크기
float ScreenSize = ProjectedBoundsSize / ScreenResolution;

// 화면 크기에 따라 적절한 LOD 선택
if (ScreenSize * LODError < PixelThreshold) {
    // 이 LOD 사용 (충분히 작음)
} else {
    // 자식 클러스터로 세분화
}
왜 삼각형 128개인가요?

128개는 GPU의 웨이브/워프 크기와 관련 있습니다. GPU는 32~64개의 스레드를 동시에 처리하므로, 128개의 삼각형은 효율적인 병렬 처리가 가능합니다.

너무 작으면 오버헤드가 크고, 너무 크면 세밀한 컬링이 어렵습니다.

Part 4: GPU 기반 컬링

Nanite의 컬링은 어떻게 동작하나요?

Nanite는 GPU에서 직접 컬링을 수행합니다. CPU의 개입 없이 수백만 개의 클러스터를 빠르게 처리할 수 있죠.

컬링 단계 설명 제거되는 것
인스턴스 컬링 전체 오브젝트 단위 화면 밖 오브젝트
클러스터 컬링 클러스터 단위 절두체/오클루전 보이지 않는 클러스터
삼각형 컬링 뒷면/서브픽셀 삼각형 렌더링 불필요한 삼각형
오클루전 컬링도 GPU에서 하나요?

네! Nanite는 2-pass 오클루전 컬링을 사용합니다.

1. 이전 프레임의 HZB(Hierarchical Z-Buffer)로 1차 컬링

2. 현재 프레임 렌더링 후 재검사 (늦게 보이게 된 것 처리)

Part 5: 소프트웨어 래스터라이저

왜 하드웨어 래스터라이저를 안 쓰나요?

기존 하드웨어 래스터라이저는 작은 삼각형에 비효율적입니다. 1픽셀보다 작은 삼각형도 최소 2x2 픽셀(쿼드) 단위로 처리하기 때문이죠.

Nanite의 소프트웨어 래스터라이저는 픽셀 단위로 정확하게 처리해서 작은 삼각형도 효율적으로 렌더링합니다.

래스터라이저 선택 로직 (개념)
// 삼각형 크기에 따라 래스터라이저 선택
if (triangleScreenSize > LARGE_TRIANGLE_THRESHOLD) {
    // 하드웨어 래스터라이저 사용
    // 큰 삼각형은 하드웨어가 더 효율적
    UseHardwareRasterizer(triangle);
} else {
    // 소프트웨어 래스터라이저 사용
    // Compute Shader로 직접 픽셀 쓰기
    UseSoftwareRasterizer(triangle);
}

Part 6: Nanite의 한계

Nanite를 못 쓰는 경우도 있나요?

Nanite는 만능이 아닙니다. 다음 경우에는 사용할 수 없습니다:

• 스켈레탈 메시 - 본 애니메이션이 있는 캐릭터

• 월드 포지션 오프셋 - 버텍스 셰이더에서 위치 변경

• 반투명 머티리얼 - 알파 블렌딩 필요

• Masked 머티리얼 (부분 지원)

항목 기존 방식 Nanite
폴리곤 한계 수백만 (성능 저하) 수십억 (실시간)
LOD 제작 수동 (노동 집약) 자동
LOD 전환 팝인 현상 부드러운 전환
애니메이션 완전 지원 제한적

📌 핵심 정리

  • Nanite는 메시를 클러스터로 분할하고 화면 기여도에 따라 적절한 LOD만 렌더링합니다
  • GPU 기반 컬링으로 CPU 병목 없이 수백만 클러스터를 처리합니다
  • 소프트웨어 래스터라이저가 작은 삼각형을 효율적으로 처리합니다
  • 스켈레탈 메시, WPO, 반투명은 지원하지 않습니다
  • 영화급 에셋을 게임에서 실시간으로 사용할 수 있게 해줍니다
