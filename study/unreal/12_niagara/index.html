Niagara 파티클 시스템

Part 1: Niagara vs Cascade

Niagara가 기존 Cascade와 다른 점은 무엇인가요?

Cascade는 노드 기반이 아닌 모듈 스택으로 제한적이었습니다. Niagara는 완전한 노드 기반 시스템으로 훨씬 유연합니다.

항목 Cascade Niagara
구조 고정 모듈 스택 완전 노드 기반
커스텀 로직 제한적 (C++ 필요) 노드로 자유롭게
GPU 시뮬레이션 제한적 지원 완전 지원
데이터 인터페이스 없음 메시, 스켈레톤 등 접근
이벤트 제한적 완전한 이벤트 시스템

Part 3: 모듈과 파라미터

모듈은 어떻게 동작하나요?

모듈은 재사용 가능한 로직 블록입니다. 내장 모듈을 조합하거나 커스텀 모듈을 만들 수 있습니다.

주요 내장 모듈
// Particle Spawn 스택
Initialize Particle      // 파티클 기본 설정
Sphere Location         // 구 형태로 위치 초기화
Add Velocity            // 초기 속도 추가
Set Color               // 초기 색상

// Particle Update 스택
Particle State          // 수명, 활성 상태
Gravity Force           // 중력 적용
Drag                    // 공기저항
Scale Color             // 수명에 따른 색상 변화
Scale Sprite Size       // 수명에 따른 크기 변화
Solve Forces and Velocity // 힘 → 속도 → 위치 계산

// Render
Sprite Renderer         // 빌보드 스프라이트
Mesh Renderer           // 3D 메시
Ribbon Renderer         // 리본/트레일
파라미터 네임스페이스는 무엇인가요?

Niagara는 네임스페이스로 파라미터를 분류합니다:

• System. - 시스템 전체에서 공유

• Emitter. - 에미터 레벨

• Particles. - 개별 파티클

• User. - 외부에서 설정 가능

Part 4: GPU 시뮬레이션

CPU vs GPU 시뮬레이션의 차이는?

CPU 시뮬레이션은 유연하지만 수천 개가 한계입니다. GPU 시뮬레이션은 수백만 개의 파티클을 처리할 수 있습니다.

항목 CPU Sim GPU Sim
파티클 수 ~수천 수백만
게임 데이터 접근 쉬움 Data Interface 필요
콜리전 정확한 물리 Depth Buffer 기반
이벤트 완전 지원 제한적
사용 예 소규모 이펙트 대규모 시뮬레이션
GPU Sim 활성화
// Emitter Properties에서
Sim Target: GPUCompute Sim

// GPU 시뮬레이션 제약사항
• 모든 모듈이 GPU 호환이어야 함
• 이벤트 발생 제한
• 게임 오브젝트 직접 접근 불가
  → Data Interface 사용

// GPU에서 Depth Collision
Modules: Collision (GPU)
  → Depth Buffer 기반 충돌
  → 정확도 낮지만 빠름

Part 5: Data Interface

외부 데이터를 Niagara에서 어떻게 사용하나요?

Data Interface를 통해 메시, 스켈레톤, 텍스처 등의 외부 데이터에 접근합니다.

Data Interface 용도
Static Mesh 메시 표면에서 파티클 스폰
Skeletal Mesh 본 위치, 스키닝된 표면 접근
Texture Sample 텍스처에서 색상/데이터 읽기
Particle Attribute Reader 다른 에미터의 파티클 읽기
Collision Query 월드 콜리전 체크
Audio Spectrum 오디오 반응 이펙트
Skeletal Mesh에서 스폰 예시
// Emitter에 Data Interface 추가
User Parameters:
  → SkeletalMeshDI (Skeletal Mesh Data Interface)

// Particle Spawn에서
Get Skinned Triangle Data
  → 스키닝된 메시 표면에서 위치, 노멀 가져오기

// 결과: 캐릭터 표면을 따라 파티클 스폰
// (불타는 캐릭터, 마법 오라 등)

Part 6: 이벤트와 통신

파티클 간, 에미터 간 통신은 어떻게 하나요?

Niagara는 강력한 이벤트 시스템을 제공합니다. 파티클 충돌, 사망 등의 이벤트로 다른 에미터를 트리거할 수 있습니다.

이벤트 기반 이펙트 체인:

┌──────────────────┐
│  Emitter: 총알   │
│                  │
│  Death Event ────┼──────────────────────────────┐
│  (파티클 사망시) │                              │
└──────────────────┘                              │
                                                  │
                               ┌──────────────────┼──────────────┐
                               │  Emitter: 폭발  │              │
                               │                  │ Event Handler│
                               │  스폰 위치 =     │ ◀────────────┘
                               │  총알 사망 위치  │
                               │                  │
                               │  Collision ──────┼─────────────┐
                               └──────────────────┘             │
                                                                │
                                               ┌────────────────┼───┐
                                               │ Emitter: 파편 │   │
                                               │                │   │
                                               │ 충돌 위치에서  │◀──┘
                                               │ 파편 생성      │
                                               └────────────────────┘

이벤트로 연결된 이펙트 체인

이벤트 설정
// Emitter A: 이벤트 발생
Particle Update:
  Generate Death Event
    → Event Name: "OnBulletDeath"
    → Payload: Position, Velocity

// Emitter B: 이벤트 수신
Event Handler:
  Source: EmitterA.OnBulletDeath

Particle Spawn:
  Set Position (Event의 Position 사용)
  Set Velocity (Event의 Velocity 기반)

📌 핵심 정리

  • Niagara는 완전한 노드 기반 파티클 시스템입니다
  • System > Emitter > Particle 계층 구조로 구성됩니다
  • GPU 시뮬레이션으로 수백만 파티클 처리 가능
  • Data Interface로 메시, 스켈레톤 등 외부 데이터 접근
  • 이벤트로 파티클 간, 에미터 간 통신 가능
