언리얼 세계의 시민권

UObject / AActor / UActorComponent 구조 이해하기

Part 1: UObject - 모든 것의 시작

UObject가 왜 중요한가요?

UObject는 언리얼 엔진의 거의 모든 클래스의 기반입니다. C++의 std::object 같은 것이지만 훨씬 강력한 기능을 제공합니다.

UObject를 상속받으면 얻는 혜택:

• 가비지 컬렉션 - 자동 메모리 관리

• 리플렉션 - 런타임 타입 정보

• 직렬화 - 저장/로드 자동화

• 에디터 통합 - 디테일 패널 노출

🎯 비유로 이해하기

UObject는 언리얼 세계의 "시민권"입니다.

시민권이 있어야 복지 혜택(GC, 리플렉션)을 받을 수 있고, 공식 시스템(에디터)에서 인정받습니다.

일반 C++ 클래스는 "외국인" - 스스로 모든 것을 관리해야 합니다.

Part 3: AActor 깊게 보기

Actor의 생명주기는 어떻게 되나요?

Actor는 명확한 생명주기를 가집니다. 각 단계에서 적절한 초기화/정리를 해야 합니다.

Actor 생명주기
// 1. 생성자 - CDO 생성 시 호출 (에디터 시작, 게임 시작 전)
AMyActor::AMyActor()
{
    // 컴포넌트 생성, 기본값 설정
    // 여기서 GetWorld()는 nullptr일 수 있음!
    RootComponent = CreateDefaultSubobject("Root");
}

// 2. PostInitializeComponents - 컴포넌트 초기화 후
void AMyActor::PostInitializeComponents()
{
    Super::PostInitializeComponents();
    // 컴포넌트 간 연결, 초기 설정
}

// 3. BeginPlay - 게임 시작 시
void AMyActor::BeginPlay()
{
    Super::BeginPlay();
    // 게임 로직 초기화
    // GetWorld() 안전하게 사용 가능
}

// 4. Tick - 매 프레임
void AMyActor::Tick(float DeltaTime)
{
    Super::Tick(DeltaTime);
    // 업데이트 로직
}

// 5. EndPlay - 파괴 또는 레벨 언로드 시
void AMyActor::EndPlay(const EEndPlayReason::Type EndPlayReason)
{
    // 정리 작업
    Super::EndPlay(EndPlayReason);
}

// 6. 소멸자 - GC에 의해 호출
AMyActor::~AMyActor()
{
    // UPROPERTY가 아닌 리소스 정리
}
생성자에서 주의할 점은 무엇인가요?

생성자는 CDO(Class Default Object) 생성 시에도 호출됩니다!

• GetWorld()가 nullptr일 수 있음

• 다른 Actor 참조 불가

• 게임 로직 금지 - 오직 기본값 설정만

Part 4: UActorComponent의 힘

컴포넌트 기반 설계의 장점은 무엇인가요?

컴포넌트는 기능의 재사용 단위입니다. 상속 대신 조합(Composition)으로 기능을 확장합니다.

상속 vs 컴포넌트
// ❌ 상속 방식의 문제
class AFlyingEnemy : public AEnemy { };
class ASwimmingEnemy : public AEnemy { };
class AFlyingSwimmingEnemy : public ??? // 다중상속 문제!

// ✅ 컴포넌트 방식
class AEnemy : public ACharacter
{
    UPROPERTY()
    UFlyingComponent* FlyingComp;  // 필요하면 추가

    UPROPERTY()
    USwimmingComponent* SwimmingComp;  // 필요하면 추가
};

// 적 유형별로 필요한 컴포넌트만 부착
// FlyingEnemy = Enemy + FlyingComponent
// SwimmingEnemy = Enemy + SwimmingComponent
// FlyingSwimmingEnemy = Enemy + FlyingComponent + SwimmingComponent
컴포넌트 유형 설명 예시
USceneComponent Transform 있음, 계층 구조 StaticMesh, Camera
UActorComponent Transform 없음, 순수 로직 Movement, Inventory

Part 5: UPROPERTY와 리플렉션

UPROPERTY를 꼭 써야 하나요?

UObject 포인터는 반드시 UPROPERTY로 선언해야 합니다! 그렇지 않으면 GC가 추적하지 못해 댕글링 포인터가 됩니다.

UPROPERTY 사용법
UCLASS()
class AMyActor : public AActor
{
    GENERATED_BODY()

public:
    // GC 추적됨, 에디터 노출
    UPROPERTY(EditAnywhere, BlueprintReadWrite, Category = "Config")
    float Health = 100.0f;

    // GC 추적됨, 에디터 숨김
    UPROPERTY()
    UMyComponent* MyComp;

    // ❌ 위험! GC가 추적하지 않음
    UMyComponent* DangerousComp;  // 절대 금지!

    // 일반 C++ 타입은 UPROPERTY 없어도 됨
    float TempValue;  // OK (UObject가 아니므로)
    TArray Values;  // OK

    // 하지만 직렬화/에디터 노출하려면 필요
    UPROPERTY(SaveGame)
    TArray SavedValues;
};
지정자 용도
EditAnywhere 에디터에서 편집 가능
VisibleAnywhere 에디터에서 보기만 가능
BlueprintReadWrite Blueprint에서 읽기/쓰기
BlueprintReadOnly Blueprint에서 읽기만
Replicated 네트워크 복제
SaveGame 게임 저장 시 포함

📌 핵심 정리

  • UObject는 GC, 리플렉션, 직렬화를 제공하는 최상위 클래스입니다
  • AActor는 월드에 존재하고 Transform과 Tick을 가집니다
  • UActorComponent는 기능의 재사용 단위로, 상속보다 조합을 권장합니다
  • UObject 포인터는 반드시 UPROPERTY로 선언해야 GC가 추적합니다
  • 클래스 선택은 "어디에 존재하고 무엇을 하는가"로 결정합니다
