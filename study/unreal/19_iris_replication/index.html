100명이 한 서버에서 싸우려면?

Iris Replication System — 언리얼 차세대 네트워크 복제 시스템의 구조, 데이터 플로우, 마이그레이션 전략

Part 1: 왜 Iris가 필요한가 — 레거시 복제의 한계

기존 언리얼 네트워크 복제가 대규모 게임에서 어떤 문제가 있었나요?

언리얼의 레거시 복제 시스템은 Pull 모델(폴링 방식)로 동작합니다. 서버가 매 틱마다 모든 복제 대상 프로퍼티를 순회하면서 "이전 값과 달라졌는가?"를 비교합니다.

플레이어가 10명이면 괜찮지만, 100명이 되면 문제가 폭발합니다:

  • N명의 플레이어 × M개의 액터 = N×M 번의 비교 연산이 매 틱마다 발생
  • 각 클라이언트 커넥션마다 독립적으로 같은 작업을 반복 — 작업 공유 불가
  • Relevancy 판정(AActor::IsNetRelevantFor())도 커넥션마다 개별 호출

CPU 비용이 클라이언트 수의 제곱에 가깝게(~N²) 증가하는 구조적 문제입니다.

🎯 비유로 이해하기

100명이 구독하는 신문사가 있다고 합시다. 레거시 방식은 구독자 한 명마다 기자가 "어떤 기사가 바뀌었지?" 하고 전체 기사를 처음부터 끝까지 다시 읽는 것과 같습니다.

Iris는 기자가 기사를 수정할 때 "이 기사 바뀌었어!"라고 깃발을 꽂아두고, 인쇄소는 깃발이 꽂힌 기사만 모아서 100명에게 공유하는 방식입니다.

ReplicationGraph로 해결할 수 없었나요?

UReplicationGraph는 레거시 시스템 위에 얹힌 최적화 레이어입니다. 액터를 공간/그룹별로 나눠서 "이 커넥션에는 이 액터들만 고려해"라고 범위를 줄여주죠.

하지만 근본적인 Pull 모델 자체는 바꾸지 못합니다:

  • 범위를 줄여도 여전히 매 틱마다 프로퍼티 비교가 필요
  • 커넥션 간 작업 공유가 불가능한 구조는 그대로
  • 대규모 게임에서는 결국 한계에 도달

Iris는 ReplicationGraph를 대체하는 것이 아니라, 복제 시스템 자체를 새로 설계한 것입니다. Iris는 자체 필터링 시스템을 갖추고 있어 ReplicationGraph와 함께 사용하지 않습니다.

Part 2: Iris란 무엇인가 — Push 모델의 탄생

Iris가 정확히 무엇인가요?

Iris는 UE 5.0에서 Experimental로 처음 등장하고, UE 5.7에서 Beta로 승격된 차세대 복제(Replication) 시스템입니다.

독립된 서버 기술이 아니라, 기존 Dedicated Server 아키텍처 안에서 프로퍼티 복제 방식만 교체하는 opt-in 시스템입니다.

에픽이 Fortnite Battle Royale(100인 서버)를 운영하면서 직접 겪은 문제를 해결하기 위해 개발했고, 2022년에 Fortnite에 내부 적용 완료했습니다.

Pull 모델 vs Push 모델 — 핵심 차이가 뭔가요?

가장 근본적인 차이는 "누가 변경을 감지하느냐"입니다:

구분 레거시 (Pull) Iris (Push)
변경 감지 서버가 매 틱 전체 프로퍼티 순회하여 비교 프로퍼티 변경 시 개발자가 MARK_PROPERTY_DIRTY로 표시
작업 공유 커넥션마다 독립적으로 처리 양자화된 상태 복사본을 커넥션 간 공유
CPU 스케일링 ~O(N²) (플레이어 × 액터) ~O(N) (변경된 프로퍼티만)
필터링 ReplicationGraph (외부 최적화) 내장 Spatial/Group/Dynamic 필터
SubObject 복제 제한적 UObject 단위 독립 복제 지원
리플레이 지원 미지원 (Beta 기준)
"양자화된 상태 복사본"이 뭔가요?

Iris는 모든 복제 대상 데이터의 양자화(Quantized) 사본을 하나 유지합니다. 이 사본은 네트워크로 전송될 때의 정밀도로 미리 변환되어 있습니다.

예를 들어 float 위치 값이 있으면, 네트워크 전송 시 반올림될 정밀도로 미리 변환해 저장합니다. 그러면:

  • 변경 감지: 양자화된 값끼리 비교하면 됨 (빠름)
  • 델타 계산: 이전 양자화 값과 현재 양자화 값의 차이만 전송
  • 커넥션 간 공유: 모든 커넥션이 같은 양자화 사본을 참조 — 커넥션마다 별도 계산 불필요

이것이 Iris의 성능 향상 핵심입니다. 레거시에서는 커넥션마다 개별적으로 수행하던 비교/직렬화 작업을 한 번만 수행하고 결과를 공유합니다.

Part 3: Iris 내부 아키텍처 — 핵심 컴포넌트

Iris를 구성하는 핵심 클래스들은 무엇인가요?

Iris는 레거시 복제 경로를 완전히 대체하는 새로운 컴포넌트 세트로 구성됩니다:

Iris 핵심 컴포넌트 계층 구조

UReplicationSystem
UActorReplicationBridge
(= UEngineReplicationBridge)
FReplicationWriter
커넥션당 1개
FNetRefHandle

Spatial / Group / Dynamic / Connection
각 컴포넌트의 역할을 자세히 설명해주세요.

1. UReplicationSystem

Iris의 중앙 제어 시스템입니다. 필터링, 객체 추적, NetId 할당, 우선순위 관리를 총괄합니다. 기존 UNetDriver의 복제 관련 책임을 흡수합니다.

2. UActorReplicationBridge (UEngineReplicationBridge)

액터 월드와 복제 시스템 사이의 다리 역할입니다. 액터/컴포넌트가 복제 시스템에 등록되고 해제되는 생명주기를 관리합니다. 액터가 스폰되면 이 브릿지를 통해 Iris에 등록되고, 고유한 FNetRefHandle을 받습니다.

3. FReplicationWriter

각 클라이언트 커넥션마다 하나씩 존재하며, 실제 패킷 인코딩과 전송을 담당합니다. 스케줄링 → 필터링 → 직렬화 → 패킷 생성의 파이프라인을 처리합니다.

4. FNetRefHandle

복제 대상 객체를 식별하는 참조 핸들입니다. 기존의 FNetworkGUID와 유사하지만, Iris의 내부 인덱싱 시스템에 최적화되어 있습니다.

필터링 시스템은 어떻게 구성되나요?

Iris는 ReplicationGraph 대신 자체적인 다층 필터링 시스템을 제공합니다:

Spatial Filter (공간 필터)

설정된 거리 내의 객체만 복제합니다. 대규모 오픈 월드에서 핵심적인 필터로, 멀리 있는 나무 1만 그루를 모든 클라이언트에 복제하지 않아도 됩니다.

Connection Filter (커넥션 필터)

TBitArray<> 기반으로 특정 커넥션에 대해 객체의 복제를 허용/차단합니다. "이 팀원에게만 보이는 객체" 같은 시나리오에 적합합니다.

Group Filter (그룹 필터)

객체를 논리적 그룹으로 묶어 일괄 관리합니다.

Dynamic Filter (동적 필터)

런타임에 조건에 따라 필터링 규칙을 변경할 수 있습니다. DefaultEngine.ini에서 액터 클래스별로 어떤 필터를 적용할지 설정합니다.

DefaultEngine.ini — 필터 설정 예시
[/Script/IrisCore.ReplicationSystem]
; TreeActor 클래스에 공간 필터 적용 — 거리 밖의 나무는 복제하지 않음
+FilterConfigs=(ClassName=/Script/TheGame.TreeActor, DynamicFilterName=Spatial)

; NPCActor에 그룹 필터 적용
+FilterConfigs=(ClassName=/Script/TheGame.NPCActor, DynamicFilterName=Group)

; 팀 전용 객체에 커넥션 필터 적용
+FilterConfigs=(ClassName=/Script/TheGame.TeamOnlyActor, DynamicFilterName=Connection)

Part 4: 데이터 플로우 파이프라인 — 틱 한 번의 여정

서버 틱 한 번에 Iris가 수행하는 전체 파이프라인을 보여주세요.

Iris의 복제 파이프라인은 4단계로 구성됩니다. 각 단계는 FReplicationWriter의 메서드에 대응합니다:

Iris 복제 파이프라인 (서버 틱 1회)

Phase 1: BeginWrite()
스케줄링 버퍼 할당 FScheduleObjectInfo 배열 준비
Phase 2: ScheduleObjects()
우선순위 계산 필터링 적용 복제 대상 결정
Phase 3: Write()
양자화 상태 비교 델타 인코딩 패킷 생성
Phase 4: FinishWrite()
리소스 해제 패킷 배치 완료 전송 큐 플러시

이 파이프라인이 각 클라이언트 커넥션마다 1사이클 실행됨

각 Phase를 상세하게 설명해주세요.

Phase 1 — BeginWrite()

한 커넥션의 복제 사이클을 시작합니다. FScheduleObjectInfo 배열을 할당하여 "이번 틱에 어떤 객체를 어떤 우선순위로 처리할지" 담을 공간을 준비합니다.

주의: 이 단계에서 MaxReplicatedObjectCount에 비례하는 메모리를 할당합니다. 기본값으로도 커넥션당 약 524KB, 101명이면 약 52.8MB의 메모리가 사용됩니다.

Phase 2 — ScheduleObjects()

복제 대상을 결정하는 핵심 단계입니다:

  1. 등록된 모든 복제 객체 순회
  2. Spatial/Group/Dynamic/Connection 필터 적용
  3. 통과한 객체에 우선순위 인덱스 할당
  4. 대역폭 예산에 맞춰 상위 N개 선택

Phase 3 — Write()

실제 직렬화 단계입니다. 양자화된 상태 사본과 현재 값을 비교해 변경된 프로퍼티만 델타 인코딩합니다. 이 비교는 양자화 사본을 커넥션 간 공유하므로, 같은 변경에 대해 한 번만 수행됩니다.

Phase 4 — FinishWrite()

사이클을 마무리합니다. 스케줄링 버퍼를 해제하고, 생성된 패킷 배치를 확정하여 전송 큐에 올립니다.

Part 5: 코드 레벨 구현 — Iris 활성화부터 프로퍼티 복제까지

프로젝트에서 Iris를 활성화하려면 어떻게 해야 하나요?

3가지 설정이 필요합니다:

1. DefaultEngine.ini — Iris 활성화
[SystemSettings]
; SubObject 복제 목록 사용 (Iris 전제 조건)
net.SubObjects.DefaultUseSubObjectReplicationList=1

; Push 모델 활성화
net.IsPushModelEnabled=1

; Iris 복제 시스템 사용
net.Iris.UseIrisReplication=1

; Push 모델 모드 (1 = Iris Push)
net.Iris.PushModelMode=1
2. YourGame.Build.cs — 모듈 의존성 추가
public class YourGame : ModuleRules
{
    public YourGame(ReadOnlyTargetRules Target) : base(Target)
    {
        // Iris 모듈 지원 설정
        SetupIrisSupport(Target);

        // 기존 의존성...
        PublicDependencyModuleNames.AddRange(new string[] {
            "Core", "CoreUObject", "Engine", "InputCore"
        });
    }
}
기존 UPROPERTY(Replicated) 코드는 어떻게 바뀌나요?

Iris에서는 Push 기반 Dirty 마킹이 핵심입니다. 기존 코드에 MARK_PROPERTY_DIRTY 매크로를 추가해야 합니다:

레거시 방식 (Pull)
UCLASS()
class AMyActor : public AActor
{
    GENERATED_BODY()

    UPROPERTY(Replicated)
    int32 Health;

    // 값을 변경하면 서버가 알아서 감지 (매 틱 비교)
    void TakeDamage(int32 Damage)
    {
        Health -= Damage;
        // 끝. 서버가 다음 틱에 변경을 감지함
    }

    void GetLifetimeReplicatedProps(
        TArray<FLifetimeProperty>& OutLifetimeProps) const override
    {
        Super::GetLifetimeReplicatedProps(OutLifetimeProps);
        DOREPLIFETIME(AMyActor, Health);
    }
};
Iris 방식 (Push)
UCLASS()
class AMyActor : public AActor
{
    GENERATED_BODY()

    UPROPERTY(Replicated)
    int32 Health;

    void TakeDamage(int32 Damage)
    {
        Health -= Damage;

        // ★ 핵심: 프로퍼티가 변경되었음을 명시적으로 알림
        MARK_PROPERTY_DIRTY_FROM_NAME(AMyActor, Health, this);
    }

    void GetLifetimeReplicatedProps(
        TArray<FLifetimeProperty>& OutLifetimeProps) const override
    {
        Super::GetLifetimeReplicatedProps(OutLifetimeProps);

        // Push 기반 파라미터 설정
        FDoRepLifetimeParams Params;
        Params.bIsPushBased = true;  // ★ Push 모드 활성화
        DOREPLIFETIME_WITH_PARAMS_FAST(AMyActor, Health, Params);
    }
};
MARK_PROPERTY_DIRTY를 빼먹으면 어떻게 되나요?

해당 프로퍼티의 변경이 클라이언트에 전파되지 않습니다. Iris는 Dirty 마킹된 프로퍼티만 처리하기 때문입니다. 이것이 Push 모델의 트레이드오프 — 성능을 얻는 대신 개발자에게 명시적 표시 책임이 생깁니다.

다만, Push 모델이 아직 불안하다면 bIsPushBased = false로 개별 프로퍼티를 레거시 방식으로 남겨둘 수도 있습니다. 점진적 마이그레이션이 가능합니다.

UObject SubObject 복제는 어떻게 하나요?

Iris의 강력한 기능 중 하나는 UObject 단위의 독립적인 SubObject 복제입니다. 액터에 종속된 일반 UObject도 자체 NetId를 갖고 독립적으로 복제할 수 있습니다:

UObject SubObject 복제 전체 코드 C++ 
// ---- SubObject 클래스 정의 ----
UCLASS()
class UMyReplicatedObject : public UObject
{
    GENERATED_BODY()

public:
    // 네트워크 지원 활성화
    bool IsSupportedForNetworking() const override { return true; }

    UPROPERTY(Replicated)
    int32 MyRepValue;

    UPROPERTY(Replicated)
    FVector Position;

    void SetMyRepValue(int32 NewValue)
    {
        MyRepValue = NewValue;
        MARK_PROPERTY_DIRTY_FROM_NAME(UMyReplicatedObject, MyRepValue, this);
    }

    void GetLifetimeReplicatedProps(
        TArray<FLifetimeProperty>& OutLifetimeProps) const override
    {
        Super::GetLifetimeReplicatedProps(OutLifetimeProps);

        FDoRepLifetimeParams Params;
        Params.bIsPushBased = true;

        DOREPLIFETIME_WITH_PARAMS_FAST(
            UMyReplicatedObject, MyRepValue, Params);
        DOREPLIFETIME_WITH_PARAMS_FAST(
            UMyReplicatedObject, Position, Params);
    }
};

// ---- 소유 액터에서 SubObject 등록 ----
UCLASS()
class AMyOwnerActor : public AActor
{
    GENERATED_BODY()

    UPROPERTY()
    TObjectPtr<UMyReplicatedObject> SubObj;

    void BeginPlay() override
    {
        Super::BeginPlay();

        if (HasAuthority())
        {
            // SubObject 생성 및 복제 시스템에 등록
            SubObj = NewObject<UMyReplicatedObject>(this);
            AddReplicatedSubObject(SubObj);
        }
    }

    void EndPlay(const EEndPlayReason::Type EndPlayReason) override
    {
        if (HasAuthority() && SubObj)
        {
            // 복제 시스템에서 해제
            RemoveReplicatedSubObject(SubObj);
        }
        Super::EndPlay(EndPlayReason);
    }
};
GameInstance에서 Iris 설정을 오버라이드할 수 있나요?

네. UGameInstance에서 최대 복제 객체 수, 프리얼로케이션 버퍼 등을 커스터마이즈할 수 있습니다:

GameInstance에서 Iris Config 오버라이드
void UTheGameInstance::OverrideIrisReplicationSystemConfig(
    FNetDriverReplicationSystemConfig& OutConfig,
    bool bIsServer) const
{
    // 최대 복제 객체 수: 131,072개
    OutConfig.MaxReplicatedObjectCount = 1 << 17;

    // 초기 객체 리스트 크기
    OutConfig.InitialNetObjectListCount =
        OutConfig.MaxReplicatedObjectCount;

    // 프리얼로케이션 메모리 버퍼 크기
    OutConfig.PreAllocatedMemoryBuffersObjectCount =
        OutConfig.MaxReplicatedObjectCount;
}

주의: PreAllocatedMemoryBuffersObjectCount를 200만처럼 과도하게 설정하면, 빈 맵에서도 ~40 FPS까지 성능이 떨어질 수 있습니다. 또한 한번 확장된 버퍼는 절대 축소되지 않아 영구적인 메모리 오버헤드가 됩니다.

Part 6: 실전 성능 벤치마크 — 진짜 빨라지나?

레거시 대비 Iris의 실제 성능 차이는 어느 정도인가요?

BorMor의 100명 플레이어 테스트 결과를 바탕으로 한 실측 데이터입니다:

측정 항목 레거시 복제 Iris (미최적화) Iris (패치 후)
서버 FPS 10–15 15–20 25–30
전체 틱 시간 83.9 ms 60.1 ms 45.5 ms
Net 틱 시간 13.3 ms 10.2 ms —
NetBroadcastTickTime 66.2 ms 45.8 ms 29.5 ms

결과: 약 31% FPS 향상, 24% 프레임 타임 단축. "패치 후"는 BeginWrite()의 메모리 할당 비효율을 수정한 버전입니다.

대규모 액터 시나리오에서는요?

10,000개의 나무 액터를 복제하는 "만 그루의 숲" 테스트에서:

  • 레거시: 약 7 FPS — 거의 플레이 불가
  • Iris + 공간 필터링: 약 40 FPS — 쾌적한 플레이 가능

5.7배의 성능 차이. 공간 필터가 거리 밖의 액터를 복제 대상에서 완전히 제외하기 때문입니다.

대역폭 사용량은 어떤가요?

대역폭은 서버 틱 레이트에 선형 비례합니다:

  • 30 FPS 기준: 5분간 약 600 MB
  • 70+ FPS 기준: 5분간 900 MB 이상

Iris 자체가 대역폭을 줄여주는 것은 아닙니다. CPU 효율 향상으로 같은 대역폭에서 더 많은 플레이어를 처리하거나, 같은 플레이어 수에서 서버 비용을 줄이는 것이 목표입니다.

Part 7: 알려진 이슈 — BeginWrite() 메모리 할당 문제

Iris에서 가장 큰 성능 병목은 어디인가요?

BorMor의 프로파일링에서 발견된 핵심 이슈는 FReplicationWriter::BeginWrite()의 과도한 메모리 할당입니다:

문제 상황 분석
// FReplicationWriter::BeginWrite() 내부에서 발생하는 할당

// MaxReplicatedObjectCount가 기본 65,535일 때:
// FScheduleObjectInfo 배열 크기 = 65,535 × 8 bytes = 524,280 bytes ≈ 512 KB

// 이 할당이 매 틱, 매 커넥션마다 발생!
// 101명 (서버 + 100 클라이언트):
// 524,280 × 101 = 52,952,280 bytes ≈ 52.8 MB / tick

// 프레임당 52.8 MB 할당 + 해제 반복
// → GC 압박 + 캐시 미스 폭증
이 문제를 어떻게 해결했나요?

두 가지 접근이 가능합니다:

  1. 버퍼 풀링/재사용: BeginWrite()에서 매번 새로 할당하지 않고, 커넥션별로 버퍼를 한 번 할당해서 재사용
  2. MaxReplicatedObjectCount 적절히 설정: 게임에 실제 필요한 복제 객체 수에 맞춰 최소화

패치 후 벤치마크에서 프레임 타임이 60.1ms → 45.5ms로 약 24% 추가 개선되었습니다. 즉 Iris 자체의 개선(레거시 대비) + 메모리 최적화의 이중 효과를 얻을 수 있습니다.

또한 한 가지 주의할 점은 확장된 버퍼가 절대 축소되지 않는다는 것입니다. 게임 도중 복제 객체가 대량 생성되었다가 파괴되어도, 메모리 풋프린트는 최고점에서 유지됩니다.

Part 8: 레거시에서 Iris로 — 마이그레이션 전략

한 번에 전부 바꿔야 하나요?

아닙니다. 점진적 마이그레이션이 가능합니다:

  • 개별 프로퍼티 단위로 bIsPushBased 플래그를 켤 수 있습니다
  • Push 모드가 아닌 프로퍼티는 Iris 안에서도 레거시 방식(폴링)으로 동작합니다
  • 한 액터의 프로퍼티 중 일부만 Push, 나머지는 Pull로 혼용 가능

권장 순서: 가장 자주 변경되는 프로퍼티(위치, 체력, 상태 등)부터 Push로 전환하면 즉각적인 성능 향상을 볼 수 있습니다.

마이그레이션 시 주의사항은?

가장 흔한 실수:

  • MARK_PROPERTY_DIRTY 누락: 프로퍼티 값은 바뀌지만 클라이언트에 전파 안 됨. 디버깅이 매우 어려움
  • ReplicationGraph 충돌: Iris와 ReplicationGraph를 동시에 사용하면 예기치 못한 동작 발생. Iris의 자체 필터링으로 전환 필요
  • 리플레이 시스템 의존: 현재 Iris는 리플레이를 지원하지 않음. 킬캠, 데모 녹화 등이 필요하면 전환 보류
  • 버퍼 과잉 설정: MaxReplicatedObjectCount를 너무 크게 잡으면 빈 맵에서도 성능 저하

Part 10: Iris의 여정 — 타임라인과 미래

Iris의 개발 히스토리는 어떻게 되나요?

Iris의 주요 마일스톤:

시기 마일스톤 상태
UE 5.0 Iris 최초 도입 Experimental
UE 5.1 ~ 5.3 문서화 및 안정화 작업 Experimental
2022 Fortnite Battle Royale에 내부 적용 프로덕션 사용
2022~2024 Fortnite 기반 버그 수정 및 최적화 지속 개선
Unreal Fest Orlando 2025 Iris 설계 철학 발표 커뮤니티 공개
UE 5.7 Beta로 승격 Beta
앞으로 Iris는 어떻게 발전할까요?

에픽의 로드맵과 커뮤니티 투표(169+ 표)를 고려하면:

  • 리플레이 지원: 현재 가장 큰 기능 갭. GA(정식 출시) 전에 해결될 가능성 높음
  • 메모리 최적화: BeginWrite() 할당 패턴 개선
  • 블루프린트 통합: Push 마킹의 BP 지원 강화
  • 정식 출시(GA): Beta에서 Production-Ready로 승격

Iris는 에픽이 Fortnite라는 자사 AAA 게임에 실전 투입한 기술이므로, 단순 실험이 아닌 장기적으로 레거시를 대체할 차세대 표준입니다.

📌 핵심 정리

  • Iris는 Push 기반 복제 시스템으로, 레거시의 매 틱 전체 폴링을 "변경 시에만 알림"으로 대체하여 CPU 비용을 근본적으로 줄입니다.
  • 양자화된 상태 사본을 커넥션 간 공유하여, N명의 플레이어에 대해 비교/직렬화 작업을 1번만 수행합니다 (N번 → 1번).
  • 내장 필터링 시스템(Spatial/Group/Dynamic/Connection)이 ReplicationGraph를 대체하며, 특히 대규모 오픈 월드에서 강력합니다.
  • 100인 테스트에서 31% FPS 향상, 1만 액터 테스트에서 5.7배 성능 개선을 실측했습니다.
  • 점진적 마이그레이션 가능: Push/Pull 혼용으로 프로퍼티 단위 전환이 가능하지만, MARK_PROPERTY_DIRTY 누락에 주의해야 합니다.
  • 현재 Beta (UE 5.7)이며, 리플레이 미지원이 가장 큰 제약입니다. Fortnite에서 실전 검증되었으므로 장기적으로 레거시를 대체할 표준입니다.
