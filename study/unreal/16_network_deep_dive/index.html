패킷은 어디로 흘러가는가

언리얼 서버 네트워크 내부 코드: 패킷 수신부터 RPC 디스패치, 보이스/대용량 데이터 전송까지

Part 1: 언리얼 네트워크 스택 — 전체 그림

언리얼 서버에서 패킷이 들어오고 나가는 전체 흐름을 먼저 보여줄 수 있나요?

언리얼의 네트워크 스택은 크게 NetDriver → NetConnection → Channel → Actor/RPC 4계층으로 구성됩니다. 소켓(UDP)에서 읽은 원시 바이트가 이 파이프라인을 따라 올라가면서 의미 있는 게임 데이터로 변환됩니다.

🎯 비유로 이해하기

우체국(NetDriver)이 우편물(패킷)을 받아서, 수신자 주소(IP:Port)로 사서함(NetConnection)을 찾고, 봉투를 열어 내용물 종류(Channel)별로 분류한 뒤, 최종 수신자(Actor)에게 전달하는 구조입니다.

송신은 반대로 진행되나요?

네, 정확히 역순입니다. 게임 로직이 RPC를 호출하거나 Property가 변경되면, Channel이 Bunch(번치)를 만들고, Connection이 Bunch들을 하나의 패킷으로 묶어 NetDriver를 통해 소켓으로 전송합니다.

Part 2: 패킷 수신 — 소켓에서 Bunch까지

서버에서 패킷 수신이 시작되는 정확한 지점은 어디인가요?

매 틱마다 엔진이 UNetDriver::TickDispatch()를 호출합니다. 이 함수 안에서 소켓의 recvfrom()을 반복 호출하여 수신된 모든 UDP 패킷을 읽어들입니다.

C++ — UIpNetDriver::TickDispatch() 핵심 흐름
void UIpNetDriver::TickDispatch(float DeltaTime)
{
    Super::TickDispatch(DeltaTime);

    // 소켓에서 패킷을 계속 읽음
    while (Socket != nullptr)
    {
        uint8 Data[MAX_PACKET_SIZE];  // 보통 1024~1200 바이트
        int32 BytesRead = 0;
        FInternetAddr FromAddr;

        // 논블로킹 recvfrom
        if (!Socket->RecvFrom(Data, sizeof(Data), BytesRead, *FromAddr))
            break;  // 더 이상 읽을 패킷 없음

        if (BytesRead > 0)
        {
            // 1) 주소로 기존 Connection 찾기
            UNetConnection* Connection = FindConnectionForAddr(FromAddr);

            if (Connection)
            {
                // 2) 해당 Connection에 패킷 전달
                Connection->ReceivedRawPacket(Data, BytesRead);
            }
            else
            {
                // 3) 새 연결 시도 (핸드셰이크)
                ProcessConnectionlessPacket(FromAddr, Data, BytesRead);
            }
        }
    }
}
ReceivedRawPacket() 이후에는 어떤 일이 벌어지나요?

패킷 헤더를 파싱하고, 시퀀스 번호 확인, ACK 처리, 그리고 패킷 내부의 Bunch(번치)들을 분리합니다. 하나의 패킷 안에 여러 Bunch가 담길 수 있습니다.

C++ — UNetConnection::ReceivedPacket() 핵심 로직
void UNetConnection::ReceivedPacket(FBitReader& Reader)
{
    // 1) 패킷 헤더 읽기
    int32 PacketSequence = MakeRelative(Reader.ReadInt(MAX_PACKETID), InPacketId);

    // 2) 시퀀스 확인 — 중복/순서 어긋남 감지
    if (PacketSequence <= InPacketId)
    {
        // 이미 처리했거나 순서가 어긋난 패킷 → 버림
        return;
    }

    InPacketId = PacketSequence;

    // 3) ACK 정보 읽기 — 상대가 내 패킷을 받았는지 확인
    ReadPacketHeader_ACK(Reader);

    // 4) 패킷 안의 Bunch들을 하나씩 분리
    while (!Reader.AtEnd())
    {
        FInBunch Bunch(this);

        // Bunch 헤더: ChIndex(채널), bReliable, bPartial, ...
        Bunch.ChIndex   = Reader.ReadInt(MAX_CHANNELS);    // 어떤 채널인지
        Bunch.bReliable = Reader.ReadBit();                 // 신뢰성 여부
        Bunch.bPartial  = Reader.ReadBit();                 // 분할 Bunch 여부

        // Bunch 데이터 읽기
        int32 BunchDataBits = Reader.ReadInt(MAX_BUNCH_DATA_BITS);
        Bunch.SetData(Reader, BunchDataBits);

        // 5) 해당 채널로 전달
        UChannel* Channel = Channels[Bunch.ChIndex];
        if (Channel)
        {
            Channel->ReceivedBunch(Bunch);
        }
    }
}

하나의 패킷 안에 여러 채널의 Bunch가 담김

Bunch가 정확히 뭔가요? 패킷과 뭐가 다른 건가요?

패킷(Packet)은 네트워크로 전송되는 물리적 단위(UDP 데이터그램)이고, 번치(Bunch)는 하나의 채널이 주고받는 논리적 데이터 단위입니다.

패킷은 여러 채널의 Bunch를 하나로 묶어서 보냅니다. 이렇게 하면 작은 데이터를 채널마다 따로 보내는 낭비를 줄입니다.

Part 3: Channel 시스템 — 데이터 종류별 통로

채널이 여러 종류가 있다고 했는데, 어떤 것들이 있나요?

언리얼은 데이터 성격에 따라 다른 채널을 사용합니다. 각 채널은 UChannel의 서브클래스입니다.

채널 타입 클래스 채널 인덱스 역할
Control UControlChannel 0 (고정) 연결 핸드셰이크, 로그인, 맵 전환
Actor UActorChannel 동적 할당 Actor 복제, Property 동기화, RPC 전송
Voice UVoiceChannel 동적 할당 음성 채팅 데이터 전송
DataStream UDataStreamChannel 동적 할당 대용량 파일/데이터 스트리밍
ActorChannel이 RPC도 처리하고 Property 복제도 처리한다고요?

맞습니다. ActorChannel은 하나의 Actor에 대한 모든 네트워크 통신을 담당합니다. Actor가 서버에서 복제 대상이 되면 해당 클라이언트의 Connection에 ActorChannel이 열립니다. 이 채널을 통해:

1) Actor 스폰 정보 전달
2) Replicated Property 동기화
3) 해당 Actor의 RPC 송수신

이 모든 것이 같은 채널을 통해 이루어집니다.

C++ — UActorChannel::ReceivedBunch() 핵심 분기
void UActorChannel::ReceivedBunch(FInBunch& Bunch)
{
    // Actor가 아직 없으면 스폰 정보를 읽어 생성
    if (Actor == nullptr)
    {
        if (!ProcessSpawnBunch(Bunch))
            return;
    }

    // Bunch 내용을 읽으며 처리
    while (!Bunch.AtEnd())
    {
        // 서브오브젝트/컴포넌트 식별
        UObject* RepObj = ReadContentBlockHeader(Bunch);

        if (RepObj)
        {
            // Property 복제 데이터인지, RPC인지 판별
            FObjectReplicator& Replicator = FindOrCreateReplicator(RepObj);

            // ▼ 이 안에서 Property vs RPC 분기가 일어남
            Replicator.ReceivedBunch(Bunch);
        }
    }
}
🎯 비유로 이해하기

ActorChannel은 "전용 우편함"입니다. 서버의 각 Actor마다 클라이언트 쪽에 하나의 우편함이 열리고, 그 Actor에 관한 모든 편지(Property 변경, RPC 호출)가 이 우편함을 통해 배달됩니다.

Part 4: RPC 내부 — 바인딩부터 큐, 디스패치까지

UFUNCTION(Server, Reliable)을 선언하면 내부적으로 어떻게 바인딩되나요?

UHT(Unreal Header Tool)가 빌드 시점에 코드를 생성합니다. 핵심은 3개의 함수가 만들어진다는 것입니다:

1) Server_Fire() — 호출자가 부르는 함수 (Thunk)
2) Server_Fire_Validate() — 서버 측 유효성 검사 (WithValidation 옵션 시)
3) Server_Fire_Implementation() — 실제 로직 (개발자가 구현)

C++ — UHT가 자동 생성하는 코드 (개념적 재현)
// ===== 개발자가 작성하는 코드 =====
UFUNCTION(Server, Reliable, WithValidation)
void Server_Fire(FVector Location, float Damage);

// ===== UHT가 .generated.h / .gen.cpp에 생성하는 코드 =====

// 1) Thunk — 호출 시 진입점
void AMyCharacter::Server_Fire(FVector Location, float Damage)
{
    // 파라미터를 직렬화용 구조체에 담음
    AMyCharacter_Server_Fire_Parms Parms;
    Parms.Location = Location;
    Parms.Damage = Damage;

    // UFunction 메타데이터에서 RPC 정보 가져옴
    UFunction* Function = FindFunctionChecked(TEXT("Server_Fire"));

    // ▼ 핵심: ProcessEvent()가 네트워크 전송을 트리거
    ProcessEvent(Function, &Parms);
}

// 2) 파라미터 직렬화 구조체
struct AMyCharacter_Server_Fire_Parms
{
    FVector Location;
    float Damage;
};

// 3) FRepLayout에 RPC 함수 등록 — 리플리케이션 시스템이 알 수 있게
static FName NAME_Server_Fire = FName(TEXT("Server_Fire"));
// → ClassReps 배열에 UFunction 포인터가 인덱스와 함께 등록됨
ProcessEvent()가 호출되면 패킷으로 어떻게 변환되나요?

여기서부터가 핵심입니다. AActor::ProcessEvent()는 해당 UFunction이 RPC 플래그를 갖고 있는지 확인하고, RPC라면 CallRemoteFunction()을 호출합니다.

C++ — RPC 호출 체인: ProcessEvent → 패킷 전송
// [1단계] AActor::ProcessEvent()
bool AActor::ProcessEvent(UFunction* Function, void* Parms)
{
    // RPC 플래그 확인
    if (Function->HasAnyFunctionFlags(FUNC_Net))
    {
        // 이 함수가 로컬에서 실행되어야 하는지, 원격으로 보내야 하는지 판단
        ERemoteMode RemoteMode = GetFunctionCallspace(Function);

        if (RemoteMode == ERF_Remote)
        {
            // 원격 전송!
            return CallRemoteFunction(Function, Parms);
        }
    }
    // 로컬 실행인 경우 일반 함수 호출
    return CallFunction(Function, Parms);
}

// [2단계] CallRemoteFunction() → FRepLayout::SendRPC()
bool AActor::CallRemoteFunction(UFunction* Function, void* Parms)
{
    UNetDriver* NetDriver = GetNetDriver();
    if (!NetDriver) return false;

    // 이 Actor의 ActorChannel을 찾음
    UActorChannel* Channel = Connection->FindActorChannel(this);

    // ▼ RPC를 Bunch로 직렬화
    FOutBunch Bunch(Channel, /* bReliable */ Function->HasFlag(FUNC_NetReliable));

    // RPC 함수 인덱스 기록 (수신 측에서 어떤 함수인지 식별)
    uint32 FunctionIndex = GetFunctionIndex(Function);
    Bunch.WriteIntWrapped(FunctionIndex, MaxRPCIndex);

    // 파라미터 직렬화
    FRepLayout& RepLayout = NetDriver->GetFunctionRepLayout(Function);
    RepLayout.SendPropertiesForRPC(Function, Bunch, Parms);

    // ▼ Bunch를 Connection의 송신 큐에 추가
    Channel->SendBunch(&Bunch, /* bMerge */ true);
    return true;
}

// [3단계] UChannel::SendBunch() — 큐잉
void UChannel::SendBunch(FOutBunch* Bunch, bool bMerge)
{
    if (Bunch->bReliable)
    {
        // Reliable Bunch는 ACK 받을 때까지 보관
        Bunch->ChSequence = ++Connection->OutReliable[ChIndex];
        OutRec.push_back(Bunch);  // 재전송 버퍼에 저장
    }

    // Connection의 SendBuffer에 Bunch 데이터 기록
    Connection->WriteBitsToSendBuffer(Bunch);

    // 여러 Bunch가 모여 하나의 패킷이 됨
    // → FlushNet()이 호출될 때 실제 전송
}

그러면 수신 측에서 RPC는 어떻게 실행되나요?

수신 측은 Bunch에서 함수 인덱스를 읽고, 해당 UFunction을 찾아 _Implementation() 함수를 호출합니다.

C++ — RPC 수신 및 디스패치
// FObjectReplicator::ReceivedBunch()에서 RPC 감지 시:
bool FObjectReplicator::ReceivedRPC(FInBunch& Bunch)
{
    // 1) 함수 인덱스 읽기
    uint32 FunctionIndex;
    Bunch.ReadIntWrapped(FunctionIndex, MaxRPCIndex);

    // 2) 인덱스로 UFunction 조회
    //    ClassReps 테이블에서 매핑된 함수 찾기
    UFunction* Function = ClassReps[FunctionIndex];

    // 3) 파라미터 역직렬화
    FRepLayout& RepLayout = OwningChannel->Connection->Driver
                            ->GetFunctionRepLayout(Function);
    void* Parms = FMemory_Alloca(Function->ParmsSize);
    RepLayout.ReceivePropertiesForRPC(Function, Bunch, Parms);

    // 4) Validation (WithValidation인 경우)
    if (Function->HasFlag(FUNC_NetValidate))
    {
        if (!Object->CallFunctionByNameWithArguments(
                TEXT("Server_Fire_Validate"), Parms))
        {
            // 유효성 검사 실패 → 연결 끊기
            Connection->Close();
            return false;
        }
    }

    // 5) 실제 실행! — _Implementation() 호출
    Object->ProcessEvent(Function, Parms);
    // → Server_Fire_Implementation(Location, Damage) 실행됨!

    return true;
}

주의: Reliable RPC가 순서대로 도착해야 하므로, 중간 시퀀스가 빠지면 뒤따르는 Reliable Bunch들은 큐에 쌓여 대기합니다. 이것이 "Reliable Buffer Overflow"의 원인이 됩니다.

Reliable Bunch의 큐잉과 재전송은 어떻게 작동하나요?

각 채널은 OutReliable 시퀀스 번호를 관리합니다. Reliable Bunch를 보낼 때마다 시퀀스를 1 증가시키고, ACK를 받을 때까지 재전송 버퍼(OutRec)에 보관합니다.

C++ — Reliable Bunch 재전송 메커니즘
// 수신 측: Reliable Bunch 순서 보장
void UChannel::ReceivedBunch(FInBunch& Bunch)
{
    if (Bunch.bReliable)
    {
        int32 ExpectedSequence = InReliable[ChIndex] + 1;

        if (Bunch.ChSequence == ExpectedSequence)
        {
            // 예상 순서와 일치 → 즉시 처리
            InReliable[ChIndex] = Bunch.ChSequence;
            ProcessBunch(Bunch);

            // 큐에 쌓여있던 후속 Bunch도 순서대로 처리
            while (PendingBunches.Num() > 0 &&
                   PendingBunches[0].ChSequence == InReliable[ChIndex] + 1)
            {
                InReliable[ChIndex]++;
                ProcessBunch(PendingBunches[0]);
                PendingBunches.RemoveAt(0);
            }
        }
        else if (Bunch.ChSequence > ExpectedSequence)
        {
            // 중간 Bunch가 빠짐 → 큐에 보관하고 대기
            PendingBunches.InsertSorted(Bunch);
        }
        // else: 이미 처리한 시퀀스 → 무시
    }
    else
    {
        // Unreliable → 즉시 처리, 순서 보장 없음
        ProcessBunch(Bunch);
    }
}

// 송신 측: ACK 타임아웃 시 재전송
void UChannel::TickReliableResend(float DeltaTime)
{
    for (FOutBunch* Bunch : OutRec)
    {
        Bunch->TimeSinceSend += DeltaTime;

        if (Bunch->TimeSinceSend > ResendTimeout)
        {
            // 재전송
            Connection->WriteBitsToSendBuffer(Bunch);
            Bunch->TimeSinceSend = 0.0f;
            Bunch->ResendCount++;

            if (Bunch->ResendCount > MaxResendCount)
            {
                // 너무 많은 재전송 → 연결 문제로 판단
                Connection->Close();
            }
        }
    }
}

Part 5: 패킷 송신 — FlushNet과 대역폭 제어

SendBunch()를 호출하면 바로 전송되나요?

아닙니다. Bunch는 Connection의 SendBuffer에 쌓이기만 합니다. 실제 전송은 틱 끝에 UNetConnection::FlushNet()이 호출될 때 일괄로 이루어집니다.

C++ — 서버 틱에서의 네트워크 처리 순서
// UWorld::Tick() 내부에서의 네트워크 관련 호출 순서

void UNetDriver::TickFlush(float DeltaTime)
{
    // 1) ServerReplicateActors — Property 변경 감지 & Bunch 생성
    if (IsServer())
    {
        ServerReplicateActors(DeltaTime);
        // 내부에서:
        //   - 각 Connection에 대해 Relevant Actor 목록 결정
        //   - 우선순위(NetPriority) 기반 정렬
        //   - 변경된 Property를 Bunch로 직렬화
        //   - ActorChannel::SendBunch() 호출
    }

    // 2) 각 Connection의 SendBuffer를 실제 전송
    for (UNetConnection* Connection : ClientConnections)
    {
        // 대역폭 제한 확인
        if (Connection->IsNetReady(/* bAutoFlush */ false))
        {
            Connection->FlushNet();
            // → SendBuffer의 내용을 UDP 패킷으로 묶어서 sendto() 호출
        }
    }
}
대역폭 제어는 어떻게 이루어지나요?

Connection마다 대역폭 버짓(QueuedBits / NetSpeed)이 있습니다. 한 틱에 보낼 수 있는 데이터양이 제한되며, 초과하면 다음 틱으로 밀립니다.

C++ — 대역폭 쓰로틀링
bool UNetConnection::IsNetReady(bool bAutoFlush)
{
    // QueuedBits: 아직 보내지 못한 비트 수
    // 매 틱마다 NetSpeed만큼 감소 (보냈으니까)
    QueuedBits -= FMath::Max(DeltaTime * CurrentNetSpeed, 0.0f);

    if (QueuedBits > 0)
    {
        // 아직 이전 데이터도 다 못 보냄 → 대기
        return false;
    }

    return true;
}

void UNetConnection::FlushNet(bool bIgnoreSimulation)
{
    if (SendBuffer.GetNumBits() > 0)
    {
        // 패킷 헤더 붙이기 (시퀀스, ACK 정보)
        WritePacketHeader(SendBuffer);

        // 실제 소켓 전송
        LowLevelSend(SendBuffer.GetData(), SendBuffer.GetNumBits());

        // 보낸 만큼 QueuedBits에 더함
        QueuedBits += SendBuffer.GetNumBits();

        // 버퍼 리셋
        SendBuffer.Reset();
    }
}

Part 6: Property 복제의 내부 — 변경 감지와 직렬화

서버가 Property 변경을 어떻게 감지하나요? 매번 전체를 보내나요?

아닙니다. 언리얼은 FRepLayout이라는 구조로 각 Actor의 복제 대상 Property 목록과 Shadow State(이전 전송값의 복사본)를 관리합니다. 매 틱마다 현재값과 Shadow를 비교해서 변경된 Property만 전송합니다.

C++ — Property 변경 감지 및 델타 직렬화
// ServerReplicateActors() 내부에서 각 Actor에 대해:

void FObjectReplicator::ReplicateProperties(FOutBunch& Bunch)
{
    FRepLayout& RepLayout = *RepLayoutPtr;

    // 1) Shadow State와 현재 값 비교
    TArray<uint16> ChangedProperties;
    RepLayout.CompareProperties(
        /* ShadowData  */ ShadowBuffer.GetData(),
        /* CurrentData */ (uint8*)Object,
        ChangedProperties   // 변경된 Property 인덱스 목록
    );

    if (ChangedProperties.Num() == 0)
        return;  // 변경 없음 → 아무것도 안 보냄

    // 2) 변경된 Property만 직렬화
    for (uint16 PropIndex : ChangedProperties)
    {
        const FRepLayoutCmd& Cmd = RepLayout.Cmds[PropIndex];

        // Property 식별자 기록
        Bunch.WriteInt(PropIndex, RepLayout.Cmds.Num());

        // Property 값 직렬화 — 타입별 처리
        switch (Cmd.Type)
        {
            case ERepLayoutCmdType::PropertyFloat:
                Bunch.WriteFloat(*(float*)(CurrentPtr + Cmd.Offset));
                break;
            case ERepLayoutCmdType::PropertyVector:
                WritePackedVector(Bunch, *(FVector*)(CurrentPtr + Cmd.Offset));
                break;
            case ERepLayoutCmdType::PropertyObject:
                // UObject*는 NetworkGUID로 변환하여 전송
                FNetworkGUID NetGUID = GuidCache->GetOrAssignNetGUID(ObjectPtr);
                Bunch.Write(NetGUID);
                break;
            // ... 각 타입별 직렬화
        }
    }

    // 3) Shadow State 업데이트 — 다음 비교를 위해
    RepLayout.CopyShadowState(ShadowBuffer.GetData(), (uint8*)Object);
}

// CompareProperties 내부 — 메모리 직접 비교
void FRepLayout::CompareProperties(
    const uint8* Shadow, const uint8* Current,
    TArray<uint16>& OutChanged) const
{
    for (int32 i = 0; i < Cmds.Num(); i++)
    {
        const FRepLayoutCmd& Cmd = Cmds[i];
        const uint8* ShadowPtr  = Shadow  + Cmd.ShadowOffset;
        const uint8* CurrentPtr = Current + Cmd.Offset;

        // 메모리 비교 — 값이 다르면 변경된 것
        if (!FMemory::Memcmp(ShadowPtr, CurrentPtr, Cmd.ElementSize) != 0)
        {
            OutChanged.Add(i);
        }
    }
}
🎯 비유로 이해하기

Shadow State는 "지난번에 보낸 사진"입니다. 매번 현재 사진과 비교해서 달라진 부분만 빨간 동그라미를 치고, 그 부분만 보내는 겁니다. 전체 사진을 매번 보내면 낭비니까요.

Part 7: 보이스 데이터 전송 — 음성이 흘러가는 경로

게임에서 보이스 채팅을 하면 음성 데이터는 어떤 경로로 전달되나요?

보이스 데이터는 일반 RPC나 Property 복제와 완전히 다른 전용 파이프라인을 탑니다. 핵심은 UVoiceChannel이라는 별도 채널을 사용한다는 것입니다.

마이크 입력
IVoiceCapture PCM Raw Audio
인코딩
IVoiceEncoder (Opus) 압축: ~6KB/s
네트워크 전송
UVoiceChannel Unreliable Bunch
디코딩 & 재생
IVoiceDecoder (Opus) USoundWaveProcedural

보이스: 캡처 → Opus 인코딩 → VoiceChannel → 디코딩 → 재생

C++ — 보이스 캡처 및 전송 흐름
// ===== 송신 측: 보이스 캡처 & 인코딩 =====

void UOnlineVoiceImpl::ProcessLocalVoicePackets()
{
    IVoiceCapture* VoiceCapture = GetVoiceCapture();
    IVoiceEncoder* VoiceEncoder = GetVoiceEncoder();

    // 1) 마이크에서 PCM 데이터 읽기
    uint32 BytesAvailable = 0;
    EVoiceCaptureState::Type CaptureState =
        VoiceCapture->GetCaptureState(BytesAvailable);

    if (CaptureState == EVoiceCaptureState::Ok && BytesAvailable > 0)
    {
        TArray<uint8> RawVoiceData;
        RawVoiceData.SetNumUninitialized(BytesAvailable);
        VoiceCapture->GetVoiceData(RawVoiceData.GetData(), BytesAvailable);

        // 2) Opus 코덱으로 압축
        //    48kHz 16bit PCM → ~6KB/s compressed
        TArray<uint8> CompressedData;
        uint32 CompressedSize = 0;
        VoiceEncoder->Encode(
            RawVoiceData.GetData(), BytesAvailable,
            CompressedData.GetData(), CompressedSize);

        // 3) 로컬 패킷 큐에 추가
        FVoicePacket Packet;
        Packet.Data = CompressedData;
        Packet.Length = CompressedSize;
        Packet.Sender = LocalPlayerUniqueId;
        LocalVoicePackets.Add(Packet);
    }
}

// ===== 서버: VoiceChannel을 통해 릴레이 =====

void UVoiceChannel::Tick()
{
    // 서버가 수신한 보이스 패킷을 다른 클라이언트에게 전달
    for (FVoicePacket& Packet : PendingVoicePackets)
    {
        FOutBunch Bunch(this, /* bReliable */ false);
        // ▲ Unreliable! 음성은 늦게 도착하면 버리는 게 나음

        // 송신자 ID
        Bunch.Write(Packet.Sender);
        // 압축된 음성 데이터
        Bunch.Write(Packet.Data.GetData(), Packet.Length);

        SendBunch(&Bunch, /* bMerge */ false);
    }
}

// ===== 수신 측: 디코딩 & 오디오 재생 =====

void UVoiceChannel::ReceivedBunch(FInBunch& Bunch)
{
    // 1) 송신자 ID 읽기
    FUniqueNetIdRepl SenderId;
    Bunch.Read(SenderId);

    // 2) 압축 데이터 읽기
    TArray<uint8> CompressedData;
    Bunch.Read(CompressedData);

    // 3) Opus 디코딩 → PCM
    IVoiceDecoder* Decoder = GetVoiceDecoder(SenderId);
    TArray<uint8> DecompressedData;
    Decoder->Decode(CompressedData.GetData(), CompressedData.Num(),
                    DecompressedData.GetData(), DecompressedSize);

    // 4) USoundWaveProcedural에 PCM 데이터 주입 → 스피커 재생
    USoundWaveProcedural* VoiceSound = GetOrCreateVoiceSound(SenderId);
    VoiceSound->QueueAudio(DecompressedData.GetData(), DecompressedSize);
}
왜 보이스는 Unreliable로 보내나요?

실시간 음성에서 지연은 품질 저하보다 나쁩니다. 0.5초 전 음성을 재전송받아 재생하면 대화 흐름이 깨집니다. 패킷이 유실되면 그냥 무시하고 다음 패킷을 재생하는 것이 훨씬 자연스럽습니다.

Opus 코덱은 패킷 손실에 대한 PLC(Packet Loss Concealment) 기능이 내장되어 있어, 빠진 구간을 추정하여 매끄럽게 이어줍니다.

Part 8: 대용량 데이터 전송 — 이미지와 파일이 전달되는 방법

만약 게임 중에 이미지(텍스처)나 큰 바이너리 데이터를 보내야 한다면 어떻게 하나요?

UDP 패킷 하나의 최대 크기는 보통 1200바이트 내외입니다. 이미지는 수십 KB~수 MB이므로 하나의 패킷에 담을 수 없습니다. 언리얼은 두 가지 메커니즘을 제공합니다:

1) Partial Bunch — 큰 Bunch를 여러 조각으로 분할
2) HTTP 기반 다운로드 — 정말 큰 에셋은 별도 HTTP 서버에서

C++ — Partial Bunch: 큰 데이터를 쪼개서 전송
// UChannel::SendBunch() 내부 — Bunch가 너무 크면 분할

void UChannel::SendBunch(FOutBunch* Bunch, bool bMerge)
{
    const int32 MAX_BUNCH_SIZE = 1024 * 64;  // 64KB (Reliable 최대)

    if (Bunch->GetNumBits() > MAX_SINGLE_BUNCH_SIZE)
    {
        // ▼ Partial Bunch로 분할
        TArray<FOutBunch*> Parts;

        int32 Offset = 0;
        while (Offset < Bunch->GetNumBits())
        {
            FOutBunch* Part = new FOutBunch(this, Bunch->bReliable);
            Part->bPartial = true;
            Part->bPartialInitial = (Offset == 0);         // 첫 조각
            Part->bPartialFinal  = (Remaining <= ChunkSize); // 마지막 조각

            // 데이터 복사
            int32 ChunkBits = FMath::Min(MAX_SINGLE_BUNCH_SIZE,
                                         Bunch->GetNumBits() - Offset);
            Part->SerializeBits(Bunch->GetData() + Offset, ChunkBits);
            Offset += ChunkBits;

            Parts.Add(Part);
        }

        // 각 조각을 순서대로 전송
        for (FOutBunch* Part : Parts)
        {
            Connection->WriteBitsToSendBuffer(Part);
        }
    }
    else
    {
        // 일반 전송
        Connection->WriteBitsToSendBuffer(Bunch);
    }
}

// ===== 수신 측: Partial Bunch 재조합 =====

void UChannel::ReceivedBunch(FInBunch& Bunch)
{
    if (Bunch.bPartial)
    {
        if (Bunch.bPartialInitial)
        {
            // 새 부분 번치 시작
            PartialBunch = new FInBunch(Bunch);
        }
        else if (PartialBunch)
        {
            // 기존에 모으던 데이터에 추가
            PartialBunch->AppendDataFromChecked(Bunch);

            if (Bunch.bPartialFinal)
            {
                // ▼ 모든 조각이 모임 → 완성된 Bunch로 처리
                ProcessBunch(*PartialBunch);
                delete PartialBunch;
                PartialBunch = nullptr;
            }
        }
    }
    else
    {
        ProcessBunch(Bunch);
    }
}

UFUNCTION(Server, Reliable)

                           int32 TotalChunks,
                           const TArray<uint8>& ChunkData);

void AMyPlayerController::SendLargeImage(const TArray<uint8>& FullImageData)
{
    const int32 CHUNK_SIZE = 16 * 1024;  // 16KB per chunk
    int32 TotalChunks = FMath::CeilToInt(
        (float)FullImageData.Num() / CHUNK_SIZE);
    int32 ImageId = FMath::Rand();

    for (int32 i = 0; i < TotalChunks; i++)
    {
        int32 Offset = i * CHUNK_SIZE;
        int32 Size = FMath::Min(CHUNK_SIZE,
                                FullImageData.Num() - Offset);

        TArray<uint8> Chunk;
        Chunk.Append(FullImageData.GetData() + Offset, Size);

        Server_SendImageChunk(ImageId, i, TotalChunks, Chunk);
    }
}

// 수신 측에서 모든 청크가 도착하면 재조립
void AMyPlayerController::Client_ReceiveImageChunk_Implementation(
    int32 ImageId, int32 ChunkIndex,
    int32 TotalChunks, const TArray<uint8>& ChunkData)
{
    FPendingImage& Pending = PendingImages.FindOrAdd(ImageId);
    Pending.TotalChunks = TotalChunks;
    Pending.ReceivedChunks.Add(ChunkIndex, ChunkData);

    if (Pending.ReceivedChunks.Num() == TotalChunks)
    {
        // 모든 청크 도착! 이미지 재조립
        TArray<uint8> FullData;
        for (int32 i = 0; i < TotalChunks; i++)
        {
            FullData.Append(Pending.ReceivedChunks[i]);
        }

        // 텍스처 생성 및 사용
        UTexture2D* Texture = CreateTextureFromCompressedData(FullData);
        ApplyEmblemTexture(Texture);

        PendingImages.Remove(ImageId);
    }
}

Part 10: 복제 스케줄링 — 누구를 먼저 보낼 것인가

서버에 Actor가 수백 개 있을 때, 매 틱마다 전부 복제하나요?

아닙니다. 대역폭이 제한되어 있으므로 우선순위 기반 스케줄링으로 "이번 틱에 어떤 Actor를 복제할지" 결정합니다. 이것이 ServerReplicateActors()의 핵심 로직입니다.

C++ — 복제 우선순위 결정
void UNetDriver::ServerReplicateActors(float DeltaTime)
{
    for (UNetConnection* Connection : ClientConnections)
    {
        // 1) 이 클라이언트에게 보여야 할 Actor 목록 수집
        TArray<FActorPriority> PriorityList;

        for (AActor* Actor : ReplicatedActors)
        {
            // Relevancy 체크 — 이 클라이언트에게 관련 있는 Actor만
            if (!Actor->IsNetRelevantFor(Connection->ViewTarget))
                continue;

            // 우선순위 계산
            float Priority = Actor->GetNetPriority(
                Connection->ViewTarget->GetActorLocation(),
                Connection->ViewTarget->GetViewRotation(),
                Connection);

            // 마지막 복제 이후 경과 시간 반영
            float TimeSinceLastRep = World->TimeSeconds -
                                     Actor->LastReplicationTime;
            Priority *= TimeSinceLastRep;

            PriorityList.Add({Actor, Priority});
        }

        // 2) 우선순위로 정렬 — 높은 것부터
        PriorityList.Sort([](const FActorPriority& A,
                             const FActorPriority& B)
        {
            return A.Priority > B.Priority;
        });

        // 3) 대역폭이 허용하는 만큼 복제
        int32 BitsWritten = 0;
        int32 MaxBits = Connection->GetMaxBitsPerTick();

        for (const FActorPriority& Entry : PriorityList)
        {
            if (BitsWritten >= MaxBits)
                break;  // 이번 틱 대역폭 소진

            UActorChannel* Channel = Connection->FindOrCreateActorChannel(
                Entry.Actor);

            int32 Written = Channel->ReplicateActor();
            BitsWritten += Written;
        }
    }
}

// Actor별 기본 우선순위 설정
float AActor::GetNetPriority(...)
{
    float Priority = NetPriority;  // 기본값: 1.0

    // 플레이어가 보고 있는 방향의 Actor에 가중치
    if (IsInViewCone(ViewLocation, ViewRotation))
        Priority *= 2.0f;

    // 가까운 Actor에 가중치
    float DistSq = FVector::DistSquared(GetActorLocation(), ViewLocation);
    if (DistSq < CloseDistanceSq)
        Priority *= 3.0f;

    return Priority;
}
🎯 비유로 이해하기

제한된 택배 트럭(대역폭)에 모든 소포(Actor)를 실을 수 없을 때, "긴급(우선순위 높음)" 소포를 먼저 싣고, 나머지는 다음 트럭(다음 틱)에 싣는 것입니다. 오래 기다린 소포(마지막 복제 이후 경과 시간)는 우선순위가 올라갑니다.

Part 11: 전체 흐름 통합 — "총 쏘기" 하나에 일어나는 모든 것

플레이어가 총을 쏘는 순간, 코드 레벨에서 패킷이 나가고 들어오는 전체 흐름을 처음부터 끝까지 보여줄 수 있나요?

좋습니다. 클라이언트 A가 마우스 클릭 → 서버 검증 → 모든 클라이언트에 이펙트 표시까지의 전체 여정을 추적해봅시다.

① Client A: 입력
마우스 클릭 Server_Fire() 호출
② Client A → Server: RPC 전송
ProcessEvent() Bunch 직렬화 SendBuffer FlushNet() UDP sendto()
③ Server: 패킷 수신
TickDispatch() recvfrom() ReceivedPacket() Bunch 분리
④ Server: RPC 실행
ActorChannel ReceivedBunch() ReceivedRPC() Server_Fire_Implementation()
⑤ Server: 결과 전파
Ammo-- (Property 변경) Multicast_PlayEffect()
⑥ Server → All Clients: 송신
ServerReplicateActors() Ammo delta + Multicast Bunch FlushNet()
⑦ All Clients: 수신 & 적용
OnRep_Ammo() → UI 갱신 Multicast_PlayEffect_Impl() → 총구 화염

"총 쏘기" 하나에 일어나는 전체 네트워크 흐름

C++ — 통합 예시 전체 코드
// ===== MyCharacter.h =====

UCLASS()
class AMyCharacter : public ACharacter
{
    GENERATED_BODY()

public:
    UPROPERTY(ReplicatedUsing = OnRep_Ammo)
    int32 Ammo = 30;

    UFUNCTION()
    void OnRep_Ammo();

    // ① 로컬 입력 처리
    void HandleFireInput();

    // ② Client → Server RPC
    UFUNCTION(Server, Reliable, WithValidation)
    void Server_Fire(FVector_NetQuantize MuzzleLocation,
                     FVector_NetQuantize AimDirection);

    // ⑤ Server → All Clients RPC
    UFUNCTION(NetMulticast, Unreliable)
    void Multicast_PlayFireEffect(FVector_NetQuantize MuzzleLocation);

    virtual void GetLifetimeReplicatedProps(
        TArray<FLifetimeProperty>& OutLifetimeProps) const override;
};

// ===== MyCharacter.cpp =====

void AMyCharacter::GetLifetimeReplicatedProps(
    TArray<FLifetimeProperty>& OutLifetimeProps) const
{
    Super::GetLifetimeReplicatedProps(OutLifetimeProps);
    DOREPLIFETIME_CONDITION(AMyCharacter, Ammo, COND_OwnerOnly);
}

// ① 클라이언트: 마우스 클릭 시 호출
void AMyCharacter::HandleFireInput()
{
    if (!HasAuthority())
    {
        // 클라이언트 → Server RPC 호출
        // 내부적으로:
        //   ProcessEvent() → CallRemoteFunction()
        //   → Bunch 생성 → SendBuffer에 추가
        //   → 틱 끝에 FlushNet()으로 UDP 전송
        Server_Fire(GetMuzzleLocation(), GetAimDirection());

        // 로컬 예측: 즉시 이펙트 재생 (반응성)
        PlayLocalFireEffect();
    }
}

// ② 유효성 검사 (서버에서 자동 호출)
bool AMyCharacter::Server_Fire_Validate(
    FVector_NetQuantize MuzzleLocation,
    FVector_NetQuantize AimDirection)
{
    // 치트 방지: 비정상적 방향이나 위치 검증
    if (AimDirection.IsNearlyZero())
        return false;  // → false 반환 시 클라이언트 강제 접속 해제

    return true;
}

// ④ 서버에서 실행되는 실제 로직
void AMyCharacter::Server_Fire_Implementation(
    FVector_NetQuantize MuzzleLocation,
    FVector_NetQuantize AimDirection)
{
    if (Ammo <= 0) return;

    // 서버에서 히트 판정
    FHitResult Hit;
    FVector TraceEnd = MuzzleLocation + AimDirection * 10000.f;
    GetWorld()->LineTraceSingleByChannel(
        Hit, MuzzleLocation, TraceEnd, ECC_Visibility);

    if (Hit.bBlockingHit)
    {
        if (AMyCharacter* Target = Cast<AMyCharacter>(Hit.GetActor()))
        {
            // 데미지 처리 (서버 권한)
            Target->TakeDamage(25.f, ...);
        }
    }

    // ⑤ Property 변경 → 다음 복제 틱에 자동 전송
    Ammo--;
    // → Shadow State와 비교 → 변경 감지 → 델타 직렬화
    // → Owner 클라이언트에게만 전송 (COND_OwnerOnly)

    // ⑤ Multicast RPC → 모든 클라이언트에 이펙트
    Multicast_PlayFireEffect(MuzzleLocation);
    // → 모든 Connection의 ActorChannel에 Unreliable Bunch 추가
}

// ⑦ 모든 클라이언트에서 실행
void AMyCharacter::Multicast_PlayFireEffect_Implementation(
    FVector_NetQuantize MuzzleLocation)
{
    // 자기 자신은 이미 로컬 예측으로 재생했으므로 스킵
    if (IsLocallyControlled()) return;

    // 총구 화염 파티클
    UGameplayStatics::SpawnEmitterAtLocation(
        GetWorld(), FireEffectParticle, MuzzleLocation);

    // 총소리
    UGameplayStatics::PlaySoundAtLocation(
        GetWorld(), FireSound, MuzzleLocation);
}

// ⑦ Owner 클라이언트: Ammo 복제 수신 시
void AMyCharacter::OnRep_Ammo()
{
    // HUD 탄약 표시 업데이트
    if (AMyHUD* HUD = GetHUD())
    {
        HUD->UpdateAmmoDisplay(Ammo);
    }
}

Part 12: 네트워크 디버깅 — 패킷을 눈으로 보는 방법

네트워크 문제가 발생했을 때 어떻게 디버깅하나요?

언리얼은 강력한 네트워크 디버깅 도구들을 제공합니다.

콘솔 명령어 — 네트워크 디버깅
// 네트워크 통계 표시
stat net              // 대역폭, 패킷 수, 채널 수 등
stat netactor         // Actor별 복제 비용

// 패킷 로스/지연 시뮬레이션 (테스트용)
net PktLoss=5         // 5% 패킷 로스
net PktLag=100        // 100ms 지연 추가
net PktLagVariance=20 // ±20ms 지터
net PktDup=2          // 2% 패킷 중복
net PktOrder=1        // 패킷 순서 뒤섞기 활성화

// 네트워크 로그 상세 출력
log LogNet Verbose
log LogNetTraffic Verbose    // 모든 Bunch 송수신 로그
log LogRep Verbose           // Property 복제 로그
log LogNetPlayerMovement Verbose  // 이동 동기화 로그

// Network Profiler (패킷 캡처 파일 생성)
net start              // 네트워크 프로파일링 시작
net stop               // 종료 → NetworkProfiling.nprof 생성
// → UnrealFrontend의 Network Profiler로 분석
도구 용도 확인 가능 정보
stat net 실시간 대역폭 모니터링 In/Out Bytes, Packets, Bunches 수
Network Profiler 패킷 내용 분석 Actor별, Property별 트래픽 비중
net PktLoss/Lag 열악한 네트워크 시뮬레이션 패킷 로스/지연 시 게임 동작 검증
LogNetTraffic Bunch 레벨 추적 어떤 채널에 무슨 데이터가 오갔는지

📌 핵심 정리

  • 네트워크 스택 4계층: NetDriver(소켓) → NetConnection(패킷/ACK) → Channel(Bunch) → Actor/RPC(게임 로직)
  • 패킷과 번치: 패킷은 물리적 UDP 데이터그램, Bunch는 채널 단위의 논리적 데이터 블록. 여러 Bunch가 하나의 패킷에 담김
  • RPC 바인딩: UHT가 Thunk/Parms 구조체를 자동 생성 → ProcessEvent()가 RPC 플래그 감지 → CallRemoteFunction()으로 Bunch 직렬화 → SendBunch()로 큐잉
  • Property 복제: FRepLayout이 Shadow State와 현재값을 메모리 비교 → 변경분만 델타 직렬화하여 전송
  • 보이스: 전용 VoiceChannel + Opus 코덱 + Unreliable 전송. 실시간성 우선, 손실 허용
  • 대용량 데이터: Partial Bunch(자동 분할), 청크 RPC(수동 분할), HTTP 다운로드(최적) 세 가지 전략
  • 복제 스케줄링: NetPriority × 경과시간으로 우선순위 산정 → 대역폭 한도 내에서 높은 순서부터 복제
