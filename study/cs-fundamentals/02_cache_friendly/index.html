"같은 코드인데 왜 100배 느리죠?"

CPU 캐시를 이해하면 성능이 보인다

Part 1: 메모리 계층 구조

CPU는 왜 직접 RAM에서 데이터를 읽지 않나요?

RAM은 CPU에 비해 너무 느립니다. 그 격차를 캐시가 메웁니다.

│                    메모리 계층 구조와 지연 시간                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   레벨        용량         지연시간      CPU 사이클     비유         │
│   ─────────────────────────────────────────────────────────────    │
│                                                                     │
│   Register    ~1KB         0.25ns        1 cycle      뇌            │
│      ▲                                                              │
│      │        ×2~4                                                  │
│   L1 Cache    64KB         1ns           4 cycles     메모장       │
│      ▲                                                              │
│      │        ×3~4                                                  │
│   L2 Cache    256KB        4ns           12 cycles    책상 위 서류  │
│      ▲                                                              │
│      │        ×10~15                                                │
│   L3 Cache    8~32MB       40ns          120 cycles   책장          │
│      ▲                                                              │
│      │        ×100~200                                              │
│   RAM         16~64GB      100ns         300 cycles   창고          │
│      ▲                                                              │
│      │        ×100,000                                              │
│   SSD/HDD     1~4TB        10-100ms      수천만 cycles  다른 도시   │
│                                                                     │

│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

캐시 히트율 90%→95%로 올리면 성능이 2배 향상될 수 있습니다!

캐시는 어떤 단위로 데이터를 가져오나요?

캐시 라인(Cache Line) 단위입니다. 보통 64바이트입니다.

│                       캐시 라인 구조                                 │

│   │ 0x0FC0      │ 0x1000 ← 요청    │ 0x1040      │ 0x1080     │   │

│   └─────────────┴──────────────────┴─────────────┴────────────┘   │
│                        │                                            │

│   Cache Line:                                                       │
│   ┌────────────────────────────────────────────────────────────┐   │

Part 2: 지역성 (Locality) 원리

캐시가 효과적인 이유는 무엇인가요?

프로그램은 지역성을 보이기 때문입니다.

지역성 유형 설명 예시
시간적 지역성 최근 접근한 데이터를 곧 다시 접근 루프 카운터, 함수 변수
공간적 지역성 인접한 메모리를 연속 접근 배열 순회, 구조체 필드
실제로 얼마나 차이가 나나요?
배열 순회 방향에 따른 성능 차이
// 2D 배열 (Row-major 저장: C/C++ 기본)
constexpr int ROWS = 10000;
constexpr int COLS = 10000;
int matrix[ROWS][COLS];

// ✅ 캐시 친화적 순회 (행 우선 - Row-major)
// 메모리: [0][0], [0][1], [0][2], ... , [1][0], [1][1], ...
void rowMajorSum() {
    int sum = 0;
    for (int i = 0; i < ROWS; ++i) {       // 행 먼저
        for (int j = 0; j < COLS; ++j) {   // 열 나중
            sum += matrix[i][j];           // 연속 메모리 접근!
        }
    }
    // 시간: ~50ms
}

// ❌ 캐시 비친화적 순회 (열 우선)
void colMajorSum() {
    int sum = 0;
    for (int j = 0; j < COLS; ++j) {       // 열 먼저
        for (int i = 0; i < ROWS; ++i) {   // 행 나중
            sum += matrix[i][j];           // 매번 10000*4=40KB 점프!
        }
    }
    // 시간: ~500ms (10배 느림!)
}

// 메모리 접근 패턴 비교:
// Row-major: [0][0] → [0][1] → [0][2]  (4바이트 간격)
// Col-major: [0][0] → [1][0] → [2][0]  (40,000바이트 간격!)

같은 연산인데 순서만 바꿔도 10배 차이!

Part 3: 데이터 지향 설계 (DOD)

게임에서 캐시 효율을 높이는 설계 방법은?

Data-Oriented Design (DOD): 데이터 배치를 최적화합니다.

전통적인 OOP와 반대로 생각합니다.

┌─────────────────────────────────────────────────────────────────────┐
│                    AoS vs SoA 비교                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  AoS (Array of Structures) - 전통적 OOP                             │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │ Entity0                │ Entity1                │ Entity2     │ │
│  │ [pos][vel][hp][name]   │ [pos][vel][hp][name]   │ [pos]...   │ │
│  │ ← 64바이트 →           │ ← 64바이트 →           │            │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  위치만 업데이트하면? → 캐시 라인마다 필요한 건 pos(12B)뿐!         │
│  나머지 52바이트는 낭비 (캐시 효율 18%)                             │
│                                                                     │
│  ─────────────────────────────────────────────────────────────────  │
│                                                                     │
│  SoA (Structure of Arrays) - DOD                                    │
│  ┌───────────────────────────────────────────────────────────────┐ │
│  │ positions[]    │ [pos0][pos1][pos2][pos3][pos4][pos5]...      │ │
│  │ velocities[]   │ [vel0][vel1][vel2][vel3][vel4][vel5]...      │ │
│  │ healths[]      │ [hp0][hp1][hp2][hp3][hp4][hp5][hp6]...       │ │
│  │ names[]        │ [name0][name1][name2]...                     │ │
│  └───────────────────────────────────────────────────────────────┘ │
│                                                                     │
│  위치만 업데이트하면? → positions 배열만 순차 접근                  │
│  캐시 라인 100% 활용!                                               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
AoS vs SoA 코드 비교
// ❌ AoS (Array of Structures) - 캐시 비효율
struct Entity {
    Vector3 position;    // 12 bytes
    Vector3 velocity;    // 12 bytes
    float health;        // 4 bytes
    char name[32];       // 32 bytes
    // 패딩 4 bytes
};  // 총 64 bytes

std::vector entities(10000);

void updatePositions_AoS() {
    for (auto& e : entities) {
        e.position += e.velocity * deltaTime;
        // 64바이트 구조체에서 24바이트만 사용
        // 캐시 효율: 37.5%
    }
}

// ✅ SoA (Structure of Arrays) - 캐시 친화적
struct Entities {
    std::vector positions;
    std::vector velocities;
    std::vector healths;
    std::vector names;
    size_t count;
};

Entities entities;

void updatePositions_SoA() {
    Vector3* pos = entities.positions.data();
    Vector3* vel = entities.velocities.data();

    for (size_t i = 0; i < entities.count; ++i) {
        pos[i] += vel[i] * deltaTime;
        // 연속된 Vector3만 접근
        // 캐시 효율: 100%
    }
}

// 벤치마크 결과 (10만 엔티티):
// AoS: 3.2ms
// SoA: 0.8ms (4배 빠름!)
항상 SoA가 좋은 건가요?

아닙니다. 접근 패턴에 따라 선택해야 합니다.

상황 추천 이유
전체 순회, 일부 필드만 사용 SoA 필요한 필드만 캐시에 로드
개별 객체의 모든 필드 접근 AoS 한 번에 전체 데이터 로드
삽입/삭제가 빈번 AoS SoA는 여러 배열 동기화 필요
SIMD 최적화 대상 SoA 벡터 연산에 최적

Part 4: 캐시 미스 유형과 대응

캐시 미스는 왜 발생하나요?
미스 유형 원인 대응 방법
Cold Miss (강제적) 첫 접근 프리페치, 웜업
Capacity Miss 캐시 용량 초과 데이터 분할, 크기 축소
Conflict Miss 같은 캐시 라인에 매핑 데이터 정렬, 패딩
프리페치(Prefetch)는 어떻게 사용하나요?
소프트웨어 프리페치
#include   // SSE intrinsics

// 미리 다음 데이터를 캐시에 로드
void processWithPrefetch(float* data, int count) {
    constexpr int PREFETCH_DISTANCE = 16;  // 캐시 라인 수

    for (int i = 0; i < count; ++i) {
        // 앞으로 접근할 데이터를 미리 로드 요청
        if (i + PREFETCH_DISTANCE < count) {
            _mm_prefetch(
                reinterpret_cast(&data[i + PREFETCH_DISTANCE]),
                _MM_HINT_T0  // L1 캐시에 프리페치
            );
        }

        // 실제 처리 (이 시점에 데이터는 이미 캐시에 있음)
        processElement(data[i]);
    }
}

// 프리페치 힌트 종류:
// _MM_HINT_T0: L1 캐시 (가장 빠름)
// _MM_HINT_T1: L2 캐시
// _MM_HINT_T2: L3 캐시
// _MM_HINT_NTA: Non-temporal (재사용 안 할 데이터)

과도한 프리페치는 오히려 역효과! 유용한 데이터를 캐시에서 밀어낼 수 있습니다.

Part 5: 핫/콜드 데이터 분리

자주 쓰는 데이터와 가끔 쓰는 데이터를 분리해야 하나요?

네! 핫/콜드 분리로 캐시 효율을 높입니다.

핫/콜드 데이터 분리
// ❌ 모든 데이터가 한 구조체에
struct GameObjectBad {
    // Hot data (매 프레임 사용)
    Vector3 position;
    Vector3 velocity;
    Quaternion rotation;
    float health;

    // Cold data (가끔 사용)
    std::string name;
    std::string description;
    int64_t creationTime;
    int64_t lastModified;
    std::vector tags;
    // ... 100바이트 이상
};
// 크기: ~200바이트, 매 프레임 접근 시 캐시 낭비

// ✅ 핫/콜드 분리
struct GameObjectHot {
    Vector3 position;
    Vector3 velocity;
    Quaternion rotation;
    float health;
    uint32_t coldDataIndex;  // 콜드 데이터 인덱스
};  // 크기: 52바이트 → 캐시 라인 1개

struct GameObjectCold {
    std::string name;
    std::string description;
    int64_t creationTime;
    int64_t lastModified;
    std::vector tags;
};

// 별도 배열로 관리
std::vector hotObjects;   // 매 프레임 순회
std::vector coldObjects; // 필요할 때만 접근

void updateAllObjects() {
    // Hot 데이터만 순회 - 캐시 효율 최대화
    for (auto& obj : hotObjects) {
        obj.position += obj.velocity * deltaTime;
    }
}

GameObjectCold& getColdData(const GameObjectHot& hot) {
    return coldObjects[hot.coldDataIndex];
}
실제 게임 엔진에서는 어떻게 적용되나요?

Unity의 ECS(DOTS), Unreal의 Mass 시스템 등이 이 원리를 활용합니다.

ECS 스타일 컴포넌트 시스템
// 컴포넌트는 데이터만
struct TransformComponent {
    Vector3 position;
    Quaternion rotation;
    Vector3 scale;
};

struct VelocityComponent {
    Vector3 linear;
    Vector3 angular;
};

struct HealthComponent {
    float current;
    float max;
};

// 시스템은 특정 컴포넌트 조합만 처리
class MovementSystem {
public:
    void update(
        std::span transforms,
        std::span velocities,
        float deltaTime
    ) {
        // 필요한 컴포넌트만 순회
        // 완벽한 캐시 활용
        for (size_t i = 0; i < transforms.size(); ++i) {
            transforms[i].position += velocities[i].linear * deltaTime;
        }
    }
};

class HealthSystem {
public:
    void update(std::span healths) {
        // Health 컴포넌트만 접근
        for (auto& h : healths) {
            if (h.current <= 0) {
                handleDeath(h);
            }
        }
    }
};

ECS는 캐시 친화적 설계의 정점입니다. 수만 개 엔티티도 빠르게 처리!

// RDPMC를 이용한 프로그래밍 방식 측정 (리눅스)
#include 
#include 

class CacheProfiler {
    int fd_;
    struct perf_event_attr pe_;

public:
    CacheProfiler() {
        memset(&pe_, 0, sizeof(pe_));
        pe_.type = PERF_TYPE_HW_CACHE;
        pe_.config = PERF_COUNT_HW_CACHE_L1D |
                    (PERF_COUNT_HW_CACHE_OP_READ << 8) |
                    (PERF_COUNT_HW_CACHE_RESULT_MISS << 16);
        pe_.disabled = 1;
        pe_.exclude_kernel = 1;

        fd_ = syscall(__NR_perf_event_open, &pe_, 0, -1, -1, 0);
    }

    void start() {
        ioctl(fd_, PERF_EVENT_IOC_RESET, 0);
        ioctl(fd_, PERF_EVENT_IOC_ENABLE, 0);
    }

    uint64_t stop() {
        ioctl(fd_, PERF_EVENT_IOC_DISABLE, 0);
        uint64_t count;
        read(fd_, &count, sizeof(count));
        return count;
    }
};

// 사용 예시
CacheProfiler profiler;
profiler.start();

// 측정할 코드
processData();

uint64_t misses = profiler.stop();
printf("L1 cache misses: %lu\n", misses);

📌 핵심 정리

  • 캐시 라인: 64바이트 단위로 데이터 이동, 인접 데이터는 "공짜"
  • 지역성: 연속 메모리 접근(공간적), 반복 접근(시간적)이 성능 핵심
  • DOD: SoA 구조로 필요한 데이터만 캐시에 로드
  • 핫/콜드 분리: 자주 쓰는 데이터는 작고 밀집되게
  • 배열 순회: Row-major 순서로 순회 (C/C++ 기준)
