멀티코어의 숨은 적, False Sharing

Part 1: False Sharing이란?

False Sharing이 뭔가요?

서로 다른 데이터지만 같은 캐시 라인에 있어서 발생하는 성능 저하입니다.

┌─────────────────────────────────────────────────────────────────────┐
│                    False Sharing 발생 상황                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Cache Line (64 bytes)                                             │
│   ┌────────────────────────────────────────────────────────────┐   │
│   │ counter[0] │ counter[1] │ counter[2] │ counter[3] │ ...    │   │
│   │  (Core 0)  │  (Core 1)  │  (Core 2)  │  (Core 3)  │        │   │
│   └────────────────────────────────────────────────────────────┘   │
│         ↓            ↓            ↓            ↓                    │
│      Core 0       Core 1       Core 2       Core 3                  │
│                                                                     │
│   Core 0이 counter[0]을 수정하면:                                   │
│   → 전체 캐시 라인이 "Invalidate"됨                                 │
│   → Core 1, 2, 3의 캐시가 무효화됨                                  │
│   → Core 1이 counter[1]을 읽으려면 메모리에서 다시 로드!            │
│                                                                     │
│   실제로는 공유하지 않는 데이터인데                                  │
│   캐시 라인 단위로 동기화되어 성능 저하 발생!                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

True Sharing: 같은 데이터 접근
False Sharing: 다른 데이터지만 같은 캐시 라인

얼마나 심각한 문제인가요?
False Sharing 벤치마크
#include 
#include 
#include 

// ❌ False Sharing 발생
struct BadCounters {
    std::atomic counter[4];  // 16바이트, 같은 캐시 라인에!
};

// ✅ False Sharing 방지
struct GoodCounters {
    alignas(64) std::atomic counter0;
    alignas(64) std::atomic counter1;
    alignas(64) std::atomic counter2;
    alignas(64) std::atomic counter3;
};

void incrementBad(BadCounters& c, int idx) {
    for (int i = 0; i < 10000000; ++i) {
        c.counter[idx]++;
    }
}

void incrementGood(GoodCounters& c, int idx) {
    std::atomic* counters[] = {
        &c.counter0, &c.counter1, &c.counter2, &c.counter3
    };
    for (int i = 0; i < 10000000; ++i) {
        (*counters[idx])++;
    }
}

// 벤치마크 결과 (4코어):
// BadCounters:  약 800ms
// GoodCounters: 약 100ms
// → 8배 성능 차이!

코어가 많을수록 False Sharing의 영향이 커집니다!

Part 2: 캐시 일관성 프로토콜

왜 한 코어가 쓰면 다른 코어 캐시가 무효화되나요?

캐시 일관성(Cache Coherence)을 유지하기 위해서입니다.

┌─────────────────────────────────────────────────────────────────────┐
│                    MESI 프로토콜 상태                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Modified (M): 이 캐시만 가지고 있고, 수정됨                        │
│                 메모리와 다른 값                                     │
│                                                                     │
│   Exclusive (E): 이 캐시만 가지고 있고, 깨끗함                      │
│                  메모리와 같은 값                                    │
│                                                                     │
│   Shared (S): 여러 캐시가 가지고 있고, 깨끗함                       │
│               읽기만 가능                                           │
│                                                                     │
│   Invalid (I): 무효 (사용 불가)                                     │
│                                                                     │
│   상태 전이 예시 (False Sharing):                                   │
│                                                                     │
│   Core 0 (counter[0])         Core 1 (counter[1])                  │
│   ────────────────────        ────────────────────                  │
│   1. Read  → S                1. Read  → S                         │
│   2. Write → M                   (S → I) 무효화!                   │
│                               3. Read  → 메모리에서 다시 로드       │
│      (M → S)                     → S                               │
│   4. Write → M                   (S → I) 무효화!                   │
│   ...무한 반복...                                                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

False Sharing = 캐시 라인 핑퐁 (Cache Line Ping-Pong)

Part 3: 해결 방법

False Sharing을 어떻게 방지하나요?
방법 1: 패딩으로 분리
#define CACHE_LINE_SIZE 64

// 패딩 구조체
struct alignas(CACHE_LINE_SIZE) PaddedCounter {
    std::atomic value;
    char padding[CACHE_LINE_SIZE - sizeof(std::atomic)];
};

// 또는 C++17 hardware_destructive_interference_size
#include 
struct alignas(std::hardware_destructive_interference_size) Counter {
    std::atomic value;
};

// 배열 사용 시
struct ThreadData {
    PaddedCounter counters[MAX_THREADS];
};

// 각 스레드는 자신의 counters[threadId]만 접근
// 서로 다른 캐시 라인이므로 False Sharing 없음
방법 2: 로컬 변수 + 최종 병합
// ❌ 매번 공유 변수 접근
std::atomic globalSum{0};

void badWorker(const int* data, int start, int end) {
    for (int i = start; i < end; ++i) {
        globalSum += data[i];  // 매번 atomic 연산!
    }
}

// ✅ 로컬에서 계산 후 한 번만 병합
void goodWorker(const int* data, int start, int end) {
    long long localSum = 0;  // 스레드 로컬 변수

    for (int i = start; i < end; ++i) {
        localSum += data[i];  // 빠른 로컬 연산
    }

    globalSum += localSum;  // 마지막에 한 번만 atomic
}
방법 3: Thread-Local Storage
// 스레드별 전용 저장소
thread_local long long tls_counter = 0;

void worker(const int* data, int count) {
    for (int i = 0; i < count; ++i) {
        tls_counter += data[i];  // 완전히 독립된 메모리
    }
}

// 결과 수집
std::vector threadCounters;  // 각 스레드의 tls_counter 주소

long long getTotal() {
    long long total = 0;
    for (auto* counter : threadCounters) {
        total += *counter;
    }
    return total;
}

Job System 통계
// ❌ 나쁜 설계: 워커 스레드 통계가 인접
struct JobSystemBad {

    }

// 추가 최적화: 청크 시작을 캐시 라인에 정렬
int alignedChunkStart(int rawStart, int particleSize) {
    int particlesPerCacheLine = 64 / particleSize;
    return (rawStart / particlesPerCacheLine) * particlesPerCacheLine;
}

코드 레벨 탐지
// 주소가 같은 캐시 라인인지 확인
bool sameCacheLine(const void* a, const void* b) {
    constexpr size_t CACHE_LINE = 64;
    return (reinterpret_cast(a) / CACHE_LINE) ==
           (reinterpret_cast(b) / CACHE_LINE);
}

// 디버그용: 멀티스레드 접근 변수 검사
template
void checkFalseSharing(const char* name, const T& array, int count) {
    for (int i = 0; i < count - 1; ++i) {
        if (sameCacheLine(&array[i], &array[i+1])) {
            printf("Warning: %s[%d] and %s[%d] share cache line!\n",
                   name, i, name, i+1);
        }
    }
}

// 사용
checkFalseSharing("threadCounters", threadCounters, numThreads);

📌 핵심 정리

  • False Sharing: 다른 데이터지만 같은 캐시 라인 공유로 인한 성능 저하
  • 원인: 캐시 일관성 유지 위한 캐시 라인 단위 무효화
  • 해결책: 64바이트 정렬, 패딩, 로컬 변수 사용, Thread-Local Storage
  • 게임 적용: 청크 분할로 연속 메모리를 한 스레드가 독점 처리
  • 탐지: perf c2c, HITM 지표, 캐시 미스율 확인
