"Core마다 캐시가 달라도 괜찮나요?"

MESI 프로토콜과 캐시 일관성

Part 1: 캐시 일관성 문제

멀티코어에서 캐시 일관성이 왜 중요한가요?

각 코어가 별도 캐시를 가지면 같은 데이터의 다른 버전이 존재할 수 있습니다.

초기 상태
Core 0 Cache: -
Core 1 Cache: -
Memory[X] = 0
→
Step 1-2: 양쪽 읽기
Core 0 Cache: X=0
Core 1 Cache: X=0
Memory[X] = 0
→
Step 3: Core 0 writes X=1
Core 0 Cache: X=1
Core 1 Cache: X=0 ← 불일치!
Memory[X] = 0

문제: Core 1이 X를 읽으면 오래된 값(0)을 봄!

Part 2: MESI 프로토콜

MESI 프로토콜은 어떻게 동작하나요?

각 캐시 라인에 4가지 상태 중 하나를 부여합니다.

M (Modified)
이 캐시만 보유, 수정됨 (dirty)
메모리와 다른 값
다른 코어가 읽으려면 이 캐시가 제공
E (Exclusive)
이 캐시만 보유, 깨끗함
메모리와 같은 값
수정하면 M으로 (버스 트래픽 없음)
S (Shared)
여러 캐시 보유, 깨끗함
메모리와 같은 값
수정하려면 다른 캐시 무효화 필요
I (Invalid)
무효 (사용 불가)
사용하려면 다시 로드 필요
Invalid (I)
시작 상태
Exclusive (E)
← Read Miss (only me)
→ Write: Modified로
↔
Shared (S)
← Read Miss (others have it)
→ Write: Invalidate + Modified로
Modified (M)
← E에서 Write
← S에서 Write + 다른 캐시 무효화
→ 다른 코어 읽기: S로 + write-back
→ eviction: Memory로 write-back
구체적인 동작 예시를 보여주세요.
MESI 동작 예시
Core 0 Cache Core 1 Cache Memory
1. Core 0 reads X (miss)
Core 0
X=5 | E
Core 1
- | I
Memory
X=5

(Exclusive: 나만 가지고 있음)

↓
2. Core 1 reads X (miss, bus snoop)
Core 0
X=5 | S
Core 1
X=5 | S
Memory
X=5

(Shared: 둘 다 가지고 있음)

↓
3. Core 0 writes X=10
Core 0
X=10 | M
(Modified)
Core 1
- | I
(Invalidated!)
Memory
X=5
↓
4. Core 1 reads X (miss)
Core 0
X=10 | S
Core 1
X=10 | S

(Core 0이 먼저 메모리에 write-back)

Part 3: 성능 영향

MESI 프로토콜이 성능에 미치는 영향은?
상황 버스 트래픽 지연
E→M (단독 쓰기) 없음 매우 빠름
S→M (공유 후 쓰기) Invalidate 브로드캐스트 느림
I→S (공유 읽기) 읽기 요청 중간
M→S (다른 코어 읽기) Write-back + 전송 매우 느림
HITM (Hit Modified) 측정
# perf c2c: cache-to-cache 트래픽 분석
perf c2c record ./game
perf c2c report

# 핵심 지표:
# - HITM: 다른 코어의 M 상태 캐시에서 데이터 가져옴
# - Remote HITM: 다른 소켓(NUMA)의 M 상태에서 가져옴
#
# HITM이 높으면 = 공유 데이터에 대한 쓰기 경쟁 심함
# → False Sharing 또는 True Sharing 최적화 필요

HITM은 캐시 일관성 유지 비용의 핵심 지표입니다.

Part 4: 게임 프로그래밍 적용

MESI를 이해하면 어떻게 코드를 개선할 수 있나요?
캐시 상태를 고려한 설계
// 상황 1: 여러 스레드가 읽기만 하는 데이터
// → 모든 캐시가 S 상태로 유지, 성능 좋음
const GameConfig& getConfig() {
    static const GameConfig config = loadConfig();
    return config;  // 모든 스레드가 공유 읽기 (S 상태)
}

// 상황 2: 한 스레드만 쓰고, 여러 스레드가 읽는 데이터
// → 쓰기 스레드: M→S 전이 발생, 읽기 스레드: S 상태
// → 쓰기 후 읽기 스레드 캐시 무효화됨
class GameState {
    // 싱글 스레드에서만 업데이트
    void update() {
        // ... 상태 변경 (M 상태)
    }

    // 여러 스레드에서 읽기
    const State& getState() const {
        return state_;  // S 상태로 공유
    }
};

// 상황 3: 여러 스레드가 쓰기 경쟁
// → S→I→M 반복, 성능 최악!
// ❌ 피해야 하는 패턴
std::atomic sharedCounter;  // 모든 스레드가 증가

// ✅ 개선: 스레드별 카운터 후 병합
thread_local int localCounter;
void increment() {
    localCounter++;  // E→M, 경쟁 없음
}
int getTotal() {
    // 마지막에 한 번만 수집
    return aggregateAllCounters();
}
Read-Mostly 데이터 최적화
// 게임 설정 같은 Read-Mostly 데이터
class ReadMostlyConfig {
private:
    // 패딩으로 다른 데이터와 캐시 라인 분리
    alignas(64) struct {
        int maxPlayers;
        int tickRate;
        float gravity;
        // ... 읽기 전용 설정
    } config_;

public:
    // 대부분의 시간 동안 S 상태 유지
    int getMaxPlayers() const { return config_.maxPlayers; }
    int getTickRate() const { return config_.tickRate; }

    // 드물게 업데이트 (핫 리로드 등)
    void reload() {
        // 모든 스레드의 캐시 무효화됨 (S→I)
        // 하지만 자주 발생하지 않으므로 괜찮음
        loadFromFile(config_);
    }
};

📌 핵심 정리

  • MESI: Modified, Exclusive, Shared, Invalid 4가지 상태
  • Exclusive: 나만 가지고 있어 쓰기 시 버스 트래픽 없음 (빠름)
  • Shared→Modified: 다른 캐시 무효화 필요 (느림)
  • HITM: M 상태에서 데이터 가져옴, 경쟁의 지표
  • 최적화: 읽기 전용 데이터 분리, 스레드별 로컬 데이터 사용
