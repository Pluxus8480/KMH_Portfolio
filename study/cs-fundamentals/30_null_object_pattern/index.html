null 체크 지옥에서 벗어나는 법

널 오브젝트 패턴(Null Object Pattern) - "아무것도 안 하는 객체"로 코드를 깔끔하게

Part 1: 문제 상황

왜 널 오브젝트 패턴이 필요한가요?

객체를 활용하는 경우, 어디선가 객체를 구해오면 항상 null인지 아닌지를 체크할 필요가 있습니다. 이 경우 코드가 상당히 지저분해집니다.

C# - null 체크가 필요한 기존 방식
Exp exp = DB.GetMonsterExp("MonsterName");
if (exp != null)
{
    exp.GiveExpToPlayer(player);
}

이런 null 체크가 코드 곳곳에 퍼지면, 가독성이 떨어지고 하나라도 빠뜨리면 NullReferenceException이 발생합니다.

Part 2: 널 오브젝트 패턴의 해결책

널 오브젝트 패턴은 어떻게 이 문제를 해결하나요?

객체를 반환하는 함수에서 null을 반환하는 대신, null과 의미가 유사한 "널 오브젝트"를 반환합니다.

이 널 오브젝트는 실제로 null이 아닌 객체이지만, 하는 일은 없는 객체입니다.

C# - 널 오브젝트 패턴 적용 후
Exp exp = DB.GetMonsterExp("MonsterName");
// NullExp 객체는 아무 일도 안 하기에 플레이어에게 경험치를 주지 않는다
exp.GiveExpToPlayer(player);

null 여부를 검사하지 않는 깔끔한 코드를 만들 수 있습니다.

비유로 설명해줄 수 있나요?
비유로 이해하기

택배 기사가 배달할 물건이 없을 때, "물건 없음"이라고 보고하는 대신 빈 상자를 들고 배달 루트를 도는 것과 같습니다.

빈 상자를 열어봐도 아무것도 없지만, 배달 시스템은 정상적으로 동작합니다. "물건이 없다"는 예외 상황을 처리할 필요가 없어지는 것이죠.

Part 3: 구현 예시

C++로 구현하면 어떻게 되나요?
C++
// 기본 인터페이스
class IExp {
public:
    virtual void GiveExpToPlayer(Player* player) = 0;
    virtual ~IExp() = default;
};

// 실제 경험치 객체
class MonsterExp : public IExp {
    int amount;
public:
    MonsterExp(int amt) : amount(amt) {}
    void GiveExpToPlayer(Player* player) override {
        player->AddExp(amount);
    }
};

// 널 오브젝트 - 아무 일도 하지 않음
class NullExp : public IExp {
public:
    void GiveExpToPlayer(Player* player) override {
        // 아무것도 하지 않는다
    }
};

// DB에서 경험치를 가져오는 함수
IExp* GetMonsterExp(const std::string& name) {
    auto it = expTable.find(name);
    if (it != expTable.end()) {
        return it->second;
    }
    // null 대신 NullExp 반환
    static NullExp nullExp;
    return &nullExp;
}

이제 호출하는 쪽에서는 null 체크 없이 바로 GiveExpToPlayer()를 호출할 수 있습니다.

Part 4: 면접 포인트

널 오브젝트 패턴의 장단점은?
구분 내용
장점 null 체크 코드 제거로 가독성 향상. NullReferenceException 방지. 다형성을 활용한 깔끔한 설계.
단점 널 오브젝트가 "진짜 null"인 상황을 감출 수 있음. 버그가 발생해도 조용히 넘어갈 위험. 클래스 수가 늘어남.

널 오브젝트 패턴을 남용하면, 실제 오류 상황이 무시되어 디버깅이 어려워질 수 있으므로 적절한 상황에서만 사용해야 합니다.

핵심 정리

  • 널 오브젝트 패턴은 null 대신 "아무것도 안 하는 객체"를 반환하여 null 체크를 제거하는 패턴이다
  • 다형성을 활용하여 인터페이스를 동일하게 유지하면서 "아무 동작도 하지 않는" 구현체를 만든다
  • 코드 가독성과 안정성이 향상되지만, 실제 오류를 감출 수 있으므로 주의가 필요하다
