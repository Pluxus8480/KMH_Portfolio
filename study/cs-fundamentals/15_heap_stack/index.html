Part 1: 스택 메모리

스택 메모리는 어떻게 동작하나요?

스택 메모리 구조

높은 주소

커널 영역
스택 (Stack) ← 아래로 성장

main() 지역변수

스택 할당 = 스택 포인터(SP)만 이동! O(1) 상수 시간

    // 함수 종료 시:
    // mov rsp, rbp       ; 스택 복원
    // pop rbp
    // ret

    // 단 3개의 명령어로 모든 지역변수 해제!
}

void heapAllocation() {
    int* a = new int(10);
    // 1. 힙 관리자 호출
    // 2. 빈 공간 탐색
    // 3. 메모리 블록 분할
    // 4. 메타데이터 업데이트
    // 5. 포인터 반환
    // → 수십~수백 사이클!

    delete a;
    // 1. 메타데이터 확인
    // 2. 병합 가능 블록 확인
    // 3. 프리 리스트 업데이트
}
특성 스택 힙
할당 속도 ~1 사이클 ~100+ 사이클
해제 속도 자동 (0 비용) delete 호출 필요
크기 제한 보통 1~8MB 가상 메모리 한도
수명 스코프에 묶임 수동 관리
캐시 효율 최고 (연속/지역성) 낮음 (단편화)

Part 2: 힙 메모리

힙은 언제 사용해야 하나요?
  • 크기가 런타임에 결정될 때
  • 수명이 함수 스코프를 넘어야 할 때
  • 대용량 데이터 (스택 오버플로우 방지)
  • 다형성(Polymorphism) 필요 시

// 2. 수명 연장

    return new Entity();  // 호출자가 관리
}

// 3. 대용량 데이터

힙 할당기는 어떻게 동작하나요?

힙 할당기 내부 구조

Free List (빈 블록 연결 리스트):

Header

size: 64

Free Space

→
Header

size: 128

Free Space

→
Header

size: 32

Free Space

→
Header

size: 256

Free Space

→ NULL

malloc(50) 호출 시:

  1. Free List에서 >= 50 + Header 크기인 블록 탐색
  2. 64바이트 블록 발견! (First Fit)
  3. 블록을 리스트에서 제거
  4. 사용자에게 데이터 영역 포인터 반환

free(ptr) 호출 시:

  1. ptr에서 Header 위치 계산
  2. Free List에 블록 추가
  3. 인접 빈 블록과 병합 (Coalescing)

Part 3: 게임 최적화 기법

게임에서 힙 사용을 줄이는 방법은?
스택 배열 활용
// ❌ 매번 힙 할당
void processBad(const std::vector& entities) {
    std::vector visible;  // 힙 할당!
    for (auto* e : entities) {
        if (isVisible(e)) visible.push_back(e);
    }
    // 처리...
}

// ✅ 스택 배열 또는 Small Buffer Optimization
void processGood(const std::vector& entities) {
    // 작은 배열은 스택에
    Entity* visible[256];
    int count = 0;

    for (auto* e : entities) {
        if (count < 256 && isVisible(e)) {
            visible[count++] = e;
        }
    }
    // 처리...
}

// ✅ std::array 또는 고정 크기 컨테이너
template
class StackVector {
    std::array data_;
    size_t size_ = 0;

public:
    void push_back(const T& v) {
        if (size_ < N) data_[size_++] = v;
    }
    // 힙 할당 없이 스택에서만 동작
};
오브젝트 풀링
// 게임에서 가장 많이 쓰는 패턴!
template
class ObjectPool {
    union Slot {
        T object;
        Slot* next;
        Slot() {}
        ~Slot() {}
    };

    Slot pool_[PoolSize];
    Slot* freeList_ = nullptr;

public:
    ObjectPool() {
        // 모든 슬롯을 free list로 연결
        for (size_t i = 0; i < PoolSize; ++i) {
            pool_[i].next = freeList_;
            freeList_ = &pool_[i];
        }
    }

    T* allocate() {
        if (!freeList_) return nullptr;

        Slot* slot = freeList_;
        freeList_ = slot->next;
        return new (&slot->object) T();  // placement new
    }

    void deallocate(T* obj) {
        obj->~T();
        Slot* slot = reinterpret_cast(obj);
        slot->next = freeList_;
        freeList_ = slot;
    }
};

// 사용 예시
ObjectPool bulletPool;

void fire() {
    Bullet* b = bulletPool.allocate();  // O(1), 힙 접근 없음!
    // ...
}

void destroy(Bullet* b) {
    bulletPool.deallocate(b);  // O(1)
}

Part 4: 스마트 포인터와 RAII

힙 메모리 누수를 방지하려면?
스마트 포인터 활용
#include 

// unique_ptr: 단독 소유권
std::unique_ptr entity = std::make_unique();
// 스코프 끝나면 자동 delete

// shared_ptr: 공유 소유권
std::shared_ptr texture = std::make_shared();
// 참조 카운트 0이 되면 자동 delete

// weak_ptr: 순환 참조 방지
class Node {
    std::shared_ptr next;
    std::weak_ptr parent;  // weak로 순환 끊기
};

// 게임에서의 적용
class Entity {
    std::unique_ptr mesh_;      // 단독 소유
    std::shared_ptr material_;       // 여러 엔티티가 공유
    std::weak_ptr target_;             // 참조만 (소유 X)

public:
    Entity() : mesh_(std::make_unique()) {}

    void setTarget(std::shared_ptr target) {
        target_ = target;  // weak로 저장
    }

    void update() {
        if (auto t = target_.lock()) {  // 아직 살아있는지 확인
            moveToward(t->position);
        }
    }
};

shared_ptr의 참조 카운팅은 원자적 연산이라 비용이 있습니다. 성능 중요한 곳에서는 unique_ptr이나 원시 포인터를 고려하세요.

스택 오버플로우는 어떻게 발생하나요?
스택 오버플로우 사례

void badRecursion(int n) {
    if (n <= 0) return;

    badRecursion(n - 1);

badRecursion(10000);  // 10MB+ 스택 사용 → 오버플로우!

📌 핵심 정리

  • 스택: ~1 사이클 할당, 자동 해제, 크기 제한 있음
  • 힙: ~100+ 사이클, 수동 관리, 큰 데이터/긴 수명용
  • 오브젝트 풀: 빈번한 할당/해제를 O(1)로 최적화
  • 스마트 포인터: 메모리 누수 방지, unique_ptr > shared_ptr
  • 스택 오버플로우: 재귀 제한, 대용량은 힙에 할당
