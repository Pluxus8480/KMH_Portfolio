옵저버 패턴(Observer Pattern)의 구독-알림 메커니즘

Part 1: 옵저버 패턴이란?

옵저버 패턴의 정의는?

옵저버 패턴은 객체의 상태 변화를 관찰하는 옵저버들을 두어, 변화가 있을 때마다 객체가 옵저버들에게 통지하는 패턴입니다.

변화가 있는 객체(Subject)에 옵저버 리스트를 두어, 변화 발생 시 리스트의 옵저버들을 호출합니다.

🎯 비유로 이해하기

유튜브 구독과 같습니다. 채널(Subject)에 새 영상이 올라오면 구독자(Observer)들에게 알림이 갑니다.

구독자가 늘거나 줄어도 채널의 영상 업로드 로직은 변하지 않습니다.

호출 흐름이 Subject에서 Observer 방향으로 흘러갑니다. 인터페이스나 델리게이트 등으로 콜백을 만들어 사용합니다.

Part 2: 장단점

옵저버 패턴의 장단점은?
장점 단점
객체와 옵저버 간 결합이 느슨해짐 멀티스레드에서 등록/취소와 실행이 겹칠 수 있음
직관적인 이해가 쉬움 너무 많은 옵저버는 성능 저하 유발
새 옵저버 추가가 쉬움 상태 관리가 복잡해질 수 있음

멀티스레드 환경에서 옵저버 등록/취소 작업과 Notify 실행이 동시에 일어나면 예상치 못한 결과가 발생할 수 있습니다. 동기화 처리에 주의해야 합니다.

📌 핵심 정리

  • 옵저버 패턴은 Subject의 상태 변화를 Observer들에게 자동 통지하는 패턴
  • Subject에 옵저버 리스트를 두고, 변화 시 순회하며 호출
  • 객체 간 느슨한 결합을 통해 유지보수성 향상
  • 멀티스레드 환경에서 등록/취소와 실행의 동기화에 주의 필요
  • 인터페이스, 델리게이트, 이벤트 등 다양한 방식으로 구현 가능
