"if문이 왜 느려요?"

CPU 파이프라인과 분기 예측의 비밀

Part 1: CPU 파이프라인 기초

파이프라인이란 무엇인가요?

명령어 처리를 여러 단계로 분할하여 동시에 처리하는 기법입니다.

🎯 비유로 이해하기

세탁을 생각해보세요. 세탁→건조→정리 3단계가 있다면:

파이프라인 없이: 첫 번째 빨래 3단계 완료 → 두 번째 시작

파이프라인 사용: 첫 번째가 건조 중일 때 두 번째가 세탁 시작!

사이클 1
I1-IF
사이클 2
I1-IDI2-IF
사이클 3
I1-EXI2-IDI3-IF
사이클 4
I1-MEMI2-EXI3-IDI4-IF
사이클 5
I1-WBI2-MEMI3-EXI4-IDI5-IF
사이클 6
I2-WBI3-MEMI4-EXI5-IDI6-IF

단계: IF(Fetch) → ID(Decode) → EX(Execute) → MEM(Memory) → WB(WriteBack)
파이프라인 없이: 25사이클 / 파이프라인 사용: 9사이클 (거의 1 IPC!) / 현대 CPU는 10~20단계

파이프라인 깊이가 깊으면 무조건 좋은 건가요?

아닙니다. 파이프라인 해저드(Hazard) 발생 시 손실이 커집니다.

해저드 유형 원인 해결책
구조적 해저드 하드웨어 리소스 충돌 리소스 복제
데이터 해저드 이전 결과가 필요 포워딩, 스톨
제어 해저드 분기 결과 미확정 분기 예측

Part 2: 분기 예측 (Branch Prediction)

분기 예측이 왜 필요한가요?

if문의 결과를 알기 전에 다음 명령어를 미리 가져와야 하기 때문입니다.

정상 실행
I1-FI1-DI1-EI1-MI1-W
분기 명령
I2-FI2-DI2-EMISSFLUSH
잘못된 예측 (버림)
I3-FI3-DXXXXXXXX
재시작 (올바른 경로)
I10-FI10-DI10-E...

예측 실패 페널티: 10~20 사이클 (파이프라인 깊이만큼) - 파이프라인 비움!

현대 CPU는 90% 이상의 예측 정확도를 가집니다!

분기 예측은 어떻게 동작하나요?

과거 패턴을 기억하고 예측합니다.

실제로 얼마나 차이가 나나요?
정렬 vs 비정렬 데이터의 분기 성능

#include 

// 0~255 범위의 랜덤 값
int data[32768];

void benchmark() {
    // 분기: 128보다 크면 더하기
    long long sum = 0;

    auto start = std::chrono::high_resolution_clock::now();

    for (int i = 0; i < 100000; ++i) {
        for (int j = 0; j < 32768; ++j) {
            if (data[j] >= 128)  // 분기!
                sum += data[j];
        }
    }

    auto end = std::chrono::high_resolution_clock::now();
    // 시간 측정
}

int main() {
    // 랜덤 데이터 생성
    for (int& d : data) d = rand() % 256;

    // 테스트 1: 비정렬 (랜덤)
    benchmark();  // 약 10초

    // 테스트 2: 정렬 후
    std::sort(data, data + 32768);
    benchmark();  // 약 3초 (3배 이상 빠름!)
}

// 왜?
// 비정렬: 128 미만/이상이 랜덤하게 섞여 있음 → 예측 50% 실패
// 정렬 후: [0,0,0,...,128,129,...,255] → 예측 거의 100% 성공

같은 연산인데 데이터 정렬 여부에 따라 3배 성능 차이!

분기를 피하는 방법은?
Branch-free 코드 기법
// ❌ 분기 있는 코드
int max_branch(int a, int b) {
    if (a > b) return a;
    else return b;
}

// ✅ 분기 없는 코드 (비트 연산)
int max_branchless(int a, int b) {
    int diff = a - b;
    int mask = diff >> 31;  // a < b면 -1(0xFFFFFFFF), 아니면 0
    return a - (diff & mask);  // a가 크면 a, b가 크면 b
}

// ✅ 조건부 이동 명령어 활용 (컴파일러 최적화)
int max_cmov(int a, int b) {
    return a > b ? a : b;  // 컴파일러가 CMOV 명령어로 변환
}

// ❌ 조건부 더하기 (분기)
void sumIf_branch(const int* data, int count, int threshold, int& sum) {
    for (int i = 0; i < count; ++i) {
        if (data[i] >= threshold)
            sum += data[i];
    }
}

// ✅ 조건부 더하기 (분기 없음)
void sumIf_branchless(const int* data, int count, int threshold, int& sum) {
    for (int i = 0; i < count; ++i) {
        // 조건을 0 또는 1로 변환
        int cond = (data[i] >= threshold);
        sum += data[i] * cond;  // 조건이 0이면 0 더함
    }
}

// ✅ SIMD 활용 (더 효율적)
void sumIf_simd(const int* data, int count, int threshold, int& sum) {
    __m128i thresh = _mm_set1_epi32(threshold);
    __m128i sumVec = _mm_setzero_si128();

    for (int i = 0; i < count; i += 4) {
        __m128i values = _mm_loadu_si128((__m128i*)(data + i));
        __m128i mask = _mm_cmpgt_epi32(values, thresh);
        sumVec = _mm_add_epi32(sumVec, _mm_and_si128(values, mask));
    }
    // ... 합산
}

Part 4: 게임에서의 분기 최적화

게임에서 분기 예측 실패가 문제가 되는 경우는?

대량의 엔티티를 처리하는 핫 루프에서 영향이 큽니다.

엔티티 타입별 처리 최적화
// ❌ 타입 검사 분기 (예측 어려움)
void updateEntities_bad(std::vector& entities) {
    for (auto& e : entities) {
        switch (e.type) {  // 매번 분기!
            case ENEMY:   updateEnemy(e);   break;
            case PLAYER:  updatePlayer(e);  break;
            case BULLET:  updateBullet(e);  break;
            case ITEM:    updateItem(e);    break;
        }
    }
}

// ✅ 타입별로 정렬하여 처리
struct EntityManager {
    std::vector enemies;
    std::vector players;
    std::vector bullets;
    std::vector items;

    void update() {
        // 같은 타입끼리 연속 처리 → 분기 예측 성공률 높음
        for (auto& e : enemies)  updateEnemy(e);
        for (auto& e : players)  updatePlayer(e);
        for (auto& e : bullets)  updateBullet(e);
        for (auto& e : items)    updateItem(e);
    }
};

// ✅ 가상 함수 대신 함수 포인터 테이블
using UpdateFunc = void(*)(Entity&);
UpdateFunc updateTable[] = {
    updateEnemy,
    updatePlayer,
    updateBullet,
    updateItem
};

void updateEntities_table(std::vector& entities) {
    // 타입별로 정렬된 상태라면 분기 예측 가능
    for (auto& e : entities) {
        updateTable[e.type](e);
    }
}
충돌 검사에서 분기를 줄이는 방법은?
충돌 검사 최적화
// ❌ Early-out 분기 많음
bool checkCollision_branch(const AABB& a, const AABB& b) {
    if (a.maxX < b.minX) return false;  // 분기
    if (a.minX > b.maxX) return false;  // 분기
    if (a.maxY < b.minY) return false;  // 분기
    if (a.minY > b.maxY) return false;  // 분기
    if (a.maxZ < b.minZ) return false;  // 분기
    if (a.minZ > b.maxZ) return false;  // 분기
    return true;
}

// ✅ 분기 없는 SIMD 버전
bool checkCollision_simd(const AABB& a, const AABB& b) {
    // min/max를 128비트 벡터로 로드
    __m128 aMin = _mm_loadu_ps(&a.minX);  // [minX, minY, minZ, ?]
    __m128 aMax = _mm_loadu_ps(&a.maxX);
    __m128 bMin = _mm_loadu_ps(&b.minX);
    __m128 bMax = _mm_loadu_ps(&b.maxX);

    // 모든 축을 동시에 비교
    __m128 test1 = _mm_cmpge_ps(aMax, bMin);  // a.max >= b.min
    __m128 test2 = _mm_cmpge_ps(bMax, aMin);  // b.max >= a.min

    // AND 연산으로 모든 조건 확인
    __m128 result = _mm_and_ps(test1, test2);

    // 모든 비트가 1이면 충돌
    return _mm_movemask_ps(result) == 0xF;
}

// ✅ 충돌 가능성이 낮을 때의 전략
void processCollisions(std::vector& entities) {
    // 1단계: 빠른 브로드페이즈 (공간 분할)
    auto pairs = spatialHash.getPotentialPairs(entities);

    // 2단계: 정밀 검사 (쌍이 적음)
    for (auto& [a, b] : pairs) {
        if (checkCollision_simd(a.aabb, b.aabb)) {
            handleCollision(a, b);
        }
    }
}

Part 5: 프로파일링

분기 예측 실패를 어떻게 측정하나요?
perf를 이용한 분기 예측 분석
# 분기 예측 관련 이벤트 측정
perf stat -e branches,branch-misses,instructions,cycles ./game

# 출력 예시:
#     1,234,567,890  branches
#        12,345,678  branch-misses  # 1.00% of all branches
#     5,000,000,000  instructions   # 1.25 IPC
#     4,000,000,000  cycles

# 핫스팟 분석
perf record -e branch-misses ./game
perf report

# 특정 함수의 분기 예측 분석
perf annotate --symbol=updateEntities

분기 실패율 1%면 양호, 5% 이상이면 최적화 대상입니다.

📌 핵심 정리

  • 파이프라인: 명령어를 단계별로 분할하여 병렬 처리, 현대 CPU는 10~20단계
  • 분기 예측: 다음 명령어를 미리 가져오기 위해 분기 결과를 예측
  • 예측 실패 비용: 파이프라인 깊이만큼 사이클 손실 (10~20 사이클)
  • 최적화 기법: 데이터 정렬, 분기 없는 코드, 같은 타입 연속 처리
  • 측정: perf로 branch-misses 이벤트 확인
