Lock-free 알고리즘과 CAS 연산

Part 1: Lock-free의 개념

Lock-free가 왜 필요한가요?

Mutex의 문제점을 해결합니다:

  • 데드락: 잘못된 lock 순서로 영원히 대기
  • Priority Inversion: 저우선 스레드가 lock 보유 시 고우선 대기
  • 컨텍스트 스위칭: 대기 시 커널 진입 오버헤드

Lock-free는 최소 하나의 스레드가 항상 진행됨을 보장합니다.

Lock-free와 Wait-free의 차이는?
구분 Lock-free Wait-free
진행 보장 전체 중 하나 이상 모든 스레드
기아(Starvation) 가능 불가능
구현 난이도 어려움 매우 어려움
성능 보통 더 좋음 오버헤드 있음

Part 2: CAS (Compare-And-Swap)

CAS 연산이란 무엇인가요?

Lock-free의 핵심 원자적 연산입니다.

┌─────────────────────────────────────────────────────────────────────┐
│                    CAS (Compare-And-Swap) 동작                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   CAS(메모리위치, 기대값, 새값)                                      │
│                                                                     │
│   if (메모리[위치] == 기대값) {                                      │
│       메모리[위치] = 새값;                                           │
│       return true;   // 성공                                        │
│   } else {                                                          │
│       return false;  // 실패 (다른 스레드가 먼저 수정)              │
│   }                                                                 │
│                                                                     │
│   ※ 이 전체 과정이 하드웨어 수준에서 원자적으로 실행됨!             │
│                                                                     │
│   예시:                                                              │
│   ┌─────────────┐                                                   │
│   │  value: 10  │  ← 현재 상태                                     │
│   └─────────────┘                                                   │
│                                                                     │
│   Thread A: CAS(&value, 10, 20)  →  성공! value = 20               │
│   Thread B: CAS(&value, 10, 30)  →  실패! (기대값 10 ≠ 현재 20)   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
C++ atomic CAS 사용
#include 

std::atomic counter{0};

// CAS 기반 안전한 증가
void safeIncrement() {
    int expected = counter.load();

    while (!counter.compare_exchange_weak(expected, expected + 1)) {
        // 실패하면 expected에 현재 값이 저장됨
        // 다시 시도
    }
}

// compare_exchange_weak vs compare_exchange_strong
// weak: 가짜 실패(spurious failure) 가능, 루프에서 사용
// strong: 값이 같으면 항상 성공, 단일 시도에 적합

Part 3: Lock-free 스택

Lock-free 스택은 어떻게 구현하나요?
Lock-free Stack 구현
template
class LockFreeStack {
private:
    struct Node {
        T data;
        Node* next;
        Node(const T& d) : data(d), next(nullptr) {}
    };

    std::atomic head{nullptr};

public:
    void push(const T& data) {
        Node* newNode = new Node(data);

        // CAS 루프
        newNode->next = head.load();
        while (!head.compare_exchange_weak(newNode->next, newNode)) {
            // 실패: newNode->next가 현재 head로 업데이트됨
            // 다시 시도
        }
    }

    bool pop(T& result) {
        Node* oldHead = head.load();

        while (oldHead != nullptr) {
            // 다른 스레드가 oldHead를 이미 pop 했을 수 있음
            if (head.compare_exchange_weak(oldHead, oldHead->next)) {
                result = oldHead->data;
                // 주의: 메모리 해제는 별도 처리 필요! (ABA 문제)
                // delete oldHead;  // 위험!
                return true;
            }
            // 실패: oldHead가 현재 head로 업데이트됨
        }
        return false;  // 스택 비어있음
    }
};

위 코드에는 ABA 문제와 메모리 해제 문제가 있습니다!

ABA 문제란 무엇인가요?
┌─────────────────────────────────────────────────────────────────────┐
│                         ABA 문제                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   초기: head → [A] → [B] → [C]                                     │
│                                                                     │
│   Thread 1:                          Thread 2:                      │
│   oldHead = A                                                       │
│   (여기서 스케줄 아웃)              pop() → A 제거                   │
│                                     pop() → B 제거                   │
│                                     push(A) → A 재사용              │
│                                                                     │
│   현재: head → [A] → [C]   (B는 해제됨)                            │
│                                                                     │
│   Thread 1 재개:                                                    │
│   CAS(head, A, A->next)                                            │
│   = CAS(head, A, B)  ← 성공! (head가 A니까)                        │
│                                                                     │
│   결과: head → [B] → ???  ← B는 이미 해제된 메모리!                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
ABA 문제 해결: Tagged Pointer
template
class LockFreeStackSafe {
private:
    struct Node {
        T data;
        Node* next;
    };

    // 포인터 + 버전 태그
    struct TaggedPointer {
        Node* ptr;
        uintptr_t tag;  // 버전 카운터
    };

    std::atomic head{{nullptr, 0}};

public:
    void push(const T& data) {
        Node* newNode = new Node{data, nullptr};
        TaggedPointer oldHead = head.load();
        TaggedPointer newHead;

        do {
            newNode->next = oldHead.ptr;
            newHead = {newNode, oldHead.tag + 1};  // 태그 증가!
        } while (!head.compare_exchange_weak(oldHead, newHead));
    }

    bool pop(T& result) {
        TaggedPointer oldHead = head.load();
        TaggedPointer newHead;

        while (oldHead.ptr != nullptr) {
            newHead = {oldHead.ptr->next, oldHead.tag + 1};

            if (head.compare_exchange_weak(oldHead, newHead)) {
                result = oldHead.ptr->data;
                // 안전한 메모리 해제 (Hazard Pointer 또는 RCU 사용)
                safeDelete(oldHead.ptr);
                return true;
            }
        }
        return false;
    }
};

Part 4: Lock-free 큐

게임에서 자주 쓰는 Lock-free 큐는?
SPSC (Single Producer Single Consumer) 큐
// Producer 1개, Consumer 1개일 때 매우 효율적
template
class SPSCQueue {
private:
    alignas(64) std::array buffer_;
    alignas(64) std::atomic head_{0};  // Consumer가 읽는 위치
    alignas(64) std::atomic tail_{0};  // Producer가 쓰는 위치

public:
    bool push(const T& item) {
        size_t currentTail = tail_.load(std::memory_order_relaxed);
        size_t nextTail = (currentTail + 1) % Capacity;

        // 큐가 가득 찼는지 확인
        if (nextTail == head_.load(std::memory_order_acquire)) {
            return false;  // Full
        }

        buffer_[currentTail] = item;
        tail_.store(nextTail, std::memory_order_release);
        return true;
    }

    bool pop(T& item) {
        size_t currentHead = head_.load(std::memory_order_relaxed);

        // 큐가 비었는지 확인
        if (currentHead == tail_.load(std::memory_order_acquire)) {
            return false;  // Empty
        }

        item = buffer_[currentHead];
        head_.store((currentHead + 1) % Capacity, std::memory_order_release);
        return true;
    }
};

// 사용 예: 오디오 스레드 ↔ 게임 스레드 통신
SPSCQueue audioQueue;
MPSC (Multiple Producer Single Consumer) 큐
// 여러 워커 스레드 → 메인 스레드 결과 전달용
template
class MPSCQueue {
private:
    struct Node {
        std::atomic next{nullptr};
        T data;
    };

    std::atomic head_;
    alignas(64) Node* tail_;  // Consumer만 접근
    Node stub_;

public:
    MPSCQueue() : head_(&stub_), tail_(&stub_) {}

    void push(const T& data) {
        Node* node = new Node();
        node->data = data;

        // Producer들이 경쟁하며 추가
        Node* prev = head_.exchange(node, std::memory_order_acq_rel);
        prev->next.store(node, std::memory_order_release);
    }

    bool pop(T& result) {
        Node* tail = tail_;
        Node* next = tail->next.load(std::memory_order_acquire);

        if (next == nullptr) {
            return false;  // Empty
        }

        result = next->data;
        tail_ = next;

        if (tail != &stub_) {
            delete tail;
        }
        return true;
    }
};

Part 5: 메모리 순서 (Memory Ordering)

memory_order는 왜 필요한가요?

CPU와 컴파일러는 성능을 위해 명령어 순서를 바꿀 수 있습니다.

Memory Order 설명 사용 상황
relaxed 순서 보장 없음, 원자성만 단순 카운터
acquire 이후 읽기가 앞으로 오지 않음 lock 획득
release 이전 쓰기가 뒤로 가지 않음 lock 해제
acq_rel acquire + release CAS 연산
seq_cst 전체 순서 보장 (기본값) 안전하지만 느림
Memory Order 사용 예시
std::atomic ready{false};
int data = 0;

void producer() {
    data = 42;  // 일반 쓰기
    ready.store(true, std::memory_order_release);
    // release: data = 42가 ready = true 이전에 완료됨을 보장
}

void consumer() {
    while (!ready.load(std::memory_order_acquire)) {
        // 대기
    }
    // acquire: ready가 true면 data = 42가 보임
    assert(data == 42);  // 항상 성공!
}

Part 6: 게임에서의 활용

게임 엔진에서 Lock-free는 어디에 쓰이나요?
Lock-free 작업 스틸링 큐
// 멀티스레드 Job System에서 사용
class WorkStealingQueue {
private:
    std::array buffer_;
    std::atomic top_{0};     // 소유자 스레드가 push/pop
    std::atomic bottom_{0};  // 다른 스레드가 steal

public:
    // 소유자: 맨 위에 추가
    void push(Job* job) {
        int64_t b = bottom_.load(std::memory_order_relaxed);
        buffer_[b % buffer_.size()] = job;
        bottom_.store(b + 1, std::memory_order_release);
    }

    // 소유자: 맨 위에서 가져옴 (LIFO)
    Job* pop() {
        int64_t b = bottom_.load(std::memory_order_relaxed) - 1;
        bottom_.store(b, std::memory_order_relaxed);

        std::atomic_thread_fence(std::memory_order_seq_cst);

        int64_t t = top_.load(std::memory_order_relaxed);

        if (t <= b) {
            Job* job = buffer_[b % buffer_.size()];
            if (t == b) {
                // 마지막 요소, steal과 경쟁
                if (!top_.compare_exchange_strong(t, t + 1)) {
                    job = nullptr;  // steal에 패배
                }
                bottom_.store(b + 1, std::memory_order_relaxed);
            }
            return job;
        }
        bottom_.store(b + 1, std::memory_order_relaxed);
        return nullptr;
    }

    // 다른 스레드: 맨 아래에서 훔쳐감 (FIFO)
    Job* steal() {
        int64_t t = top_.load(std::memory_order_acquire);
        int64_t b = bottom_.load(std::memory_order_acquire);

        if (t < b) {
            Job* job = buffer_[t % buffer_.size()];
            if (top_.compare_exchange_strong(t, t + 1)) {
                return job;
            }
        }
        return nullptr;
    }
};

Work Stealing은 코어 간 부하 균형을 자동으로 맞춰줍니다!

📌 핵심 정리

  • Lock-free: 최소 하나의 스레드가 항상 진행, 데드락 없음
  • CAS: Lock-free의 핵심, compare_exchange_weak/strong 사용
  • ABA 문제: Tagged Pointer나 Hazard Pointer로 해결
  • SPSC Queue: 단일 Producer-Consumer에 최적, 매우 빠름
  • Memory Order: 성능을 위해 적절한 순서 지정 필요
