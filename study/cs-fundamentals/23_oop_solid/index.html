좋은 설계의 다섯 가지 약속

객체지향 프로그래밍 SOLID 원칙 완전 정복

Part 1: SRP - 단일 책임 원칙

Single Responsibility Principle이란?

객체는 단 하나의 책임만을 가져야 합니다. 어떤 변화에 의해 수정되어야 하는 이유는 오직 하나뿐이어야 합니다.

🎯 비유로 이해하기

스위스 아미 나이프가 아니라, 용도별 전문 도구를 쓰는 것과 같습니다.

칼은 자르는 일만, 드라이버는 조이는 일만 담당해야 합니다.

C++ - SRP 위반 vs 준수
// 위반: Player가 너무 많은 책임을 가짐
class Player {
    void Move();
    void Attack();
    void SaveToDatabase();  // DB 저장까지?
    void RenderUI();        // UI 렌더링까지?
};

// 준수: 책임을 분리
class Player { void Move(); void Attack(); };
class PlayerDataManager { void Save(Player& p); };
class PlayerUI { void Render(Player& p); };

Part 2: OCP - 개방-폐쇄 원칙

Open-Closed Principle이란?

소프트웨어 개체는 확장에는 열려 있어야 하고, 수정에는 닫혀 있어야 합니다.

기존 코드를 수정하지 않고도 새로운 기능을 추가할 수 있도록 설계해야 합니다.

인터페이스와 상속을 활용하면, 새로운 기능 추가 시 기존 코드를 건드리지 않고 새 클래스를 추가하기만 하면 됩니다.

Part 3: LSP - 리스코프 치환 원칙

Liskov Substitution Principle이란?

자식 클래스는 언제나 부모 클래스의 역할을 대체할 수 있어야 합니다.

부모 클래스를 사용하는 코드에 자식 클래스를 넣어도 정상 동작해야 합니다.

C++ - LSP 위반 예시
class Bird {
public:
    virtual void Fly() { /* 날기 */ }
};

class Penguin : public Bird {
public:
    void Fly() override {
        // 펭귄은 못 날아! - LSP 위반
        throw std::runtime_error("Can't fly!");
    }
};

부모 클래스의 계약(Fly)을 자식이 깨뜨리면 LSP 위반입니다.

Part 4: ISP - 인터페이스 분리 원칙

Interface Segregation Principle이란?

클라이언트가 필요하지 않는 기능에 의존하도록 강요하지 않는 원칙입니다.

클라이언트가 필요로 하는 기능만 인터페이스로 분리하면, 불필요한 재컴파일과 의존을 막을 수 있습니다.

C++ - ISP 적용
// 위반: 하나의 거대한 인터페이스
class ICharacter {
    virtual void Move() = 0;
    virtual void Attack() = 0;
    virtual void Swim() = 0;    // 모든 캐릭터가 수영?
    virtual void Fly() = 0;     // 모든 캐릭터가 비행?
};

// 준수: 인터페이스 분리
class IMovable { virtual void Move() = 0; };
class IAttackable { virtual void Attack() = 0; };
class ISwimmable { virtual void Swim() = 0; };
class IFlyable { virtual void Fly() = 0; };

Part 5: DIP - 의존성 역전 원칙

Dependency Inversion Principle이란?

상위 수준 모듈은 하위 수준 모듈에 의존하면 안 되며, 둘 모두 추상화에 의존해야 합니다.

추상화는 구체적인 사항에 의존해서는 안 되고, 구체적인 사항이 추상화에 의존해야 합니다.

추상화된 인터페이스나 추상 클래스를 통해 상호작용하도록 설계하면, 하위 모듈의 변경이 상위 모듈에 영향을 미치지 않습니다.

📌 핵심 정리

  • SRP: 하나의 클래스는 하나의 책임만 (변경 이유는 하나)
  • OCP: 확장에 열림, 수정에 닫힘 (인터페이스/상속 활용)
  • LSP: 자식은 부모를 완전히 대체 가능해야 함
  • ISP: 불필요한 인터페이스 의존 강요 금지
  • DIP: 상위/하위 모듈 모두 추상화에 의존
