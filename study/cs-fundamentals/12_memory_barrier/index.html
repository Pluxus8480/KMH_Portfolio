"코드 순서대로 실행 안 된다고요?"

메모리 순서와 배리어의 이해

Part 1: 메모리 재정렬 문제

코드가 순서대로 실행 안 되는 경우가 있나요?

네! 컴파일러와 CPU 모두 성능을 위해 순서를 바꿀 수 있습니다.

재정렬 문제 예시
// 원래 코드
int data = 0;
bool ready = false;

// Thread 1 (Producer)
void producer() {
    data = 42;      // (1)
    ready = true;   // (2)
}

// Thread 2 (Consumer)
void consumer() {
    while (!ready);  // (3) ready가 true가 될 때까지 대기
    assert(data == 42);  // (4) 항상 성공할까?
}

// 문제: 컴파일러 또는 CPU가 (1)과 (2)를 재정렬할 수 있음!
// 실행 순서가 (2) → (1)이 되면:
// - Consumer가 ready=true를 보고 루프 탈출
// - 하지만 data는 아직 42가 아닐 수 있음!
// - assert 실패!

Part 2: 메모리 배리어 종류

메모리 배리어는 어떻게 동작하나요?

배리어 이전/이후의 메모리 연산 순서를 강제합니다.

배리어 종류 효과 사용 상황
Load Barrier 이전 읽기 완료 보장 데이터 읽기 후 조건 확인
Store Barrier 이전 쓰기 완료 보장 데이터 쓰기 후 플래그 설정
Full Barrier 모든 연산 완료 보장 완전한 동기화 필요 시

Release 이전 쓰기가 뒤로 안 감 Lock 해제 시

Part 3: C++ 메모리 순서

C++에서 메모리 순서는 어떻게 사용하나요?
std::memory_order 사용
#include 

std::atomic data{0};
std::atomic ready{false};

// Producer
void producer() {
    data.store(42, std::memory_order_relaxed);  // 순서 상관없이 저장
    ready.store(true, std::memory_order_release);  // release: 이전 쓰기 완료 보장
}

// Consumer
void consumer() {
    while (!ready.load(std::memory_order_acquire));  // acquire: 이후 읽기 보호
    int d = data.load(std::memory_order_relaxed);
    assert(d == 42);  // 항상 성공!
}

// memory_order 종류:
// memory_order_relaxed  : 순서 보장 없음, 원자성만
// memory_order_acquire  : 이후 읽기가 앞으로 안 옴
// memory_order_release  : 이전 쓰기가 뒤로 안 감
// memory_order_acq_rel  : acquire + release
// memory_order_seq_cst  : 전체 순서 보장 (기본값, 가장 안전하지만 느림)
실제 적용 예시
// Double-Checked Locking Pattern
class Singleton {
    static std::atomic instance;
    static std::mutex mutex;

public:
    static Singleton* getInstance() {
        Singleton* tmp = instance.load(std::memory_order_acquire);
        if (tmp == nullptr) {
            std::lock_guard lock(mutex);
            tmp = instance.load(std::memory_order_relaxed);
            if (tmp == nullptr) {
                tmp = new Singleton();
                instance.store(tmp, std::memory_order_release);
            }
        }
        return tmp;
    }
};

// Spinlock 구현
class Spinlock {
    std::atomic locked{false};

public:
    void lock() {
        while (locked.exchange(true, std::memory_order_acquire)) {
            // busy wait
        }
        // acquire: 이후 임계구역 코드가 앞으로 안 옴
    }

    void unlock() {
        locked.store(false, std::memory_order_release);
        // release: 임계구역 코드가 이 뒤로 안 감
    }
};
seq_cst를 항상 쓰면 안 되나요?

seq_cst는 가장 안전하지만 성능 비용이 있습니다!

Memory Order x86 비용 ARM 비용
relaxed 최소 최소
acquire/release 거의 무료 배리어 명령어
seq_cst MFENCE or LOCK 전체 배리어

x86은 강한 메모리 모델이라 acquire/release가 거의 무료지만, ARM은 약한 모델이라 비용이 큽니다.

Part 4: 컴파일러 배리어

CPU 배리어 외에 컴파일러 배리어도 있나요?
컴파일러 배리어
// volatile은 컴파일러 최적화만 방지, CPU 재정렬은 막지 못함!
volatile int flag;  // 매번 메모리에서 읽기는 하지만 순서 보장 없음

// 컴파일러 배리어 (순서 보장 없이 최적화만 방지)
#ifdef _MSC_VER
    _ReadWriteBarrier();  // MSVC
#else
    asm volatile("" ::: "memory");  // GCC/Clang
#endif

// 올바른 동기화: atomic + memory_order
std::atomic flag{false};
flag.store(true, std::memory_order_release);

// std::atomic_thread_fence
int data = 0;
std::atomic ready{false};

void producer() {
    data = 42;
    std::atomic_thread_fence(std::memory_order_release);
    ready.store(true, std::memory_order_relaxed);
}

void consumer() {
    while (!ready.load(std::memory_order_relaxed));
    std::atomic_thread_fence(std::memory_order_acquire);
    assert(data == 42);
}

Part 5: 게임 프로그래밍 패턴

게임에서 메모리 순서가 중요한 경우는?
스레드 간 데이터 전달
// 렌더 스레드와 게임 스레드 간 프레임 데이터 전달
struct FrameData {
    Matrix4x4 viewMatrix;
    std::vector commands;
    // ... 많은 데이터
};

class DoubleBuffer {
    FrameData buffers[2];
    std::atomic readyIndex{-1};

public:
    // 게임 스레드: 데이터 준비 후 공개
    void submitFrame(int index) {
        // 모든 데이터 쓰기가 완료된 후 인덱스 공개
        readyIndex.store(index, std::memory_order_release);
    }

    // 렌더 스레드: 데이터 사용
    FrameData* getReadyFrame() {
        int index = readyIndex.load(std::memory_order_acquire);
        if (index < 0) return nullptr;
        // acquire 덕분에 buffers[index]의 모든 데이터가 보임
        return &buffers[index];
    }
};

// Lock-free 메시지 큐
template
class SPSCQueue {
    // ... (이전 예제 참조)

    void push(const T& item) {
        buffer[tail] = item;
        // 데이터 쓰기 후 tail 업데이트
        tail_.store(tail + 1, std::memory_order_release);
    }

    bool pop(T& item) {
        size_t tail = tail_.load(std::memory_order_acquire);
        if (head == tail) return false;
        // acquire 덕분에 buffer[head] 데이터가 보임
        item = buffer[head];
        head_.store(head + 1, std::memory_order_release);
        return true;
    }
};

📌 핵심 정리

  • 재정렬: 컴파일러와 CPU 모두 성능을 위해 순서 변경 가능
  • Acquire: 이후 읽기가 배리어 이전으로 이동 방지 (lock 획득)
  • Release: 이전 쓰기가 배리어 이후로 이동 방지 (lock 해제)
  • seq_cst: 가장 안전하지만 성능 비용 있음
  • 게임 적용: 스레드 간 데이터 전달 시 release/acquire 쌍 사용
