가상 메모리와 운영체제의 마법

Part 1: 가상 메모리의 필요성

왜 프로그램마다 메모리 주소가 0부터 시작하나요?

각 프로세스는 가상 주소 공간을 갖기 때문입니다.

물리 메모리와 가상 메모리는 별개입니다. OS가 둘을 매핑해줍니다.

┌─────────────────────────────────────────────────────────────────────┐
│                      가상 메모리 개념                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Process A (가상)         Process B (가상)         Physical RAM    │
│   ┌─────────────┐         ┌─────────────┐         ┌─────────────┐  │
│   │ 0x00000000  │─────┐   │ 0x00000000  │──────┐  │ 0x00001000  │  │
│   │    Code     │     │   │    Code     │      │  ├─────────────┤  │
│   ├─────────────┤     └──▶│─────────────┤──────┼─▶│ Process A   │  │
│   │ 0x00400000  │         │ 0x00400000  │      │  ├─────────────┤  │
│   │    Data     │─────────┼─────────────┼──────┼─▶│ Process B   │  │
│   ├─────────────┤         ├─────────────┤      │  ├─────────────┤  │

│   └─────────────┘         └─────────────┘         │   (Free)    │  │
│                                                   └─────────────┘  │
│                                                                     │
│   - 각 프로세스는 동일한 가상 주소 체계 사용                        │
│   - MMU(Memory Management Unit)가 실시간 변환                       │
│   - 프로세스 간 메모리 격리 (보안)                                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
가상 메모리의 핵심 장점은 무엇인가요?
장점 설명 게임에서의 의미
메모리 격리 프로세스 간 침범 불가 치트 방지의 기본

Part 2: 페이징 시스템

가상 주소가 물리 주소로 어떻게 변환되나요?

페이지 테이블을 통해 변환됩니다. 보통 4KB 단위입니다.

┌─────────────────────────────────────────────────────────────────────┐
│                      주소 변환 과정 (4KB 페이지)                     │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   가상 주소 (64비트 예시: 0x00007FFF12345ABC)                        │
│   ┌──────────────────────────────────────────────────────────────┐ │
│   │ 사용 안 함 │  PML4  │  PDPT  │   PD   │   PT   │  Offset  │ │
│   │   16bit   │  9bit  │  9bit  │  9bit  │  9bit  │   12bit  │ │
│   └──────────────────────────────────────────────────────────────┘ │
│                   │         │        │        │         │          │
│                   ▼         ▼        ▼        ▼         ▼          │
│               ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐              │
│   CR3 ───────▶│ PML4  │─▶│ PDPT  │─▶│  PD   │─▶│  PT   │──┐        │
│               │ Table │ │ Table │ │ Table │ │ Table │  │        │
│               └───────┘ └───────┘ └───────┘ └───────┘  │        │
│                                                        │        │
│                                     Physical Frame ◀───┘        │
│                                     Number (PFN)                │
│                                           │                     │
│                                           ▼                     │
│   물리 주소 = (PFN × 4096) + Offset                              │
│            = 0x0000000087654ABC                                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
수동 주소 변환 예시
// 가상 주소에서 페이지 번호와 오프셋 추출
#define PAGE_SIZE 4096
#define PAGE_SHIFT 12
#define PAGE_MASK (~(PAGE_SIZE - 1))

uint64_t virtualAddress = 0x00007FFF12345ABC;

// 페이지 번호 (상위 비트)
uint64_t pageNumber = virtualAddress >> PAGE_SHIFT;
// = 0x7FFF12345

// 페이지 내 오프셋 (하위 12비트)
uint64_t offset = virtualAddress & (PAGE_SIZE - 1);
// = 0xABC (2748)

// 물리 주소 계산 (페이지 테이블에서 PFN 조회 후)
uint64_t physicalFrameNumber = 0x87654;  // 페이지 테이블에서 조회
uint64_t physicalAddress = (physicalFrameNumber << PAGE_SHIFT) | offset;
// = 0x87654ABC
4단계 페이지 테이블은 왜 필요한가요?

64비트 주소 공간을 단일 테이블로 관리하면 테이블 크기가 너무 큽니다.

🎯 비유로 이해하기

전화번호부를 생각해보세요.

모든 번호를 한 페이지에 나열하면 책이 너무 두꺼워집니다. 대신 "시→구→동→번지" 순으로 계층화하면 필요한 부분만 펼치면 됩니다.

방식 필요 메모리 특징
단일 테이블 256TB (불가능) 이론상 불가
4단계 페이징 수 KB ~ 수 MB 사용 영역만 할당

Part 3: 스와핑과 페이지 폴트

RAM보다 큰 메모리를 쓸 수 있는 원리는?

스와핑(Swapping): 당장 안 쓰는 메모리를 디스크에 저장합니다.

┌─────────────────────────────────────────────────────────────────────┐
│                      페이지 폴트 처리 과정                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   1. CPU가 가상 주소 접근                                           │
│      ┌─────────────┐                                               │
│      │   CPU       │───▶ 가상 주소 0x12345000                      │
│      └─────────────┘                                               │
│             │                                                       │
│             ▼                                                       │
│   2. MMU가 페이지 테이블 조회                                       │
│      ┌─────────────┐                                               │
│      │ Page Table  │  Present bit = 0 (메모리에 없음!)             │
│      │  Entry      │───────────────────────────────────────┐       │

│                                                            │       │

│      │ (Swap)      │     │ (Free Frame)│     │ Update      │      │
│      └─────────────┘     └─────────────┘     └─────────────┘      │

└─────────────────────────────────────────────────────────────────────┘

페이지 폴트는 매우 느립니다! RAM 접근: ~100ns, 디스크: ~10ms (10만배 차이)

게임에서 스와핑이 발생하면 어떻게 되나요?

스터터링(Stuttering)이 발생합니다. 프레임이 갑자기 끊기는 현상입니다.

스와핑 방지 기법 (Windows)
// 중요한 메모리를 물리 RAM에 고정 (Pinning)
#include 

void* criticalBuffer = VirtualAlloc(
    NULL,
    1024 * 1024 * 100,  // 100MB
    MEM_COMMIT | MEM_RESERVE,
    PAGE_READWRITE
);

// 물리 메모리에 고정 (스왑 아웃 방지)
BOOL success = VirtualLock(criticalBuffer, 1024 * 1024 * 100);

if (!success) {
    // 권한 부족 또는 메모리 부족
    DWORD error = GetLastError();
}

// 사용 완료 후 해제
VirtualUnlock(criticalBuffer, 1024 * 1024 * 100);
VirtualFree(criticalBuffer, 0, MEM_RELEASE);
Linux에서의 메모리 잠금
#include 

void* buffer = mmap(NULL, size, PROT_READ | PROT_WRITE,
                    MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);

// 물리 메모리에 잠금
if (mlock(buffer, size) == -1) {
    perror("mlock failed");  // 권한 필요 (CAP_IPC_LOCK)
}

// 또는 전체 프로세스 메모리 잠금
mlockall(MCL_CURRENT | MCL_FUTURE);

// 해제
munlock(buffer, size);
munmap(buffer, size);

게임에서는 텍스처, 사운드 등 중요 에셋을 mlock/VirtualLock으로 고정합니다.

게임에서 주로 사용하는 할당 패턴은?
풀 할당자 (Pool Allocator)
// 동일 크기 객체를 위한 고속 할당자
template
class PoolAllocator {
private:
    struct Block {
        Block* next;
    };

    alignas(T) char memory_[sizeof(T) * PoolSize];
    Block* freeList_ = nullptr;

public:
    PoolAllocator() {
        // 모든 블록을 free list로 연결
        for (size_t i = 0; i < PoolSize; ++i) {
            Block* block = reinterpret_cast(&memory_[i * sizeof(T)]);
            block->next = freeList_;
            freeList_ = block;
        }
    }

    T* allocate() {
        if (!freeList_) return nullptr;  // 풀 고갈

        Block* block = freeList_;
        freeList_ = block->next;
        return reinterpret_cast(block);
    }

    void deallocate(T* ptr) {
        Block* block = reinterpret_cast(ptr);
        block->next = freeList_;
        freeList_ = block;
    }
};

// 사용 예시
PoolAllocator bulletPool;

void SpawnBullet() {
    Bullet* bullet = bulletPool.allocate();
    new (bullet) Bullet();  // placement new
}

void DestroyBullet(Bullet* bullet) {
    bullet->~Bullet();
    bulletPool.deallocate(bullet);
}
프레임 할당자 (Frame/Linear Allocator)
// 매 프레임 리셋되는 임시 메모리용
class FrameAllocator {
private:
    char* buffer_;
    size_t capacity_;
    size_t offset_ = 0;

public:
    explicit FrameAllocator(size_t capacity)
        : capacity_(capacity) {
        buffer_ = static_cast(malloc(capacity));
    }

    ~FrameAllocator() { free(buffer_); }

    void* allocate(size_t size, size_t alignment = 8) {
        // 정렬 맞추기
        size_t aligned = (offset_ + alignment - 1) & ~(alignment - 1);

        if (aligned + size > capacity_) {
            return nullptr;  // 메모리 부족
        }

        void* ptr = buffer_ + aligned;
        offset_ = aligned + size;
        return ptr;
    }

    // 프레임 끝에 한 번에 리셋!
    void reset() { offset_ = 0; }

    // 개별 해제는 없음 (No-op)
};

// 사용 예시
FrameAllocator frameAlloc(1024 * 1024);  // 1MB

void GameLoop() {
    // 이번 프레임 임시 데이터
    auto* tempData = frameAlloc.allocate(sizeof(TempStruct));

    // ... 프레임 로직 ...

    // 프레임 끝: 한 번에 정리
    frameAlloc.reset();
}

풀 할당자는 동일 객체 대량 생성에, 프레임 할당자는 임시 데이터에 적합합니다.

단편화를 줄이는 방법은?
Slab Allocator (크기별 풀)
// 크기별로 별도 풀을 관리하여 단편화 최소화
class SlabAllocator {
private:
    // 크기별 풀 (16, 32, 64, 128, 256, 512, 1024 바이트)
    static constexpr size_t SLAB_SIZES[] = {16, 32, 64, 128, 256, 512, 1024};
    static constexpr size_t NUM_SLABS = sizeof(SLAB_SIZES) / sizeof(size_t);

    struct Slab {
        void* freeList;
        size_t blockSize;
        size_t capacity;
    };

    Slab slabs_[NUM_SLABS];

    size_t findSlabIndex(size_t size) {
        for (size_t i = 0; i < NUM_SLABS; ++i) {
            if (size <= SLAB_SIZES[i]) return i;
        }
        return NUM_SLABS;  // 대형 할당
    }

public:
    void* allocate(size_t size) {
        size_t idx = findSlabIndex(size);

        if (idx == NUM_SLABS) {
            // 대형 할당은 직접 mmap
            return mmap(NULL, size, PROT_READ | PROT_WRITE,
                       MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);
        }

        return allocateFromSlab(idx);
    }

    void deallocate(void* ptr, size_t size) {
        size_t idx = findSlabIndex(size);

        if (idx == NUM_SLABS) {
            munmap(ptr, size);
            return;
        }

        returnToSlab(idx, ptr);
    }
};

Part 6: 실전 메모리 최적화

게임에서 메모리 사용을 프로파일링하는 방법은?
커스텀 할당 추적기
// 메모리 할당 추적을 위한 래퍼
class MemoryTracker {
private:
    struct AllocationInfo {
        size_t size;
        const char* file;
        int line;
        const char* category;
    };

    std::unordered_map allocations_;
    std::unordered_map categoryTotals_;
    std::mutex mutex_;
    size_t totalAllocated_ = 0;
    size_t peakAllocated_ = 0;

public:
    static MemoryTracker& instance() {
        static MemoryTracker tracker;
        return tracker;
    }

    void* allocate(size_t size, const char* file, int line,
                   const char* category = "default") {
        void* ptr = malloc(size);

        std::lock_guard lock(mutex_);
        allocations_[ptr] = {size, file, line, category};
        totalAllocated_ += size;
        categoryTotals_[category] += size;
        peakAllocated_ = std::max(peakAllocated_, totalAllocated_);

        return ptr;
    }

    void deallocate(void* ptr) {
        std::lock_guard lock(mutex_);

        auto it = allocations_.find(ptr);
        if (it != allocations_.end()) {
            totalAllocated_ -= it->second.size;
            categoryTotals_[it->second.category] -= it->second.size;
            allocations_.erase(it);
        }

        free(ptr);
    }

    void printReport() {
        std::lock_guard lock(mutex_);

        printf("=== Memory Report ===\n");
        printf("Current: %zu bytes\n", totalAllocated_);
        printf("Peak: %zu bytes\n", peakAllocated_);
        printf("\nBy Category:\n");

        for (const auto& [category, size] : categoryTotals_) {
            printf("  %s: %zu bytes\n", category.c_str(), size);
        }

        if (!allocations_.empty()) {
            printf("\nPotential Leaks:\n");
            for (const auto& [ptr, info] : allocations_) {
                printf("  %p: %zu bytes at %s:%d\n",
                       ptr, info.size, info.file, info.line);
            }
        }
    }
};

// 매크로로 편리하게 사용
#define TRACKED_NEW(type, category) \
    new (MemoryTracker::instance().allocate(sizeof(type), __FILE__, __LINE__, category)) type

#define TRACKED_DELETE(ptr, category) \
    do { (ptr)->~decltype(*ptr)(); \
         MemoryTracker::instance().deallocate(ptr); } while(0)

📌 핵심 정리

  • 가상 메모리: 프로세스 격리, 물리 메모리 초과 사용 가능
  • 페이징: 4KB 단위로 가상→물리 주소 변환, 다단계 테이블로 효율화
