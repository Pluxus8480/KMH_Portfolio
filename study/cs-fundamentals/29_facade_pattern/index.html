퍼사드 패턴(Facade Pattern) - 복잡한 서브시스템을 단순한 인터페이스로 감싸는 기법

Part 1: 퍼사드 패턴이란?

퍼사드 패턴은 어떤 패턴인가요?

퍼사드 패턴의 주 사용처는 '객체 그룹에 간단하지만 구체적인 인터페이스'를 제공하는 경우입니다.

복잡한 서브시스템들을 하나의 간단한 인터페이스 뒤에 숨겨서, 클라이언트가 쉽게 사용할 수 있도록 만드는 구조 패턴입니다.

쉽게 비유하면?
비유로 이해하기

온라인 쇼핑몰에서 "물건 구매" 버튼을 누르면, 내부적으로는 창고에서 물건 꺼내기, 송장번호 붙이기, 발송지로 보내기, 배달 등의 복잡한 과정이 진행됩니다.

하지만 사용자는 단순히 "구매" 버튼만 누르면 됩니다. 이 "구매" 버튼이 바로 퍼사드입니다.

Part 2: 퍼사드 패턴의 구조

구조를 그림으로 보면 어떻게 되나요?
Client
간단한 인터페이스 호출
→
Facade
물건구매() 인터페이스
→
서브시스템들
창고 / 송장 / 발송 / 배달

퍼사드는 내부의 하위 클래스들을 조합해서, 사용자가 알 수 없게 복잡한 과정을 처리합니다.

코드 예시를 보여주세요.
C++
// 서브시스템 클래스들
class Warehouse {
public:
    void PickItem(const std::string& item) { /* 창고에서 물건 꺼내기 */ }
};

class Shipping {
public:
    void AttachInvoice(int orderId) { /* 송장번호 붙이기 */ }
    void SendToDestination() { /* 발송지로 보내기 */ }
};

class Delivery {
public:
    void Deliver() { /* 배달 수행 */ }
};

// 퍼사드 클래스 - 단순한 인터페이스 제공
class ShopFacade {
private:
    Warehouse warehouse;
    Shipping shipping;
    Delivery delivery;
public:
    void Purchase(const std::string& item, int orderId) {
        warehouse.PickItem(item);
        shipping.AttachInvoice(orderId);
        shipping.SendToDestination();
        delivery.Deliver();
    }
};

// 클라이언트는 이것만 호출하면 됨
ShopFacade shop;
shop.Purchase("GamePad", 12345);

Part 3: 장단점과 면접 포인트

퍼사드 패턴의 장단점은?
구분 내용
장점 복잡한 시스템으로부터 간단한 인터페이스를 만들 수 있다. 인터페이스 분리를 통해 클라이언트와 실제 역할을 하는 클래스 간 결합도를 낮출 수 있다.
단점 퍼사드 클래스가 지나치게 비대해질 경우 유지보수가 어려워진다. 사용자에게 하위 클래스(일을 하는 클래스)를 완전히 숨길 수는 없다.

퍼사드가 너무 많은 책임을 지면 "God Object"가 될 수 있으므로, 적절한 크기를 유지하는 것이 중요합니다.

어댑터 패턴과는 뭐가 다른가요?
구분 어댑터 패턴 퍼사드 패턴
목적 인터페이스 변환 인터페이스 단순화
대상 하나의 클래스 여러 서브시스템
핵심 호환성 확보 복잡도 감소

핵심 정리

  • 퍼사드 패턴은 복잡한 서브시스템들을 하나의 간단한 인터페이스로 감싸는 구조 패턴이다
  • 클라이언트와 서브시스템 간의 결합도를 낮추고, 사용 편의성을 높인다
  • 퍼사드 클래스가 비대해지지 않도록 주의해야 한다
  • 어댑터 패턴은 "인터페이스 변환", 퍼사드 패턴은 "인터페이스 단순화"가 목적이다
