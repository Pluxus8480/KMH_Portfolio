프록시 패턴(Proxy Pattern) - 객체 접근을 제어하는 대리인 클래스

Part 1: 프록시 패턴이란?

프록시 패턴은 어떤 패턴인가요?

Proxy는 "대리인, 대리권, 대리"라는 뜻입니다. 프록시 클래스는 실제 클래스를 이어주는 대리인 역할을 합니다.

객체를 직접 호출하는 것이 아닌, 프록시 클래스를 통해 간접적으로 접근하는 구조 패턴입니다.

비유로 설명해주세요.
비유로 이해하기

연예인에게 직접 연락하는 대신 매니저(프록시)를 통해 연락하는 것과 같습니다.

매니저는 스케줄 확인, 접근 제어, 사전 준비 등의 역할을 하고, 실제 업무는 연예인이 수행합니다.

Part 2: 프록시 패턴의 구조

구조는 어떻게 되나요?
Client
Subject 인터페이스 사용
→
Proxy
접근 제어 + RealSubject 호출
→
RealSubject
실제 기능 수행

클라이언트는 Proxy를 통해 RealSubject에 접근하며, Proxy가 접근 제어, 캐싱, 지연 로딩 등의 부가 기능을 수행합니다.

Part 3: 장단점

프록시 패턴의 장단점은?
구분 내용
장점 실제 일을 하는 클래스와 구분 지을 수 있음. 사이즈가 큰 객체가 있을 시 로딩 전에 프록시 객체를 통해 참조 가능.
단점 로직이 복잡해지면서 코드 읽기가 좋지 않을 수 있음. 객체 생성 시 단계가 늘어나 빈번한 생성/삭제에서 메모리 부담 증가.

가급적 다른 해결법이 있다면 프록시 패턴보다 단순한 방법을 먼저 고려하는 것이 좋습니다.

Part 4: C++에서의 프록시 패턴 사례

C++ 표준 라이브러리에도 프록시 패턴이 있나요?

네, 대표적으로 두 가지가 있습니다.

1. vector<bool>

vector<bool>은 일반 vector와 달리 비트 단위로 저장하기 때문에, operator[]가 실제 bool 참조가 아닌 프록시 객체를 반환합니다.

2. 스마트 포인터

스마트 포인터는 일반 포인터의 인터페이스를 유지하면서(operator->, operator*) 내부적으로 참조 카운팅, 메모리 해제 등의 부가 작업을 수행합니다.

스마트 포인터는 실제 포인터의 "대리인"으로, 메모리 관리라는 부가 기능을 투명하게 제공하는 프록시 패턴의 대표적 사례입니다.

C++ - 스마트 포인터가 프록시 패턴인 이유
// 일반 포인터처럼 사용하지만 내부적으로 메모리 관리 수행
std::shared_ptr<GameObject> obj = std::make_shared<GameObject>();

// operator-> 를 통해 실제 객체에 접근 (프록시 패턴)
obj->Update();

// 스코프를 벗어나면 자동으로 메모리 해제 (부가 기능)

핵심 정리

  • 프록시 패턴은 실제 객체에 대한 접근을 제어하는 대리인 클래스를 사용하는 구조 패턴이다
  • 접근 제어, 지연 로딩, 캐싱 등의 부가 기능을 투명하게 추가할 수 있다
  • C++의 스마트 포인터와 vector<bool>이 프록시 패턴의 대표적 사례이다
  • 로직이 복잡해질 수 있으므로, 단순한 대안이 있다면 먼저 고려해야 한다
