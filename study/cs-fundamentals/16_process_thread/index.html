프로세스와 스레드의 차이, 멀티 프로세스 vs 멀티 스레드

Part 1: 프로세스와 스레드, 무엇이 다른가?

프로세스와 스레드의 차이가 뭔가요?

프로세스는 OS에게서 받은 작업의 단위이며, 스레드는 할당받은 자원을 사용하는 실행 단위입니다.

프로세스는 코드, 데이터, 힙, 스택 자원을 각자 할당받으며 다른 프로세스와 아무 관계가 없습니다. 다른 프로세스가 종료되더라도 공유하는 부분의 손상이 없다면 아무 영향을 주지 않습니다.

스레드의 경우 프로세스 안에서 실행되는 단위로 스택만을 따로 가지며 코드, 데이터, 힙은 다른 스레드들과 공유합니다.

🎯 비유로 이해하기

프로세스는 각자 독립된 사무실을 가진 회사원이고, 스레드는 같은 사무실(프로세스)에서 일하는 직원들입니다.

같은 사무실의 직원들은 프린터, 냉장고 등을 공유하지만, 각자의 책상(스택)은 따로 씁니다.

Part 2: 멀티 프로세스 vs 멀티 스레드

왜 멀티 스레드를 더 자주 사용하나요?

프로세스는 무겁고 스레드는 상대적으로 가볍기 때문입니다.

구분 멀티 프로세스 멀티 스레드
자원 생성 코드/데이터/힙/스택 모두 새로 할당 새 스택만 할당 (시스템 콜 감소)
Context Switching 무거움 (모든 영역 교체) 가벼움 (스택만 교체)
안정성 독립적이라 안정적 하나의 스레드 오류가 전체에 영향
동기화 IPC 필요 동기화 문제 관리 필요

멀티 스레드 환경에서는 여러 동기화 문제가 있으므로 이에 대한 관리를 잘 해줘야 하며, 면접에서도 관련 질문이 많이 나옵니다.

Part 3: 면접 포인트

프로세스와 스레드에서 자주 나오는 면접 질문은?

스레드가 스택만 따로 가지는 이유: 스레드는 독립적인 실행 흐름을 가지므로 함수 호출 스택이 독립적이어야 합니다. 반면 코드/데이터/힙을 공유함으로써 효율적인 협력이 가능합니다.

Context Switching이란: CPU가 현재 작업 중인 프로세스/스레드의 상태를 저장하고 다른 프로세스/스레드의 상태를 로드하는 과정입니다.

📌 핵심 정리

  • 프로세스는 OS로부터 받은 작업 단위, 스레드는 실행 단위
  • 프로세스는 코드/데이터/힙/스택을 독립 할당, 스레드는 스택만 독립
  • 멀티 스레드는 Context Switching이 가볍고 시스템 콜이 적어 효율적
  • 멀티 스레드는 동기화 문제(경쟁 조건, 데드락 등) 관리가 필수
