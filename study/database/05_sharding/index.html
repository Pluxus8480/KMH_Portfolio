Part 4: 샤딩 전략

Range 샤딩과 Hash 샤딩의 차이는?
Range Sharding
// 유저 ID 범위로 샤드 결정
int GetShardByRange(int userId) {
    if (userId < 1000000)  return 0;  // Shard 0: 1~100만
    if (userId < 2000000)  return 1;  // Shard 1: 100만~200만
    if (userId < 3000000)  return 2;  // Shard 2: 200만~300만
    return 3;
}

// 장점: 범위 쿼리 효율적
// 단점: 최신 샤드에 부하 집중, 불균형
Hash Sharding
// 해시 함수로 샤드 결정
int GetShardByHash(int userId, int numShards) {
    return hash(userId) % numShards;
}

// 장점: 균등 분산
// 단점: 범위 쿼리 어려움, 샤드 추가 시 리밸런싱
Consistent Hashing (권장)
// 샤드 추가/제거 시 최소한의 데이터만 이동
class ConsistentHash {
    std::map ring;  // hash → shard_id
    int virtualNodes = 100;

public:
    void AddShard(int shardId) {
        for (int i = 0; i < virtualNodes; i++) {
            uint32_t hash = hashFunc("shard_" + to_string(shardId) +
                                     "_" + to_string(i));
            ring[hash] = shardId;
        }
    }

    int GetShard(int userId) {
        uint32_t hash = hashFunc(to_string(userId));
        auto it = ring.lower_bound(hash);
        if (it == ring.end()) it = ring.begin();
        return it->second;
    }
};

// 샤드 추가 시: 1/N만 이동 (vs 기존 해시: 거의 전부 이동)

Part 5: 크로스 샤드 문제

다른 샤드의 유저와 거래하려면 어떻게 하나요?

크로스 샤드 트랜잭션은 2PC(Two-Phase Commit) 또는 Saga 패턴이 필요합니다.

크로스 샤드 거래 (Saga 패턴):

Shard A
1. A 골드 차감 (-1000)
(임시 차감, 보상 트랜잭션 준비)
3. A 커밋 (차감 확정)
Shard B
2. B 아이템 이동
(성공 시 커밋)
↓
완료!
양쪽 모두 커밋 성공

만약 2단계 실패 시:

보상 트랜잭션
A 골드 복구 (+1000)

Saga 패턴: 각 단계가 실패하면 보상 트랜잭션으로 롤백

Saga 패턴 구현 예시
class CrossShardTrade {
public:
    void ExecuteTrade(int userA, int userB, int itemId, int gold) {
        try {
            // Step 1: A의 골드 임시 차감
            shardRouter.Execute(userA,
                "UPDATE users SET gold = gold - ?, pending_gold = ? WHERE id = ?",
                gold, gold, userA);

            // Step 2: 아이템 이동 (A → B)
            shardRouter.Execute(userA,
                "DELETE FROM inventory WHERE user_id = ? AND item_id = ?",
                userA, itemId);

            shardRouter.Execute(userB,
                "INSERT INTO inventory (user_id, item_id) VALUES (?, ?)",
                userB, itemId);

            // Step 3: B의 골드 추가, A의 pending 정리
            shardRouter.Execute(userB,
                "UPDATE users SET gold = gold + ? WHERE id = ?",
                gold, userB);

            shardRouter.Execute(userA,
                "UPDATE users SET pending_gold = 0 WHERE id = ?",
                userA);

        } catch (Exception& e) {
            // 보상 트랜잭션 실행
            CompensateTrade(userA, userB, itemId, gold);
            throw;
        }
    }

    void CompensateTrade(int userA, int userB, int itemId, int gold) {
        // 실패 지점에 따라 롤백
        // A 골드 복구
        shardRouter.Execute(userA,
            "UPDATE users SET gold = gold + pending_gold, pending_gold = 0 WHERE id = ?",
            userA);
        // 아이템 복구 등...
    }
};

크로스 샤드 트랜잭션은 복잡하고 느립니다. 가능하면 피하는 설계가 좋습니다.

// 주기적으로 각 샤드에서 데이터 수집
void AggregateRankingDaily() {
    vector allScores;

    for (auto& shard : shards) {
        auto scores = shard.Query(
            "SELECT char_id, level, exp FROM characters ORDER BY level DESC LIMIT 1000"
        );
        allScores.insert(allScores.end(), scores.begin(), scores.end());
    }

    // 전체 정렬 후 글로벌 DB에 저장
    sort(allScores);
    globalDb.ReplaceRanking(allScores);
}

📌 핵심 정리

  • 파티셔닝: 단일 DB 내 테이블 분할, 쿼리 최적화 목적
  • 샤딩: 여러 DB 서버에 데이터 분산, 용량/부하 분산 목적
  • 샤딩 키: 유저 ID가 보통 최선 (균등 분산, 데이터 지역성)
  • Consistent Hashing: 샤드 추가/제거 시 최소 데이터 이동
  • 크로스 샤드: Saga 패턴으로 처리, 가능하면 피하는 설계
  • 글로벌 데이터: 별도 DB 또는 Redis로 집계
