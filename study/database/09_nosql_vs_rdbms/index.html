"우리 게임에는 어떤 DB가 맞을까?"

NoSQL vs RDBMS 게임별 선택 전략

Part 1: 왜 선택이 어려운가?

게임마다 최적의 DB가 다른 이유는 무엇인가요?

게임 장르에 따라 데이터 특성이 완전히 다릅니다.

┌─────────────────────────────────────────────────────────────────────┐
│                     게임 장르별 데이터 특성                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  MMORPG                           │  캐주얼/소셜 게임               │
│  ├── 복잡한 관계 (길드, 거래, 퀘스트)  │  ├── 단순한 구조             │
│  ├── 트랜잭션 중요 (아이템 거래)       │  ├── 읽기 중심               │
│  ├── 스키마 안정적                    │  ├── 빠른 스키마 변경 필요    │
│  └── → RDBMS 유리                    │  └── → NoSQL 유리            │
│                                                                     │
│  실시간 PvP                         │  분석/로그                     │
│  ├── 초고속 읽기/쓰기                  │  ├── 대량 쓰기               │
│  ├── 단순 키-값 접근                   │  ├── 집계 쿼리               │
│  ├── 일시적 데이터                     │  ├── 스키마리스              │
│  └── → Redis/메모리 DB               │  └── → Document DB           │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

은탄환은 없습니다. 데이터 특성을 먼저 분석하세요.

RDBMS와 NoSQL의 근본적 차이는?
특성 RDBMS NoSQL
스키마 고정 (Schema-on-Write) 유연 (Schema-on-Read)
관계 JOIN으로 표현 비정규화/임베딩
일관성 ACID 보장 Eventually Consistent
확장 수직 확장 (Scale-Up) 수평 확장 (Scale-Out)
쿼리 SQL (강력한 조회) 제한적 (키 기반)

MMORPG 아이템 거래 - RDBMS가 필수인 이유
-- 아이템 거래는 ACID가 보장되어야 함
BEGIN TRANSACTION;

-- 1. 판매자 인벤토리에서 아이템 제거
UPDATE inventory
SET quantity = quantity - 1
WHERE character_id = @seller_id AND item_id = @item_id;

-- 2. 구매자 인벤토리에 아이템 추가
INSERT INTO inventory (character_id, item_id, quantity)
VALUES (@buyer_id, @item_id, 1)
ON DUPLICATE KEY UPDATE quantity = quantity + 1;

-- 3. 골드 이동
UPDATE currency SET gold = gold + @price WHERE character_id = @seller_id;
UPDATE currency SET gold = gold - @price WHERE character_id = @buyer_id;

-- 4. 거래 기록
INSERT INTO trade_log (seller_id, buyer_id, item_id, price, trade_time)
VALUES (@seller_id, @buyer_id, @item_id, @price, NOW());

COMMIT;

Redis로 매치 상태 관리
// 매치 상태 저장 (Redis)
class MatchStateManager {
    constructor(redisClient) {
        this.redis = redisClient;
    }

    // 플레이어 상태 업데이트 (초당 수십회)
    async updatePlayerState(matchId, playerId, state) {
        const key = `match:${matchId}:player:${playerId}`;
        await this.redis.hSet(key, {
            x: state.x,
            y: state.y,
            health: state.health,
            kills: state.kills,
            alive: state.alive ? 1 : 0
        });
        await this.redis.expire(key, 3600); // 1시간 후 자동 삭제
    }

    // 생존자 수 확인
    async getAlivePlayers(matchId) {
        const pattern = `match:${matchId}:player:*`;
        const keys = await this.redis.keys(pattern);

        let aliveCount = 0;
        for (const key of keys) {
            const alive = await this.redis.hGet(key, 'alive');
            if (alive === '1') aliveCount++;
        }
        return aliveCount;
    }

    // 매치 종료 시 결과 영구 저장
    async finalizeMatch(matchId) {
        const results = await this.getMatchResults(matchId);

        // MySQL에 영구 저장
        await this.saveToMySQL(results);

        // Redis 정리
        const keys = await this.redis.keys(`match:${matchId}:*`);
        if (keys.length > 0) {
            await this.redis.del(keys);
        }
    }
}

여러 DB 간 데이터 동기화는 어떻게 하나요?

이벤트 기반 동기화가 일반적입니다.

Kafka를 이용한 DB 동기화
// 이벤트 발행 (MySQL 변경 시)
class UserService {
    async updateUserLevel(userId, newLevel) {
        // 1. MySQL 업데이트 (Source of Truth)
        await this.mysql.query(
            'UPDATE users SET level = ? WHERE id = ?',
            [newLevel, userId]
        );

        // 2. 이벤트 발행
        await this.kafka.produce('user-events', {
            type: 'USER_LEVEL_CHANGED',
            userId: userId,
            newLevel: newLevel,
            timestamp: Date.now()
        });
    }
}

// 이벤트 소비 (다른 DB들)
class CacheSync {
    async handleUserEvent(event) {
        if (event.type === 'USER_LEVEL_CHANGED') {
            // Redis 캐시 갱신
            await this.redis.hSet(
                `user:${event.userId}`,
                'level', event.newLevel
            );
        }
    }
}

class AnalyticsSync {
    async handleUserEvent(event) {
        if (event.type === 'USER_LEVEL_CHANGED') {
            // ClickHouse에 레벨업 이벤트 기록
            await this.clickhouse.insert('level_up_events', {
                user_id: event.userId,
                new_level: event.newLevel,
                event_time: new Date(event.timestamp)
            });
        }
    }
}

class SocialSync {
    async handleUserEvent(event) {
        if (event.type === 'USER_LEVEL_CHANGED') {
            // Neo4j 유저 노드 속성 업데이트
            await this.neo4j.run(
                'MATCH (u:User {id: $userId}) SET u.level = $level',
                {userId: event.userId, level: event.newLevel}
            );
        }
    }
}

이벤트 기반은 결국적 일관성(Eventual Consistency)입니다. 즉시 일관성이 필요하면 동기 호출을 사용하세요.

Part 5: 선택 체크리스트

실무에서 DB를 선택할 때 체크해야 할 것들은?
체크 항목 RDBMS 선택 NoSQL 선택
데이터 관계 복잡한 JOIN 필요 관계 단순/없음
트랜잭션 ACID 필수 일관성 양보 가능
스키마 변경 안정적, 변경 적음 자주 변경됨
쿼리 패턴 복잡한 검색, 집계 키 기반 접근
확장성 수직 확장으로 충분 대규모 수평 확장
팀 경험 SQL 숙련도 높음 NoSQL 경험 있음
결정 플로우차트 (의사 코드)
function chooseDatabase(requirements) {
    // 1. 트랜잭션 요구사항 체크
    if (requirements.needsACID) {
        if (requirements.complexJoins) {
            return "MySQL/PostgreSQL";
        }
        if (requirements.needsHorizontalScale) {
            return "CockroachDB/TiDB"; // NewSQL
        }
    }

    // 2. 데이터 구조 체크
    if (requirements.dataStructure === "key-value") {
        if (requirements.persistence === false) {
            return "Redis";
        }
        return "DynamoDB/Redis with AOF";
    }

    if (requirements.dataStructure === "document") {
        if (requirements.flexibleSchema) {
            return "MongoDB";
        }
    }

    if (requirements.dataStructure === "graph") {
        return "Neo4j/Amazon Neptune";
    }

    // 3. 접근 패턴 체크
    if (requirements.writeHeavy && requirements.timeSeriesData) {
        return "Cassandra/TimescaleDB";
    }

    if (requirements.analyticsQueries) {
        return "ClickHouse/BigQuery";
    }

    // 4. 기본값
    return "PostgreSQL"; // 범용적이고 기능이 풍부함
}

📌 핵심 정리

  • RDBMS: 복잡한 관계, 트랜잭션 필수, 안정적 스키마 → MMORPG 핵심 데이터
  • Document DB: 유연한 스키마, 단순 관계 → 모바일 게임, 설정 데이터
  • Key-Value: 초고속 접근, 단순 구조 → 캐시, 세션, 실시간 상태
  • Polyglot Persistence: 목적에 맞는 여러 DB 조합이 현실적 해답
  • 선택 기준: 데이터 관계, 트랜잭션 요구, 쿼리 패턴, 확장성 고려
