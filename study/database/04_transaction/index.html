트랜잭션 ACID와 게임에서의 동시성 제어

Part 1: 트랜잭션과 ACID

트랜잭션이 왜 필요한가요?

여러 작업을 하나의 단위로 묶어 안전하게 처리합니다.

트랜잭션 없을 때:

단계작업결과
1A의 골드 -1000성공
2B의 골드 +1000실패 (DB 에러)
3A의 아이템 제거실행 안 됨
4B의 아이템 추가실행 안 됨

결과: A만 골드 손실!

트랜잭션으로 처리:

BEGIN
→
A 골드 -1000
성공
→
B 골드 +1000
실패
→
ROLLBACK
모든 변경 취소, 원상복구
ACID가 무엇인가요?
속성 의미 게임 예시
Atomicity (원자성) 전부 성공 또는 전부 실패 거래: 양쪽 다 반영 or 둘 다 안 반영
Consistency (일관성) 규칙 위반 불가 골드 음수 불가, 아이템 중복 불가
Isolation (격리성) 동시 작업 간 간섭 없음 두 거래가 동시에 같은 아이템 접근
Durability (영속성) 커밋 후 영구 저장 서버 크래시 후에도 데이터 유지

Part 2: 동시성 문제들

동시에 같은 데이터를 수정하면 어떤 문제가 생기나요?

1. Lost Update (갱신 손실)

시간Transaction ATransaction B
t1READ gold = 1000
t2READ gold = 1000
t3WRITE gold = 1100 (+100)
t4WRITE gold = 800 (-200)

결과: A의 +100이 사라짐! (금액 오류)

2. Dirty Read (더티 리드)

시간Transaction ATransaction B
t1WRITE gold = 500
t2READ gold = 500
t3ROLLBACK (취소!)
t4gold = 500 사용...

결과: B가 존재하지 않는 데이터 사용!

3. Non-Repeatable Read (반복 불가능한 읽기)

시간Transaction ATransaction B
t1READ gold = 1000
t2WRITE gold = 500, COMMIT
t3READ gold = 500 ???

결과: 같은 트랜잭션에서 다른 값 읽음!

Part 3: 격리 수준 (Isolation Levels)

격리 수준이란?

동시성 문제를 얼마나 방지할지 결정하는 설정입니다.

격리 수준 Dirty Read Non-Repeatable Phantom 성능
READ UNCOMMITTED ⚠️ 가능 ⚠️ 가능 ⚠️ 가능 ⭐⭐⭐
READ COMMITTED ✅ 방지 ⚠️ 가능 ⚠️ 가능 ⭐⭐
REPEATABLE READ ✅ 방지 ✅ 방지 ⚠️ 가능 ⭐
SERIALIZABLE ✅ 방지 ✅ 방지 ✅ 방지 느림

MySQL InnoDB 기본값: REPEATABLE READ, 대부분의 게임에 적합

Part 4: 락(Lock) 전략

비관적 락(Pessimistic Lock)이란?

"충돌할 것"이라 가정하고, 먼저 락을 걸고 작업합니다.

비관적 락 예시 (FOR UPDATE)
-- 아이템 강화 (동시 강화 방지)
BEGIN TRANSACTION;

-- 아이템 행 락 획득 (다른 트랜잭션은 대기)
SELECT * FROM inventory
WHERE item_id = 12345
FOR UPDATE;

-- 강화 로직 실행
-- 강화석 확인, 성공/실패 판정 등...

UPDATE inventory
SET enchant_level = enchant_level + 1
WHERE item_id = 12345;

COMMIT;

-- FOR UPDATE: 해당 행을 쓰기 락
-- FOR SHARE: 해당 행을 읽기 락 (다른 쓰기 차단)
낙관적 락(Optimistic Lock)이란?

"충돌은 드물다"고 가정하고, 커밋 시 충돌 감지합니다.

낙관적 락 예시 (Version)
-- 테이블에 version 컬럼 필요
ALTER TABLE characters ADD COLUMN version INT DEFAULT 1;

-- 읽기
SELECT char_id, gold, version FROM characters
WHERE char_id = 1;
-- 결과: gold = 1000, version = 5

-- 업데이트 (version 조건 포함)
UPDATE characters
SET gold = 900, version = version + 1
WHERE char_id = 1 AND version = 5;

-- 영향받은 행 수 확인
-- 0이면 다른 트랜잭션이 먼저 수정함 → 재시도!

if (affected_rows == 0) {
    // 충돌 발생! 재시도 또는 에러 처리
    throw OptimisticLockException();
}
특성 비관적 락 낙관적 락
락 시점 읽기 전 쓰기 시
충돌 시 대기 재시도
적합 충돌 많을 때 충돌 드물 때
게임 예시 재화 거래, 강화 캐릭터 정보 수정

Part 5: 데드락 방지

데드락이 뭔가요?

두 트랜잭션이 서로의 락을 기다리며 영원히 멈추는 상황입니다.

데드락 시나리오 (A-B 거래)

단계Transaction ATransaction B
1LOCK 아이템1 (성공)LOCK 아이템2 (성공)
2LOCK 아이템2 대기...LOCK 아이템1 대기...
3B가 해제해야 함A가 해제해야 함
4무한 대기무한 대기

DEADLOCK! 서로가 서로의 락을 기다리며 영원히 멈춤

데드락을 어떻게 방지하나요?
데드락 방지: 락 순서 고정
-- BAD: 락 순서가 일정하지 않음
-- Transaction A: Lock(아이템1) → Lock(아이템2)
-- Transaction B: Lock(아이템2) → Lock(아이템1)

-- GOOD: 항상 ID 오름차순으로 락
void TradeItems(int itemA, int itemB) {
    int first = min(itemA, itemB);
    int second = max(itemA, itemB);

    BEGIN TRANSACTION;

    SELECT * FROM inventory WHERE item_id = first FOR UPDATE;
    SELECT * FROM inventory WHERE item_id = second FOR UPDATE;

    // 거래 로직...

    COMMIT;
}

-- 모든 트랜잭션이 같은 순서로 락 → 데드락 불가!
데드락 방지: 타임아웃
-- MySQL: 락 대기 타임아웃 설정
SET innodb_lock_wait_timeout = 5;  -- 5초

-- 5초 후에도 락 획득 못하면 에러 발생
-- 에러: Lock wait timeout exceeded

-- 애플리케이션에서 재시도
try {
    ExecuteTransaction();
} catch (LockTimeoutException) {
    // 잠시 후 재시도
    sleep(random(100, 500));
    ExecuteTransaction();
}

Part 6: 게임에서의 실전 패턴

재화 차감은 어떻게 안전하게 하나요?
안전한 재화 차감
-- BAD: Race Condition 가능
SELECT gold FROM characters WHERE char_id = 1;  -- 1000
-- 다른 요청이 여기서 끼어들 수 있음!
UPDATE characters SET gold = 1000 - 500 WHERE char_id = 1;

-- GOOD: 원자적 업데이트 + 조건 체크
UPDATE characters
SET gold = gold - 500
WHERE char_id = 1 AND gold >= 500;

-- affected_rows 확인
if (affected_rows == 0) {
    // 골드 부족 또는 존재하지 않는 캐릭터
    throw InsufficientGoldException();
}

-- BETTER: 트랜잭션 + 명시적 에러 처리
BEGIN TRANSACTION;

SELECT gold FROM characters WHERE char_id = 1 FOR UPDATE;

if (gold < 500) {
    ROLLBACK;
    throw InsufficientGoldException();
}

UPDATE characters SET gold = gold - 500 WHERE char_id = 1;

-- 로그 기록
INSERT INTO gold_log (char_id, change_type, amount, balance_after)
VALUES (1, 'spend', -500, gold - 500);

COMMIT;
선착순 아이템 지급은 어떻게 하나요?
선착순 100명 아이템 지급
-- 이벤트 참여 테이블
CREATE TABLE event_participants (
    event_id INT,
    char_id BIGINT,
    participated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (event_id, char_id)
);

-- 참여 시도 (동시 요청 안전하게 처리)
BEGIN TRANSACTION;

-- 현재 참여자 수 확인 (락)
SELECT COUNT(*) as cnt FROM event_participants
WHERE event_id = 1
FOR UPDATE;  -- 테이블 레벨 락 주의!

if (cnt >= 100) {
    ROLLBACK;
    throw EventFullException();
}

-- 참여 등록 (중복 시 에러)
INSERT INTO event_participants (event_id, char_id)
VALUES (1, @char_id);

-- 아이템 지급
INSERT INTO inventory (char_id, item_template_id, quantity)
VALUES (@char_id, 9999, 1);

COMMIT;

-- 더 효율적: INSERT 실패로 중복/수량 체크
INSERT INTO event_participants (event_id, char_id)
SELECT 1, @char_id
WHERE (SELECT COUNT(*) FROM event_participants WHERE event_id = 1) < 100;

-- affected_rows로 성공 여부 확인

📌 핵심 정리

  • ACID: 원자성, 일관성, 격리성, 영속성 - 트랜잭션의 4대 속성
  • Lost Update: 동시 수정 시 한쪽 변경이 사라짐 → 락으로 방지
  • 비관적 락: FOR UPDATE로 먼저 락, 충돌 많을 때 적합
  • 낙관적 락: version 체크, 충돌 시 재시도, 충돌 드물 때 적합
  • 데드락 방지: 락 순서 고정, 타임아웃 설정
  • 안전한 차감: WHERE 조건에 잔액 체크 포함
