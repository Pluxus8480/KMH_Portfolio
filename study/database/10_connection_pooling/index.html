"왜 DB 연결이 이렇게 느려요?"

커넥션 풀링의 원리와 최적화

Part 1: 커넥션의 비용

매 요청마다 DB 연결하면 왜 문제가 되나요?

DB 연결 수립은 생각보다 비용이 큽니다.

┌─────────────────────────────────────────────────────────────────────┐
│                    TCP 연결 + DB 인증 과정                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│    Client                                    DB Server              │
│       │                                          │                  │
│       │──────── 1. TCP SYN ─────────────────────▶│                  │
│       │◀─────── 2. TCP SYN-ACK ─────────────────│   TCP 3-Way      │
│       │──────── 3. TCP ACK ─────────────────────▶│   Handshake     │
│       │                                          │   (~1ms LAN)    │
│       │──────── 4. TLS ClientHello ────────────▶│                  │
│       │◀─────── 5. TLS ServerHello ─────────────│   TLS Handshake  │
│       │──────── 6. TLS Finished ────────────────▶│   (~5-10ms)     │
│       │                                          │                  │
│       │──────── 7. Auth Request ────────────────▶│                  │
│       │◀─────── 8. Auth Challenge ──────────────│   인증 과정      │
│       │──────── 9. Auth Response ───────────────▶│   (~5-10ms)     │
│       │◀─────── 10. Auth OK ────────────────────│                  │
│       │                                          │                  │
│       │──────── 11. Query ──────────────────────▶│   실제 쿼리     │
│       │◀─────── 12. Result ─────────────────────│   (~1ms)        │
│       │                                          │                  │
│   총 시간: 연결 15-20ms + 쿼리 1ms = 약 20ms                        │
│   커넥션 풀 사용 시: 쿼리 1ms만 소요!                                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

쿼리 자체보다 연결 수립이 10배 이상 오래 걸릴 수 있습니다!

Part 2: 커넥션 풀 기본 원리

커넥션 풀은 어떻게 동작하나요?

미리 연결을 만들어두고 재사용합니다.

┌─────────────────────────────────────────────────────────────────────┐
│                      커넥션 풀 동작 원리                             │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Application                    Connection Pool         Database   │
│                                                                     │
│   Request 1 ─────▶ ┌──────────────────────┐                        │
│                    │  ┌────┐ ┌────┐ ┌────┐ │    ┌──────────┐       │
│   Request 2 ─────▶ │  │Conn│ │Conn│ │Conn│ │───▶│          │       │
│                    │  │ 1  │ │ 2  │ │ 3  │ │    │    DB    │       │
│   Request 3 ─────▶ │  └────┘ └────┘ └────┘ │◀───│  Server  │       │
│                    │   Idle   Busy   Busy  │    │          │       │
│   Request 4 ─────▶ │                       │    └──────────┘       │
│     (대기)         │  Pool Size: 3         │                       │
│                    └──────────────────────┘                        │
│                                                                     │
│   흐름:                                                             │
│   1. Request 1 도착 → Conn 1 획득 → 쿼리 실행 → Conn 1 반환        │
│   2. Request 2 도착 → Conn 2 획득 → 쿼리 실행 중...                │
│   3. Request 3 도착 → Conn 3 획득 → 쿼리 실행 중...                │
│   4. Request 4 도착 → 풀 가득 참 → 대기열에서 기다림                │
│   5. Conn 2 반환됨 → Request 4가 Conn 2 획득                       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
🎯 비유로 이해하기

매번 새 차를 사서 목적지 가고 폐차하는 것 vs 렌터카 회사에서 차를 빌려 쓰고 반납하는 것

커넥션 풀은 렌터카 회사처럼 미리 차(연결)를 확보해두고 필요할 때 빌려주는 방식입니다.

C++ 커넥션 풀 구현 (게임 서버용)
class ConnectionPool {
private:
    std::queue<MYSQL*> available_;
    std::unordered_set<MYSQL*> inUse_;
    std::mutex mutex_;
    std::condition_variable cv_;

    size_t minSize_;
    size_t maxSize_;
    std::string host_, user_, password_, database_;

public:
    ConnectionPool(size_t minSize, size_t maxSize,
                   const std::string& host,
                   const std::string& user,
                   const std::string& password,
                   const std::string& database)
        : minSize_(minSize), maxSize_(maxSize)
        , host_(host), user_(user)
        , password_(password), database_(database)
    {
        // 최소 커넥션 미리 생성
        for (size_t i = 0; i < minSize_; ++i) {
            available_.push(createConnection());
        }
    }

    // 커넥션 획득 (RAII 패턴)
    class ScopedConnection {
        ConnectionPool* pool_;
        MYSQL* conn_;
    public:
        ScopedConnection(ConnectionPool* pool, MYSQL* conn)
            : pool_(pool), conn_(conn) {}

        ~ScopedConnection() {
            if (conn_) pool_->release(conn_);
        }

        MYSQL* get() { return conn_; }

        // Move only
        ScopedConnection(ScopedConnection&& other) noexcept
            : pool_(other.pool_), conn_(other.conn_) {
            other.conn_ = nullptr;
        }
    };

    ScopedConnection acquire(std::chrono::milliseconds timeout = 3000ms) {
        std::unique_lock<std::mutex> lock(mutex_);

        // 사용 가능한 커넥션 대기
        bool available = cv_.wait_for(lock, timeout, [this] {
            return !available_.empty() ||
                   (available_.size() + inUse_.size() < maxSize_);
        });

        if (!available) {
            throw std::runtime_error("Connection pool timeout");
        }

        MYSQL* conn = nullptr;
        if (!available_.empty()) {
            conn = available_.front();
            available_.pop();

            // 유효성 검사
            if (mysql_ping(conn) != 0) {
                mysql_close(conn);
                conn = createConnection();
            }
        } else {
            // 새 커넥션 생성 (maxSize 이내)
            conn = createConnection();
        }

        inUse_.insert(conn);
        return ScopedConnection(this, conn);
    }

    void release(MYSQL* conn) {
        std::lock_guard<std::mutex> lock(mutex_);
        inUse_.erase(conn);
        available_.push(conn);
        cv_.notify_one();
    }

private:
    MYSQL* createConnection() {
        MYSQL* conn = mysql_init(nullptr);
        if (!mysql_real_connect(conn,
                host_.c_str(), user_.c_str(),
                password_.c_str(), database_.c_str(),
                3306, nullptr, 0)) {
            throw std::runtime_error(mysql_error(conn));
        }
        return conn;
    }
};

// 사용 예시
void gameLogic(ConnectionPool& pool) {
    auto conn = pool.acquire();  // RAII - 스코프 벗어나면 자동 반환

    MYSQL_RES* result = nullptr;
    mysql_query(conn.get(), "SELECT * FROM users WHERE id = 1");
    result = mysql_store_result(conn.get());
    // ... 결과 처리
    mysql_free_result(result);
}  // 여기서 자동으로 release() 호출

Part 4: 풀 크기 최적화

풀 크기는 어떻게 정하나요?

유명한 공식이 있습니다: connections = (core_count * 2) + effective_spindle_count

하지만 게임 서버는 특수한 상황이 많아 실제 테스트가 중요합니다.

풀 크기 계산 가이드
# 기본 공식
pool_size = (CPU 코어 수 * 2) + (SSD면 1, HDD면 디스크 수)

# 예시: 8코어 CPU, SSD 사용
pool_size = (8 * 2) + 1 = 17

# 하지만 게임 서버는 추가 고려 필요:

# 1. 동시 요청 수 기반
# - 게임 서버당 동시 접속자: 1000명
# - 초당 평균 DB 요청: 접속자당 0.5회
# - 쿼리 평균 실행 시간: 5ms
#
# 필요 커넥션 = (500 요청/초) * (0.005초) = 2.5
# 여유 포함 = 2.5 * 4 = 10개

# 2. 피크 타임 고려
# - 피크 시 요청량 3배 증가
# 피크 필요 = 10 * 3 = 30개

# 최종: min=10, max=30 으로 시작 후 모니터링하며 조정

        Gauge.builder("hikari.connections.total", poolMXBean,
                HikariPoolMXBean::getTotalConnections)
            .register(registry);
    }
}

잘못된 코드 vs 올바른 코드
// ❌ 잘못된 코드 - 예외 발생 시 커넥션 누수
async function badCode(userId) {
    const conn = await pool.getConnection();
    const [user] = await conn.query('SELECT * FROM users WHERE id = ?', [userId]);

    if (!user) {
        throw new Error('User not found');  // conn.release() 호출 안 됨!
    }

    const [items] = await conn.query('SELECT * FROM inventory WHERE user_id = ?', [userId]);
    conn.release();  // 여기까지 도달 못 할 수 있음
    return { user, items };
}

// ✅ 올바른 코드 - try-finally로 항상 반환
async function goodCode(userId) {
    const conn = await pool.getConnection();
    try {
        const [user] = await conn.query('SELECT * FROM users WHERE id = ?', [userId]);

        if (!user) {
            throw new Error('User not found');
        }

        const [items] = await conn.query('SELECT * FROM inventory WHERE user_id = ?', [userId]);
        return { user, items };
    } finally {
        conn.release();  // 예외 발생해도 항상 실행
    }
}

// ✅ 더 좋은 코드 - pool.query 사용 (자동 반환)
async function betterCode(userId) {
    const [user] = await pool.query('SELECT * FROM users WHERE id = ?', [userId]);

    if (!user) {
        throw new Error('User not found');
    }

    const [items] = await pool.query('SELECT * FROM inventory WHERE user_id = ?', [userId]);
    return { user, items };
}
C++ RAII 패턴으로 누수 방지
// RAII (Resource Acquisition Is Initialization)
// 스코프 벗어나면 자동으로 리소스 해제

class ScopedConnection {
    ConnectionPool& pool_;
    MYSQL* conn_;

public:
    explicit ScopedConnection(ConnectionPool& pool)
        : pool_(pool), conn_(pool.acquire()) {}

    ~ScopedConnection() {
        if (conn_) {
            pool_.release(conn_);
        }
    }

    // 복사 금지
    ScopedConnection(const ScopedConnection&) = delete;
    ScopedConnection& operator=(const ScopedConnection&) = delete;

    // 이동만 허용
    ScopedConnection(ScopedConnection&& other) noexcept
        : pool_(other.pool_), conn_(other.conn_) {
        other.conn_ = nullptr;
    }

    MYSQL* operator->() { return conn_; }
    MYSQL* get() { return conn_; }
};

// 사용 예시 - 어떤 경로로 함수를 빠져나가도 안전
void processUser(ConnectionPool& pool, int userId) {
    ScopedConnection conn(pool);  // 획득

    // 쿼리 실행
    std::string query = "SELECT * FROM users WHERE id = " + std::to_string(userId);
    mysql_query(conn.get(), query.c_str());

    // 조건에 따른 조기 반환 - 문제 없음!
    if (someCondition) {
        return;  // ~ScopedConnection() 호출됨
    }

    // 예외 발생 - 문제 없음!
    if (errorCondition) {
        throw std::runtime_error("Error");  // ~ScopedConnection() 호출됨
    }

    // 정상 종료
}  // ~ScopedConnection() 호출됨

Spring 읽기/쓰기 분리 설정
@Configuration
public class ReadWriteSplitConfig {

    @Bean
    @Primary
    public DataSource routingDataSource() {
        Map targetDataSources = new HashMap<>();
        targetDataSources.put("master", masterDataSource());
        targetDataSources.put("replica", replicaDataSource());

        ReplicationRoutingDataSource routingDataSource =
            new ReplicationRoutingDataSource();
        routingDataSource.setTargetDataSources(targetDataSources);
        routingDataSource.setDefaultTargetDataSource(masterDataSource());

        return routingDataSource;
    }

    @Bean
    public DataSource masterDataSource() {
        HikariConfig config = new HikariConfig();
        config.setJdbcUrl("jdbc:mysql://master:3306/game_db");
        config.setMaximumPoolSize(20);  // 쓰기는 적게
        return new HikariDataSource(config);
    }

    @Bean
    public DataSource replicaDataSource() {
        HikariConfig config = new HikariConfig();
        // 여러 Replica 로드밸런싱
        config.setJdbcUrl("jdbc:mysql:loadbalance://replica1,replica2/game_db");
        config.setMaximumPoolSize(50);  // 읽기는 많이
        return new HikariDataSource(config);
    }
}

// 라우팅 결정 로직
public class ReplicationRoutingDataSource extends AbstractRoutingDataSource {

    @Override
    protected Object determineCurrentLookupKey() {
        // @Transactional(readOnly = true) 이면 replica
        return TransactionSynchronizationManager.isCurrentTransactionReadOnly()
            ? "replica" : "master";
    }
}

// 사용 예시
@Service
public class UserService {

    @Transactional(readOnly = true)  // → replica 풀 사용
    public User getUser(Long id) {
        return userRepository.findById(id);
    }

    @Transactional  // → master 풀 사용
    public void updateUser(User user) {
        userRepository.save(user);
    }
}

📌 핵심 정리

  • 연결 비용: TCP + TLS + 인증으로 20ms+. 커넥션 풀로 1ms 쿼리만 소요
  • 풀 크기: (CPU코어*2)+1 공식 참고, 실제론 모니터링하며 조정
  • 누수 방지: try-finally, RAII 패턴 필수. 개발 환경에서 leak detection 활성화
  • 모니터링: pending(대기 스레드)가 핵심 지표. 지속 발생 시 풀 확장
  • 읽기/쓰기 분리: 별도 풀 구성, @Transactional(readOnly)로 라우팅
