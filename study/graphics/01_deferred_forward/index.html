빛을 그리는 두 가지 철학

Deferred vs Forward 렌더링

Part 1: Forward Rendering의 원리

Forward Rendering은 어떻게 동작하나요?

Forward Rendering은 가장 직관적인 방식입니다. 각 오브젝트를 그릴 때 모든 라이트의 영향을 한 번에 계산합니다.

마치 화가가 캔버스에 그림을 그릴 때, 한 붓터치로 색상과 명암을 모두 표현하는 것과 같습니다.

Forward Rendering 파이프라인:

For each Object:
    For each Light:
        ┌─────────────────────────────────────────────┐
        │  Vertex Shader                              │
        │  → 정점 변환                                │
        └─────────────────┬───────────────────────────┘
                          ▼
        ┌─────────────────────────────────────────────┐
        │  Fragment Shader                            │
        │  → 텍스처 샘플링                            │
        │  → 라이팅 계산 (모든 라이트)               │
        │  → 최종 색상 출력                          │
        └─────────────────┬───────────────────────────┘
                          ▼
                    Frame Buffer

Forward Rendering의 기본 흐름

Forward Rendering Fragment Shader (개념)
// 각 픽셀마다 모든 라이트 계산
float4 ForwardPS(VSOutput input) : SV_Target
{
    float3 albedo = texAlbedo.Sample(sampler, input.UV);
    float3 normal = normalize(input.Normal);
    float3 finalColor = float3(0, 0, 0);

    // 모든 라이트 순회
    for (int i = 0; i < numLights; i++)
    {
        float3 lightDir = normalize(lights[i].Position - input.WorldPos);
        float NdotL = max(dot(normal, lightDir), 0);
        finalColor += albedo * lights[i].Color * NdotL;
    }

    return float4(finalColor, 1.0);
}

Part 2: Forward의 한계

Forward Rendering의 문제점은 무엇인가요?

가장 큰 문제는 복잡도가 O(오브젝트 × 라이트)라는 것입니다.

오브젝트 1000개, 라이트 100개면? 100,000번의 라이팅 계산이 필요합니다. 게다가 오버드로우 문제도 있습니다 - 보이지 않는 픽셀도 계산됩니다.

🎯 오버드로우 문제

Forward는 뒤에서 앞으로 그리든 앞에서 뒤로 그리든, 겹쳐서 가려지는 픽셀도 모두 계산합니다.

마치 벽 뒤에 있는 그림도 먼저 완벽하게 그린 뒤에 벽으로 덮어버리는 것과 같습니다. 낭비죠!

Part 3: Deferred Rendering의 등장

Deferred Rendering은 어떻게 이 문제를 해결하나요?

Deferred는 렌더링을 두 단계로 분리합니다:

1. Geometry Pass: 지오메트리 정보만 G-Buffer에 저장

2. Lighting Pass: G-Buffer를 읽어 라이팅 계산

이렇게 하면 보이는 픽셀만 라이팅을 계산합니다!

Deferred Rendering 파이프라인:

┌─────────────────────────────────────────────────────────┐
│                    Geometry Pass                         │
│  For each Object:                                       │
│    → G-Buffer에 기하 정보 저장                          │
│                                                          │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐   │
│  │ Albedo   │ │ Normal   │ │ Depth    │ │ Specular │   │
│  │ (RGB)    │ │ (XYZ)    │ │ (Z)      │ │ (RGB)    │   │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘   │
│                    G-Buffer                              │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│                    Lighting Pass                         │
│  For each Light:                                        │
│    → G-Buffer 샘플링                                    │
│    → 라이팅 계산                                        │
│    → 결과 누적                                          │
│                                                          │
│  최종 결과 = Σ(각 라이트의 기여)                       │
└─────────────────────────────────────────────────────────┘

Deferred Rendering의 2단계 구조

Part 4: G-Buffer 구조

G-Buffer에는 무엇이 저장되나요?

라이팅에 필요한 모든 표면 정보를 저장합니다:

버퍼 내용 포맷 예시
Albedo 기본 색상 (RGB) R8G8B8A8
Normal 표면 법선 (XYZ) R16G16B16A16F
Depth 깊이 값 D24S8 / D32F
Specular/Roughness 반사 특성 R8G8B8A8
Metallic 금속성 R8 (또는 패킹)
G-Buffer 출력 (Geometry Pass)
struct GBufferOutput
{
    float4 Albedo   : SV_Target0;  // RGB = Color, A = AO
    float4 Normal   : SV_Target1;  // RGB = Normal (encoded)
    float4 Material : SV_Target2;  // R = Metallic, G = Roughness
};

GBufferOutput GeometryPS(VSOutput input)
{
    GBufferOutput output;

    output.Albedo = texAlbedo.Sample(sampler, input.UV);
    output.Normal = float4(EncodeNormal(input.Normal), 1);
    output.Material.r = texMetallic.Sample(sampler, input.UV);
    output.Material.g = texRoughness.Sample(sampler, input.UV);

    return output;
}

Part 5: 성능 비교

어떤 상황에서 어떤 방식이 유리한가요?

간단한 규칙: 라이트가 많으면 Deferred, 적으면 Forward

항목 Forward Deferred
복잡도 O(오브젝트 × 라이트) O(오브젝트) + O(라이트)
대역폭 낮음 높음 (G-Buffer)
반투명 쉬움 별도 패스 필요
MSAA 쉬움 어려움 (비용 높음)
머티리얼 다양성 자유로움 G-Buffer 제한
메모리 적음 G-Buffer 크기
🎯 언제 무엇을 선택?

Forward: 모바일, VR, 반투명 많은 씬, 라이트 적은 실외

Deferred: PC/콘솔, 라이트 많은 실내, 복잡한 조명 효과

Part 6: 하이브리드 접근법

둘 중 하나만 써야 하나요?

아니요! 현대 엔진들은 하이브리드 방식을 사용합니다.

• 불투명 오브젝트: Deferred

• 반투명 오브젝트: Forward

• 특수 머티리얼: Forward (Deferred 제한 회피)

하이브리드 파이프라인
// 1. Deferred Pass - 불투명 오브젝트
RenderOpaqueToGBuffer();
DeferredLightingPass();

// 2. Forward Pass - 반투명 + 특수 머티리얼
EnableDepthRead();  // Deferred의 깊이 재사용
for (Object obj : forwardObjects)
{
    if (obj.IsTranslucent())
    {
        RenderForwardTranslucent(obj);
    }
    else if (obj.NeedsForward())  // 커스텀 셰이더 등
    {
        RenderForwardOpaque(obj);
    }
}

// 3. Post Processing
ApplyPostProcessing();

📌 핵심 정리

  • Forward: 오브젝트별로 모든 라이트 계산, 단순하지만 라이트 많으면 느림
  • Deferred: G-Buffer에 정보 저장 후 라이팅, 라이트 많을 때 유리
  • Deferred의 G-Buffer는 대역폭을 많이 사용하고 반투명 처리가 어려움
  • 현대 엔진은 하이브리드: 불투명은 Deferred, 반투명은 Forward
  • 선택 기준: 라이트 수, 반투명 양, 타겟 플랫폼
