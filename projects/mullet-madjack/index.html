01 프로젝트 개요

2주 만에 완성한 80년대 사이버펑크 FPS 모작

인디 슈터 Mullet MadJack을 DirectX 9로 모작한 팀 프로젝트입니다. 원작의 핵심인 10초 카운트다운 안에 적을 처치해 시간을 벌어가는 빠른 템포 슈팅과 비주얼을 재현했습니다.

저는 카메라 / 커스텀 물리 / 플레이어 제어 / ActiveUI 시스템 / 최종 보스(비홀더)를 담당했고, 추가로 팀 전체의 작업 속도를 가로막던 이미지 사이즈 정규화 문제를 OpenCV 자동화 툴로 해결했습니다.

03 ActiveUI 트리 시스템

왜 만들었나 — 매번 Clone 하는 비용

처음에는 UI를 띄울 때마다 프로토타입을 Clone 해서 띄우고, 사라지면 Dead 처리하는 구조였습니다. 그런데 상점 / HP바 / 콤보 / 보스HP / 크로스헤어처럼 자주 켜졌다 꺼지는 UI까지 매번 Clone → Delete를 반복하니, 1초에 수십 번 토글되는 UI에서 메모리 단편화와 끊김이 발생했습니다.

해결: 한 번 만들어둔 인스턴스를 활성/비활성 플래그로만 토글하는 ActiveUI 시스템을 추가했습니다.

5개 컨테이너로 분리된 UI 매니저

UI 의 수명과 렌더 방식이 다르기 때문에, CUi_Manager는 5개의 컨테이너로 UI를 분리해서 관리합니다.

CUi_Manager
m_Ui_LifePrototypes — 원본 보관
m_Ui_LifeClonelist — 수명 있는 NonBlend (피격 데미지 등)
m_Ui_LifeBlendClonelist — 수명 있는 Blend (이펙트성 UI)
m_Ui_Active / m_Ui_ActiveBlend — 영구 인스턴스, 플래그 토글
m_Ui_ShopActive — 상점 전용 (별도 렌더 패스)
CUi.h — Active 플래그가 바뀔 때마다 Initialize 호출 C++ 
void Set_Active(bool _isActive)
{
    if (_isActive == true && m_bActive == true)
        return;

    if (_isActive == true)
        Initialize_Active();   // 켤 때만 호출되는 초기화
    else
        Initialize_InActive(); // 끌 때만 호출되는 정리

    m_bActive = _isActive;
}

트리 구조 — 부모를 켜면 자식이 따라온다

상점 UI를 예로 들면, Ui_Shop 하나를 켜는 순간 안에 있는 업그레이드 선택 슬롯, 가격 표시, 설명창, 포커스 효과 UI가 같이 활성화되어야 합니다.

05 커스텀 물리

PhysX 없이 직접 만든 가벼운 Rigidbody

이 프로젝트는 PhysX 같은 외부 물리 엔진을 쓰지 않고, FPS 에서 필요한 최소 기능만 가진 CRigidbody 컴포넌트를 직접 구현했습니다. 중력 / 속도 / 마찰 / 지면 판정만 다루지만, 대시 / 슬라이드 / 슬로프 / 점프는 모두 이 위에서 만들어졌습니다.

Rigidbody.cpp — XZ 평면 마찰만 분리해서 적용 C++ 
void CRigidbody::Update(_float fTimeDelta)
{
    if (false == m_bGround && true == m_bUseGravity)
        m_vVelocity += m_vGravity * fTimeDelta;

    if (m_fFriction != 0.f)
    {
        _float3 vInvVelocity = m_vVelocity;
        vInvVelocity.y = 0.f;            // Y축 마찰 제외 (점프/낙하 영향 없음)
        vInvVelocity *= m_fFriction * fTimeDelta;

        _float3 vXZVelocity = m_vVelocity;
        vXZVelocity.y = 0.f;

        if (D3DXVec3Length(&vXZVelocity) < D3DXVec3Length(&vInvVelocity)) {
            m_vVelocity.x = 0.f;         // 마찰이 속도보다 크면 0으로 클램프
            m_vVelocity.z = 0.f;
        } else {
            m_vVelocity -= vInvVelocity;
        }
    }

    _float3 vOwnerPos = m_pOwnerTransform->Get_State(CTransform::STATE_POSITION);
    vOwnerPos += m_vVelocity * fTimeDelta;
    m_pOwnerTransform->Set_State(CTransform::STATE_POSITION, &vOwnerPos);
}

핵심 디테일: 마찰을 XZ 평면에만 적용하고 Y축은 제외했습니다. 이렇게 안 하면 점프해서 공중에 떠 있는 동안에도 마찰이 작용해 낙하 속도가 줄어드는 비현실적인 결과가 나옵니다. 또한 마찰이 현재 속도보다 크면 빼는 대신 0으로 클램프해서 부호가 뒤집혀 떨리는 현상을 막았습니다.

원형 지뢰 링 — 360° / 12° 간격 30발

가장 단순하지만 시각적 임팩트가 큰 패턴입니다. cos(θ)·r, sin(θ)·r로 원 둘레를 30분할해서 한 번에 30개의 지뢰를 깔아두고, 플레이어를 좁은 영역으로 밀어붙입니다.

Beholder.cpp — 360도 원형 지뢰 배치 C++ 
void CBeholder::RoundAirStrike()
{
    float f_radius = m_fRoundStrikeRadius;
    for (int i = 0; i < 360; i = i + 12)
    {
        float f_x = (float)(cos(deg2rad(i)) * f_radius);
        float f_y = (float)(sin(deg2rad(i)) * f_radius);

        _bool bExplode = true;
        CGameObject* pObj = m_pGameInstance->Add_Clone(
            m_pGameInstance->Get_CurrentLevelID(),
            L"LandMine", L"Prototype_LandMine", &bExplode);
        pObj->Get_Transform()->Add_Pos(_float3(f_x, -0.3f, f_y + 10.f));
    }
}

07 리사주 커브 응용

왜 리사주 커브였나

비홀더 패턴의 핵심 문제: "같은 보스를 여러 번 잡아도 매번 다르게 느껴지게 하려면 어떻게 해야 할까?"

처음에는 발사 위치만 랜덤하게 뽑았는데, 플레이어 입장에서는 결국 "어디서 날아올지 모르는 직선"일 뿐이었습니다. 그래서 궤적 자체가 매번 다른 곡선으로 그려지도록 리사주 커브를 도입했습니다.

🎯 비유로 이해하기

리사주 커브는 두 개의 사인파를 X / Y 축에 직교시켜 그린 도형입니다. 오실로스코프에서 자주 보이는 8자, 별모양, 나선 같은 패턴이 모두 같은 공식에서 주파수와 위상만 바꾼 결과입니다.

비홀더는 레이저를 쏠 때마다 이 주파수/위상/속도를 새로 추첨해서, 같은 공식이지만 매번 다른 곡선으로 플레이어를 추적합니다.

MathManager.cpp — 리사주 커브 공식 C++ 
_float2 CMath_Manager::Lissajous_Curve(
    _float fTimeDelta, _float& fLissajousTime,
    _float fWidth, _float fHeight,
    _float fLagrangianX, _float fLagrangianY,
    _float fPhaseDelta, _float fSpeed)
{
    fLissajousTime += fTimeDelta * fSpeed;

    _float fPosX = fWidth  * sin(fLagrangianX * fLissajousTime + 3.14f / fPhaseDelta);
    _float fPosY = fHeight * cos(fLagrangianY * fLissajousTime);

    return { fPosX, fPosY };
}

08 OpenCV 이미지 자동화

팀이 2주째 묶여 있던 문제

원본 에셋의 스프라이트는 크기가 제각각이었습니다. 같은 애니메이션 안에서도 프레임마다 가로세로 픽셀이 달라, 그대로 텍스처로 올리면 크기 보정용 매트릭스를 매 프레임 다시 만들어야 하고, 로드 시간도 느려지고, 캐릭터가 프레임마다 떨려 보이는 현상이 나타났습니다.

기존 팀원들은 이걸 포토샵으로 한 장씩 캔버스 확장 후 가운데 정렬로 풀고 있었고, 이 작업에만 약 2주가 소요되었습니다. 게다가 새 에셋이 추가되면 다시 같은 작업을 반복해야 했습니다.

해결 — OpenCV 일괄 변환 + 게임 시작 시 자동 실행

OpenCV 기반 파이썬 스크립트를 만들어 폴더를 통째로 훑게 했습니다. 각 이미지의 알파 채널과 콘텐츠 영역을 분석하고, 애니메이션 그룹 내에서 가장 큰 크기를 기준 캔버스로 잡은 뒤, 모든 프레임을 같은 캔버스 안에서 위치를 보존하며 패딩하는 방식입니다. 이렇게 하면 캐릭터의 발 위치나 무게중심이 프레임 사이에 흔들리지 않습니다.

이 스크립트를 빌드 후처리 / 게임 부트 단계에 연결해서, 새 에셋이 들어와도 사람이 손댈 필요 없이 자동 정규화되도록 만들었습니다.

Before
아티스트가 새 에셋 전달 → 프로그래머가 포토샵 수작업 → 빌드 (2주)
→
After
아티스트가 새 에셋 전달 → 자동 정규화 → 빌드 (0분)

부수 효과 — 로딩 시간과 런타임 안정성

크기를 통일하니까 같은 애니메이션 그룹의 텍스처들을 동일 사이즈 스프라이트 시트처럼 다룰 수 있게 되어, 매 프레임 크기 보정 매트릭스를 다시 만들 필요가 없어졌습니다. 결과적으로:

  • 레벨 로딩 시간 감소 (텍스처 로더가 크기를 추측하지 않아도 됨)
  • 애니메이션 프레임 사이 떨림 제거
  • 새 에셋 추가 시 프로그래머 개입 0
  • 아티스트의 작업 사이클이 짧아져 후반 폴리싱 시간 확보

09 유사 스켈레톤 애니메이션

스프라이트 시트로 만든 가짜 본 애니메이션

DX9 에 본 기반 스켈레탈 애니메이션을 도입할 시간이 없어, 스프라이트 시트 프레임을 정해진 간격으로 넘기는 방식으로 캐릭터 애니메이션을 만들었습니다. 위에서 만든 OpenCV 정규화 덕분에 모든 프레임 크기가 같아서, 단순 인덱스만 바꿔주면 자연스러운 애니메이션이 재생됩니다.

CAnimation 컴포넌트는 태그별로 텍스처 배열을 보관하고, Play_Animation(tag, fps, loop) 한 줄로 애니메이션을 교체합니다. 보스 / 적 / 무기 / UI 가 전부 이 컴포넌트 위에서 동작합니다.

Animation.cpp — 프레임 간격마다 인덱스 증가 C++ 
void CAnimation::Update(_float fTimeDelta)
{
    if (m_bAnimEnd) return;

    m_fTimeAcc += fTimeDelta;

    if (m_fTimeAcc > m_fFrameGap)
    {
        if (m_iTextureNum < m_iMaxTextureNum)
            ++m_iTextureNum;
        else if (m_bLoop)
            m_iTextureNum = 0;
        else if (m_iTextureNum == m_iMaxTextureNum)
            m_bAnimEnd = true;

        m_fTimeAcc = 0.f;
    }
}

10 트러블슈팅 & 회고

자주 토글되는 UI 에서 발생한 메모리 단편화
매번 Clone & Dead 처리하던 UI 중 자주 호출되는 것들을 영구 인스턴스로 분리하고, Active 플래그 토글로만 표시/숨김을 제어하는 ActiveUI 시스템을 도입했습니다. 또한 부모 UI 가 Tick 안에서 자식들을 직접 토글하는 트리 구조로 호출부 코드를 단순화했습니다.
호출부에서 UI 한 번만 켜면 연관 UI 가 일괄 활성화되어, 보스전처럼 UI 가 격하게 바뀌는 상황에서도 안정적으로 동작합니다.
팀의 작업 속도를 가로막던 이미지 정규화 수작업
포토샵으로 한 장씩 처리하던 작업을 OpenCV 일괄 변환 스크립트로 대체했습니다. 애니메이션 그룹 단위로 최대 크기 캔버스를 잡고, 콘텐츠 위치를 보존하며 패딩하는 로직을 만들어 게임 시작 시 자동 실행되도록 연동했습니다.
2주 분량의 수작업이 사라졌고, 새 에셋 추가 시에도 프로그래머 개입이 필요 없어졌습니다. 로딩 시간 단축과 프레임 떨림 제거는 덤이었습니다.
같은 보스 패턴이 반복되며 단조롭게 느껴지는 문제
레이저의 발사 위치만 랜덤화하던 방식에서, 리사주 커브의 주파수/위상/속도를 매 발사마다 난수로 추첨하는 방식으로 바꿨습니다. 같은 공식이 같은 패턴 안에서도 매번 다른 곡선을 그리게 됩니다.
한 패턴 안에서 모든 레이저의 궤적이 달라져, 플레이어가 외워서 회피하는 것이 불가능해졌습니다. 동일 공식을 보스 / 무기 이펙트 / 처형 연출까지 재사용해 코드 중복도 줄였습니다.
공중에서 마찰이 작용해 낙하 속도가 줄어드는 문제
Rigidbody 마찰을 XZ 평면 속도에만 적용하고 Y 축은 분리했습니다. 또한 마찰값이 현재 속도보다 큰 경우 부호 반전으로 떨리는 현상을 막기 위해 0 으로 클램프하는 분기를 추가했습니다.
점프 / 낙하 / 대시가 자연스럽게 연결되고, 정지 상태에서 떨리는 jitter 가 사라졌습니다.
